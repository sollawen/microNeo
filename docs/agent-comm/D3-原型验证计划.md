# 原型验证计划（D3）

> 本文档是 `D2-通信协议方案.md` 的**执行文档**：把已经定稿的协议用最小成本端到端跑通，在写进 microNeo 核心之前验证风险点。
>
> **性质**：原型分两部分。`agents/pi/` 是仓库长期资产（当前处于原型验证 M1→M2）；`proto/agentbridge/` 是一次性原型（模拟发送端），验证完退役，但其中的 `registry.go`/`message.go` 将来搬进 `internal/agentbridge/`。

---

## 一、为什么现在做原型

`D2` 已经把协议层完整定稿（注册表、传输、报文 schema、递送语义、版本、接收端契约），§14 的开放问题全是 UX 或可选项，**没有阻塞原型的协议决策**。

但有两类风险靠"读文档"无法消除，必须实测：

1. **pi 扩展能不能起 socket server 并活一整个 session**——D2 §8.3 的生命周期设计（session_start 注册 / session_shutdown 注销）从未实测过。
2. **从 socket 回调里调 `pi.sendUserMessage` 能不能真的触发 agent 对话**——这是 D2 §6 "带 prompt 立即递送" 的核心，官方示例只在命令处理里用过该 API。

原型的目标就是把这两点验证掉，同时把协议骨干（注册表/发现/传输/分帧/schema）全链路跑通。

---

## 二、原型范围

三个独立部件，**完全不碰 microNeo 核心**（符合项目"对 micro 原生零侵入"原则）：

| 部件 | 形态 | 验证什么 |
|------|------|---------|
| **① pi 扩展（TS）** | 仓库内 `agents/pi/`（自描述 pi 包，`pi install` 装载） | session_start 起注册 + socket server、收报文、session_shutdown 清理、按 D2 §6 分流递送 |
| **② Go discover** | 独立小程序 | registryDir 算法、注册文件解析、PID + connect 存活检测、僵尸 GC |
| **③ Go send** | 独立小程序 | 报文组装、connect → write 一行 JSON → close、目标选择 |

为什么 Go 这两个值得认真写而不是一次性脚本：它们的 `registry` / `message` 核心代码将来几乎可以**原样搬进 `internal/agentbridge/`**，不是丢掉的东西。

---

## 三、两个里程碑

风险密度不同，拆成 M1 / M2。M1 是必须项；M2 紧接着做，因为它才是"agent 真的能用上"的证明，且最不确定的 pi 集成点就在这里。

### M1 — 协议骨干验证

pi 扩展收到报文后只做 **UI 展示**（`ctx.ui.notify` + `setWidget`），**不进 LLM**。Go discover/send 跑通。

**验收清单**：

- [ ] pi 扩展：`session_start` 时算 registryDir、起 Unix socket server、写注册文件 `<name>.json`
- [ ] pi 扩展：accept 连接、按 `\n` 分帧、`JSON.parse`、校验信封 `v`/`type`
- [ ] pi 扩展：收到报文 → `ctx.ui.notify` 显示摘要 + `ctx.ui.setWidget` 显示待用提示
- [ ] pi 扩展：`session_shutdown` 时删注册文件 + 删 socket + 关 server
- [ ] Go discover：算 registryDir、glob `*.json`、解析、PID 检测 + connect 试探、GC 僵尸
- [ ] Go send：组装信封 + payload（含 prompt 与纯上下文两种）、connect → 写一行 → close
- [ ] 联调：discover 看到注册的 pi；send 发"带 prompt"和"纯上下文"各一条；pi 侧 notify/widget 正确显示来源、文件、行号、选区、prompt

### M2 — 递送验证（杀手级 demo）

pi 扩展按 D2 §6 分流真正把上下文送进 LLM。

**验收清单**：

- [ ] 带 prompt 的报文 → `pi.sendUserMessage(formatContextWithPrompt(p))`，agent **真的开口回复**
- [ ] agent 正在 streaming 时收到带 prompt 报文 → 用 `{ deliverAs: "followUp" }` 排队，不打断
- [ ] 纯上下文报文 → 暂存 `pendingContext` + widget 提示；不自动触发对话
- [ ] 挂 `before_agent_start`：用户下次发言时若 `pendingContext` 非空，prepend 进 prompt 后清空 + 清 widget
- [ ] `/mnab paste` / `/mnab send` / `/mnab discard` / `/mnab status` 命令可用

> **M2 退路**：如果从 socket 回调直接调 `sendUserMessage` 有问题，退化为"纯上下文 + before_agent_start 隐式带上"也能证明 agent 可用上上下文——只丢掉"带 prompt 立即触发"这一条，其余价值都在。

## 四、目录结构

两部分，定位不同：

- **`agents/pi/`** —— pi 接收端扩展，**仓库一等公民资产**（当前处于原型验证 M1→M2，将来随协议演进，并按 D2 §8.2 拆 core/adapter）。未来 opencode 接收端放 `agents/opencode/`，以此类推。
- **`proto/agentbridge/`** —— **一次性原型**，模拟 microNeo 发送端（Go discover/send）。验证完退役，但其中的 `registry.go`/`message.go` 将来搬进 `internal/agentbridge/`。

```
agents/
├── pi/                       # pi 接收端扩展（长期资产，当前原型阶段）
│   ├── package.json          # { "name": "microneo-agentbridge-pi", "pi": { "extensions": ["./index.ts"] } }
│   ├── index.ts              # 接收端扩展（M1 → M2）
│   └── README.md             # 含 pi install 安装说明
└── opencode/                 # （未来 Phase 3）opencode 接收端 TUI 插件

proto/agentbridge/            # 一次性原型：模拟 microNeo 发送端
├── README.md                 # 标注"一次性原型"、运行步骤
└── go/
    ├── go.mod                # 独立 module（agentbridge-proto），不进 microNeo 的 make build
    ├── registry.go           # 包 agentbridge：registryDir / discover / 存活检测 / GC
    ├── message.go            # 包 agentbridge：Envelope / ContextPayload 类型 + JSON
    └── cmd/
        ├── discover/main.go
        └── send/main.go
```

**关键点**：
- `proto/agentbridge/go` 有**独立的 go.mod**，根目录的 `make build` 不会编译它——对 microNeo 主线零干扰。验证完可整个删除或保留作参考。
- `agents/pi/` 带 `package.json` 声明 `pi.extensions`，是**自描述的 pi 包**，未来可扩展为多文件（core/adapter 拆分、README、依赖声明）。
- proto 的 `registry.go`/`message.go` 用包名 `agentbridge`，与生产包 `internal/agentbridge/` 同名——验证完是**字面意义的搬文件**。

## 五、已定的技术决策（默认值）

以下沿用 D2，无异议则按此实现：

- **registryDir 算法**：`$XDG_RUNTIME_DIR` → `$TMPDIR` → `/tmp`，目录名 `microneo-agent-bridge-${UID}`，权限 `0700`，socket 文件 `0600`（D2 §3.1）
- **传输**：Unix domain socket（SOCK_STREAM），每次发送一次连接→写一行→关闭（D2 §4）
- **分帧**：逐行 JSON，`\n` 结束，JSON 内部不得有裸 `\n`（D2 §4.3）
- **版本**：注册文件 `protocol: "mnab-1"`，信封 `v: 1`（D2 §七）
- **报文类型**：v1 只实现 `context`（`bye` 预留）（D2 §5.2）
- **命名约定**：结构名（目录/包/module）用 `agentbridge`——`internal/agentbridge/`、Go 包 `agentbridge`、pi 包 `microneo-agentbridge-pi`、proto module `agentbridge-proto`；用户面对的短字符串用 `mnab`（**m**icro**N**eo **a**gent **b**ridge）——协议 `mnab-1`、slash 命令 `/mnab`、配置项 `mnab.*`、环境变量 `MNAB_*`、widget key `"mnab"`
- **pi 扩展源码**：仓库内 `agents/pi/`（自描述 pi 包，含 `package.json` 声明 `pi.extensions`）
- **pi 扩展安装**：`pi install <agents/pi 绝对路径>` —— pi 把本地路径登记进 `~/.pi/agent/settings.json`，**原地引用不拷贝**，`/reload` 即时生效。任何项目下启动 pi 都自动加载，满足"任何项目都能收 microNeo 的消息"
  - 必须用**绝对路径**（相对路径是相对 settings 文件解析的，跟仓库无关）
  - 省略 `-l` 即登记到用户全局 settings（正是我们要的）
  - 安装后用 `pi list` 可审计；`pi remove microneo-agentbridge-pi` 卸载
  - 详见 pi 文档 [packages.md](…/packages.md) 的 "Local Paths"

## 七、技术风险与对策

### 风险 1：pi 扩展的 socket 生命周期会"闪"

pi 在 `/new`、`/resume`、`/reload` 时会重新加载扩展（先 `session_shutdown` 旧实例，再重跑 factory + `session_start`）。所以 socket server 在 session 切换时**会拆了重建**，有一个短窗口。

**对策（原型阶段）**：按 D2 §8.3 老实做（session_start 注册、session_shutdown 清理）即可。发送端的 connect 探测 + GC 能自愈这个窗口（连不上就 GC 重选）。生产阶段如果想消除闪烁再优化，不阻塞原型。

### 风险 2：从 socket 回调调 `sendUserMessage` 是未经验证的用法

D2 §8.3 设计从 socket data 回调触发递送。`sendUserMessage` 挂在顶层 `pi` 对象（非 `ctx`），理论上任何地方可调，但官方示例只在命令处理里用过。

**对策**：这正是 **M2 要验证的核心未知项**。M2 验收清单第 1 条就是它。退路见 §三 M2 末尾的"退路"。

### 风险 3：多 pi 同时运行时的 name 冲突

D2 §3.3 要求后启动者检测旧 pid 已死则覆盖、仍活则自改名。原型阶段先实现最简单的：默认 name = `pi-<pid>`（pid 天然唯一，几乎不会冲突）。冲突处理留到生产前补全。

## 八、执行顺序建议

1. 建 `agents/pi/`（package.json + index.ts 骨架）+ `proto/agentbridge/` 骨架 + 两处 README
2. 写 Go `registry.go` + `message.go`（包 agentbridge，纯逻辑，可单测）
3. 写 pi 扩展 M1 版（session_start 起 server + 注册；收报文 notify/widget；session_shutdown 清理）
4. `pi install <agents/pi 绝对路径>` 装载，启动 pi 验证扩展加载、注册表落地
5. 写 Go `cmd/discover` + `cmd/send`
6. **M1 联调**，过验收清单
7. pi 扩展升级到 M2（分流 + sendUserMessage + before_agent_start + `/mnab` 命令）
8. **M2 联调**，过验收清单
9. 归档：把验证结论、踩坑记录回写进本文档或 D2 的开放问题
