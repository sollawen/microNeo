# Editor-Agent Bridge：架构设计


---

## 2. 总体架构

```
 ┌─────────────────────┐   用户主动触发     ┌──────────────────────┐
 │   microNeo  (进程A) │   ── push 1 JSON ─►│   AI Agent  (进程X)  │
 │   编辑 a.md          │   （Unix socket）  │   pi 扩展            │
 │   选中 + 按键发送     │                    │   收到→呈现/触发对话 │
 └─────────────────────┘                    └──────────────────────┘
       │                                              ▲
       │        ┌───────────────────────────┐         │
       │   读   │   注册表 (Registry)        │   写    │（agent 启动时登记，
       └───────►│   $RUNTIME_DIR/            │◄────────┘ 发送前读取）
                │   microneo-agent-bridge-UID│
                │   <name-or-pid>.json       │
                └───────────────────────────┘
```

三个角色：
1. **发送端（Sender）** = microNeo 里的 Lua 插件。用户触发 → 采集当前上下文 → 发一次。
2. **接收端（Receiver）** = ai agent 里的扩展程序（pi 用 TS 扩展实现）。监听 → 收到报文 → 呈现给用户 / 注入对话。
3. **注册表（Registry）** = 一个固定的目录，接收端启动时在这里登记自己，发送端发送前在这里发现接收端。

> 协议（JSON 报文格式 + 注册表格式）是 **agent 无关** 的。这样 pi、opencode、未来任何 agent 只要按协议实现"接收端"，都能被 microNeo 识别。发送端不需要为每种 agent 写不同代码。

---

## 3. 关键设计决策

### 决策 1：发送端 = 纯 Lua 插件，不碰 micro 原生 Go 代码

这是最重要的决策，直接满足 AGENTS.md "对 micro 原生代码侵入越小越好" 的约束。

micro 的 Lua 插件系统已经暴露了我们需要的全部能力（见 `runtime/help/plugins.md` 和 `internal/lua/lua.go`）：

| 能力 | 来源 | 用途 |
|------|------|------|
| `net.DialUnix` / `net.Dial` | `import("net")` | 连接接收端的 Unix socket |
| `os.WriteFile` / `os.ReadFile` / `os.Stat` / `os.Getpid` | `import("os")` | 读写注册表、检测存活 |
| `filepath.Glob` / `filepath.Join` | `import("filepath")` | 扫描注册表目录 |
| `micro.InfoBar()` / `micro.CurPane()` | `import("micro")` | 与用户交互、取当前 bufpane |
| `config.MakeCommand` | `import("micro/config")` | 注册 `agent-bridge` 命令 |
| `bp.Buf:GetActiveCursor()` → `.X`/`.Y` | BufPane/Buffer API | 读取光标位置 |
| `c:GetSelectionStart()` / `:GetSelectionEnd()` | Cursor API | 读取选区 |
| `bp.Buf:GetName()` / `bp.Buf:Path()` | Buffer API | 读取文件路径 |

**结论：发送端是一个放在 `~/.config/microNeo/plug/agent-bridge/agent-bridge.lua` 的插件，配合 `repo.json`，零 Go 代码改动。**

### 决策 2：IPC 机制 = Unix domain socket

候选对比：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| **Unix socket** | 本地、零网络暴露、快、支持双向、跨语言成熟 | Windows 支持较弱（v1 可先 Unix-only） | ✅ 采用 |
| TCP localhost | 跨平台、实现简单 | 要管理端口冲突、有网络暴露风险 | ❌ |
| 文件 + 轮询 | 极简单 | 延迟高、磁盘 IO、并发竞争 | ❌ |
| HTTP | 标准 | 每次 POST 开销大、要起 server | ❌ |

- macOS / Linux：原生支持，v1 只覆盖这两个平台（microNeo 主要平台）。
- 连接策略：**每次发送一次"连接 → 写一行 JSON → 关闭"**。Unix socket 握手开销极小（微秒级），换来无需管理长连接状态、对 agent 重启天然鲁棒。按需发送频率低，这个开销完全可忽略。

### 决策 3：接收端如何处理到达的上下文（按需 vs 实时的根本区别）

> 因为是**用户主动按需发送**（不是高频实时推送），所以**不存在淹没 LLM 的问题**。每条报文都是用户明确想发送的，直接当成一条有意义的消息处理即可。

收到报文后，接收端根据报文是否附带 `prompt` 决定行为：

| 报文形态 | 接收端行为 | 用户体验 |
|----------|-----------|----------|
| **带 prompt**（用户在 microNeo 里附了一句话，如"这段帮我改改"） | 作为一条**用户消息**发给 LLM（`pi.sendUserMessage`），立即触发对话 | 在 agent 里直接看到 AI 在回复这段 |
| **纯上下文**（只送了文件/光标/选区，没附话） | 把上下文**暂存为"待用上下文"**，widget 提示用户"有来自 microNeo 的上下文待用"；用户下次发消息时自动带上，或用 `/eab` 命令主动粘贴/丢弃 | 用户可以在 agent 侧补一句话再决定要不要发给 AI |

> **上表是 pi adapter 的行为**。**带 prompt** 的处理各 agent 一致（都是"发一条用户消息立即触发对话"）；**纯上下文**的处理因 agent 而异——pi 走"隐式注入"（`before_agent_start`，agent 偷偷知道但 transcript 不显眼），opencode 走"预填输入框"（用户可见、可编辑，更透明）。这是两个 adapter 必须各自实现、不能强求统一的地方。协议层（报文格式、带不带 prompt 字段）两者完全一致。详见 §7 与 `opencode调研.md` §6.5。

这样：
- **带 prompt** = 编辑器里直接发起一次协作，最顺。
- **纯上下文** = 把编辑器状态"挂"到 agent，用户决定怎么用，不强行打断 LLM。
- 两种都很轻量，**不需要**之前实时方案里那套内存快照持续刷新 / debounce / 周期性注入的复杂机制。

### 决策 4：协议与注册表 agent 无关

发送端只认"一个在监听的 Unix socket 地址"，不关心背后是 pi 还是 opencode。每个接收端按相同协议实现即可。

## 4. 服务发现与注册（Registry）

### 4.1 注册表目录

跨平台、用户私有、重启不残留：

```
${XDG_RUNTIME_DIR:-$TMPDIR:-/tmp}/microneo-agent-bridge-${UID}/
```

- 优先 `$XDG_RUNTIME_DIR`（Linux 下 `/run/user/$UID`，进程退出自动清理）。
- macOS 没有 XDG_RUNTIME_DIR，退回 `$TMPDIR`（通常是 `/var/folders/.../T/`）。
- 目录名带 `$UID`，多用户机器上隔离。
- 由首个启动的进程（agent 或 editor）按 `0700` 权限创建。

> 命名算法建议集中在一处：发送端（Lua）和接收端（TS）各自用同样算法算出该路径，避免不一致。v1 写死算法，两端实现各自复刻。

### 4.2 注册文件

每个接收端在该目录下写一个文件 `<name>.json`（文件名 = name，name 唯一）。内容：

```json
{
  "name": "pi-main",
  "pid": 12345,
  "transport": "unix",
  "socket": "/tmp/.../pi-main.sock",
  "protocol": "eab-1",
  "startedAt": 1717000000,
  "cwd": "/Users/me/project",
  "labels": ["default"]
}
```

字段说明：
- `name`：人类可读的唯一名（用户起或自动生成，如 `pi-<pid>`）。文件名即 name，保证唯一。
- `protocol`：协议版本号，发送端发现版本不匹配时跳过该接收端。
- `labels`：可选标签，发送端可按标签筛选。

### 4.3 存活检测

发送端发送前发现接收端时，按优先级判断存活：
1. **首选：试探连接**——对 `socket` 地址做一次 `DialUnix` + 立即关闭。连得上即认为存活（最可靠）。
2. **兜底：PID 检测**——读 `pid`，判断进程是否存在。
3. **垃圾回收**——连接失败 / PID 不存在的注册文件，由发送端顺手删除。

接收端正常退出时：删除自己的注册文件 + 删除 socket 文件。

## 5. 通信协议（Wire Protocol）

### 5.1 传输层

- **Unix socket，stream，每次发送一次连接**。
- 编码：**逐行 JSON**（每条报文一行 `\n` 结束），便于未来扩展为长连接多报文。
- 方向：单向 microNeo → agent。协议预留双向扩展（agent 回 `ack`，或后续发 `goto` 指令），但 v1 不实现。

### 5.2 报文公共信封

所有报文共享头部字段：

```json
{
  "v": 1,
  "type": "context | bye",
  "editor": { "pid": 54321, "name": "microNeo", "instance": "win-1" },
  "ts": 1717000123.456,
  "payload": { ... }
}
```

- `v`：协议版本，等于注册表里的 `protocol`。
- `editor.instance`：发送端给本 microNeo 实例起的标识（窗口/tab 区分），多实例时让 agent 区分来源。

### 5.3 报文类型（payload）

按需发送模式下，报文类型非常少：

| type | payload | 触发时机 |
|------|---------|----------|
| `context` | `{ path, basename, filetype, line, col, selection?, prompt?, visible_lines? }` | **用户主动触发发送**（核心报文） |
| `bye` | `{}` | 发送端实例退出（可选，用于让 agent 清理对应来源） |

`context.payload` 字段说明：

| 字段 | 必需 | 说明 |
|------|------|------|
| `path` | ✅ | 文件绝对路径 |
| `basename` | ✅ | 文件名（展示用） |
| `filetype` | ✅ | micro 识别的文件类型 |
| `line` / `col` | ✅ | 光标位置，**0-based**（与 micro 内部 `Loc.X/Y` 一致） |
| `selection` | ❌ | 选区对象 `{ start:{line,col}, end:{line,col}, text? }`，无选区则省略 |
| `selection.text` | ❌ | 选中文本（选区较小时带，如 < 2KB） |
| `prompt` | ❌ | 用户附的一句话提问；**有则 agent 立即触发对话，无则暂存待用** |
| `visible_lines` | ❌ | 可见区域文本（可选，方便 agent 看上下文） |

示例（带选区 + 提问，最典型场景）：

```json
{"v":1,"type":"context","editor":{"pid":54321,"name":"microNeo","instance":"win-1"},
 "ts":1717000123.5,
 "payload":{
   "path":"/Users/me/a.md","basename":"a.md","filetype":"markdown",
   "line":42,"col":8,
   "selection":{"start":{"line":42,"col":8},"end":{"line":45,"col":0},"text":"## 标题\n正文..."},
   "prompt":"这段标题感觉怪，帮我换个说法"
 }}
```

## 6. 发送端实现（microNeo Lua 插件）

### 6.1 文件结构

```
~/.config/microNeo/plug/agent-bridge/
├── repo.json            # 插件元信息（name/description/website/...）
├── agent-bridge.lua     # 主逻辑
└── help/
    └── agent-bridge.md
```

放在用户配置目录，属于"用户级插件"，不进 micro 原生 runtime，**完全不动 micro 仓库代码**。

### 6.2 插件骨架（伪代码，说明思路）

```lua
-- agent-bridge.lua
VERSION = "0.1.0"

local net = import("net")
local os  = import("os")
local fp  = import("filepath")
local micro = import("micro")
local config = import("micro/config")

-- 1) 算出注册表目录（与接收端 TS 同一算法）
local function registry_dir() ... end

-- 2) 发现接收端：Glob 目录下 *.json，逐个试探 socket 连接，返回存活列表
local function discover_listeners() ... end

-- 3) 选择目标：读配置项 agentbridge.target；没有就让用户用命令选
local current_target = nil  -- {name=, socket=}

-- 4) 发送：connect → 写一行 JSON → close
local function send(msg)
  if not current_target then
    micro.InfoBar():Message("agent-bridge: 未选择目标，请运行 > agent-bridge select")
    return
  end
  local ok, conn = pcall(net.Dial, "unix", current_target.socket)
  if not ok then
    micro.InfoBar():Error("agent-bridge: 连接失败，重新选择目标")
    refresh_target(); return
  end
  conn:Write(json_line(msg) .. "\n"); conn:Close()
end

-- 5) 采集当前编辑上下文（用户触发时调用一次）
local function capture_context(bp, prompt)
  local buf = bp.Buf
  local c   = buf:GetActiveCursor()
  local sel = nil
  if c:HasSelection() then
    local s, e = c:GetSelectionStart(), c:GetSelectionEnd()
    sel = { start={line=s.Y, col=s.X}, end={line=e.Y, col=e.X},
            text = selection_text_if_small(buf, s, e) }
  end
  return {
    path=buf:Path(), basename=buf:GetName(), filetype=buf:FileType(),
    line=c.Y, col=c.X, selection=sel, prompt=prompt
  }
end

-- 6) 命令：用户在 microNeo 里触发
function init()
  config.MakeCommand("agent-bridge", cmd_agent_bridge, config.NoComplete)
  config.TryBindKey("CtrlAlt s", "command:agent-bridge send", true)  -- 可选快捷键
end

function cmd_agent_bridge(bp, args)
  local sub = args[1] or "send"
  if     sub == "list"   then show_listeners()
  elseif sub == "select" then prompt_select_target(bp)
  elseif sub == "send"   then
    -- args[2..] 拼成 prompt（可选）
    local prompt = table.concat({select(2, table.unpack(args))}, " ")
    send({v=1, type="context", editor=me(),
          ts=os.Time(), payload=capture_context(bp, prompt ~= "" and prompt or nil)})
    micro.InfoBar():Message("agent-bridge: 已发送给 " .. (current_target and current_target.name or "?"))
  elseif sub == "status" then show_status()
  end
end
```

### 6.3 用户交互（命令）

通过 `config.MakeCommand` 注册，统一入口 `agent-bridge <sub>`：

| 命令 | 作用 |
|------|------|
| `agent-bridge send [prompt...]` | **核心**：采集当前上下文发送给目标；可选附一句话 |
| `agent-bridge list` | 列出当前所有存活的接收端 |
| `agent-bridge select` | 弹出选择，挑一个作为目标（多个时） |
| `agent-bridge status` | 在 InfoBar 显示当前目标 / 连接状态 |
| `agent-bridge clear-target` | 清除当前目标，下次发送时重新选择 |

> 快捷键：可选绑定一个组合键（如 `CtrlAlt-s`）直接触发 `agent-bridge send`（无 prompt），选中即发，最顺。是否默认绑定由用户决定，避免与现有绑定冲突。

### 6.4 配置项

通过 `config.RegisterGlobalOption` 注册：

| 选项 | 默认 | 说明 |
|------|------|------|
| `agentbridge.target` | `""` | 指定目标 name；空=唯一则自动用 / 多个则提示选择 |
| `agentbridge.send_selection_text` | `true` 且 <2KB | context 报文是否带选中文本 |
| `agentbridge.send_visible_lines` | `false` | 是否带上可见区域文本（更费，按需开） |

## 7. 接收端实现（共享 core + agent adapter）

> 详细调研（尤其 opencode 选型论证）见 `docs/agent-comm/opencode调研.md`。本节给出实现结构。

**整体结构**：接收端代码分两层——
- **core（agent 无关，pi/opencode 共用一份）**：socket server、注册表读写、报文解析/分帧、payload 类型、context→text 格式化、"带 prompt / 不带 prompt"的分流判定。两边都是 TS 且都能用 `node:net`/`node:fs`，core 一行不改即可复用。
- **adapter（各 agent 实现）**：把 core 的 `onContext(payload)` 回调接到各自的能力上。pi adapter 与 opencode adapter 各 ~100-150 行，差异主要在三处：**发消息的方式、纯上下文的处理、UI/slash 命令**。

| 能力 | pi adapter | opencode adapter（TUI 插件） |
|------|-----------|------------------------------|
| 发消息 | `pi.sendUserMessage(text)` 一行 | 渲染 SolidJS 组件进 `session_prompt` Slot 拿 `promptRef`，再 `ref.set()+submit()` |
| 纯上下文 | `before_agent_start` **隐式注入** | **预填进输入框**（`ref.set` 不 submit，用户可见）/ `/eab paste` |
| UI | `ctx.ui.notify` / `setWidget` | `api.ui.toast` / `api.ui.Slot`(widget) |
| slash 命令 | pi 命令 API | `api.command.register`(带 `slash`) |

下面 §7.1–7.6 是 **pi adapter** 的细节；§7.7 是 **opencode adapter**。

### 7.1 位置（pi adapter）

`~/.pi/agent/extensions/agent-bridge/index.ts`（全局扩展）。这样每个 pi 会话都自动加载。也可做项目级 `.pi/extensions`。

### 7.2 生命周期与监听

关键事件挂钩：

| pi 事件 | 接收端做的事 |
|---------|--------------|
| `session_start` | 创建 Unix socket server；算出 socket 路径；写注册表 JSON |
| 收到 socket 报文 | 解析 JSON → 见 §7.3 处理 |
| `session_shutdown` | 删注册表文件 + socket 文件；关 server |

> 不需要 `before_agent_start` 周期注入，也不需要内存快照持续刷新——按需发送的每条报文都是独立的一次性事件。

### 7.3 收到报文的处理（决策 3 的落地）

```typescript
// 待用上下文：只保留最后一次纯上下文（无 prompt）的，被新报文覆盖
let pendingContext: ContextPayload | null = null;

server.on("connection", (conn) => {
  // 读一行 JSON，解析出 { v, type, editor, payload }
  onMessage(msg);
});

function onMessage(msg) {
  if (msg.type !== "context") return;
  const p = msg.payload;

  if (p.prompt) {
    // —— 带 prompt：立即作为用户消息发给 LLM，触发对话 ——
    const text = formatContextWithPrompt(p);  // 见下
    pi.sendUserMessage(text);
    pendingContext = null;  // 已用掉
    ctx.ui.setWidget("eab", []);  // 清掉待用提示
  } else {
    // —— 纯上下文：暂存待用 ——
    pendingContext = { ...p, from: msg.editor };
    ctx.ui.setWidget("eab", [
      `📡 来自 microNeo (${msg.editor.instance}): ${p.basename} L${p.line + 1}`,
      p.selection ? `   选区 [L${p.selection.start.line+1} → L${p.selection.end.line+1}]` : "   无选区",
      `   下次发言自动带上；/eab paste 手动粘贴；/eab discard 丢弃`,
    ]);
  }
}
```

`formatContextWithPrompt(p)` 把上下文格式化成给 LLM 的消息，示例：

```
[microNeo 上下文]
文件：/Users/me/a.md (markdown)
光标：L43:8
选区（L43:8 → L46:0）：
  ## 标题
  正文...

[/microNeo 上下文]
用户提问：这段标题感觉怪，帮我换个说法
```

### 7.4 命令

| pi 命令 | 作用 |
|---------|------|
| `/eab paste` | 把当前 `pendingContext` 插入到输入框（用户可编辑后再发） |
| `/eab send` | 把 `pendingContext` 作为一条用户消息立即发给 LLM |
| `/eab discard` | 丢弃当前 `pendingContext` |
| `/eab status` | 显示当前 pendingContext 摘要 |
| `/eab rename <name>` | 改自己在注册表里的 name |

### 7.5 "下次发言自动带上"怎么实现

挂 `before_agent_start`：如果 `pendingContext` 非空，且这是**用户**消息（不是 tool result 流），就把它格式化后 prepend 到本次 prompt，然后清空 `pendingContext` + widget。

```typescript
pi.on("before_agent_start", async (event, ctx) => {
  if (!pendingContext) return;
  const ctxText = formatContextStandalone(pendingContext);
  pendingContext = null;
  ctx.ui.setWidget("eab", []);
  return { message: { customType: "eab-context", content: ctxText, display: true } };
});
```

### 7.6 UI 反馈

- `ctx.ui.setWidget("eab", lines)`：编辑器上方显示"📡 有来自 microNeo 的上下文待用"。
- `ctx.ui.setStatus("eab", "...")`：底部状态栏同步。
- 这些只在 `ctx.mode === "tui"` 时做。

### 7.7 opencode adapter（TUI 插件）

> opencode 用 **TUI 插件**实现接收端（不是 server 插件）。选型论证见 `opencode调研.md` §6.5/§6.6。核心思路：走 TUI 原生输入通道发送，opencode 的 server 完全无感——它收到的就是"用户敲了一段话回车了"。

**位置**：`~/.config/opencode/plugins/agent-bridge.ts`（全局，启动自动加载；也可发 npm 包在 `opencode.json` 的 `plugin` 声明）。

**拿输入通道**：TUI 插件渲染一个 SolidJS 组件进 `session_prompt`（或 `home_prompt`）宿主插槽，拿到 `promptRef`。该 ref 提供 `set(prompt)` / `submit()` / `focus()` 等，等价于操作用户输入框。

```typescript
let promptRef: TuiPromptRef | undefined;
api.ui.Slot("session_prompt", (props) => {
  props.ref?.((ref) => { promptRef = ref ?? undefined; });
  return null;  // 组件可不渲染任何可见内容，只为拿 ref
});
```

**收到报文的处理**（复用 core，差异只在分流后的动作）：

| 报文形态 | opencode adapter 行为 |
|----------|----------------------|
| 带 prompt | `promptRef.set({input: formatContextWithPrompt(p)})` + `promptRef.submit()` —— 走原生输入流，server 当作用户手敲，立即触发对话 |
| 纯上下文 | `promptRef.set({input: formatContextStandalone(p)})`，**不 submit** —— 把上下文**预填进输入框**。用户切到 opencode 看到已有一段编辑器上下文，自己接着敲问题再回车。（备选：暂存到 `api.kv`，用 `/eab paste` 粘贴） |

**与 pi adapter 的结构性差异**（重要，不可因子化）：
- pi 的纯上下文是**隐式注入**（`before_agent_start` 把上下文 prepend，agent 知道但 transcript 里不一定显眼）；opencode 是**预填可见**（上下文就在输入框里，用户可编辑）。后者更透明，但语义不同。
- opencode 拿 `promptRef` 需要一坨 SolidJS Slot 脚手架，pi 没有；这是 opencode adapter 比 pi 重的部分（~多 30-50 行）。

**UI / slash 命令 / 生命周期**：
- `api.ui.toast({message})` 反馈；`api.ui.Slot("session_prompt_right", ...)` 可常驻显示"待用上下文"状态 widget。
- `api.command.register(() => [{slash:{name:"eab"}, ...}])` 注册 `/eab paste` 等。
- `api.lifecycle.signal`（AbortSignal）管 socket server；opencode 是单进程（实测，见 `opencode调研.md` §5.1），TUI 在 = opencode 在 = socket 在。

**实现前必须实测确认**（见 `opencode调研.md` §8 #3/#4/#5，命脉项）：
1. `promptRef.set()+submit()` 是否真能发出一条用户消息并触发 LLM；
2. 多 session 时如何拿到"当前活动 session"的 ref；home 页（无 session）怎么办；
3. 预填（set 不 submit）的内容在用户切换窗后是否保留、是否会被清掉。

## 8. 多 Agent 选择（发送端定位目标）

发送端触发 `send` 时，定位目标的决策流：

```
读配置 agentbridge.target
  │
  ├─ 有值 name ──► 在注册表找该 name，验证 socket 可连 ──► 设为 target，发送
  │                    └─ 连不上：提示用户，走下面的"选择"
  │
  └─ 空 ──► discover_listeners()
              ├─ 0 个：InfoBar 提示"没有运行中的 ai agent 接收端"，放弃本次发送
              ├─ 1 个：自动设为 target，直接发送
              └─ 多个：触发 "agent-bridge select" 让用户挑
                        （InfoBar 列出 name + pid + cwd，用户选完即发）
```

- 选定后写入运行态 `current_target`，可持久化到 `agentbridge.target`。
- 每次发送失败（连接错误）要重新选择。
- "多个时每次都问"可能烦——所以一旦选过，持久化到配置，下次直接用，直到失效。

## 9. 边界情况与生命周期

| 场景 | 处理 |
|------|------|
| **agent 崩溃**（没清注册表） | socket 文件残留但连不上；发送端试探连接失败 → 删除该注册文件，提示重新选择 |
| **microNeo 崩溃** | 无影响。按需模式下没有持续连接，agent 端 `pendingContext` 是用户主动送来的，本来就等用户处理 |
| **多 microNeo → 同一 agent** | 报文带 `editor.instance` 区分；agent 的 `pendingContext` 被新报文覆盖（只保留最后一次） |
| **agent 正在 streaming** | 带 prompt 的报文用 `sendUserMessage(..., {deliverAs:"followUp"})`，等当前回合结束再触发，不打断 |
| **用户切换目标 / 切走文件** | 每次发送都是独立的，自然处理 |
| **协议版本不匹配** | 发送端读 `protocol` 字段，不匹配则跳过该 listener 并告警 |

## 10. 安全性

- **只走本地 Unix socket**，无网络监听、无端口暴露。
- 注册表目录 `0700`，socket 文件 `0600`，仅本用户可访问。
- 报文里 **默认只带选中文本**（可见区域文本默认关闭），降低敏感信息意外进入 agent 上下文的风险。
- 可考虑加"敏感文件路径前缀"黑名单（见开放问题），命中时不发送。
- 未来双向扩展（agent → editor，如 `goto line`）必须显式经用户/配置授权后才允许执行。

## 12. 开放问题（待讨论）
