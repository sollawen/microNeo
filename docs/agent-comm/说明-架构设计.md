## 一、系统定位

**一句话**：让用户在 microNeo 里编辑文件时，能**按需、一次性**地把当前编辑上下文（光标位置 / 选区 / 可附一句话提问）推送给一个正在运行的 ai agent（pi、opencode…）。

**关键限定**（来自 `通信的原始需求.md`）：
- 通信方向：**microNeo → agent 单向**（v1）
- 触发方式：**用户主动、按需、一次性**（不是对光标/选区变化的持续追踪）
- 多对多：一台机器上可能有多个 microNeo 进程（编辑不同文件）+ 多个 agent 进程

**为什么存在**：用户在编辑器里写代码/文档，想让 agent 看"我现在在看的这里"，免去复制粘贴上下文、切窗口、描述位置的繁琐。

## 二、三个角色

```
┌──────────────────────────────────────────────┐
│  microNeo（全 Go，写死，与具体 agent 无关）    │
│  ┌─────────────┐   ┌──────────────────────┐  │
│  │ notePane    │   │ 发送管线（Go）        │  │
│  │ 浮窗 UI     │──→│ 采集·组装·发现·选择·发│  │
│  └─────────────┘   └──────────┬───────────┘  │
└───────────────────────────────┼──────────────┘
                                 │ EABP 协议（见说明-EABP.md）
            ┌────────────────────┼────────────────────┐
            ▼                    ▼                    ▼
     ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
     │ pi 扩展（TS）│    │ opencode 插件│    │ XXX 扩展     │
     │ 注册/监听/   │    │（TUI 插件）  │    │（社区写）    │
     │ 递送 LLM     │    │              │    │              │
     └──────────────┘    └──────────────┘    └──────────────┘
            ▲                    ▲                    ▲
            │              注册表（Registry）
            │         固定目录，无独立进程
            └──────────── 退出时登记/注销 ────────────┘
```

| 角色 | 实体 | 形态 | 核心职责 |
|------|------|------|---------|
| **发送端 Sender** | microNeo 进程 | Go（写死在 microNeo 核心） | 采集上下文 → 发现接收端 → 选目标 → 发报文 |
| **接收端 Receiver** | agent 内的扩展程序 | 各 agent 自有机制（pi=TS 扩展 / opencode=TUI 插件） | 启动注册 → 监听 → 解析报文 → 递送到自己的 LLM 通道 |
| **注册表 Registry** | 一个固定目录 | 文件系统（无进程） | 接收端登记自己；发送端发现接收端 |

### 一次完整发送经过哪些角色

```
用户在 microNeo 按热键
  → notePane 浮窗打开（冻结主 pane），用户在其中写下提问/注解
  → 发送命令把 notePane 内容 + 主 pane 上下文交给发送管线
  → 发送管线：采集 → 组装报文 → 查注册表发现接收端 → 选目标 → connect socket → 写一行 JSON → close
  → 接收端：accept → 分帧解析 → 按 message 有无分流 → 递送到 LLM 通道 / 暂存待用
  → agent 里用户看到结果（AI 开始回复 / 提示有上下文待用）
```

详细流程：协议层见 `说明-EABP.md §五/六/九`；发送端实现见 `说明-发送端.md`；接收端实现见 `说明-接收端.md`。

## 三、分层与职责边界

把系统切成四层，每层职责单一、边界清晰：

```
┌─────────────────────────────────────────────────────────┐
│  ① UI 层（发送端）                                       │
│     notePane 浮窗：让用户在光标下方写下提问/注解           │
│     → 输出：content（提问）+ 主编辑器上下文               │
├─────────────────────────────────────────────────────────┤
│  ② 发送管线层（发送端，Go）                               │
│     采集上下文·组装报文·查注册表·选目标·连接发送·反馈      │
│     → 输出：一行 JSON 报文写到目标 socket                 │
├─────────────────────────────────────────────────────────┤
│  ③ 协议层（跨两端，契约）                                 │
│     注册表格式·传输方式（Unix socket + 逐行 JSON）         │
│     报文 schema·版本·递送语义                             │
│     → 输出：两端都遵守的契约                              │
├─────────────────────────────────────────────────────────┤
│  ④ 接收适配层（接收端，各 agent）                         │
│     socket 监听·注册·分帧·按 message 分流                 │
│     → 递送到 agent 自己的 LLM 通道                        │
└─────────────────────────────────────────────────────────┘
```

各层的归属与文档：

| 层 | 跑在哪 | 语言 | 设计/实现文档 |
|----|--------|------|-------------|
| ① UI 层 | microNeo 进程 | Go | `说明-notepane.md` |
| ② 发送管线层 | microNeo 进程 | Go | `说明-发送端.md` |
| ③ 协议层 | 跨两端（契约） | 语言无关 | `说明-EABP.md` |
| ④ 接收适配层 | agent 进程 | 各 agent 自有（TS） | `说明-接收端.md`；opencode 见 `opencode调研.md` |

**层间依赖方向**：
- **发送端**：①→②→③（层层下沉到协议）
- **接收端**：④ 依赖 ③（实现协议契约）
- **③ 是唯一的跨端契约**——两端都不直接依赖对方的具体实现

### 3.1 为什么这样分层

- **UI 与发送管线分离**：notePane 只管"让用户舒服地写下文字"，发送管线只管"把东西可靠地送到"。两者通过 `(content, 主编辑器上下文)` 解耦——notePane 提供 content，被冻结的主 BufPane 提供路径/光标/选区。未来可以加其他入口（如"快速发送选区"热键），共用同一条发送管线。对接见 `说明-EABP.md §6.1` 与 `说明-发送端.md §三-四`。
- **协议层独立成契约**：这是 LSP 式架构的支撑点。协议层不依赖任何一端的实现细节，两端各自实现它。
- **接收端自成一层、各 agent 独立**：接收适配层是"差异集中区"——每个 agent 的实现都不同，但都满足同一个协议契约。这一层可以抽 core（agent 无关）+ adapter（agent 特定）两小层，见 `说明-EABP.md §8.2`。

## 四、架构原则

下列五条原则是整个系统设计的硬约束，任何分文档的设计都必须服从。

### 原则 1：LSP 式架构——协议 agent 无关

microNeo 只定义协议（注册表格式、传输方式、报文 schema、递送语义），**不关心背后是 pi 还是 opencode**。每个 agent 按协议实现"接收端"即可被识别。

> **价值**：接入新 agent 时**完全不用碰 microNeo 的代码**。这是"精力有限、只亲自接 pi + opencode、长尾留给社区"的最佳支撑。详见 `说明-EABP.md §1.2 铁律 1`。

### 原则 2：发送端全 Go，写死在 microNeo 核心

发送端的所有核心代码（UI、采集、发现、选择、发送）都用 Go 写在 microNeo 核心包内。**microNeo 自身不包含任何针对具体 agent 的 Lua 插件**。

> 满足 micro 原生零侵入（AGENTS.md「对 micro 原生代码的侵入和改动越小越好」），且通信逻辑属于核心能力。详见 `说明-EABP.md §1.2 铁律 1`。

### 原则 3：Lua 不参与通信逻辑

Lua 只在"用户级个性化"层面保留极小空间（自定义热键等 micro 原生能力），**不参与任何 agent 通信**。通信发送管线是 microNeo 的 Go 核心，不存在任何 agent 通信相关的 Lua 钩子。

> 详见 `说明-EABP.md §1.2 铁律 3`。

### 原则 4：能力差异由接收端吸收

协议只规定"**递送什么**"，不规定"agent 怎么用"。各 agent 的扩展机制负责把递送物落到自己的 LLM 通道。

> **典型例**：纯上下文报文，pi 用 `setEditorText` 填入输入框（用户接管）；opencode 可能用"预填输入框"。两者都是合规的协议实现，协议不强求统一。详见 `说明-EABP.md §六（递送语义）` 与 `说明-接收端.md §六`（pi 实际选择）。

### 原则 5：按需单向，不做持续追踪

v1 只做 microNeo → agent 单向、用户主动、一次性推送。不做光标/选区变化的持续追踪、不做内存快照周期刷新。

> 这决定了系统的全部轻量性——没有 debounce、没有长连接状态管理、没有高频事件流。详见 `通信的原始需求.md`。

## 五、职责矩阵

明确"谁做什么、不做什么"，避免边界模糊。

### 5.1 microNeo（发送端，①②层）

**做**：
- notePane 浮窗 UI（让用户在光标下方写下提问/注解；内容存 BTScratch buffer，关闭即丢）
- 采集编辑上下文（路径、光标、选区）——见 `说明-发送端.md §四`
- 组装协议报文（构造 envelope + payload → `MarshalLine`）
- 发现接收端（`eabp.Discover`：查注册表、试探连接、GC 僵尸注册）
- 多目标时让用户选择（**当前 MVP**：恰好 1 个 receiver 才发，0 或多个则报错）
- 连接 socket 发送报文（connect → write 一行 → close）
- InfoBar 反馈发送结果

**不做**：
- 不针对任何具体 agent 写代码（无 pi 插件、无 opencode 插件）——**零 Lua 钩子**
- 不维护长连接、不做持续推送、不做事件追踪
- 不决定"agent 怎么用这份上下文"（那是接收端的事）

### 5.2 agent 扩展（接收端，④层）

**做**：
- 启动时写注册表（起 socket + 登记自己）——见 `说明-接收端.md §四`
- 监听 socket、分帧解析报文（`\n` 分帧 + `JSON.parse` + 校验 `v`/`type`）
- 按协议递送语义处理（带 `message` → 触发对话；纯上下文 → 填入输入框 / 待用）
- 退出时注销注册（关 server + 删注册文件 + 删 socket）

**做（因 agent 而异）**：
- 带 `message` 时怎么把消息送进 LLM（pi：`pi.sendUserMessage(text, {deliverAs: "steer"})`；opencode：TBD）
- 纯上下文怎么"待用"（pi：`ctx.ui.setEditorText`；opencode：TBD）
- UI 反馈形式（pi：notify；opencode：TBD）

**不做**：
- 不修改 microNeo 的任何代码
- 不依赖 microNeo 的内部 API（只遵守协议契约）

### 5.3 协议（③层，契约）

**规定**（详见 `说明-EABP.md`）：
- 注册表目录算法、注册文件格式（§三）
- 传输方式（Unix socket + 逐行 JSON）（§四）
- 报文信封与 payload schema、版本号（§五、§七）
- 递送语义（`message` 有无的分流规则）（§六）
- 接收端合规要求（§八）

**不规定**：
- agent 用什么扩展机制实现
- agent 收到后具体怎么递送给 LLM
- UI 怎么呈现

## 六、关键设计决策

每条只给决策 + 一句话理由，细节看对应分文档。

| # | 决策 | 理由 | 详见 |
|---|------|------|------|
| 1 | 发送端全 Go 写死，非 Lua 插件 | 满足 micro 原生零侵入，且通信逻辑属于核心能力 | 本文 §四 原则 2 / `说明-EABP.md` §1.2 |
| 2 | LSP 式架构，协议 agent 无关 | 接入新 agent 不改 microNeo；精力有限只亲自接 pi+opencode | 本文 §四 原则 1 / `说明-EABP.md` §1.2 |
| 3 | IPC = Unix domain socket | 本地零网络暴露、快、跨语言成熟；v1 覆盖 macOS/Linux，Windows 因 AF_UNIX（Go 1.17+ 原生支持）留 v2 | `说明-EABP.md` §四 |
| 4 | 每次发送一次连接→写一行→关闭 | 握手开销极小，换无状态管理 + 对 agent 重启鲁棒 | `说明-EABP.md` §4.2 |
| 5 | 注册表 = 固定目录，无独立进程 | 最简、无新依赖、文件系统天然持久 | `说明-EABP.md` §三 |
| 6 | 发送端负责 GC 僵尸注册 | 发送端是唯一"会去连"的角色，顺手 GC 最自然 | `说明-EABP.md` §3.5 |
| 7 | 按 `message` 有无分两条递送路径 | 对应"用户问 vs 挂上下文"两种意图；协议层一致，实现层各 agent 自由 | `说明-EABP.md` §六 |
| 8 | 用"递送"而非"注入"措辞 | opencode 是预填输入框非隐式改写；措辞要诚实反映两种实现 | `opencode调研.md` §7.2 |
| 9 | UI = notePane 浮窗 | 嵌入 BufPane 获得完整编辑能力（白名单 bindings），内容存 BTScratch buffer（关闭即丢）；浮窗位于光标下方，打开时冻结主 pane | `说明-notepane.md` / `用户界面-V2.md` |
| 10 | opencode 接收端用 TUI 插件 | 走原生输入通道，server 无感；无头模式是伪需求 | `opencode调研.md` §6.5/§6.6 |
| 11 | line/col 1-based | 与 LLM 工具链（sed/ripgrep/read）天然对齐，不需 ±1 转换 | `说明-EABP.md` §5.3 + §11 |
| 12 | 选区文本阈值 = 30 行 | 行数对用户可见可预测，字节阈值不可见；超阈值 LLM 读文件 | `说明-发送端.md` §3.3 / `说明-接收端.md` §七 |
| 13 | Alt-Enter 在两个上下文做不同事 | 主编辑器侧 Alt-Enter = 打开 notePane；notePane 内 Alt-Enter = 发送 | `说明-notepane.md` §七 |
| 14 | 私有 `notePaneActions` map | notePane 专属 action 不进 `BufKeyActions`（全局），`bufpane.go` 净减一行 | `说明-notepane.md` §6.3 |

## 七、边界与约束（明确不做什么）

| 边界 | v1 现状 | 备注 |
|------|---------|------|
| **不做双向通信** | ✅ 不做 | agent → microNeo 的 `ack`/`goto`/高亮留待后续，且必须显式授权 |
| **不做持续追踪** | ✅ 不做 | 不追踪光标移动、选区变化；不做内存快照周期刷新 |
| **不做长连接** | ✅ 不做 | 每次发送独立连接，换无状态管理 |
| **不做 Windows** | ✅ 不做 | 接收端（Node）支持需验证，整体留 v2 |
| **不做无头 agent 接收端** | ✅ 不做 | `opencode serve` 等无用户、无对话的场景不在范围内 |
| **不做多 receiver 选择 UI** | ✅ 不做（MVP） | 当前实现：0 或多个 receiver → 报错；恰好 1 个 → 发。产品功能后续迭代 |
| **microNeo 不内置任何 agent 专用代码** | ✅ 做到 | 所有 agent 接入都是外挂扩展 |
| **不持久化 notePane 内容** | ✅ 做到 | BTScratch buffer，关闭即丢（避免隐私风险 + 与 agent 那边的对话历史重复） |
| **不发送 `visible_lines` 字段** | ✅ 做到 | v1 不发送可见区域文本，降低敏感信息意外进入 agent 上下文的风险 |

## 九、开放问题（架构级）

1. **新 agent 接入的标准化程度**：协议契约（§5.3）够不够清晰让社区独立实现？是否需要一份独立的"接收端实现规范"文档？（详见 `说明-EABP.md §12.1`）
2. **空 content 发送的语义**：当前 notePane 链路**直接拦截不发**（D10 决策 1）；其他入口（如未来"快速发送选区"热键）可能需要不同决策（详见 `说明-EABP.md §12.1`）
3. **多份待用上下文**：v1 多来源覆盖只保留最后一次，未来是否按来源保留多份？（详见 `说明-EABP.md §12.1`）

6. **`sender.instance` 的真实实现**：v1 MVP 固定 `"default"`。microNeo 多实例/多窗口支持需要先做"窗口/tab ID"概念（详见 `说明-EABP.md §12.1`）
