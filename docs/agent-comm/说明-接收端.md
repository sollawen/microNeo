## 三、registryDir()（与 Go 端一致）

```typescript
function registryDir(): string {
  const override = process.env.MNAB_REG_DIR;
  if (override) return override;
  const base = process.env.XDG_RUNTIME_DIR
    || process.env.TMPDIR
    || "/tmp";
  return path.join(base, `microneo-agent-bridge-${process.getuid?.() ?? 0}`);
}
```

**逐字符对齐** `internal/eabp/registry.go:RegistryDir()`：

- `MNAB_REG_DIR` 覆盖优先
- `$XDG_RUNTIME_DIR` → `$TMPDIR` → `/tmp` 优先级回落
- `process.getuid?.() ?? 0`：`getuid` 仅 Unix 可用，`?? 0` 兜底

**`process.getuid?.()` 是 TS 的 optional call 语法**——只在方法存在时调用，Node Windows 版本不报错。

## 四、生命周期

### 4.1 注册（session_start）

`pi.on("session_start", async (_e, ctx) => { ... })`：

```typescript
pi.on("session_start", async (_e, ctx) => {
  name = `pi-${process.pid}`;        // PID 保证唯一
  const dir = registryDir();
  fs.mkdirSync(dir, { recursive: true, mode: 0o700 });
  socketPath = path.join(dir, `receiver-${name}.sock`);
  regFile    = path.join(dir, `receiver-${name}.json`);

  server = net.createServer((conn) => {
    let buf = "";
    conn.on("data", (chunk) => {
      buf += chunk.toString("utf8");
      let nl;
      while ((nl = buf.indexOf("\n")) >= 0) {
        handleLine(buf.slice(0, nl), ctx);
        buf = buf.slice(nl + 1);
      }
    });
  });
  await new Promise<void>((res, rej) => {
    server!.listen(socketPath, () => res());
    server!.on("error", rej);
  });
  fs.chmodSync(socketPath, 0o600);
  fs.writeFileSync(regFile, JSON.stringify({
    name, pid: process.pid, transport: "unix",
    socket: socketPath, protocol: PROTOCOL,
    startedAt: Math.floor(Date.now()/1000), cwd: process.cwd(), labels: ["default"],
  }));
});
```

**关键步骤**：

1. **name = `pi-${process.pid}`**——PID 天然唯一，规避冲突处理（说明-EABP §3.3）
2. **mkdirSync recursive mode 0o700**——注册表目录，权限仅本用户
3. **net.createServer** 监听 socket，**手写 buffer 分帧**（Node 无内置逐行 socket reader）
4. **fs.chmodSync(socketPath, 0o600)**——socket 文件 0600
5. **写注册文件**——含全部说明-EABP §3.2 字段

**会话切换**：`/new` / `/resume` / `/reload` 会先 `session_shutdown` 旧实例再 `session_start` 新实例。每次都重新走一遍注册流程。microNeo 发送端靠 connect 试探自动 GC 旧注册（`说明-发送端.md §2.2`）。

### 4.2 注销（session_shutdown）

```typescript
pi.on("session_shutdown", async () => {
  if (server) {
    await new Promise<void>((res) => server!.close(() => res()));
    server = null;
  }
  try { fs.unlinkSync(regFile); } catch {}
  try { fs.unlinkSync(socketPath); } catch {}
});
```

**关键步骤**：

1. `server.close()` 等所有 conn 关闭（**不**强制断开，由 TCP 自然结束）
2. `unlinkSync` 注册文件（包 `try/catch`——文件可能已不存在）
3. `unlinkSync` socket 文件（同上）
4. 清 `server = null`

> **M1 → M2 演进**：M1 时还有 `let pending: any = null; pending = null;` 在 shutdown 里清空状态——M2 改用 `setEditorText` 后无状态，删了。

### 4.3 状态变量

```typescript
let server: net.Server | null = null;
let name = "", socketPath = "", regFile = "";
```

**闭包变量**——`session_start` 写，`session_shutdown` 读/清。**没有 M2 那种 pending 状态**——M2 改用 `setEditorText` 后无服务端状态。

## 五、报文解析

### 5.1 分帧

```typescript
server = net.createServer((conn) => {
  let buf = "";
  conn.on("data", (chunk) => {
    buf += chunk.toString("utf8");
    let nl;
    while ((nl = buf.indexOf("\n")) >= 0) {
      handleLine(buf.slice(0, nl), ctx);
      buf = buf.slice(nl + 1);
    }
  });
});
```

**不定长 buffer** + `indexOf("\n")` 循环——按需增长，无定长截断风险。理论上报文可以任意大（虽然实际场景不会）。

### 5.2 解析 + 版本校验

```typescript
function handleLine(line: string, ctx: any) {
  let env: any;
  try { env = JSON.parse(line); } catch { return; }
  if (env.v !== 1 || env.type !== "context") return;
  onMessage(env, ctx);
}
```

**校验规则**：

- JSON 解析失败 → **静默忽略**（catch 后 return）
- `env.v !== 1` → 静默忽略（协议主版本不匹配）
- `env.type !== "context"` → 静默忽略（说明-EABP §7.1「忽略未知 type」前向兼容；v1 的 `bye` 报文自然走这条路径）

**不告警**：避免报文洪泛时噪音爆炸；前向兼容的体现。

## 六、递送语义（协议核心）

```typescript
function onMessage(env: any, ctx: any) {
  const p = env.payload;
  const text = formatText(p);

  if (p.message) {
    // 带 message → 走 sendUserMessage（永远带 deliverAs: "steer"）
    pi.sendUserMessage(text, { deliverAs: "steer" });
  } else {
    // 纯上下文 → 填入输入框，不触发对话
    ctx.ui.setEditorText(text);
    ctx.ui.notify(`📌 已放入输入框，可编辑后发送`, "info");
  }
}
```

**两条路径**（说明-EABP §六）：

| `p.message` 存在？ | 递送方式 | 用户体验 |
|--------------------|---------|----------|
| **是**（非空字符串） | `pi.sendUserMessage(text, {deliverAs: "steer"})` | 立即/排队触发 LLM 对话 |
| **否**（字段缺省或空） | `ctx.ui.setEditorText(text)` + `notify` | 填入输入框，用户编辑后手动发送 |

### 6.1 带 message 路径：永远用 `deliverAs: "steer"`

**为什么不需要 `isIdle()` 分流**：

`pi.sendUserMessage` 内部调 `prompt()`，唯一硬约束是「streaming 时**不带** `streamingBehavior` 会抛错」。只要**永远带上** `deliverAs`，idle/streaming 两种状态都不会抛错：

- **idle 时**：`steer` 参数被忽略（`prompt()` 的 streaming 分支不进入），走正常立即提交
- **streaming 时**：进入 steer 队列，等当前回合工具链跑完、下次 LLM 调用前插入

**`steer` vs `followUp`**：

- `steer`：当前回合工具链跑完、下次 LLM 调用前插入（**快**）
- `followUp`：等到 agent 完全空闲（多轮工具全部结束）（**慢**）

`steer` 适合"用户发来新问题"语义——尽快让用户从 microNeo 发来的问题被处理。

### 6.2 纯上下文路径：填入输入框

```typescript
ctx.ui.setEditorText(text);
ctx.ui.notify(`📌 已放入输入框，可编辑后发送`, "info");
```

**设计原则**：

- **不触发对话**（说明-EABP §6.4「不得自动触发」）
- **不暂存**（无 `pending` 状态、M2 时删了）——文本直接进输入框
- **用户完全控制**：可编辑、可删、可补充、可丢弃
- **多来源覆盖**：发两份纯上下文 → 后者**直接覆盖**输入框内容（说明-EABP §6.5）
- **notify 提示**：告知"有上下文待用"

> **为什么不用 `before_agent_start` 隐式注入**：参考 `pi-in-zellij/editor/editor.ts` 已验证——外部内容直接填输入框更透明、更简单、更符合「由用户决定怎么用」。无状态、无泄漏、无意外。

## 七、文本格式化（formatText）

**当前实现**（D10 决策 4 方案 B 后）：

```typescript
function formatText(p: any): string {
  const sel = p.selection;
  const selText = sel?.text && sel.text.length > 0 ? sel.text : "";

  if (sel && selText) {
    // 有选区且文字未截断：内联文字（用自然语言标位置，不用 @ 语法）
    const header = `来自 ${p.path} 第 ${sel.start.line}-${sel.end.line} 行的选中内容：`;
    return p.message ? `${header}\n\n${selText}\n\n${p.message}` : `${header}\n\n${selText}`;
  }

  // 无选区 / 选区文字被截断（超过 MaxSelectionLines）：走 @ 引用
  const focus = sel
    ? `line${sel.start.line}-line${sel.end.line}`
    : `${p.cursor.line}`;
  const base = `@${p.path} :line${focus}`;
  return p.message ? `${base}\n\n${p.message}` : base;
}
```

### 7.1 决策点：`Selection.Text` 是否为空

| 情况 | `selText` | 走的分支 | LLM 行为 |
|------|-----------|---------|---------|
| 无 selection | `""` | @ 引用 | 读文件 |
| 有 selection，Text 非空（≤30 行） | 实际文字 | **内联 + 自然语言标位置** | **不读文件**（用内联文字） |
| 有 selection，Text 被发送端截断（>30 行） | `""` | @ 引用 | 读文件 |

**两端协调**：

- **发送端**（`internal/action/notepane.go:lowestCursorScreenRow`）决定发不发 Text：`lineSpan > MaxSelectionLines` 时 `selText = ""`
- **接收端**（`formatText`）只看 Text 是否为空：是 → 内联；否 → @ 引用
- 接收端**不做行数判断**——行数是发送端决策的结果，已经体现在 Text 是否为空上

### 7.2 四种渲染形态

| 场景 | 输出 | 走哪条路径 |
|------|------|-----------|
| 有选区 + Text 非空 + message | `来自 /path 第 A-B 行的选中内容：\n\n<selText>\n\n<message>` | `sendUserMessage` |
| 有选区 + Text 非空 + 无 message | `来自 /path 第 A-B 行的选中内容：\n\n<selText>` | `setEditorText` |
| 无选区 / Text 空 + message | `@/path :line<focus>\n\n<message>` | `sendUserMessage` |
| 无选区 / Text 空 + 无 message | `@/path :line<focus>` | `setEditorText` |

### 7.3 `@path :lineX` 语法

LLM 工具链（Cursor/Claude/Copilot）通用约定。**pi 系统提示会触发 LLM 用 read 工具去读这个文件**。

**D10 关键洞察**：只要消息里有 `@path :lineX` 格式，**LLM 一定会去 read 文件**——哪怕 `Selection.Text` 已经把内容内联在消息里。所以"既放 @ 引用又内联文字"反而**双倍消耗**（read + 内联都在）。

**省 token 的唯一办法**：内联时**彻底不放** `@` 引用格式。改用自然语言标位置（"来自 X 第 N-M 行"）——LLM 不触发 read，直接用内联文字。

### 7.4 行号基址

`p.cursor.line` / `p.selection.start.line` / `p.selection.end.line` 都是 **1-based**（v1 协议），与 LLM 工具（sed/ripgrep/read）天然对齐。formatText 输出直接用，不做 ±1 转换。

### 7.5 为什么不用 `@` 引用 + 内联组合

见 §7.3。`@` 语法会强制 read → 内联文字被无视 → 双倍消耗。

## 八、错误处理

| 场景 | 处理 |
|------|------|
| JSON parse 失败 | `handleLine` catch 后 return（不告警） |
| `env.v !== 1` | 静默 return（协议主版本不匹配） |
| `env.type !== "context"` | 静默 return（未知 type，前向兼容） |
| `socket listen` 失败 | `session_start` 的 Promise reject（pi 报错扩展加载失败） |
| `socket close` 失败 | `try/catch` 吞掉（unlink 文件可能已不在） |
| `unlinkSync` 失败 | `try/catch` 吞掉（文件可能已被外部清理） |

**v1 不做的事**（明确出界）：

- ❌ `pending` 暂存 / widget / slash 命令（M2 删了 pending，纯上下文走 `setEditorText`）
- ❌ `before_agent_start` 注入（M2 时考虑过，D10 改 formatText 后不需要）
- ❌ `bye` 报文处理（microNeo 不发；说明-EABP §7.1「忽略未知 type」天然前向兼容）
- ❌ 接收端读文件补全上下文（违反「只递送事实」原则 + 有文件竞态）
- ❌ 输入框内容持久化（用户没发送就丢——符合无状态设计）
- ❌ 未知 `type` 报文的告警（前向兼容，不告警）
- ❌ 协议版本不匹配的告警（静默忽略）
