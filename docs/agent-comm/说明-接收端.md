# AIBP 接收端说明（pi 实现）

> 本文档描述 pi AIBP 接收端（`aibp-receivers/aibp-pi/`）的**当前实现状态**——以最终代码为准。
>
> **覆盖范围**：
> - pi 扩展的 socket 生命周期（注册/注销）
> - 报文解析与版本校验
> - 按 `message` 有无分流的递送语义
> - `formatText` 文本格式化（D10 决策 4 纯内联 + 尖括号标签分隔）
>
> **不覆盖**：
> - AIBP 协议本身（见 `说明-AIBP.md`）
> - microNeo 发送端实现（见 `说明-发送端.md`）
> - opencode / 其他 agent 接收端（未实现，调研见 `opencode调研.md`）
>
> **与历史文档的关系**：D5 是 pi 接收端初版设计（M2 形态）；D10 决策 4 改写 `formatText`（从 `@path :lineX` + 内联组合 → 纯内联，触发端读文件被关闭）；后续又加尖括号标签分隔三段（`<selection: ...>` / `<user input>`），进一步提升 LLM 语义分类准确度。本文档以最终代码为准。

## 三、registryDir()（与 Go 端一致）

```typescript
function registryDir(): string {
  const override = process.env.MNAB_REG_DIR;
  if (override) return override;
  const base = process.env.XDG_RUNTIME_DIR
    || process.env.TMPDIR
    || "/tmp";
  return path.join(base, `microneo-agent-bridge-${process.getuid?.() ?? 0}`);
}
```

**逐字符对齐** `internal/aibp/registry.go:RegistryDir()`：

- `MNAB_REG_DIR` 覆盖优先
- `$XDG_RUNTIME_DIR` → `$TMPDIR` → `/tmp` 优先级回落
- `process.getuid?.() ?? 0`：`getuid` 仅 Unix 可用，`?? 0` 兜底

**`process.getuid?.()` 是 TS 的 optional call 语法**——只在方法存在时调用，Node Windows 版本不报错。

## 四、生命周期

### 4.1 注册（session_start）

名字生成走 **D11 名字池方案**（详见 §4.4）：从池子里顺序占用一个干净名字，listen 抢锁，写注册文件，footer 显示。

```typescript
pi.on("session_start", async (_e, ctx) => {
  // 1. 加载名字池（§4.4）：返回干净 string[]，或 null（池子文件坏）
  const names = loadNamePool(ctx);
  if (names === null) return;             // 池子文件坏，notify 已在 loadNamePool 完成

  // 2. connection handler 闭包（捕获 ctx）
  const connectionHandler = (conn: net.Socket) => {
    let buf = "";
    conn.on("data", (chunk) => {
      buf += chunk.toString("utf8");
      let nl;
      while ((nl = buf.indexOf("\n")) >= 0) {
        handleLine(buf.slice(0, nl), ctx);
        buf = buf.slice(nl + 1);
      }
    });
  };

  // 3. 分配名字（§4.4）：listen 在内部抢锁；server 通过闭包赋值给外层变量
  const got = await allocateName(names, connectionHandler);
  if (got === null) {                     // 池子耗尽
    ctx.ui.notify("⚠ 名字池已满，本次不接收消息", "warning");
    return;
  }
  name = got.name;                        // 形如 "Alpha"
  socketPath = got.socketPath;

  // 4. 写注册文件（server 已在 allocateName 内 listen 完成）
  regFile = path.join(registryDir(), `ai-${name}.json`);
  fs.writeFileSync(regFile, JSON.stringify({
    name, pid: process.pid, transport: "unix",
    socket: socketPath, protocol: PROTOCOL,
    startedAt: Math.floor(Date.now()/1000), cwd: process.cwd(), labels: ["default"],
  }));

  // 5. footer 显示自己的名字（注册成功后才显示）
  ctx.ui.setStatus("aibp", ctx.ui.theme.fg("accent", `● ${name}`));
});
```

**关键步骤**：

1. **loadNamePool**——读池子文件（`$XDG_CONFIG_HOME/aibp/aibp-names.json`），首次自动种子默认 15 词；解析失败 → notify + 放弃注册（不覆盖用户文件）
2. **allocateName**——listen 原子抢锁占用一个名字（§4.4）；池子耗尽 → notify + return
3. **connectionHandler 由调用方传入**——确保 listen 抢锁创建的 server 就是最终处理消息的 server（避免「listen 成功但消息没人处理」的空 server）
4. **写注册文件**——含全部说明-AIBP §3.2 字段
5. **setStatus 显示 footer**——`ctx.ui.theme.fg("accent", "● Alpha")`

**会话切换**：`/new` / `/resume` / `/reload` 会先 `session_shutdown` 旧实例再 `session_start` 新实例。名字按池子顺序分配——第一个接收端叫 `Alpha`，第二个 `Bravo`，依此类推；旧实例退出后释放名字，后续接收端可重用。

**与 microNeo 发送端的协作**：microNeo 扫描 `registryDir` 下所有 `ai-*.json`，并发往每个 socket。多 pi 并存时每个 pi 各自收到消息（说明-AIBP §3.3）。pi 这边靠 `process.kill(pid, 0)` + listen 抢锁自管占用集合，不依赖发送端 GC。

### 4.2 注销（session_shutdown）

```typescript
pi.on("session_shutdown", async (_e, ctx) => {
  ctx.ui.setStatus("aibp", undefined);    // 清除 footer 显示
  if (server) {
    await new Promise<void>((res) => server!.close(() => res()));
    server = null;
  }
  try { fs.unlinkSync(regFile); } catch {}
  try { fs.unlinkSync(socketPath); } catch {}
});
```

**关键步骤**：

1. **setStatus(key, undefined)**——清除 footer 的 `aibp` 条目；对不存在的 key 幂等安全
2. `server.close()` 等所有 conn 关闭（**不**强制断开，由 TCP 自然结束）
3. `unlinkSync` 注册文件（包 `try/catch`——文件可能已不存在）
4. `unlinkSync` socket 文件（同上）
5. 清 `server = null`

> **注**：所有事件（含 `session_shutdown`）都接收 `ctx`——`ExtensionHandler<E,R> = (event, ctx) => ...` 类型签名保证。无需额外守卫。

### 4.3 状态变量

```typescript
let server: net.Server | null = null;
let name = "", socketPath = "", regFile = "";
```

**闭包变量**——`session_start` 写（`server` 在 `allocateName` 内通过闭包赋值），`session_shutdown` 读/清。**无服务端业务状态**——M2 改用 `setEditorText` 后，连接 handler 不再携带 pending 状态。

### 4.4 名字池机制（D11）

D11 方案详见 `docs/agent-comm/D11-名字分配方案.md`。本节列实现要点。

#### 4.4.1 池子文件

位置：`$XDG_CONFIG_HOME/aibp/aibp-names.json`（fallback `~/.config/aibp/aibp-names.json`）。

- 持久化配置目录（与 `registryDir` 的运行时目录分离）
- 首次启动自动种子默认 15 词（NATO 音标 `Alpha`–`Oscar`，`X` 用 `Xray` 避连字符）
- 用户可手改，重载即生效

#### 4.4.2 三步规范化（normalizeNames）

顺序固定：**截断 → 去重 → 字符过滤**（在 load 时一次性做完，allocateName 无需再判合法性）：

1. **截断到 10 字符**——控制 socket 路径总长（最坏 ~68 字符 vs Linux 108 上限）；非 string 元素 → 空串
2. **保留首次出现去重**（不排序）——保持「有序占用」语义
3. **字符过滤**——跳过含 `/` `\0` `:` 空格 `-` 的 name（`-` 保持禁止：避免 name 自带 `-` 让 socket 文件名 `ai-<name>.sock` 里的 `ai-` 分隔标记产生视觉歧义）；记 warn 日志，不报错

#### 4.4.3 加载三分支（loadNamePool）

| 情况 | 行为 |
|------|------|
| 文件存在 + 合法非空数组 | 读取 → 规范化 → 返回 |
| 文件不存在 / 内容空 / 非数组 | **种子默认值**（原地写文件）→ 返回默认 10 词 |
| 解析失败（非法 JSON） | `ctx.ui.notify("⚠ ... 格式错误")` + 返回 null（**不覆盖**用户文件，留给用户修） |

返回 null 时调用方应直接放弃注册——不要因为池子坏就静默不接收，也不要覆盖用户数据。

#### 4.4.4 分配名字（allocateName）

签名：`allocateName(names, connectionHandler) → { name, socketPath } | null`

核心思路：**listen 是原子锁**——成功 listen 即占用，无需文件锁或 PID 仲裁。

```
for (name in names):
  if name in occupied:           # 扫描 registryDir + kill(pid, 0) 探活构建
    continue
  sockPath = .../ai-<name>.sock
  if tryListen(sockPath):        # listen 成功 = 抢到锁
    return { name, socketPath }
  # listen 失败（EADDRINUSE）→ connect 试探区分真活 vs 僵尸 socket
  if not connect_probe(sockPath):   # 200ms 超时；connect 失败 = 僵尸
    unlink sockPath
    if tryListen(sockPath):
      return { name, socketPath }
  # connect 成功（真活）OR 重 listen 失败 → 换下一个
return null                       # 池子耗尽
```

**三道占用检测**（层层兜底）：

1. **occupied 集合**——扫 `ai-*.json`，`process.kill(pid, 0)` 探活；死 PID 的注册顺手 GC（删 json + sock）
2. **listen 抢锁**——EADDRINUSE 即说明 socket 被占
3. **connect 试探**——listen 失败时判断占用的 socket 是真有接收端 vs 僵尸文件；僵尸则删后重 listen

> **关键 API 修正**：探活用 `process.kill(pid, 0)`（Node 全局），**不是** `child_process.kill`——后者不存在，抛 TypeError 会被 catch 吞掉，导致所有 PID 被判死、所有注册被 GC。

**connectionHandler 由调用方传入**：listen 创建 server 时即挂上分帧逻辑——抢锁的 server 就是最终处理消息的 server，不会出现「listen 成功但消息没人处理」的空 server。

**listen 成功后的状态切换**：抢锁阶段用 `once("listening")` + `once("error")` + `settled` guard 封装成 Promise；成功后 `removeAllListeners("error")` 换成持久日志 handler，避免运行时 error 变成 unhandled rejection。

## 五、报文解析

### 5.1 分帧

```typescript
server = net.createServer((conn) => {
  let buf = "";
  conn.on("data", (chunk) => {
    buf += chunk.toString("utf8");
    let nl;
    while ((nl = buf.indexOf("\n")) >= 0) {
      handleLine(buf.slice(0, nl), ctx);
      buf = buf.slice(nl + 1);
    }
  });
});
```

**不定长 buffer** + `indexOf("\n")` 循环——按需增长，无定长截断风险。理论上报文可以任意大（虽然实际场景不会）。

### 5.2 解析 + 版本校验

```typescript
function handleLine(line: string, ctx: any) {
  let env: any;
  try { env = JSON.parse(line); } catch { return; }
  if (env.v !== 1 || env.type !== "context") return;
  onMessage(env, ctx);
}
```

**校验规则**：

- JSON 解析失败 → **静默忽略**（catch 后 return）
- `env.v !== 1` → 静默忽略（协议主版本不匹配）
- `env.type !== "context"` → 静默忽略（说明-AIBP §7.1「忽略未知 type」前向兼容；v1 的 `bye` 报文自然走这条路径）

**不告警**：避免报文洪泛时噪音爆炸；前向兼容的体现。

## 六、递送语义（协议核心）

```typescript
function onMessage(env: any, ctx: any) {
  const p = env.payload;
  const text = formatText(p);

  if (p.message) {
    // 带 message → 走 sendUserMessage（永远带 deliverAs: "steer"）
    pi.sendUserMessage(text, { deliverAs: "steer" });
  } else {
    // 纯上下文 → 填入输入框，不触发对话
    ctx.ui.setEditorText(text);
    ctx.ui.notify(`📌 已放入输入框，可编辑后发送`, "info");
  }
}
```

**两条路径**（说明-AIBP §六）：

| `p.message` 存在？ | 递送方式 | 用户体验 |
|--------------------|---------|----------|
| **是**（非空字符串） | `pi.sendUserMessage(text, {deliverAs: "steer"})` | 立即/排队触发 LLM 对话 |
| **否**（字段缺省或空） | `ctx.ui.setEditorText(text)` + `notify` | 填入输入框，用户编辑后手动发送 |

### 6.1 带 message 路径：永远用 `deliverAs: "steer"`

**为什么不需要 `isIdle()` 分流**：

`pi.sendUserMessage` 内部调 `prompt()`，唯一硬约束是「streaming 时**不带** `streamingBehavior` 会抛错」。只要**永远带上** `deliverAs`，idle/streaming 两种状态都不会抛错：

- **idle 时**：`steer` 参数被忽略（`prompt()` 的 streaming 分支不进入），走正常立即提交
- **streaming 时**：进入 steer 队列，等当前回合工具链跑完、下次 LLM 调用前插入

**`steer` vs `followUp`**：

- `steer`：当前回合工具链跑完、下次 LLM 调用前插入（**快**）
- `followUp`：等到 agent 完全空闲（多轮工具全部结束）（**慢**）

`steer` 适合"用户发来新问题"语义——尽快让用户从 microNeo 发来的问题被处理。

### 6.2 纯上下文路径：填入输入框

```typescript
ctx.ui.setEditorText(text);
ctx.ui.notify(`📌 已放入输入框，可编辑后发送`, "info");
```

**设计原则**：

- **不触发对话**（说明-AIBP §6.4「不得自动触发」）
- **不暂存**（无 `pending` 状态、M2 时删了）——文本直接进输入框
- **用户完全控制**：可编辑、可删、可补充、可丢弃
- **多来源覆盖**：发两份纯上下文 → 后者**直接覆盖**输入框内容（说明-AIBP §6.5）
- **notify 提示**：告知"有上下文待用"

> **为什么不用 `before_agent_start` 隐式注入**：参考 `pi-in-zellij/editor/editor.ts` 已验证——外部内容直接填输入框更透明、更简单、更符合「由用户决定怎么用」。无状态、无泄漏、无意外。

## 七、文本格式化（formatText）

**当前实现**（D10 决策 4 纯内联 + 尖括号标签分隔）：

```typescript
function formatText(p: any): string {
  const sel = p.selection;
  const selText = sel?.text && sel.text.length > 0 ? sel.text : "";

  if (sel && selText) {
    // 有选区且文字未截断：内联 + 尖括号标签分隔（不用 @ 语法 → 不触发 LLM 读文件）
    const header = `<selection: ${p.path} lines ${sel.start.line}-${sel.end.line}>`;
    return p.message ? `${header}\n\n${selText}\n\n<user input>\n\n${p.message}` : `${header}\n\n${selText}`;
  }

  // 无选区 / 选区文字被截断（超过 MaxSelectionLines）：走 @ 引用，让 LLM 自己读
  const focus = sel
    ? `line${sel.start.line}-line${sel.end.line}`
    : `${p.cursor.line}`;
  const base = `@${p.path} :line${focus}`;
  return p.message ? `${base}\n\n${p.message}` : base;
}
```

### 7.1 决策点：`Selection.Text` 是否为空

| 情况 | `selText` | 走的分支 | LLM 行为 |
|------|-----------|---------|---------|
| 无 selection | `""` | @ 引用 | 读文件 |
| 有 selection，Text 非空（≤30 行） | 实际文字 | **内联 + 尖括号标签** | **不读文件**（用内联文字） |
| 有 selection，Text 被发送端截断（>30 行） | `""` | @ 引用 | 读文件 |

**两端协调**：

- **发送端**（`internal/action/notepane.go:lowestCursorScreenRow`）决定发不发 Text：`lineSpan > MaxSelectionLines` 时 `selText = ""`
- **接收端**（`formatText`）只看 Text 是否为空：是 → 内联；否 → @ 引用
- 接收端**不做行数判断**——行数是发送端决策的结果，已经体现在 Text 是否为空上

### 7.2 四种渲染形态

| 场景 | 输出 | 走哪条路径 |
|------|------|-----------|
| 有选区 + Text 非空 + message | `<selection: /path lines A-B>\n\n<selText>\n\n<user input>\n\n<message>` | `sendUserMessage` |
| 有选区 + Text 非空 + 无 message | `<selection: /path lines A-B>\n\n<selText>` | `setEditorText` |
| 无选区 / Text 空 + message | `@/path :line<focus>\n\n<message>` | `sendUserMessage` |
| 无选区 / Text 空 + 无 message | `@/path :line<focus>` | `setEditorText` |

**示例**（有选区 + Text 非空 + message）：

```
<selection: /Users/sollawen/notes/idea.md lines 12-14>

这里是被选中的原文第一行
这里是被选中的原文第二行
这里是被选中的原文第三行

<user input>

帮我润色
```

### 7.3 `@path :lineX` 语法

LLM 工具链（Cursor/Claude/Copilot）通用约定。**pi 系统提示会触发 LLM 用 read 工具去读这个文件**。

**D10 关键洞察**：只要消息里有 `@path :lineX` 格式，**LLM 一定会去 read 文件**——哪怕 `Selection.Text` 已经把内容内联在消息里。所以"既放 @ 引用又内联文字"反而**双倍消耗**（read + 内联都在）。

**省 token 的唯一办法**：内联时**彻底不放** `@` 引用格式。改用尖括号标签标位置（`<selection: /path lines A-B>` + `<user input>` 三段分隔）——LLM 不触发 read，直接用内联文字。

### 7.4 行号基址

`p.cursor.line` / `p.selection.start.line` / `p.selection.end.line` 都是 **1-based**（v1 协议），与 LLM 工具（sed/ripgrep/read）天然对齐。formatText 输出直接用，不做 ±1 转换。

### 7.5 为什么不用 `@` 引用 + 内联组合

见 §7.3。`@` 语法会强制 read → 内联文字被无视 → 双倍消耗。

### 7.6 为什么用尖括号标签

- **LLM 高 attention**：`<tag>...</tag>` 是 LLM 训练数据里的高频结构化标记，attention 权重明显高于纯自然语言前缀（如"来自…的选中内容："）。
- **跨语言友好**：标签用英文（`<selection>` / `<user input>`），与路径、文件名、用户消息的语言无关。
- **段落分隔清晰**：在选区内容长、或用户消息主题与原文相近时（如"这段话什么意思"），明确的三段边界能防止 LLM 误把选区文字当成用户指令的一部分。
- **selText / message 原样透传**：标签只起分隔作用，不对原文做任何修改。

## 八、错误处理

| 场景 | 处理 |
|------|------|
| JSON parse 失败 | `handleLine` catch 后 return（不告警） |
| `env.v !== 1` | 静默 return（协议主版本不匹配） |
| `env.type !== "context"` | 静默 return（未知 type，前向兼容） |
| 池子文件解析失败 | `loadNamePool` notify + 返回 null → 不注册（session_start 正常返回，扩展仍加载） |
| 池子耗尽（所有名字占用） | `allocateName` 返回 null → `ctx.ui.notify("⚠ 名字池已满")` + 不注册（同上） |
| `server` 运行时 error | 抢锁成功后接的持久日志 handler 记 warn（不 crash pi） |
| `socket close` 失败 | `try/catch` 吞掉（unlink 文件可能已不在） |
| `unlinkSync` 失败 | `try/catch` 吞掉（文件可能已被外部清理） |

**v1 不做的事**（明确出界）：
