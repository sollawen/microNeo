# D11 — 接收端名字分配方案

> **状态**：方案设计（待实施）
>
> **解决的问题**：多接收端时，`name = pi-<pid>` 对用户无意义，发送端选目标时无法区分。
>
> **范围**：**仅接收端**（pi 扩展 / 未来的 opencode 扩展）。发送端不改，EABP 协议不改。
>
> **与历史文档的关系**：D11 是"D 系列决策"的延续（D5/D10 已被 `说明-接收端.md` 吸收）。本方案落地后，实现细节会被吸收进 `说明-接收端.md`；本文作为决策与算法依据留存。

---

## 一、问题

当前接收端 `name = pi-<pid>`（`eabp-pi/index.ts:30`）。

| 症状 | 后果 |
|------|------|
| PID 对用户无意义 | 发送端列表里 `pi-16934` / `pi-32972`，用户对不上"哪个是我哪个 pi" |
| 多接收端并存场景常见 | 用户同时开多个 pi（不同任务）很自然 |
| 自动启发式全部失败 | cwd 可能相同（同项目开两个 pi）、session 名可能未设、终端窗口名也会重复 |

**本质矛盾**：单一自动维度无法既"必然唯一"又"必然人类可读"。

---

## 二、核心思路

**把问题从"探测会不会重复"转成"保证不重复"。**

预先准备一份**有序名字池**，接收端启动时**按顺序分配一个没人用的**名字。

- 唯一性靠**分配**（而非探测），池够大就必然不撞
- 可读性靠**池子里是人类词**（NATO 音标字母，Alpha / Bravo / …）
- 跨 agent 共享同一池子，全局统一占用（无前缀，name = `Alpha` / `Bravo`）

> **认知转换**：不再纠结"name 字段既要唯一又要可读"，而是承认 name 字段的唯一性职责由**分配机制**保证，可读性由**池子内容**保证——两者由同一机制同时满足，不再相互掣肘。

---

## 三、设计原则

| 原则 | 说明 |
|------|------|
| **分配而非探测** | 接收端主动"占"一个名字，而不是从自身属性推一个可能撞的 |
| **顺序占用** | 第 1 个启动的 pi 拿池子第 1 个名字；第 2 个拿第 2 个 |
| **填第一个空缺** | 已占用的跳过；前面有人退出释放了名字，新启动者填回（池子前段保持连续占用） |
| **池子 agent 无关** | `eabp-names.json` 内容是裸名字；接收端直接用裸名注册（`Alpha`），不带 agent 前缀 |
| **全局统一占用** | 所有接收端（pi / opencode / …）扫全部 `ai-*.json` 判占用；第一个启动者拿 Alpha，第二个拿 Bravo，**不区分 agent 类型** |
| **协议零改动** | name 本就是自由字符串（`说明-EABP §3.2`），`pi-Alpha` 合法 |

## 四、池子文件（`eabp-names.json`）

### 4.1 位置

接收端实现规范（**所有接收端必须一致**，类似 `registryDir()` 的两端对齐约束）：

```
默认：$XDG_CONFIG_HOME/eabp/eabp-names.json（$XDG_CONFIG_HOME 未设则回落 ~/.config）
```

- `$XDG_CONFIG_HOME/eabp/` 是**独立目录**，不依附于 microNeo 或任何 agent —— 否则 opencode 扩展会依赖 microNeo 配置路径，破坏 agent 无关性
- 持久路径（注册表目录是 `$TMPDIR` 会随重启清，不能放那）
- **首次运行自动种子**：该文件不存在时，接收端用代码内硬编码的默认词表原地创建它（见 §4.5），用户打开即见、即改

> **不在协议范围**：这是接收端之间的实现约定，类似 `registryDir()` 的两端对齐，**不写进 `说明-EABP.md`**。属"接收端实现规范"，写进 `说明-接收端.md` + 未来 opencode 接收端文档。

### 4.2 格式

**池子文件是 JSON 数组**（最简）：

```json
["Alpha","Bravo","Charlie","Delta","Echo","Foxtrot","Golf","Hotel","India","Juliet","Kilo","Lima","Mike","November","Oscar"]
```

> **词表来源**：标准 **NATO 音标字母表**前 15 个（A–O 子集）。用户可自行替换为任何喜欢的词。
>
> **关于 `X-ray`**：本子集不含 X；若用户自行在 `eabp-names.json` 里写 `X-ray`（原拼写含 `-`），会被 §4.3 合法性过滤跳过（不报错）。如确需 X，写为 `Xray`。

### 4.3 名字规范化规则

名字会进 socket 文件名 `ai-pi-<name>.sock`，进入分配前由 `loadNamePool` **统一规范化**（§4.5 伪代码步骤 4），`allocateName` 收到的就是干净池子、无需再判合法性。规则三条，**按顺序**作用：

1. **截断**：每个 name 截断到前 10 字符
   - 控制最终 socket 路径长度（最坏 ≈ 68 字符，远低于 Linux 108 限制）
   - 用户写超长词造成的不适（footer 显示名即截断版，如 `WashingtonMonument` → `pi-Washington`）由用户自己承担
2. **去重**：截断后撞名的，**保留首次出现**、丢掉后续（保持池子"有序占用"语义，不排序）
3. **字符过滤**：截断+去重后的 name，若含 `/` `\0` `:` 空格 `-` 等路径/文件名非法字符（`-` 保持禁止：避免 name 自带 `-` 让 socket 文件名 `ai-<name>.sock` 里的 `ai-` 分隔标记产生视觉歧义），跳过（不报错，记日志）

> **完整示例**：`["WashingtonMonument", "WashingtonState", "X-ray", "Alpha"]` → 截断 `["Washington", "Washington", "X-ray", "Alpha"]` → 去重 `["Washington", "X-ray", "Alpha"]` → 字符过滤（`X-ray` 含 `-`）→ `["Washington", "Alpha"]`

> **NATO 拼写里的 `X-ray`**：标准音标字母表中 X 的原拼写含 `-`，与上述字符强约束冲突。本方案在代码内默认值里**写为 `Xray`**（去连字符）以满足约束。用户在 `eabp-names.json` 里若坚持写 `X-ray`，截断不变（5 字符），会在 §4.5 规范化的字符过滤步骤被跳过（不报错，仅记日志）。

### 4.4 用户编辑

用户随时可改 `eabp-names.json`（加/减/换词）。**生效时机**：下次接收端启动（新 session_start）。

> **不改运行中接收端的名字**：name 在 session_start 时定，整个会话期间不变（见 §7.1）。

### 4.5 默认池：硬编码 + 首次自动种子

**默认词表硬编码在每个接收端的代码里**（一个 TS **字符串常量**，以单空格分隔名字；不随包发布单独文件）：

```typescript
const DEFAULT_NAMES_STR = "Alpha Bravo Charlie Delta Echo Foxtrot Golf Hotel India Juliet Kilo Lima Mike November Oscar";
```

> **为什么是 string 而不是 array**：代码内默认值是给人/调试时一眼能看清单的纯文本形式；运行时需要 array 时再 `split(' ')` 得之。一行写完、可直接复制到终端 `echo` 验证。

#### `loadNamePool()` 伪代码

```typescript
/**
 * 加载名字池。返回 string[] 时进入正常分配（已规范化，干净）；
 * 返回 null 表示池子文件坏了——session_start 见 null 应放弃注册。
 */
function loadNamePool(ctx: Ctx): string[] | null {
    let raw: string[];

    // 1. 有文件 + 合法非空数组 → 读出来
    if (file 存在 && JSON.parse(内容) 是非空数组) {
        raw = parsed;
    }
    // 2. 没文件 或 空数组 → 等同"想重置"，原地创建 + 用默认
    else if (!file 存在 || parsed 是空数组) {
        mkdir -p 父目录;
        写 JSON.stringify(DEFAULT_NAMES_STR.split(' ')) 到文件;
        raw = DEFAULT_NAMES_STR.split(' ');
    }
    // 3. 有文件但解析失败 → 通知用户 + 返回 null（不覆盖用户文件，留给用户修）
    else {
        ctx.notify("⚠ eabp/eabp-names.json 格式错误，本次不接收消息");
        return null;
    }

    // 4. 规范化（§4.3）：截断到 10 字符 → 保留首次出现顺序去重 → 字符过滤
    //    返回的池子已干净，allocateName 无需再判合法性
    return normalize(raw);
}
```

**读 + 种子规则**（关键：永不破坏用户文件，**除非用户主动清空成 `[]`**）：

| 文件状态 | 行为 |
|---------|------|
| **不存在** | 先 `mkdir -p` 父目录，再 `JSON.stringify(DEFAULT_NAMES_STR.split(' '))` 原地创建文件；用 split 结果跑分配 |
| **存在但内容是 `[]`**（空数组） | **等同不存在**：按"不存在"分支处理，原地覆盖写入默认 10 词（用户清空 = 想要重置，尊重其意图） |
| **存在且是非空 JSON 数组** | 直接用 |
| **存在但 JSON 解析失败** | `ctx.notify` 一次性提示用户 + `loadNamePool` 返回 `null`；**不覆盖**用户文件（保护用户手改坏的内容，让用户能修） |

> **为什么硬编码而非随包文件**：默认池只读一次（`session_start`），一个 TS 常量足够；少一个要随 `pi install` 部署/维护的文件。自动种子让真实可编辑文件**主动出现**在用户 config 目录——比"包里藏个 default 文件"更可发现。
>
> **为什么不放 microNeo runtime**：消费者是接收端扩展（pi/opencode，独立 TS 包），不是 microNeo。接收端跑在 pi/opencode 扩展运行时里，读不到 microNeo 二进制 embed 的 Go 资产；若放 microNeo runtime 磁盘路径，opencode 接收端就得反向依赖 microNeo 安装——正是 §4.1 要避免的 agent 耦合。每个接收端**各自硬编码自己的默认池**，自含、互不依赖（与 `registryDir()`「两端各算各的、结果对齐」同类约束）。
>
> **发送端完全不关心此文件**：发送端只读注册表里的 `ai-*.json`、取其 `name` 字段（`internal/eabp/registry.go:48/32`），不读 `eabp-names.json`、也不知道名字怎么选出来的。eabp-names.json 纯属接收端内部分配用的池子。

> **首次运行 vs 坏文件的区分原则**：
> - **不存在 / 空数组** = 用户没配置或想重置 → 替他写默认，零配置上手
> - **存在但解析失败** = 用户**改过**且改坏了 → 通知他修，**绝不静默 fallback**（静默 fallback 容易让用户困惑"我改了 eabp-names.json 怎么没用"）

## 五、分配算法（接收端 `session_start`）

```
输入：（无 prefix 参数）
输出：最终 name（如 "Alpha"）+ 占用成功的 socket

1. 读池子：names = loadNamePool(ctx)   // §4.5
   ↓
   if (names === null) {
     // loadNamePool 内部已 ctx.notify 提示用户
     return;   // session_start 跳过本节后续——不注册、不调 setStatus、正常退出
   }

2. 算注册目录：dir = registryDir()
3. 已占用集合 occupied = {}：
   for each ai-*.json in dir:
     解析出 rid（裸名，去掉 ai- 前缀）
     若该注册文件的 PID 存活（kill(pid,0)==0）→ occupied.add(rid)
     （PID 死 → 僵尸注册，顺手 GC 删掉 json+sock，不计入 occupied）
4. for name in names（顺序）:    // names 已由 loadNamePool 规范化（§4.3），无需再判合法性
     if name in occupied: continue
     socketPath = dir/ai-<name>.sock
     尝试 listen(socketPath):    // §5.1 竞态锁
       成功 → 占用成功，返回 (<name>, socketPath)
       失败（EADDRINUSE）→ 见 §5.1 处理
5. 池子耗尽（全部占用/非法）→ 返回 null（池子耗尽信号）   // §5.2
```

> **两个 `null` 出口汇总**（§五算法可能"早返 null"两次）：
>
> | 出口 | 含义 | 用户感知 |
> |------|------|---------|
> | 步骤 1 `loadNamePool` 返回 null | §4.5 池子文件坏 | `ctx.notify` 一次性提示 + 不注册 |
> | 步骤 5 池子全占满 | §5.2 池子耗尽 | `ctx.notify` 一次性提示 + 不注册 |
>
> 两种"不注册"形式上都不显示名字、都用 `ctx.notify` 告知用户，**但归因不同**（详见 §5.2）：
> - §4.5 池子文件坏 = 用户配置问题（需修文件才能恢复）
> - §5.2 池子耗尽 = 临时状态（关掉别的 pi 后下次就有名字）

### 5.1 竞态处理：socket 文件 = 原子锁

两个接收端同一毫秒启动，都看到 Alpha 空缺。**用 socket 文件创建做原子锁**——listen 系统调用本身是原子的：

```
尝试 listen(socketPath):
  成功 → 我抢到了，占用 Alpha
  失败（EADDRINUSE）→
    试 connect(socketPath, 200ms timeout):
      成功 → 真有活接收端在用 → 换下一个 name
      失败 → 上次崩溃的僵尸 socket 文件 →
        删 socketPath → 重 listen:
          成功 → 我抢到了
          失败 → 换下一个 name（异常情况，记日志）
```

**零额外锁机制**——复用 Unix socket 的 listen 原子性 + connect 试探（与发送端 `Discover` 的同款判据，`说明-EABP §3.5`）。

### 5.2 池子耗尽：notify 提示 + 不注册

同时开 >N 个接收端（N = 池子大小），全占满：

- **含义**：N 个名字都被占 → 当前已有 N 个接收端活着在通信，再多一个就是"凑热闹"
- **行为**：`allocateName` 返回 `null`；`session_start` 看到 `null` → `ctx.notify("⚠ 名字池已满，本次不接收消息")` 一次性提示，然后 **不注册、不监听 socket、不调 setStatus、正常退出**（return 0）
- **为什么不静默**：池子耗尽对用户是非预期状态（"我刚启的接收端怎么没出现在列表里？"），完全静默会让用户困惑。notify 一次指出原因足够，不打扰后续编辑
- **不退到 PID 凑数**：凑数 PID 名会污染发送端列表（用户看到一堆 PID 项更糟），不如缺席

> 池子默认 15 个（NATO 音标字母表前 15），日常够用。用户若常开 >15 个接收端：
> 1. 先想想是不是需要那么多同时活着的 pi（通常 3–5 个已经够）
> 2. 如果确实要，加大 `eabp-names.json`（§4.4）

### 5.3 名字释放

接收端 `session_shutdown` 现有流程（删 json + 删 sock）**天然释放** name：

- 正常退出 → 名字立即回池子，可被下次复用
- 崩溃 → 残留 json/sock，下次接收端启动时被 GC（§5.1 的 connect 失败 → 删）

**无需额外释放机制**。

## 六、各接收端实现

### 6.1 pi 接收端（`eabp-receivers/eabp-pi/index.ts`）

改动点（**仅 `session_start` 里 name 生成那段**）：

```typescript
// 现状（第 30 行）
name = `pi-${process.pid}`;

// 改为
const names = loadNamePool(ctx);
if (names === null) return;        // §4.5 池子文件坏，notify 已在内部完成
name = await allocateName(names);   // §五 算法；name 形如 "Alpha"
```

**配套**：

- 代码内硬编码 `const DEFAULT_NAMES_STR = "Alpha Bravo ..."`（15 字母 NATO 音标 A–O，§4.5）
- `loadNamePool(ctx)` 负责读文件 / 首次自动种子 / 空数组重置 / 坏文件 notify / 规范化（§4.3：截断 10 字符 + 保留首次顺序去重 + 字符过滤）
- `allocateName(names)` 实现 §五 算法（含 §5.1 竞态锁、§5.2 池子耗尽 —— 返回 `null` 时由调用方 `ctx.notify` 提示 + 不注册）
- 注册成功后调用 `ctx.ui.setStatus("eabp", "● " + name)` 在 footer 显示（§7.2）
- `session_shutdown` 调 `setStatus("eabp", undefined)` 清理

### 6.2 opencode 接收端（**未实现，预留**）

opencode 接收端实现时，**与 pi 接收端完全同一份代码**——同一份算法、同一份 eabp-names.json、同一个 `allocateName(names)` 调用，零 agent 特定逻辑：

```typescript
name = await allocateName(names);   // → "Alpha" / "Bravo" / ...（与 pi 共享全局占用）
```

**这是本方案的核心价值**：agent 无关的名字分配机制，新 agent 接入零额外设计。pi 和 opencode 共享同一个全局名字空间，第一个启动者（无论哪个 agent）拿 Alpha，第二个拿 Bravo。

## 七、边界与决策点

### 7.1 名字粘性

| 选项 | 行为 | 代价 |
|------|------|------|
| **A. 不粘性（已确认）** | pi 重启后重新分配，可能换名（Alpha→Bravo） | 0；符合协议"无状态、按需"语义 |
| B. 跨会话粘性 | 记录"上次我是 Alpha"，重启优先取回 | 增加持久状态 + 关联键（用什么关联？PID 会变） |

**已确认 A**（v1）。理由：

- EABP 协议本就是无状态、按需、一次性的
- name 只是"本次会话期间"的标识，不需跨会话稳定
- 粘性的关联键难选（PID 重启会变、cwd 可能撞、session id 又太长）

> 若用户反馈困扰再加 B。v1 按 A 实现。

### 7.2 footer 显示自己的名字

**只有注册成功后才在 footer 显示名字**；注册失败（§4.5 池子文件坏 / §5.2 池子耗尽）或主动退出（session_shutdown）都不显示——详见本节末尾两段。

注册成功时，用 pi 扩展 SDK 的 `ctx.ui.setStatus()` 把名字写进 pi 的状态注册表：**写进去就持续保留**（pi 每次重画 footer 都会把它拼回去，不像 `ctx.notify` 是一次性闪过），任何时候打开 pi 都看得到，直到主动清掉。

```typescript
// session_start 末尾（allocateName 返回名字后）
ctx.ui.setStatus("eabp", ctx.ui.theme.fg("accent", `● ${name}`));
```

**效果**（pi 窗口底部 footer 会多一项）：

```
[模型: claude-3] [git: main] [eabp: ● Alpha]  [其他信息...]
```

> **API 依据**：`ctx.ui.setStatus(extKey, text)` 是 pi 扩展的官方 TUI hook（见 `pi-coding-agent/docs/tui.md` Pattern 4 "Persistent Status Indicator"，例子 `examples/extensions/status-line.ts`）。extKey 在 pi 内部是 namespace，重复设置同名 key 会覆盖（安全、幂等）。
>
> **为什么选 footer 而不是 notify / title / widget**：
> - 持续显示名字的需求 → 一次性 `ctx.notify` 满足不了（之前测试时 notify 被反馈"没注意到"）
> - 不覆盖 pi 默认 footer（用 `setStatus` 是加一项，不替换整条）
> - 与 `plan-mode` / `status-line` 等其他扩展同款做法，习惯一致
> - 不依赖 terminal 支持 title 协议
> - `setWidget` 会顶到输入框位置，过重

> **与不注册场景协调**：无论 `loadNamePool` 返回 `null`（§4.5 池子文件坏）还是 `allocateName` 返回 `null`（§5.2 池子耗尽），本次 session **都不调 setStatus**（"eabp" 键空闲，footer 无指示），各自由对应流程 `ctx.notify` 一次性提示原因——未注册时 footer 显示任何东西都名不副实，notify 一次告知原因足够。
>
> **session_shutdown 清理**：

```typescript
pi.on("session_shutdown", async () => {
  ctx.ui.setStatus("eabp", undefined);  // 避免下次 pi 启动时短暂残留旧名字
  // ... 现有 sock/json 清理逻辑 ...
});
```

### 7.3 名字与发送端 UI 的衔接

本方案**只解决接收端 name 生成**。发送端列表展示（`pi-Alpha · microNeo · 3min`）是另一件事，属发送端 UI 工作，不在 D11 范围。

但本方案是发送端 UI 的**前置依赖**——有了可读 name，发送端展示才有意义。

## 十一、决策记录

> 本方案所有决策点均已确认，下表留档备查。

1. ~~**§7.1 名字粘性**：v1 先做"不粘性"（推荐 A）还是跨会话粘性 B~~ —— ✅ **已确认**：用「不粘性」（A），pi 重启后重新分配（可能换名），符合协议无状态语义；v1 按 A 实现，若用户反馈困扰再加 B
2. ~~**§4.1 池子默认位置**：独立目录 `$XDG_CONFIG_HOME/eabp/` 还是复用 `~/.config/microNeo/`~~ —— ✅ **唯一可选**：`eabp-names.json` 的消费者是 EABP 接收端扩展（pi/opencode），与 microNeo 本身无关，复用 `~/.config/microNeo/` 会把 opencode 接收端反着绑定到 microNeo 安装，直接违背 §三 「agent 无关」原则。独立目录是唯一可选，无须决策。
3. ~~**§4.2 默认池子词表**：用 NATO 口语（Alice/Baker/…）还是其他？用户可改，但默认要定一份~~ —— ✅ **已确认**：用标准 NATO 音标字母表前 15 个（A–O，§4.2），用户自行加 `X-ray` 会被合法性过滤跳过（§4.3）
4. ~~**§7.2 notify 形式**：notify 不够显眼时的备选（状态栏 / banner）—— 留作体验优化，还是 D11 内一起解决？~~ —— ✅ **已确认**：§7.2 用 `ctx.ui.setStatus()` 在 footer **持续显示名字**（不是 notify）；`ctx.notify` 仍保留给一次性错误提示（§4.5 池子文件坏 / §5.2 池子耗尽）——两者各司其职
5. ~~**§三 占用模型**：per-prefix 独立占用（`pi-Alpha` / `oc-Alpha` 可并存）还是全局统一占用（所有 agent 共享一个名字空间）？~~ —— ✅ **已确认**（2026-06 反转）：改用「全局统一占用」。理由：用户不需要在名字层面分辨"哪个是 pi 哪个是 opencode"，前缀对用户是冗余信息；全局占用让名字更短、发送端列表更干净。代价：pi 和 opencode 不能同时都叫 Alpha（第一个拿 Alpha，第二个拿 Bravo），但这正是"用户不需要分辨"的体现。池子随之扩到 15（A–O）以吸收全局并发的容量需求。
