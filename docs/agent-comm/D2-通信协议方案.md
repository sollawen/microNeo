## 一、架构原则（来自三分屏方案，不可动摇）

本文的一切设计服从 `三分屏方案.md` §7.1 确立的原则：

> **microNeo 发送端的所有核心代码都用 Go 写死。microNeo 自身不包含任何针对具体 agent（pi、opencode、…）的 Lua 插件。**
> 不同的 ai agent 通过各自遵守 microNeo 通信协议的**扩展程序**（运行在 agent 侧）来接入。

这是 **LSP 式架构**：

```
┌──────────────────────────────────────────────┐
│  microNeo（全 Go，写死，与具体 agent 无关）    │
│  ┌─────────────┐   ┌──────────────────────┐  │
│  │ InlineInput │   │ Agent 注册表 + 选择   │  │
│  │ 撕纸 UI     │──→│ + 上下文打包 + 发送   │  │   ← 本文档定义这一段
│  └─────────────┘   └──────────┬───────────┘  │
└───────────────────────────────┼──────────────┘
                                 │ 本协议（Go 定义，本文档）
            ┌────────────────────┼────────────────────┐
            ▼                    ▼                    ▼
     ┌──────────────┘    ┌──────────────┐    ┌──────────────┐
     │ pi 扩展（TS）│    │ opencode 插件│    │ XXX 扩展     │
     │ 注册/监听/   │    │（TUI 插件）  │    │（社区写）    │
     │ 递送 LLM     │    │              │    │              │
     └──────────────┘    └──────────────┘    └──────────────┘
```

由此推出三条铁律（**协议设计必须满足**）：

1. **协议 agent 无关**：报文格式、注册表格式、传输方式都不含任何 pi/opencode 特有概念。接入新 agent 不需要改 microNeo 一行代码。
2. **Lua 零参与通信**：三分屏方案 §7.2 已明确作废 `onInit`/`listAgents`/`onSend`/`formatContext`/`onInlineInputClose` 等 Lua 事件。发送管线是 Go 核心，不存在"发送端 Lua 插件"。
3. **能力差异由接收端吸收**：协议只规定"递送什么"，不规定"agent 怎么用"。各 agent 的扩展机制（pi 的 TS 扩展 / opencode 的 TUI 插件）负责把递送物落到自己的 LLM 通道。

> **术语约定**：本文一律用"**递送**（deliver）"描述协议把上下文交给接收端这个动作，**不**用"注入"。原因见 opencode 调研 §7.2——opencode 的接收是"预填输入框"而非隐式改写消息流，"注入"会误导。协议层只递送，是否/如何进入 LLM 上下文由各 agent 自决。

## 二、三个角色

| 角色 | 实体 | 语言/形态 | 职责 |
|------|------|----------|------|
| **发送端 Sender** | microNeo 进程 | Go（写死在 microNeo 核心） | 采集编辑上下文 → 查注册表 → 选目标 → 发报文 |
| **接收端 Receiver** | ai agent 内的扩展程序 | 各 agent 自有机制（pi=TS 扩展，opencode=TUI 插件） | 启动注册 → 监听 socket → 解析报文 → 递送到自己的 LLM 通道 |
| **注册表 Registry** | 一个固定目录 | 文件系统（无独立进程） | 接收端登记自己；发送端发现接收端 |

- 一个用户机器上：可能有**多个 microNeo 进程**（编辑不同文件）= 多个发送端；可能有**多个 agent 进程**（pi、opencode、或多个 pi session）= 多个接收端。
- 通信方向：**单向 microNeo → agent**（v1）。协议预留双向扩展（agent → microNeo 的 `ack` / `goto`），v1 不实现。
- 通信触发：**用户主动、按需、一次性**。不是对光标/选区变化的持续追踪。

## 三、服务发现与注册表（Registry）

> 继承 `架构设计.md` §4，机制不变，细化字段语义。

### 3.1 注册表目录

跨平台、用户私有、重启不残留：

```
${XDG_RUNTIME_DIR:-${TMPDIR:-/tmp}}/microneo-agent-bridge-${UID}/
```

解析优先级：
1. `$XDG_RUNTIME_DIR`（Linux `/run/user/$UID`，进程退出自动清理）——首选
2. `$TMPDIR`（macOS 通常是 `/var/folders/.../T/`）——macOS 兜底
3. `/tmp`——最后兜底

属性：
- 目录名带 `$UID`，多用户机器天然隔离
- 由首个需要它的进程（发送端或接收端）按 `0700` 权限创建
- socket 文件就放在该目录下，权限 `0600`

> **实现约束（必须两端一致）**：发送端（Go）和接收端（TS）各自用**完全相同的算法**算出该路径。本文固定算法，两端复刻。Go 侧建议集中在一个函数 `func registryDir() string`；TS 侧建议抽公共 core 模块（pi 扩展与 opencode 插件共用一份）。

### 3.2 注册文件

每个接收端在该目录下写一个文件 `<name>.json`（**文件名 = name，name 全局唯一**）。内容：

```json
{
  "name": "pi-main",
  "pid": 12345,
  "transport": "unix",
  "socket": "/tmp/.../microneo-agent-bridge-501/pi-main.sock",
  "protocol": "mnabs-1",
  "startedAt": 1717000000,
  "cwd": "/Users/me/project",
  "labels": ["default"]
}
```

字段说明：

| 字段 | 必需 | 说明 |
|------|------|------|
| `name` | ✅ | 人类可读的唯一名。用户起或自动生成（默认 `<agent>-<pid>`，如 `pi-12345`）。**等于文件名**，保证唯一 |
| `pid` | ✅ | 接收端进程 PID，用于存活检测 |
| `transport` | ✅ | 传输类型，v1 固定 `"unix"` |
| `socket` | ✅ | 该接收端监听的 Unix socket 绝对路径 |
| `protocol` | ✅ | 协议版本号，见 §七。发送端发现版本不匹配时跳过该接收端 |
| `startedAt` | ✅ | 启动时间（Unix 秒），用于展示和排序 |
| `cwd` | ❌ | 接收端工作目录，发送端可据此判断"是不是同一个项目" |
| `labels` | ❌ | 自由标签数组，发送端可按标签筛选（如 `["default"]`、`["frontend"]`） |

### 3.3 接收端注册流程

接收端（agent 扩展）启动时：

1. 算出注册表目录（§3.1），不存在则 `0700` 创建
2. 决定自己的 `name`：用户可配（如 pi 的 `/mnabs rename`），否则默认 `<agent>-<pid>`
3. 算出 socket 路径：`<注册表目录>/<name>.sock`
4. 在该路径**创建并监听** Unix socket（stream 类型）
5. 写注册文件 `<name>.json`（§3.2 全字段）

> **name 冲突处理**：若 `<name>.json` 已存在（上次崩溃残留，或真的撞名），接收端应检测：读旧文件的 `pid`，若该 pid 已死 → 覆盖注册；若仍活 → 给自己改名（追加 `pid` 或序号）后再注册。避免两个接收端写同一 socket 路径。

### 3.4 接收端注销流程

接收端正常退出时：
1. 删除自己的注册文件 `<name>.json`
2. 删除 socket 文件 `<name>.sock`
3. 关闭 socket server

### 3.5 存活检测与垃圾回收（发送端职责）

发送端在发送前发现接收端时，按优先级判断存活：

1. **首选——试探连接**：对 `socket` 地址做一次 `connect` + 立即关闭。连得上即认为存活（最可靠）
2. **兜底——PID 检测**：读注册文件 `pid`，判断进程是否存在（`os.FindProcess` + signal 0 / `kill -0`）
3. **顺手 GC**：连接失败且 PID 不存在的注册文件，由发送端**直接删除**（防止僵尸注册堆积）

> 设计理由：接收端崩溃会留下僵尸注册文件（来不及清理）。发送端是唯一"会去连"的角色，由它顺手 GC 最自然，且能自我修正——下次发送时不再看到失效目标。

## 四、传输层（Transport）

> 继承 `架构设计.md` §5.1，机制不变。

### 4.1 选型：Unix domain socket

| 方案 | 结论 | 理由 |
|------|------|------|
| **Unix socket（采用）** | ✅ | 本地、零网络暴露、快、跨语言成熟；macOS/Linux 原生支持，Windows 10 1803+ 经 AF_UNIX 也可用 |
| TCP localhost | ❌ | 要管理端口冲突、有网络暴露风险 |
| 文件 + 轮询 | ❌ | 延迟高、磁盘 IO、并发竞争 |
| HTTP | ❌ | 每次 POST 开销大、要起 server |

- **平台范围（v1）**：实现 `unix` transport，覆盖 macOS / Linux（microNeo 主要平台）。
- **Windows 留待 Phase 4**：技术上 AF_UNIX 自 Windows 10 1803 起原生可用，Go 1.17+ 的 `net.Listen("unix")` / `net.Dial("unix")` 在 Windows 上直接可用（本项目 Go ≥1.19），**发送端预期可零额外代码运行**；接收端取决于运行时（Bun 直接支持，Node 需 v22+ 才稳定支持）。Phase 4 优先验证 AF_UNIX 路线，若 Node 版本不满足再考虑 named pipe 兜底。详见 §十三 Phase 4。
- **协议层抽象已就位**：注册表的 `transport` 字段（§3.2）本就为此设计——两端各自实现自己的 transport，v1 只做 `unix`，不阻塞主线。
- **socket 类型**：`SOCK_STREAM`（流式），便于按行分帧。

### 4.2 连接策略：每次发送一次"连接 → 写一行 → 关闭"

```
发送端                                接收端
  │                                     │
  │ ── connect(socket) ──────────────► │ accept
  │ ── write(json_line + "\n") ──────► │ read 一行 → 解析 → 处理
  │ ── close ─────────────────────────► │ (conn 关闭)
  │                                     │
```

- **为什么不用长连接**：Unix socket 握手开销极小（微秒级），换来无需管理连接状态、对 agent 重启天然鲁棒。按需发送频率低（人手触发），开销完全可忽略。
- **接收端实现**：每个连接读到一个 `\n` 就解析一条报文处理；连接关闭即结束本次。未来若扩展为长连接多报文，分帧逻辑不变。

### 4.3 编码：逐行 JSON

- 每条报文是一行 JSON，以 `\n` 结束
- 一条报文一行，**不允许跨行**（JSON 内部不得有裸 `\n`；字符串里的换行必须用 `\n` 转义）
- 接收端按 `\n` 分帧，逐行 `JSON.parse`
- 字符编码：UTF-8

### 4.4 方向

- v1：**单向 microNeo → agent**
- 协议预留双向扩展（agent 回 `ack`；后续 agent → microNeo 的 `goto`/高亮），v1 不实现，详见 §十

## 五、报文协议（Wire Protocol）

> 继承 `架构设计.md` §5.2/5.3，机制不变，细化 schema。

### 5.1 公共信封（Envelope）

所有报文共享头部字段：

```json
{
  "v": 1,
  "type": "context",
  "editor": { "pid": 54321, "name": "microNeo", "instance": "win-1" },
  "ts": 1717000123.456,
  "payload": { ... }
}
```

| 字段 | 必需 | 说明 |
|------|------|------|
| `v` | ✅ | 协议主版本号（整数）。v1 = `1`。等于注册表里的 `protocol` 的主版本 |
| `type` | ✅ | 报文类型，见 §5.2 |
| `editor` | ✅ | 发送端标识对象（见下） |
| `ts` | ✅ | 发送时刻（Unix 浮点秒，毫秒精度）。用于接收端排序/展示 |
| `payload` | ✅ | 报文负载，类型取决于 `type` |

`editor` 对象：

| 字段 | 必需 | 说明 |
|------|------|------|
| `pid` | ✅ | 发送端（microNeo）进程 PID |
| `name` | ✅ | 固定 `"microNeo"`（预留未来其他编辑器） |
| `instance` | ✅ | 本 microNeo 实例标识（用户/系统起的窗口/tab 名，如 `win-1`）。多实例时让接收端区分来源 |

### 5.2 报文类型

按需发送模式下，报文类型极少：

| `type` | `payload` | 触发时机 | 方向 |
|--------|-----------|----------|------|
| `context` | 见 §5.3 | **用户主动触发发送**（核心报文） | microNeo → agent |
| `bye` | `{}` | 发送端实例退出（可选，让 agent 清理对应来源的待用上下文） | microNeo → agent |

> v1 只实现 `context`。`bye` 是预留项，发送端退出时可选发送，帮助接收端 GC（如清掉来自该 `editor.instance` 的 `pendingContext`）。

### 5.3 `context` 报文 payload

```json
{
  "path": "/Users/me/a.md",
  "basename": "a.md",
  "filetype": "markdown",
  "line": 42,
  "col": 8,
  "selection": {
    "start": { "line": 42, "col": 8 },
    "end":   { "line": 45, "col": 0 },
    "text":  "## 标题\n正文..."
  },
  "prompt": "这段标题感觉怪，帮我换个说法",
  "visible_lines": null
}
```

字段说明：

| 字段 | 必需 | 类型 | 说明 |
|------|------|------|------|
| `path` | ✅ | string | 文件绝对路径 |
| `basename` | ✅ | string | 文件名（展示用，避免 agent 显示超长路径） |
| `filetype` | ✅ | string | micro 识别的文件类型（如 `markdown`、`go`、`python`） |
| `line` | ✅ | int | 光标行，**0-based**（与 micro 内部 `Loc.Y` 一致） |
| `col` | ✅ | int | 光标列，**0-based**（与 micro 内部 `Loc.X` 一致） |
| `selection` | ❌ | object | 选区对象，无选区则**省略整个字段**（不是 `null`） |
| `selection.start` | ✅* | `{line,col}` | 选区起点（0-based） |
| `selection.end` | ✅* | `{line,col}` | 选区终点（0-based） |
| `selection.text` | ❌ | string | 选中文本。选区较小时带（默认 < 2KB），超阈值省略由接收端按 `start/end` 自取或提示 |
| `prompt` | ❌ | string | **用户附的一句话提问**。有则接收端作为一条用户消息递送给 LLM（立即触发对话）；无则递送为"待用上下文"。详见 §六 |
| `visible_lines` | ❌ | string | 可见区域文本（可选，更费，默认关闭）。方便 agent 直接看屏幕上下文 |

> *`selection.start` / `selection.end` 在 `selection` 存在时必需。

**关键语义：`prompt` 的有无决定递送形态**（这是整个协议最核心的语义分歧点，详见 §六）：

- `prompt` 存在（非空字符串）→ 用户明确想问 → 接收端递送为**一条用户消息并触发对话**
- `prompt` 缺省/空 → 用户只想"挂载"编辑上下文 → 接收端递送为**待用上下文**（用户决定怎么用）

### 5.4 完整报文示例

**示例 A：带选区 + 提问（最典型，来自 InlineInput）**

```json
{"v":1,"type":"context","editor":{"pid":54321,"name":"microNeo","instance":"win-1"},
 "ts":1717000123.5,
 "payload":{
   "path":"/Users/me/a.md","basename":"a.md","filetype":"markdown",
   "line":42,"col":8,
   "selection":{"start":{"line":42,"col":8},"end":{"line":45,"col":0},"text":"## 标题\n正文..."},
   "prompt":"这段标题感觉怪，帮我换个说法"
 }}
```

**示例 B：纯上下文（无选区、无提问，来自快速发送热键）**

```json
{"v":1,"type":"context","editor":{"pid":54321,"name":"microNeo","instance":"win-1"},
 "ts":1717000123.5,
 "payload":{
   "path":"/Users/me/main.go","basename":"main.go","filetype":"go",
   "line":128,"col":3
 }}
```

**示例 C：仅光标位置 + 提问（无选区）**

```json
{"v":1,"type":"context","editor":{"pid":54321,"name":"microNeo","instance":"win-1"},
 "ts":1717000123.5,
 "payload":{
   "path":"/Users/me/a.md","basename":"a.md","filetype":"markdown",
   "line":10,"col":0,
   "prompt":"我现在在看这里，帮我看看这段逻辑对不对"
 }}
```

## 六、递送语义（Delivery Semantics）—— 协议最核心的部分

> 本节定义"接收端拿到 `context` 报文后该干什么"。**协议层只规定语义分歧点（prompt 有无），不规定具体实现机制**——后者因 agent 能力而异，由各 adapter 自决（见 §八）。

### 6.1 两条递送路径

接收端根据 `payload.prompt` 是否存在分流：

| 报文形态 | 语义 | 接收端职责（协议要求） | 用户体验 |
|----------|------|----------------------|----------|
| **带 `prompt`** | 用户明确发起一次协作 | 把 `上下文 + prompt` 格式化为一条用户消息，**递送给 LLM 并触发对话回合** | 在 agent 里直接看到 AI 开始回复 |
| **纯上下文**（无 `prompt`） | 用户把编辑器状态"挂"到 agent | 把上下文**递送为待用上下文**（pending）；用户下次发消息时带上，或用户主动粘贴/丢弃 | agent 提示"有来自 microNeo 的上下文待用"，用户决定怎么用 |

### 6.2 为什么是这两条路径（不是别的）

回到需求本质（`原始需求.md`）：**用户主动、按需、一次性**推送。

- 因为是按需（不是高频实时），**不存在淹没 LLM 的问题**——每条报文都是用户明确想发的。
- 带 prompt = 编辑器里直接发起协作，最顺。
- 纯上下文 = 把编辑器状态"挂"到 agent，用户决定怎么用，**不强行打断 LLM**。
- 两者都很轻量，**不需要**实时方案里那套内存快照持续刷新 / debounce / 周期性注入的复杂机制。

### 6.3 "带 prompt" 的递送：协议层一致

带 prompt 的处理**所有 agent 一致**：递送为一条用户消息，触发对话。差异只在 agent 用什么 API 把消息送进去（pi 的 `sendUserMessage` / opencode 的 `promptRef.set+submit`），这是 §八 adapter 的职责，不是协议的事。

**冲突处理（协议建议，非强制）**：若 agent 正在 streaming（上一回合未结束），带 prompt 的报文应**排队等当前回合结束**再触发，不打断。各 agent 用自己的机制实现（pi 的 `deliverAs:"followUp"`；opencode 的 submit 天然排队）。

### 6.4 "纯上下文" 的递送：协议层一致，实现因 agent 而异

**协议要求**（所有 agent 必须满足）：
- 收到纯上下文报文后，**不得**自动触发 LLM 对话（用户没问，不该答）
- 应有可见提示告知用户"有来自 microNeo 的上下文待用"
- 应提供手段让用户在下次发言时带上，或主动粘贴/丢弃

**实现机制因 agent 而异**（协议不强求统一，这正是 agent 无关性的体现）：

| agent | 纯上下文实现 | 透明度 |
|-------|-------------|--------|
| **pi** | 暂存 `pendingContext`；挂 `before_agent_start`，用户下次发言时把上下文 prepend 进 prompt，然后清空 | 半透明（agent 偷偷知道，transcript 里 prepend 可见但可能不显眼） |
| **opencode** | `promptRef.set({input: 上下文})` **不 submit**——预填进输入框，用户切过去看到已有上下文，自己接着敲问题再回车；或 `/mnabs paste` 从 `api.kv` 取暂存粘贴 | **完全透明**（上下文就在输入框，用户可编辑） |

> 这两种都是合法的协议实现。**协议只要求"不自动触发 + 可见提示 + 用户可控"**，至于"隐式带上"还是"预填可见"，是各 agent 能力的自然落点，不可也不必强求统一。详见 §八。

### 6.5 多来源覆盖

多个 microNeo 实例向同一 agent 发纯上下文时，接收端的 `pendingContext` **被新报文覆盖**（只保留最后一次）。报文带 `editor.instance` 区分来源；覆盖时应在提示里更新来源标注。

> 这是 v1 的简化策略。未来若需保留多份待用上下文，可按 `editor.instance` 做 map，但 v1 不做。

## 七、协议版本

### 7.1 版本号方案

- 注册文件 `protocol` 字段格式：`mnabs-<major>`，如 `mnabs-1`（"microNeo agent bridge protocol v1"）
- 信封 `v` 字段：主版本整数，如 `1`
- **主版本**：报文结构不兼容变更时 +1（增删必需字段、改语义）
- **次版本/兼容 additions**：新增可选字段、新增报文类型——**不改主版本**。接收端应忽略未知字段、忽略未知 `type`（前向兼容）

### 7.2 版本协商

- 发送端读注册文件 `protocol`，主版本不匹配 → 跳过该接收端 + 告警提示
- 接收端读信封 `v`，主版本不匹配 → 忽略该报文 + 可选告警
- v1 不做复杂的 capabilities 协商（YAGNI）

### 7.3 当前版本

**v1（`mnabs-1`）**——本文定义的全部内容。报文类型仅 `context`（`bye` 预留）。

## 八、接收端契约（Receiver Contract）与 adapter

> 本节定义"一个 agent 想接入 microNeo，必须实现什么"。这是 LSP 式架构的接入规范。

### 8.1 协议合规要求（所有接收端必须满足）

一个合规的接收端必须：

1. **启动时注册**：算注册表目录（§3.1）→ 起 Unix socket server（§4）→ 写注册文件（§3.2）
2. **监听并分帧**：accept 连接 → 按 `\n` 分帧 → `JSON.parse` 每行 → 校验信封 `v`/`type`
3. **处理 `context` 报文**：按 §6 分流（prompt 有无）
4. **退出时注销**：删注册文件 + 删 socket + 关 server（§3.4）
5. **提供基础用户控制**（建议）：
   - 查看当前待用上下文（`/mnabs status`）
   - 手动粘贴待用上下文（`/mnabs paste`）
   - 丢弃待用上下文（`/mnabs discard`）
   - 改自己的 name（`/mnabs rename <name>`）

> 命令名前缀建议统一用 `/mnabs`（microNeo agent bridge），便于用户跨 agent 形成肌肉记忆。各 agent 可按自己习惯调整。

### 8.2 通用 core + agent adapter（实现结构）

接收端代码建议分两层（pi 与 opencode 都是 TS，可大量复用）：

- **core（agent 无关，pi/opencode 共用一份纯 TS 模块）**：
  - 注册表目录算法
  - Unix socket server（`node:net`）
  - 注册文件读写（`node:fs`）
  - 报文分帧 + JSON 解析 + 信封校验
  - payload 类型定义
  - `context` → 给 LLM 的文本格式化（`formatContextWithPrompt` / `formatContextStandalone`）
  - "prompt 有无"的分流判定
- **adapter（各 agent 实现，~100-150 行）**：把 core 的 `onContext(payload)` 回调接到各自能力上

差异主要在三处：**发消息的方式**、**纯上下文的处理**、**UI/slash 命令**。

### 8.3 pi adapter（TS 扩展）

**位置**：`~/.pi/agent/extensions/mnabs/index.ts`（全局扩展，每个 pi 会话自动加载）。

**生命周期挂钩**：

| pi 事件 | 接收端做的事 |
|---------|--------------|
| `session_start` | 起 socket server；算 socket 路径；写注册表 JSON |
| 收到 socket 报文 | core 分帧解析 → §6 分流 |
| `session_shutdown` | 删注册文件 + socket；关 server |

**收到报文的处理**（决策 §6 的落地）：

```typescript
let pendingContext: ContextPayload | null = null;  // 只保留最后一次纯上下文

function onContext(p: ContextPayload, from: EditorInfo) {
  if (p.prompt) {
    // 带 prompt：立即作为用户消息发给 LLM，触发对话
    const text = formatContextWithPrompt(p);
    pi.sendUserMessage(text);              // followUp 模式，不打断 streaming
    pendingContext = null;
    clearWidget();
  } else {
    // 纯上下文：暂存待用
    pendingContext = { ...p, from };
    showWidget(`📡 来自 microNeo (${from.instance}): ${p.basename} L${p.line+1}`);
  }
}
```

**"下次发言自动带上"**（pi 特有，半透明）：挂 `before_agent_start`，若 `pendingContext` 非空且是用户消息，把上下文 prepend 到本次 prompt，然后清空 + 清 widget。

**pi 命令**：

| 命令 | 作用 |
|------|------|
| `/mnabs paste` | 把 `pendingContext` 插入输入框（用户可编辑后再发） |
| `/mnabs send` | 把 `pendingContext` 作为一条用户消息立即发给 LLM |
| `/mnabs discard` | 丢弃 `pendingContext` |
| `/mnabs status` | 显示 `pendingContext` 摘要 |
| `/mnabs rename <name>` | 改注册表里的 name |

**UI 反馈**：`ctx.ui.setWidget("mnabs", lines)` 编辑器上方显示待用提示；`ctx.ui.setStatus` 状态栏同步。仅 `ctx.mode === "tui"` 时做。

### 8.4 opencode adapter（TUI 插件）

> 选型论证见 `opencode调研.md` §6.5/§6.6。**用 TUI 插件，不做 server 插件**（无头模式是伪需求）。走 TUI 原生输入通道，server 完全无感。

**位置**：`~/.config/opencode/plugins/mnabs.ts`（全局，启动自动加载；也可发 npm 包在 `opencode.json` 的 `plugin` 声明）。

**拿输入通道**：TUI 插件渲染一个 SolidJS 组件进 `session_prompt`（或 `home_prompt`）宿主插槽，拿到 `promptRef`。该 ref 提供 `set(prompt)` / `submit()` / `focus()`：

```typescript
let promptRef: TuiPromptRef | undefined;
api.ui.Slot("session_prompt", (props) => {
  props.ref?.((ref) => { promptRef = ref ?? undefined; });
  return null;  // 组件可不渲染可见内容，只为拿 ref
});
```

**收到报文的处理**（复用 core，差异只在分流后的动作）：

| 报文形态 | opencode adapter 行为 |
|----------|----------------------|
| 带 prompt | `promptRef.set({input: formatContextWithPrompt(p)})` + `promptRef.submit()` —— 走原生输入流，server 当作用户手敲，立即触发对话 |
| 纯上下文 | `promptRef.set({input: formatContextStandalone(p)})` **不 submit** —— 预填进输入框。用户切到 opencode 看到已有上下文，自己接着敲问题再回车（备选：暂存 `api.kv`，`/mnabs paste` 粘贴） |

**与 pi adapter 的结构性差异**（重要，不可因子化）：
- pi 的纯上下文是**半透明隐式带上**（`before_agent_start` prepend）；opencode 是**预填可见**（上下文就在输入框，用户可编辑）。后者更透明。
- opencode 拿 `promptRef` 需要一坨 SolidJS Slot 脚手架，pi 没有——这是 opencode adapter 比 pi 重的部分（~多 30-50 行）。

**UI / slash 命令 / 生命周期**：
- `api.ui.toast({message})` 反馈；`api.ui.Slot("session_prompt_right", ...)` 可常驻显示"待用上下文"状态 widget
- `api.command.register(() => [{slash:{name:"mnabs"}, ...}])` 注册 `/mnabs paste` 等
- `api.lifecycle.signal`（AbortSignal）管 socket server；opencode 是**单进程**（实测，见 opencode调研.md §5.1），TUI 在 = opencode 在 = socket 在

**实现前必须实测确认**（见 opencode调研.md §8 #3/#4/#5，命脉项）：
1. `promptRef.set()+submit()` 是否真能发出一条用户消息并触发 LLM
2. 多 session 时如何拿到"当前活动 session"的 ref；home 页（无 session）怎么办
3. 预填（set 不 submit）的内容在用户切换窗后是否保留、是否会被清掉

### 8.5 新 agent 接入清单

任何新 agent（XXX）想接入 microNeo，按本节做：

- [ ] 选定自己的扩展机制（能起 socket、能向自己的 LLM 发消息、能有 UI 反馈）
- [ ] 实现 core（直接复用 pi/opencode 的 TS core，或按 §8.1 重写）
- [ ] 实现 adapter：`onContext` 分流到"发消息"和"待用上下文"两条路径
- [ ] 注册 `/mnabs status|paste|discard|rename` 命令（建议）
- [ ] **不需要改 microNeo 任何代码**——这正是 LSP 式架构的价值

## 九、发送端管线（Sender Pipeline，Go）

> 本节取代 `架构设计.md` §6（Lua 插件实现）。**发送端是 Go 核心，不是 Lua 插件**。本节定义 Go 侧的发送管线，它是 InlineInput 的 `onClose` 回调的落点。

### 9.1 与 InlineInput 的对接

`三分屏方案.md` §4.2 定义的 InlineInput：

```go
type InlineInput struct {
    ...
    onClose func(content string, bp *BufPane)  // 关闭回调
}
```

`onClose(content, bp)` 把**输入内容 + 编辑上下文**交给 Go 核心的发送管线。对接关系：

| InlineInput 提供 | 发送管线用途 |
|------------------|-------------|
| `content`（用户在 input pane 敲的字） | → 报文 `payload.prompt`（若非空） |
| `bp`（原 BufPane） | → 报文 `payload` 的 path/basename/filetype/line/col/selection |

> **空 content 的语义**：用户打开 InlineInput 但什么都没敲就关闭——`prompt` 缺省，发送的是**纯上下文**报文（§6.4）。协议支持，但 UX 上建议 InlineInput 在 content 为空时提示确认"不附提问，仅发送上下文？"。

> **其他入口（未来）**：协议不绑定 InlineInput。未来可加"快速发送选区"热键（不打开 InlineInput，直接 `bp` → 纯上下文报文），共用同一条发送管线。InlineInput 是"带 prompt 发送"的主入口，不是唯一入口。

### 9.2 发送管线流程

```
onClose(content, bp)
   │
   ▼
┌─────────────────────────────────────────┐
│ 1. 采集上下文 captureContext(bp)          │   path/basename/filetype/line/col/selection
└──────────────────┬──────────────────────┘
                   ▼
┌─────────────────────────────────────────┐
│ 2. 组装报文 buildMessage(payload)        │   envelope + payload（prompt 有无由 content 决定）
└──────────────────┬──────────────────────┘
                   ▼
┌─────────────────────────────────────────┐
│ 3. 查注册表 discoverReceivers()          │   Glob 目录 → 试探连接 → GC 僵尸 → 返回存活列表
└──────────────────┬──────────────────────┘
                   ▼
              ┌──────┴──────┐
   0 个       │             │ 1 个          多个
   ▼          ▼             ▼               ▼
 InfoBar 提示  直接用       自动选           selectTarget() UI 让用户挑
 "无接收端"   target        target
                   │
                   ▼
┌─────────────────────────────────────────┐
│ 4. 发送 send(target, jsonLine)           │   connect → write 一行 JSON → close
└──────────────────┬──────────────────────┘
                   ▼
┌─────────────────────────────────────────┐
│ 5. 反馈 InfoBar:Message("已发送给 X")    │   失败则提示重选
└─────────────────────────────────────────┘
```

### 9.3 目标选择决策流（详细）

```
读配置 mnabs.target
  │
  ├─ 有值 name ──► 在注册表找该 name，试探 socket 可连 ──► 设为 target，发送
  │                    └─ 连不上：InfoBar 提示，走下面"选择"
  │
  └─ 空 ──► discoverReceivers()
              ├─ 0 个：InfoBar 提示"没有运行中的 ai agent 接收端"，放弃本次发送
              ├─ 1 个：自动设为 target，直接发送
              └─ 多个：弹出 selectTarget UI（列出 name + pid + cwd），用户选完即发
```

- 选定后写入运行态 `currentTarget`，可持久化到配置 `mnabs.target`（下次直接用，直到失效）
- 每次发送失败（连接错误）→ 清 `currentTarget`，下次重选

### 9.4 上下文采集（Go）

从 `bp *BufPane` 提取：

| payload 字段 | Go 来源 |
|-------------|---------|
| `path` | `bp.Buf.Path` |
| `basename` | `bp.Buf.GetName()` / `filepath.Base` |
| `filetype` | `bp.Buf.Settings["filetype"]` |
| `line` / `col` | `bp.Buf.GetActiveCursor().X / .Y`（0-based，与 micro 内部一致） |
| `selection` | `cursor.HasSelection()` → `GetSelectionStart()` / `GetSelectionEnd()` |
| `selection.text` | 选区文本（< 2KB 时带，超阈值省略） |

> 这些 API 都是 micro 原生 Lua 插件已暴露的能力（见 `架构设计.md` §3 决策 1 的 API 表），Go 核心直接调原生 Go 方法即可，无需经 Lua。

### 9.5 配置项

通过 micro 的 `config.RegisterGlobalOption` 注册（Go 侧，非 Lua）：

| 选项 | 默认 | 说明 |
|------|------|------|
| `mnabs.target` | `""` | 指定目标 name；空=唯一则自动用 / 多个则提示选择 |
| `mnabs.send_selection_text` | `true`（< 2KB） | context 报文是否带选中文本 |
| `mnabs.send_visible_lines` | `false` | 是否带可见区域文本（更费，按需开） |
| `mnabs.privacy_blacklist` | `""` | 敏感文件路径前缀（逗号分隔），命中时不发送（见 §十一） |

### 9.6 Go 包结构建议

```
internal/action/
├── inlineinput.go        ← 三分屏方案定义的 UI（onClose 在这里）
└── mnabs/                ← 本节定义的发送管线（新建子包）
    ├── registry.go       ← registryDir / discoverReceivers / GC
    ├── message.go        ← Envelope / ContextPayload 类型 + JSON 序列化
    ├── capture.go        ← captureContext(bp)
    ├── send.go           ← send(target, msg) = connect+write+close
    ├── select.go         ← selectTarget UI（多目标时）
    └── pipeline.go       ← SendFromInlineInput(content, bp) 总入口，接 onClose
```

> 包名 `mnabs`（microNeo agent bridge）。零侵入 micro 原生代码——全部在新文件内，唯一例外是 InlineInput 在 `internal/action/tab.go` 的 `SetActive` 加焦点锁守卫（三分屏方案 §5.1/§六 已述）。

## 十、生命周期与边界情况

| 场景 | 处理 |
|------|------|
| **agent 崩溃**（没清注册表） | socket 文件残留但连不上；发送端试探连接失败 → 删该注册文件（GC）→ 提示重新选择 |
| **microNeo 崩溃** | 无影响。按需模式下没有持续连接，agent 端 `pendingContext` 是用户主动送来的，本来就等用户处理 |
| **多 microNeo → 同一 agent** | 报文带 `editor.instance` 区分；agent 的 `pendingContext` 被新报文覆盖（只保留最后一次，§6.5） |
| **agent 正在 streaming** | 带 prompt 的报文排队等当前回合结束再触发（`deliverAs:"followUp"` / submit 天然排队），不打断 |
| **用户切换目标 / 切走文件** | 每次发送都是独立的，自然处理 |
| **协议版本不匹配** | 发送端读注册文件 `protocol`，主版本不匹配 → 跳过该 listener + 告警 |
| **InlineInput 活跃时切 Tab** | 三分屏方案 §9.4：初版限制 InlineInput 只在当前 Tab 活跃，切 Tab 前自动 close（触发或不触发发送，按配置） |
| **name 冲突**（两个接收端撞名） | §3.3：后启动者检测旧 pid 已死则覆盖，仍活则自改名 |
| **socket 文件被外部删除** | 接收端 accept 失败/下次 listen 重建；发送端连接失败 → GC 注册文件 |

## 十一、安全性

- **只走本地 Unix socket**，无网络监听、无端口暴露
- 注册表目录 `0700`，socket 文件 `0600`，仅本用户可访问
- 报文里**默认只带选中文本**（`send_visible_lines` 默认关闭），降低敏感信息意外进入 agent 上下文的风险
- **隐私黑名单**（可选配置 `mnabs.privacy_blacklist`）：编辑密钥/凭证文件时，路径命中黑名单前缀则拒绝发送 + InfoBar 提示。v1 可不实现，但协议预留
- 未来双向扩展（agent → microNeo，如 `goto line`、高亮某行）必须**显式经用户/配置授权**后才允许执行——v1 不实现，但这是双向化的硬门槛

## 十四、开放问题（待讨论）

1. **空 content 发送**：InlineInput 里用户什么都没敲就关闭，是否触发纯上下文发送？还是提示确认？还是放弃？——倾向"提示确认"，避免误触。
2. **是否给"快速发送选区"独立热键**：不打开 InlineInput，直接把选区作为纯上下文发给当前 target。共用 §9 管线，只是入口不同。
3. **多份待用上下文**：v1 多来源覆盖只保留最后一次（§6.5）。未来是否按 `editor.instance` 保留多份？多份时 UI 怎么呈现？
4. **隐私黑名单**：是否 v1 就要？密钥/凭证文件不希望进 agent 上下文。
5. **pi 多会话命名**：用户开多个 pi（多 session），每个都注册成 receiver，发送端怎么区分？——用 `name`，默认 `pi-<pid>`，用户可 `/mnabs rename`。
6. **opencode 优先级**：已定 Phase 3（与 pi 同构，core 零改动只加 adapter）。用 TUI 插件，不做 server 插件（无头模式是伪需求，见 opencode调研.md §6.6）。
7. **registry 目录命名**：现为 `microneo-agent-bridge-${UID}`（沿用旧文档，跨文档一致）。是否简化为 `microneo-agent-${UID}`？——倾向不动，保持跨文档一致。
