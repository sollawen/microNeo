# microNeo ↔ Agent 通信协议

> 本文档定义 **microNeo 与 ai agent 之间的通信协议**，是整个 agent 通信需求的"协议层"权威文档。
>
> **文档家族与定位**
> - `原始需求.md` —— 需求源头（microNeo → agent 单向按需推送）
> - `架构设计.md`（D0）—— 总体架构与决策。本文是其协议层的细化与权威
> - `用户界面-V2.md` / `notePane实施计划.md` —— 发送端 UI 设计。**与协议层正交**：协议不规定消息从哪来（见 §一）
> - `opencode调研.md` —— opencode 接收端选型论证（实现层，非协议）
> - `D3-原型验证计划.md` —— 原型执行计划
>
> **本文只规定通信协议本身**（注册表、传输、报文 schema、递送语义、版本、接收端契约）。microNeo 发送端如何实现、各 agent 扩展如何落地，都不在本文范围。

## 一、架构原则（不可动摇）

这是 **LSP 式架构**：microNeo 与各 ai agent 是平等的两端，靠一份共同协议对接；任何一方都不感知对方内部实现。

```
┌──────────────────────────────────────────────┐
│  microNeo（与具体 agent 无关）                 │
│  ┌─────────────┐   ┌──────────────────────┐  │
│  │ 发送入口     │   │ Agent 注册表          │  │
│  │ 消息   +    │──→│ + 上下文打包 + 发送   │  │   ← 本文档定义这一段
│  │ 编辑上下文   │   └──────────┬───────────┘  │
│  └──────┬──────┘              │              │
└─────────┼─────────────────────┼──────────────┘
          │ 消息   + 上下文      │ 本协议（本文档）
          │ 从哪来不在协议范围   │
          │ (UI / 热键 / 命令)   │
            ┌────────────────────┼────────────────────┐
            ▼                    ▼                    ▼
     ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
     │ pi 扩展       │    │ opencode 插件│    │ XXX 扩展     │
     │ 注册/监听/   │    │              │    │（社区写）    │
     │ 递送 LLM     │    │              │    │              │
     └──────────────┘    └──────────────┘    └──────────────┘
```

由此推出三条铁律（**协议设计必须满足**）：

1. **协议 agent 无关**：报文格式、注册表格式、传输方式都不含任何 pi/opencode 特有概念。接入新 agent 不需要改 microNeo 一行代码。
2. **协议 UI 无关**：协议只定义"有一段文字（消息）+ 编辑上下文要发给 agent"之后的全部流程。**这段文字从哪来、由什么 UI/热键/命令产生，不是协议关心的事**。
3. **能力差异由接收端吸收**：协议只规定"递送什么"，不规定"agent 怎么用"。各 agent 的扩展程序负责把递送物落到自己的 LLM 通道。

> **术语约定**：本文一律用"**递送**（deliver）"描述协议把上下文交给接收端这个动作，**不**用"注入"——"注入"暗示隐式改写消息流，而协议层只递送，是否/如何进入 LLM 上下文由接收端自决。

## 二、三个角色

| 角色 | 实体 | 职责 |
|------|------|------|
| **发送端 Sender** | microNeo 进程 | 拿到 `(消息, 编辑上下文)` → 查注册表 → 选目标 → 发报文 |
| **接收端 Receiver** | ai agent 内的扩展程序 | 启动注册 → 监听 socket → 解析报文 → 递送到自己的 LLM 通道 |
| **注册表 Registry** | 一个固定目录 | 接收端登记自己；发送端发现接收端 |

- 一个用户机器上：可能有**多个 microNeo 进程**（编辑不同文件）= 多个发送端；可能有**多个 agent 进程**（pi、opencode、或多个 pi session）= 多个接收端。
- 通信方向：**单向 microNeo → agent**（v1）。协议预留双向扩展（agent → microNeo 的 `ack` / `goto`），v1 不实现。
- 通信触发：**用户主动、按需、一次性**。不是对光标/选区变化的持续追踪。

## 三、服务发现与注册表（Registry）

### 3.1 注册表目录

跨平台、用户私有、重启不残留：

```
${XDG_RUNTIME_DIR:-${TMPDIR:-/tmp}}/microneo-agent-bridge-${UID}/
```

解析优先级：
1. `$XDG_RUNTIME_DIR`（Linux `/run/user/$UID`，进程退出自动清理）——首选
2. `$TMPDIR`（macOS 通常是 `/var/folders/.../T/`）——macOS 兜底
3. `/tmp`——最后兜底

属性：
- 目录名带 `$UID`，多用户机器天然隔离
- 由首个需要它的进程（发送端或接收端）按 `0700` 权限创建
- socket 文件就放在该目录下，权限 `0600`
- 目录本身不主动删除，靠运行时目录随系统重启自动清理；应用只负责清理自己名下的注册文件（见 §3.4 / §九）

> **实现约束（必须两端一致）**：发送端和接收端各自用**完全相同的算法**算出该路径。本文固定算法，两端复刻。

### 3.2 注册文件

每个接收端在该目录下写一个文件 `receiver-<name>.json`，其中 `<name>` 是该接收端的 name 字段值（如 `pi-main` → `receiver-pi-main.json`）。文件名 = 固定前缀 `receiver-` + name，**去掉前缀即 name**，name 全局唯一。内容：

```json
{
  "name": "pi-main",
  "pid": 12345,
  "transport": "unix",
  "socket": "/tmp/.../microneo-agent-bridge-501/receiver-pi-main.sock",
  "protocol": "eabp-1",
  "startedAt": 1717000000,
  "cwd": "/Users/me/project",
  "labels": ["default"]
}
```

字段说明：

| 字段 | 必需 | 说明 |
|------|------|------|
| `name` | ✅ | 人类可读的唯一名。用户起或自动生成（默认 `<agent>-<pid>`，如 `pi-12345`）。注册文件名为 `receiver-<name>.json`，去前缀即 name，保证唯一 |
| `pid` | ✅ | 接收端进程 PID，用于存活检测 |
| `transport` | ✅ | 传输类型，v1 固定 `"unix"` |
| `socket` | ✅ | 该接收端监听的 Unix socket 绝对路径 |
| `protocol` | ✅ | 协议版本号，见 §七。发送端发现版本不匹配时跳过该接收端 |
| `startedAt` | ✅ | 启动时间（Unix 秒）。发送端展示“选择目标”列表时的稳定排序依据 |
| `cwd` | ❌ | 接收端工作目录，发送端可据此判断"是不是同一个项目" |
| `labels` | ❌ | 自由标签数组，发送端可按标签筛选（如 `["default"]`、`["frontend"]`） |

> **为什么加 `receiver-` 前缀**：该目录以后可能混入别的角色文件。前缀即角色命名空间，发送端扫描时按前缀过滤（`receiver-*.json`），未来新增角色不改动现有命名规则。socket 文件同理用 `receiver-<name>.sock`。两端 strip 前缀的算法必须一致。

### 3.3 接收端注册流程

接收端（agent 扩展）启动时：

1. 算出注册表目录（§3.1），不存在则 `0700` 创建
2. 决定自己的 `name`：用户可配，否则默认 `<agent>-<pid>`
3. 算出 socket 路径：`<注册表目录>/receiver-<name>.sock`
4. 在该路径**创建并监听** Unix socket（stream 类型）
5. 写注册文件 `receiver-<name>.json`（§3.2 全字段）

> **name 冲突处理**：若 `receiver-<name>.json` 已存在（上次崩溃残留，或真的撞名），接收端应检测：读旧文件的 `pid`，若该 pid 已死 → 覆盖注册；若仍活 → 给自己改名（追加 `pid` 或序号）后再注册。避免两个接收端写同一 socket 路径。

### 3.4 接收端注销流程

接收端正常退出时：
1. 删除自己的注册文件 `receiver-<name>.json`
2. 删除 socket 文件 `receiver-<name>.sock`
3. 关闭 socket server

### 3.5 存活检测与垃圾回收（发送端职责）

发送端在发送前发现接收端时，按优先级判断存活：

1. **首选——试探连接**：对 `socket` 地址做一次 `connect` + 立即关闭。连得上即认为存活（最可靠）
2. **旁证——PID 检测**：读注册文件 `pid`，判断进程是否存在
3. **顺手 GC**：连接失败且 PID 不存在的注册文件，由发送端**直接删除**（防止僵尸注册堆积）

> **connect 为权威判据，PID 为旁证**：进程崩溃后 socket 无人 listen，connect 必失败——这是 GC 的主路径，PID 复用问题根本进不了这条判断链（复用旧 PID 的新进程不会监听旧 socket）。PID 仅在 connect 成功时作为辅助确认。

## 四、传输层（Transport）

### 4.1 选型：Unix domain socket

| 方案 | 结论 | 理由 |
|------|------|------|
| **Unix socket（采用）** | ✅ | 本地、零网络暴露、快、跨语言成熟；macOS/Linux 原生支持，Windows 10 1803+ 经 AF_UNIX 也可用 |
| TCP localhost | ❌ | 要管理端口冲突、有网络暴露风险 |
| 文件 + 轮询 | ❌ | 延迟高、磁盘 IO、并发竞争 |
| HTTP | ❌ | 每次 POST 开销大、要起 server |

- **v1 范围**：仅 `unix` transport，覆盖 macOS / Linux（microNeo 主要平台）。其他平台/传输方式（如 Windows named pipe）由注册表的 `transport` 字段预留，v1 不实现。
- **socket 类型**：`SOCK_STREAM`（流式），便于按行分帧。

### 4.2 连接策略：每次发送一次"连接 → 写一行 → 关闭"

```
发送端                                接收端
  │                                     │
  │ ── connect(socket) ──────────────► │ accept
  │ ── write(json_line + "\n") ──────► │ read 一行 → 解析 → 处理
  │ ── close ─────────────────────────► │ (conn 关闭)
  │                                     │
```

- **为什么不用长连接**：按需发送模式下发送频率极低（人手触发），长连接的状态维护成本大于每次建连的微秒级开销；且无状态设计对 agent 重启天然鲁棒。
- **接收端实现**：每个连接读到一个 `\n` 就解析一条报文处理；连接关闭即结束本次。未来若扩展为长连接多报文，分帧逻辑不变。

### 4.3 编码：逐行 JSON

- 每条报文是一行 JSON，以 `\n` 结束
- 一条报文一行，**不允许跨行**（JSON 内部不得有裸 `\n`；字符串里的换行必须用 `\n` 转义）
- 接收端按 `\n` 分帧，逐行 `JSON.parse`
- 字符编码：UTF-8
- **“单行”是分帧约束，不是数据量约束**：单行 JSON 可以任意长、任意嵌套（大段选区文本、可见缓冲区、长消息都合法），代价只是人类肉眼可读性差
- 接收端因报文可能很大，需用**不定长 buffer**读（按需增长直到遇到 `\n`），不要用定长 buffer 截断

### 4.4 方向

- v1：**单向 microNeo → agent**
- 协议预留双向扩展（agent 回 `ack`；后续 agent → microNeo 的 `goto`/高亮），v1 不实现，相关边界情况见 §九

## 五、报文协议（Wire Protocol）

### 5.1 公共信封（Envelope）

所有报文共享头部字段：

```json
{
  "v": 1,
  "type": "context",
  "sender": { "pid": 54321, "name": "microNeo", "instance": "win-1" },
  "ts": 1717000123.456,
  "payload": { ... }
}
```

| 字段 | 必需 | 说明 |
|------|------|------|
| `v` | ✅ | 协议主版本号（整数）。v1 = `1`。等于注册表里的 `protocol` 的主版本 |
| `type` | ✅ | 报文类型，见 §5.2 |
| `sender` | ✅ | 发送端标识对象（见下） |
| `ts` | ✅ | 发送时刻（Unix 浮点秒，毫秒精度）。用于接收端排序/展示 |
| `payload` | ✅ | 报文负载，类型取决于 `type` |

`sender` 对象：

| 字段 | 必需 | 说明 |
|------|------|------|
| `pid` | ✅ | 发送端（microNeo）进程 PID |
| `name` | ✅ | 固定 `"microNeo"`（预留未来其他编辑器） |
| `instance` | ✅ | 本 microNeo 实例标识（用户/系统起的窗口/tab 名，如 `win-1`）。多实例时让接收端区分来源 |

### 5.2 报文类型

按需发送模式下，报文类型极少：

| `type` | `payload` | 触发时机 | 方向 |
|--------|-----------|----------|------|
| `context` | 见 §5.3 | **用户主动触发发送**（核心报文） | microNeo → agent |
| `bye` | `{}` | 发送端实例退出（可选，让 agent 清理对应来源的待用上下文） | microNeo → agent |

> v1 只实现 `context`。`bye` 是预留项，发送端退出时可选发送，帮助接收端 GC（如清掉来自该 `sender.instance` 的待用上下文）。

### 5.3 `context` 报文 payload

```json
{
  "path": "/Users/me/a.md",
  "cursor": { "line": 42, "col": 8 },
  "selection": {
    "start": { "line": 42, "col": 8 },
    "end":   { "line": 45, "col": 0 },
    "text":  "## 标题\n正文..."
  },
  "message": "这段标题感觉怪，帮我换个说法",
  "visible_lines": null
}
```

字段说明：

| 字段 | 必需 | 类型 | 说明 |
|------|------|------|------|
| `path` | ✅ | string | 文件绝对路径 |
| `cursor` | ✅ | Position | 光标位置（0-based） |
| `selection` | ❌ | object | 选区对象，无选区则**省略整个字段**（不是 `null`） |
| `selection.start` | ✅* | Position | 选区起点（0-based） |
| `selection.end` | ✅* | Position | 选区终点（0-based） |
| `selection.text` | ❌ | string | 选中文本。选区较小时带（默认 < 2KB），超阈值省略由接收端按 `start/end` 自取或提示 |
| `message` | ❌ | string | **消息：用户附的一段文字**。有则接收端作为一条用户消息递送给 LLM（立即触发对话）；无则递送为"待用上下文"。详见 §六 |
| `visible_lines` | ❌ | string | 可见区域文本（可选，更费，默认关闭）。方便 agent 直接看屏幕上下文 |

> **Position 类型**：`{ "line": int, "col": int }`，坐标 **0-based**（行从 0 起、列从 0 起）。被 `cursor`、`selection.start`、`selection.end` 复用。
>
> **合法性约束（协议契约）**：由**发送端保证合法**——`line`/`col` 非负，`col` 不超出该行长度，`selection.end` 不小于 `selection.start`（不允许反向选区）。**接收端不做校验**，遇非法值行为未定义。这呼应原则④——发送端只递送它已知为真的事实。

> *`selection.start` / `selection.end` 在 `selection` 存在时必需。

**关键语义：`message`（消息）的有无决定递送形态**（这是整个协议最核心的语义分歧点，详见 §六）：

- `message` 存在（非空字符串）→ 用户明确想问 → 接收端递送为**一条用户消息并触发对话**
- `message` 缺省/空 → 用户只想"挂载"编辑上下文 → 接收端递送为**待用上下文**（用户决定怎么用）

### 5.4 完整报文示例

**示例 A：带选区 + 消息（最典型的带消息场景）**

```json
{"v":1,"type":"context","sender":{"pid":54321,"name":"microNeo","instance":"win-1"},
 "ts":1717000123.5,
 "payload":{
   "path":"/Users/me/a.md",
   "cursor":{"line":42,"col":8},
   "selection":{"start":{"line":42,"col":8},"end":{"line":45,"col":0},"text":"## 标题\n正文..."},
   "message":"这段标题感觉怪，帮我换个说法"
 }}
```

**示例 B：纯上下文（无选区、无提问）**

```json
{"v":1,"type":"context","sender":{"pid":54321,"name":"microNeo","instance":"win-1"},
 "ts":1717000123.5,
 "payload":{
   "path":"/Users/me/main.go",
   "cursor":{"line":128,"col":3}
 }}
```

**示例 C：仅光标位置 + 消息（无选区）**

```json
{"v":1,"type":"context","sender":{"pid":54321,"name":"microNeo","instance":"win-1"},
 "ts":1717000123.5,
 "payload":{
   "path":"/Users/me/a.md",
   "cursor":{"line":10,"col":0},
   "message":"我现在在看这里，帮我看看这段逻辑对不对"
 }}
```

> 三个示例的区别只在 payload 字段组合，与消息/上下文的**来源无关**——协议不关心它们由什么 UI/热键产生。

## 六、递送语义（Delivery Semantics）—— 协议最核心的部分

> 本节定义"接收端拿到 `context` 报文后该干什么"。**协议层只规定语义分歧点（`message` 有无）与必须满足的行为约束，不规定具体实现机制**——后者因 agent 能力而异，由各接收端自决（见 §八）。

### 6.1 两条递送路径

接收端根据 `payload.message` 是否存在分流：

| 报文形态 | 语义 | 接收端职责（协议要求） | 用户体验 |
|----------|------|----------------------|----------|
| **带消息**（`message`） | 用户明确发起一次协作 | 把 `上下文 + 消息` 格式化为一条用户消息，**递送给 LLM 并触发对话回合** | 在 agent 里直接看到 AI 开始回复 |
| **纯上下文**（无 `message`） | 用户把编辑器状态"挂"到 agent | 把上下文**递送为待用上下文**（pending）；用户下次发消息时带上，或用户主动粘贴/丢弃 | agent 提示"有来自 microNeo 的上下文待用"，用户决定怎么用 |

### 6.2 为什么是这两条路径（不是别的）

回到需求本质（`原始需求.md`）：**用户主动、按需、一次性**推送。

- 因为是按需（不是高频实时），**不存在淹没 LLM 的问题**——每条报文都是用户明确想发的。
- 带消息 = 编辑器里直接发起协作，最顺。
- 纯上下文 = 把编辑器状态"挂"到 agent，用户决定怎么用，**不强行打断 LLM**。
- 两者都很轻量，**不需要**实时方案里那套内存快照持续刷新 / debounce / 周期性注入的复杂机制。

### 6.3 "带消息" 的递送：协议层一致

带消息的处理**所有接收端一致**：递送为一条用户消息，触发对话。具体用什么机制把消息送进 LLM 通道，是各接收端的实现职责，不是协议的事。

> **冲突处理（协议建议，非强制）**：若 agent 正在 streaming（上一回合未结束），带消息的报文应**排队等当前回合结束**再触发，不打断。具体排队机制由接收端自决。
>
> **协议层边界**：v1 是单向通信，不传 receiver 状态（是否正在 streaming、队列长度等），发送端无法获知冲突是否发生。因此**冲突的识别与排队完全由接收端自主处理**，协议不协调——这属于原则③「能力差异由接收端吸收」的必然结果。

### 6.4 "纯上下文" 的递送：协议层一致，实现因 agent 而异

**协议要求**（所有接收端必须满足）：
- 收到纯上下文报文后，**不得**自动触发 LLM 对话（用户没问，不该答）
- 应有可见提示告知用户"有来自 microNeo 的上下文待用"
- 应提供手段让用户在下次发言时带上，或主动粘贴/丢弃

> 至于"隐式带上"还是"预填可见"等具体形态，由接收端按自身能力自决，协议不强求统一——这正是协议 agent 无关性的体现。

### 6.5 多来源覆盖

多个 microNeo 实例向同一 agent 发纯上下文时，接收端的 `pendingContext` **被新报文覆盖**（只保留最后一次）。报文带 `sender.instance` 区分来源；覆盖时应在提示里更新来源标注。

> 这是 v1 的简化策略。未来若需保留多份待用上下文，可按 `sender.instance` 做 map，但 v1 不做。

## 七、协议版本

### 7.1 版本号方案

- 注册文件 `protocol` 字段格式：`eabp-<major>`，如 `eabp-1`（"Editor Agent Bridge Protocol v1"）
- 信封 `v` 字段：主版本整数，如 `1`
- **主版本**：报文结构不兼容变更时 +1（增删必需字段、改语义）
- **次版本/兼容 additions**：新增可选字段、新增报文类型——**不改主版本**。接收端应忽略未知字段、忽略未知 `type`（前向兼容）

### 7.2 版本协商

- 发送端读注册文件 `protocol`（字符串如 `eabp-1`），解析取主版本整数（`1`），与信封 `v` 比较；不匹配 → 跳过该接收端 + 告警提示
- 接收端读信封 `v`（整数），与自身版本的主版本比较；不匹配 → 忽略该报文 + 可选告警
- v1 不做复杂的 capabilities 协商（YAGNI）

### 7.3 当前版本

**v1（`eabp-1`）**——本文定义的全部内容。报文类型仅 `context`（`bye` 预留）。

## 八、接收端契约（Receiver Contract）

> 本节定义"一个 agent 想接入 microNeo，必须实现什么"。这是 LSP 式架构的接入规范——协议不只规定报文格式，也规定接收端的行为义务。

### 8.1 协议合规要求（所有接收端必须满足）

一个合规的接收端必须：

1. **启动时注册**：算注册表目录（§3.1）→ 起 Unix socket server（§四）→ 写注册文件（§3.2）
2. **监听并分帧**：accept 连接 → 按 `\n` 分帧 → `JSON.parse` 每行 → 校验信封 `v`/`type`
3. **处理 `context` 报文**：按 §六 分流（`message` 有无）
4. **退出时注销**：删注册文件 + 删 socket + 关 server（§3.4）

> 接入新 agent **不需要改 microNeo 任何代码**——这正是 LSP 式架构的价值。各 agent 按自身扩展机制实现上述四点即可。

## 九、生命周期与边界情况

| 场景 | 处理 |
|------|------|
| **agent 崩溃**（没清注册表） | socket 文件残留但连不上；发送端试探连接失败 → 删该注册文件（GC，§3.5）→ 提示重新选择 |
| **microNeo 崩溃** | 无影响。按需模式下没有持续连接，agent 端 `pendingContext` 是用户主动送来的，本来就等用户处理 |
| **多 microNeo → 同一 agent** | 报文带 `sender.instance` 区分；agent 的 `pendingContext` 被新报文覆盖（只保留最后一次，§6.5） |
| **agent 正在 streaming** | 接收端自主排队：v1 单向不传状态，冲突的识别与处理全由接收端负责（§6.3） |
| **协议版本不匹配** | 发送端读注册文件 `protocol`，主版本不匹配 → 跳过该接收端 + 告警 |
| **name 冲突**（两个接收端撞名） | §3.3：后启动者检测旧 pid 已死则覆盖，仍活则自改名 |
| **socket 文件被外部删除** | 接收端 accept 失败/下次 listen 重建；发送端连接失败 → GC 注册文件 |

## 十、安全性

- **只走本地 Unix socket**，无网络监听、无端口暴露
- 注册表目录 `0700`，socket 文件 `0600`，仅本用户可访问
- 报文里 `visible_lines`（可见区域文本）为**可选字段，默认不发送**，降低敏感信息意外进入 agent 上下文的风险
- 未来双向扩展（agent → microNeo，如 `goto line`、高亮某行）必须**显式经用户/配置授权**后才允许执行——v1 不实现，但这是双向化的硬门槛
