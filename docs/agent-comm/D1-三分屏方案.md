# InlineInput：撕纸式内联分屏

## 一、需求与界面意象

> **本方案在整体需求中的位置**：InlineInput 是 `原始需求.md`（microNeo → ai agent 单向推送）的 **UI 部分**。它承担“用户输入一句话提问”这一环节；用户关闭输入区时，InlineInput 触发一次发送——把输入内容连同编辑上下文（光标位置 / 选区 / 文件）打包，发给用户选定的 agent。具体通信协议见独立的 `microNeo-Agent 通信协议.md`（另文）。本文只聚焦发送端的 **UI 机制**。

用户在主编辑区编辑文件，光标停在某一行（设为 `line10`）。按下一个热键：

- 当前文件像一张纸**在 line10 之下被撕开**，分成上下两半
- 上半纸显示 `line0–line10`（原文件）
- 下半纸显示 `line11–end`（原文件）
- 撕开的**夹缝**里出现一个输入区，用户在里面编辑**全新的内容**
- 输入区的内容**不进入原文件的 buffer**——原文件完全不受影响
- 用户按某个键（如 `Esc` / 确认）后，**触发发送**：把输入内容 + 编辑上下文打包，发给用户选定的 agent；随后输入区消失，上下两半纸重新接合，恢复成单一窗口。原文件不受输入的任何影响

```
   ┌──────────────────────────┐
   │  上半 pane   原文件 line0–10   │  非活跃
   ├──────────────────────────┤  ← 撕开线
   │  input pane 新 buffer，可编辑  │  活跃（焦点锁定）
   ├──────────────────────────┤  ← 撕开线
   │  下半 pane   原文件 line11–end │  非活跃
   └──────────────────────────┘
```

## 二、核心设计决策：这不是浮层，是一次临时三分屏

初看这个需求像是"浮动窗口"，但反复推敲后发现：**浮动是错误的抽象**。

它本质上就是 micro 已经具备的能力——**打开多个窗口、在其中一个里编辑独立 buffer**。只是普通 split 是"二分平铺"，而我们要的是"三分 + 夹缝编辑"。

因此 InlineInput 的核心决策是：

> **InlineInput = 一次临时性的三分屏。复用 micro 原生的 `views.Node` 布局树机制，在光标处把当前 pane 一分为三。**

这个决策带来三个决定性好处：

1. **机制零成本**：布局、resize 重排、渲染顺序、事件路由，全部白拿，与普通 split 完全一致
2. **视觉即需求**：撕纸效果是三分屏的天然形态，不是特效
3. **最小侵入**：不碰 BufWindow、BufPane、views.Node 的任何核心逻辑

## 三、关键架构事实（代码佐证）

### 3.1 views.Node 天然支持三分屏

`views.Node` 的子节点是切片，**数量不限**（`internal/views/splits.go:36`）：

```go
type Node struct {
    ...
    children []*Node   // 任意多个
    ...
}
```

`vHSplit` 在非叶子分支里就是往 `children` 里 `append` 一个（`splits.go:355`）。因此"一个 STVert 节点带三个 STHoriz 子节点 `[上半, input, 下半]`"是这套数据结构的**原生形态**，不是硬凑。`Resize`（`splits.go:209`）遍历所有子节点按 `propH` 比例铺排，三个子节点一视同仁。

### 3.2 渲染是"顺序覆盖"，三分屏自然成立

主循环（`cmd/micro/micro.go:491`）：

```go
for _, ep := range action.MainTab().Panes {
    ep.Display()
}
```

每个 pane 按 `Panes` 数组顺序画到 tcell 后缓冲区，后画的盖在先画的上面。三个 pane 在布局树里占据各自不相交的区域，顺序绘制就得到完整的撕纸画面。**无需任何层叠/合成层逻辑。**

### 3.3 稳态下 Display 不滚动视口——焦点锁的关键依据

`BufWindow.Display()`（`bufwindow.go:285`）只画，不碰 `StartLine`：

```go
func (w *BufWindow) Display() {
    w.updateDisplayInfo()
    w.displayStatusLine()
    w.displayScrollBar()
    w.displayBuffer()   // 画，不调 Relocate
}
```

`Relocate`（滚动）只在三个时刻被触发：**创建时**、**Resize 时**、**收到事件时**。只要堵住"事件"这条路，稳态下上下两个 pane 的视口纹丝不动。这正是焦点锁的设计依据。

### 3.4 焦点切换只有一个入口

所有切焦点的路（键盘 `next-split`、鼠标点击、Lua 调用）最终都汇到 `Tab.SetActive`（`tab.go:341`）：

```go
func (t *Tab) SetActive(i int) {
    t.active = i
    for j, p := range t.Panes {
        if j == i { p.SetActive(true) } else { p.SetActive(false) }
    }
}
```

在它开头加一个守卫判断，就能一处封死所有焦点逃逸。

**代码级确证（焦点锁严密性的关键）**：全工程对 `t.active` 的赋值**仅此一处**（`tab.go:342`，即 `Tab.SetActive` 内部）。键盘 `next-split`（`actions.go:2058`）、鼠标点击 pane（`tab.go:312`）、新建 split 自动设焦（`bufpane.go:666`）、终端 pane（`termpane.go:226`）等所有焦点切换，全部经由 `Tab.SetActive`。pane 级的 `BufPane.SetActive`（`bufpane.go:700`）只改 `active` 标志位、**不动 `t.active`**，无法独立改变焦点归属。因此守卫只加在 `Tab.SetActive` 开头即可覆盖全部路径——不存在绕过它的旁路。这是整个"原文件零干扰"承诺的根基，已由代码证实成立。

### 3.5 非活跃窗口不画光标

`BufWindow.showCursor`（`bufwindow.go:397`）有 `if w.active` 守卫。上下两个 pane 设为非活跃后，原 buffer 的光标不会画在它们身上，视觉干净。

## 四、整体方案

### 4.1 数据结构

InlineInput 活跃时，当前 Tab 的布局树从单节点变成三分屏：

```
撕开前：                     撕开后：
                            
Node (leaf, buffer B)        Node (STVert, 三子节点)
                              ├── Node (上半, buffer B, 非活跃)
                              ├── Node (input, buffer 新建, 活跃)
                              └── Node (下半, buffer B, 非活跃)
```

- **上半 pane**：显示原 buffer B，视口锁定 `line0–line10`
- **input pane**：全新的空 buffer，独享活动光标，是唯一的活跃 pane
- **下半 pane**：显示原 buffer B，视口锁定 `line11–end`

上下两个 pane 共享原 buffer B，但都不活跃。原 buffer 的活动光标停留在 line10（上半 pane 的视口内），这一点对上半无害，对下半需要处理（见 5.2）。

### 4.2 组件结构

新建 `internal/action/inlineinput.go`，零侵入 micro 原生代码：

```go
type InlineInput struct {
    active     bool
    origPane   *BufPane   // 撕开前的原 pane（用于接合时还原）
    origSplitID uint64    // 原 pane 在布局树中的 id
    topPane    *BufPane   // 上半
    inputPane  *BufPane   // 输入（活跃）
    bottomPane *BufPane   // 下半

    // input 的配置
    height     int        // input 区高度（默认 10）
    onClose    func(content string, bp *BufPane) // 关闭回调：把 输入内容 + 编辑上下文 交给 Go 核心的发送管线（见通信协议文档）
}

// 生命周期
func ToggleInlineInput()             // 热键入口
func (i *InlineInput) open(bp *BufPane)   // 撕开
func (i *InlineInput) close()             // 接合 + 回调
```

### 4.3 三分屏的构造方式

完全复用原生 split API，连续两次 `HSplit`。具体调用路径参考 `BufPane.HSplitIndex`（`bufpane.go:671`）：

1. 记录原 pane `bp`、它的 `splitID`、光标行 `curY`
2. 在 `bp` 下方 HSplit 出 **input pane**（新空 buffer）
3. 在 input pane 下方再 HSplit 出 **下半 pane**（绑定原 buffer B）
4. 上半 pane 即原 `bp` 自身（视口保持在 `line0–curY`）
5. input 高度可配置；上下两半按剩余高度比例分配
6. `SetActive` 指向 input pane，上下两半设为非活跃

**两次 HSplit 的实际树形演变**（代码确证，`views/splits.go`）——这是"三个平级兄弟"成立的推导过程：

- 起点：单叶子节点（同时是根）= 原 `bp`
- 第一次 HSplit（在 `bp` 上调用）：根节点是叶子且 `parent == nil` → 被改为 `STVert` 容器，`vHSplit`（`splits.go:339`）在其 `children` 里 append 两个 `STHoriz` 子节点 → 树变为 `[top, input]`
- 第二次 HSplit（在 input 上调用）：input 是 `STHoriz` 叶子、父节点是 `STVert`，于是走 `hHSplit`（`splits.go:274`）**委托父节点**，父节点的 `vHSplit` 在 `children` 里 insert 第三个子节点 → 树变为 `[top, input, bottom]`

最终得到的正是 §4.1 图中"一个 `STVert` 带三个 `STHoriz` 平级兄弟"的形态。这一步是整个方案成立的基础——它证明三分屏不需要任何特殊布局代码，完全是 micro split 机制的天然结果。

### 4.4 关闭与接合

关闭时反向操作。**关键：三个 pane 是上述的平级兄弟，`Node.Unsplit` 一次只删一个叶子节点，因此接合需要两次 unsplit——逐个删掉 input 和下半，最终保留上半 = 原 `origPane`**：

1. 调 `onClose(inputPane.Buf.String(), origPane)`：把输入内容 + 编辑上下文（光标 / 选区 / 文件）交给 Go 核心的发送管线（注册表查询 → 目标选择 → 发送，见通信协议文档）
2. 第一次 unsplit：对 input 的 split 节点调 `Node.Unsplit`（`splits.go:464`），将其从父节点 `children` 删除 → 树从 `[top, input, bottom]` 变为 `[top, bottom]`；同步 `tab.RemovePane`（`tab.go:373`）移除 input pane
3. 第二次 unsplit：对 bottom 的 split 节点调 `Node.Unsplit` → 树变为 `[top]`；父节点只剩单子，`flatten`（`splits.go:485`）自动把 top 收编为根，布局回到撕开前的单 pane 形态；同步 `RemovePane` 移除 bottom pane
4. `SetActive` 指回 `origPane`，视口还原到撕开前的位置
5. 清理 InlineInput 状态

> **不会误伤原 buffer**：`Node.unsplit`（`splits.go:442`）只操作布局树的 `children` 切片，`Tab.RemovePane`（`tab.go:373`）只操作 `Panes` 数组，两者都不碰 buffer。只有 `BufPane.Close`（`bufpane.go:695`）才会关 buffer，本路径不调用它。原文件 buffer B 在整个接合过程中始终存活、内容零修改。

> **避开一个坑**：micro 自带的 `BufPane.Unsplit` action（`actions.go:2039`）注释写 "closes all splits in the current tab except the active one"，**与实际行为不符**——代码只删除调用者自身这一个叶子节点，并非删除其余所有 split。本方案不直接复用该 action，按上面精确步骤操作，避免被注释误导。

## 五、三个技术焦点

### 5.1 焦点锁 —— 堵死事件路

input 活跃期间，禁止焦点转移到上下两半（以及任何其它 pane）。入口在 `Tab.SetActive`（`tab.go:341`）加一处守卫：

```go
func (t *Tab) SetActive(i int) {
    if InlineInputInstance != nil && InlineInputInstance.active {
        if target := t.Panes[i]; target != InlineInputInstance.inputPane {
            return   // 焦点锁：只允许留在 input pane
        }
    }
    // ...原逻辑
}
```

一处封死：键盘 `next-split`、鼠标点击上下半 pane、Lua `pane:SetActive` 全部失效。

**焦点锁的连锁效果**：因为事件只进 input pane，上下两半永不收到事件 → 永不触发它们的 Relocate → 稳态视口天然稳定。这是整个设计的点睛之笔，它让大半个问题（稳态视口）自动消失。

### 5.2 下半 pane 的视口稳定 —— re-stamp

焦点锁解决了稳态，但还有两个瞬间会触发 Relocate，且都来自同一个根因：**buffer B 的活动光标停在 line10，而下半 pane 的视口想停在 line11**。

`Relocate`（`bufwindow.go:236`）用 buffer 的活动光标决定视口：

```go
activeC := w.Buf.GetActiveCursor()   // line10
...
if c.LessThan(w.StartLine) {         // line10 < line11 成立
    w.StartLine = c                  // ← 把下半 pane 视口拽回 line10
}
```

对下半 pane，`BufWindow.Resize`（`bufwindow.go:100`）和 `initialRelocate`（`bufpane.go:384`）都会触发这个拽回。

**解法：re-stamp（重新盖章）** —— 让框架按它的逻辑干完，立刻覆盖回来。不动 Relocate、不动 BufWindow：

| 时机 | 框架行为 | 我们的补刀 |
|------|---------|-----------|
| 三分屏构造完成后 | `initialRelocate` 把下半拽到 line10 | 立刻 `bottomPane.StartLine = SLoc{Line: curY+1}` |
| 终端 resize 后 | `BufWindow.Resize` 内部 Relocate 又拽回 | 在 resize 钩子末尾再盖一次 |

集中在一个 `relayoutInlineInput()` 函数里，两行核心逻辑。符合 AGENTS.md"不修改原生代码"原则——我们不改 Relocate，只在它干完之后覆盖结果。

### 5.3 resize —— 完全免费

三分屏的三个 pane 都在布局树里。终端尺寸变化时：

```
TabList.Resize() (tab.go:82)
  └── Node.Resize() (splits.go:209)       ← 按 propH 自动重排三个子节点
        ├── 上半 pane.Resize()
        ├── input pane.Resize()
        └── 下半 pane.Resize()
```

比例分配、位置重算全部由 `views.Node` 自动完成，与普通 split 行为一致。**唯一的后续动作**是 resize 完成后对下半 pane 做一次 re-stamp（5.2）。这就是相对"浮动窗口"方案的巨大优势——浮动方案要自己维护 X/Y/Width 三个坐标，这里只剩 Y 一个 re-stamp。

### 5.4 动态高度 —— 像聊天框一样随内容生长

输入区的高度跟随内容行数变化，从 1 行长起，回车就长高一格，删到只剩 1 行就收回。设硬上限 `maxLines`（默认 10），长过上限后高度封顶，转为内部滚动。

**核心机制**：micro 的布局树自带动态调整单个 split 高度的能力——`Node.ResizeSplit`（`splits.go:187`）。它的 `vResizeSplit`（`splits.go:143`）正是处理 STVert 三子节点的逻辑：

```go
c1, c2 = n.children[i], n.children[i+1]   // i=input → c1=input, c2=下半
toth := c1.H + c2.H                         // input+下半 的总高
c1.Resize(c1.W, size)                       // input 设为新高度
c2.Resize(c2.W, toth-size)                  // 下半 自动让位/回收差额
```

调一次 `inputNode.ResizeSplit(newH)`：input 区变成 newH 行，下半区同步缩/扩，上半区纹丝不动。这正是"撕口随内容变宽变窄"的精确实现。内置保护 `if size >= toth { return false }` 会拒绝把下半挤没。

**高度公式**：

```go
desired := clamp(inputBuf.LinesNum(), 1, effectiveMax)
if desired != currentInputHeight {
    inputNode.ResizeSplit(desired)   // 下半自动让位/回收
    relayoutInlineInput()            // re-stamp 下半视口（5.2 那个）
}
```

触发时机：**每次 input 事件处理完**查一次行数。不 hook 回车键——查行数比 hook 按键更健壮，粘贴多行、撤销、删除都能正确响应。

**达到上限后的滚动是免费的**。假设用户输入到第 11 行：

| 关注点 | 机制 | 成本 |
|------|------|------|
| input 区高度 | 停在 10，`ResizeSplit` 不再调用 | 我们的判断 |
| 第 11 行怎么看到 | input pane 是活跃 pane，`Relocate` 正常把光标滚进视口 | **白送**（活跃 pane 的原生行为） |
| 滚动条 | `displayScrollBar` 条件 `LinesNum() > Height` 成立 | **白送**（scrollbar 开启时自动出现/消失） |

关键回顾 5.1 的焦点锁：它堵死的是上下两个**非活跃** pane 的事件路。input pane **是活跃的**，它本来就该收事件、该触发 Relocate、该跟随光标滚动。所以"上限后内部滚动 + 滚动条"不是新功能，是活跃 BufPane 的原生行为，零自定义滚动逻辑。

滚动条遵循 micro 的 buffer 级 `scrollbar` setting（默认 false），跟随用户全局设置即可。

**终端太小时的降级**：10 行是硬上限，但终端本身可能更小。安全公式：

```go
effectiveMax := min(maxLines, 总可用高度 - minBottomReserve)
```

`minBottomReserve` 取 2~3 行（保证下半区至少露几行，不至于被 input 完全吞掉）。这样 `ResizeSplit` 的内置保护永远不会被触发，我们用 `effectiveMax` 提前挡住。

**配置化**：`maxLines` 默认 10，做成可配置选项 `inlineinput.maxlines`，用户嫌小可调。

**与 5.2/5.3 的关系**：动态高度不引入新机制，它就是把 5.3 的 resize 重排从"终端尺寸变化触发"扩展到"内容行数变化也触发"。善后动作完全一样——都是 re-stamp 下半 pane。代码上是同一个 `relayoutInlineInput()` 函数被多一个调用点调用。

## 六、侵入性清单

| 环节 | 方案 | 侵入性 |
|------|------|--------|
| 三分屏布局 | 复用 `views.Node`（STVert 三子节点） | **零** |
| resize 重排 | 复用 `Node.Resize` 比例分配 | **零** |
| 编辑能力 | input pane 是全新 buffer + 原生 BufPane | **零** |
| 渲染顺序 | 三个 pane 自然顺序绘制 | **零** |
| 稳态视口稳定 | 焦点锁堵死事件路 | **零** |
| 非活跃光标隐藏 | 复用 `showCursor` 的 `if w.active` | **零** |
| 焦点锁死 | `Tab.SetActive` 加一处守卫 | **极小**（原生文件加 ~4 行） |
| 下半 pane 视口 | re-stamp（构造后 + resize 后覆盖 StartLine） | **零**（不动原生代码） |

唯一需要触碰的原生文件是 `internal/action/tab.go` 的 `SetActive`，加一个守卫判断。其余全部在新文件 `inlineinput.go` 内完成。

> 注：`Tab.SetActive` 的守卫如果坚持零侵入，也可以用"事件拦截"替代——在主循环事件分发处，当 input 活跃时把所有非 input 目标的事件改投给 input pane。但相比之下，`SetActive` 守卫更集中、更不容易漏（鼠标点击等不经过事件分发的事件也能挡住）。这是一个可权衡的点。

## 七、Go/Lua 边界与架构定位

### 7.1 核心原则：microNeo 全 Go，协议中心

本方案以及它服务的 agent 通信需求，遵循一个明确的架构原则：

> **microNeo 发送端的所有核心代码都用 Go 写死。microNeo 自身不包含任何针对具体 agent（pi、opencode、…）的 Lua 插件。**不同的 ai agent 通过各自遵守 microNeo 通信协议的**扩展程序**（运行在 agent 侧）来接入。

这是 **LSP 式架构**：编辑器只定义协议，每个 agent 用自己的扩展机制实现协议的接收端。接入新 agent 时**完全不用碰 microNeo 的代码**——这正是“精力有限、只亲自接 pi + opencode、长尾留给社区”的最佳支撑。

```
┌──────────────────────────────────────────────┐
│  microNeo（全 Go，写死，与具体 agent 无关）    │
│  ┌─────────────┐   ┌──────────────────────┐  │
│  │ InlineInput │   │ Agent 注册表 + 选择   │  │
│  │ 撕纸 UI     │──→│ + 上下文打包 + 发送   │  │
│  └─────────────┘   └──────────┬───────────┘  │
└───────────────────────────────┼──────────────┘
                                 │ 通信协议（Go 定义，另文）
            ┌────────────────────┼────────────────────┐
            ▼                    ▼                    ▼
     ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
     │ pi 扩展      │    │ opencode 扩展│    │ XXX 扩展     │
     │ 注册/监听/   │    │（社区/用户） │    │（社区写）    │
     │ 转发 LLM     │    │              │    │              │
     └──────────────┘    └──────────────┘    └──────────────┘
```

职责划分：

- **三分屏 UI、焦点锁、re-stamp、动态高度、上下文打包、agent 注册表、目标选择 UI、实际发送** —— 全部在 Go 核心
- **通信协议**（注册 / 监听 / 消息格式 / 传输方式）—— Go 定义，见独立的 `microNeo-Agent 通信协议.md`
- **agent 侧的注册、监听、转发给 LLM** —— 各 agent 自己的扩展机制（如 pi extension / opencode plugin）

### 7.2 Lua 的剩余空间（很小）

在此架构下，**Lua 不参与任何 agent 通信逻辑**。它只在“用户级个性化”层面保留极小的空间，且与 agent 是谁完全无关，例如：

- 自定义触发热键（micro 原生 keybinding 能力，不需要专门暴露）
- 用户想固定附加的提问前缀 / 后缀（可选的轻量配置）

也就是说：**agent 通信这块的 Lua 事件 = 零**。原先设想的 `onInit` / `listAgents` / `onSend` / `formatContext` / `onInlineInputClose` 等 Lua 事件全部作废——它们的职责由 Go 核心和通信协议承担。

是否暴露 `micro.ToggleInlineInput` 给 Lua 仅用于热键绑定，是一个**可选的最小暴露**，第一版可不暴露（默认热键直接在 Go `defaults.go` 绑定即可）。

## 九、边界情况与风险

### 9.1 光标在文件首/末行
- 光标在 line0：上半 pane 高度为 0 → 应隐藏上半，退化为二分屏（input + 下半）
- 光标在末行：下半 pane 高度为 0 → 同理退化为二分屏
- **处理**：构造时检测，单边为 0 时跳过对应 HSplit

### 9.2 终端高度过小
- 剩余高度不足以容纳 input + 至少 1 行上下半
- **处理**：设最小可用高度阈值，不足时拒绝打开并提示

### 9.3 原文件被外部修改
- input 活跃期间原 buffer 若被外部进程改动（autosave、文件 watch）
- **处理**：input 活跃期间原 buffer 视为只读快照，关闭时检测是否变化并提示用户

### 9.4 多 Tab 场景
- InlineInput 绑定到具体 Tab，切换 Tab 时应如何处理
- **处理**：初版限制 InlineInput 只在当前 Tab 活跃，切 Tab 前自动 close；后续可考虑跨 Tab 保持

### 9.5 性能
- 主循环每帧多 Display 两个 pane，但它们都不活跃、无事件，开销可忽略
- re-stamp 仅在构造和 resize 时执行，非热路径

## 十、备选方案对比

| 方案 | 可行性 | resize 复杂度 | 视觉 | 侵入性 |
|------|--------|--------------|------|--------|
| 纯 Lua | ❌ | — | — | — |
| 浮动窗口（脱离布局树） | ✅ | 高（X/Y/Width 全自管） | 浮在上方 | 中 |
| 撕纸式三分屏（本方案） | ✅ | **零**（白拿） | 真正撕开 | **极小** |

浮动窗口方案在 BufWindow 渲染机制上技术可行（BufWindow 只认 X/Y/W/H，顺序覆盖可叠加），但 resize 时要独立维护全部坐标，且不符合"复用原生机制"的原则。撕纸式三分屏在 resize、布局、事件路由上全面复用原生能力，是更契合 micro 架构的方案。
