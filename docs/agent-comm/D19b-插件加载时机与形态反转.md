# D19b · aibp-opencode 插件形态反转：改用 TUI 插件

> **状态**：基于 D19「源码核对」+ 本次「启动时机」追查的修订。
> **一句话结论**：D19 §二把插件形态从 TUI 翻成 server 的反转**不成立**，应**翻回 TUI 插件**。
> 直接诱因是用户实测症状：**server 插件要等用户先发一句 hi（触发 instance bootstrap）才启动**，而 TUI 插件在「主界面就绪」时立即加载。

---

## 一、症状与根因

### 1.1 现象
- opencode 启动进入主界面后，aibp server 插件**没有**跟着启动（注册表里没有 `ai-*.json`，socket 没起）。
- 用户先发一句 hi、开启对话后，插件才启动。

### 1.2 根因：server 插件受 instance bootstrap gating

源码铁证链：

1. **server 插件体跑在 `Plugin.state` 的 `InstanceState.make(init)` 里**（`packages/opencode/src/plugin/index.ts`），而 `InstanceState.make` = `ScopedCache.make({lookup})`（`effect/instance-state.ts`）—— **惰性缓存，首次 `get` 才执行 `init`**。
2. **`init` 里跑 plugin 主体**，但 `init` 只被 `InstanceBootstrap.run` → `plugin.init()` 调用（`project/bootstrap.ts`）。
3. **`InstanceBootstrap.run` 只在 `InstanceStore.load` → `boot` 里触发**（`project/instance-store.ts:boot`）。
4. **`InstanceStore.load` 的真实触发点**（`grep store.load`）：
   - HTTP `InstanceContextMiddleware`（`server/.../middleware/instance-context.ts:29`）——**每条 instance-scoped 路由（含 `/tui/*`、`/session/*`）首次被访问时**才 load；
   - `effect-cmd.ts:88`（CLI 命令跑时）；
   - worker 的 `checkUpgrade`（`cli/tui/worker.ts:60`）——一个 **`setTimeout(...,1000).unref()`** 的 eager preload。
5. 那个 `+1s checkUpgrade` preload 在 v1.17.x 才有（commit `fe0c4f8c7`，2026-06-07）。**它不是插件契约的一部分**，是个 1 秒定时器的隐式副作用：
   - 用户用的发布版可能没有它 / 被其它启动路径阻塞；
   - 即便有，它是个 `.unref()` 定时器，进程忙时可能延迟；
   - 上游随时可能改掉。

> **结论**：server 插件的「启动时机」本质是「**首次访问 instance-scoped 路由**」——而发第一句 hi 正是第一次密集访问 `/session` 路由的时刻。**用户看到的「发 hi 才起」就是 instance bootstrap 的 gating 表现**，不是偶发 bug。

## 二、决定性事实：TUI 插件在「主界面就绪」时立即加载

### 2.1 TUI 插件加载链（全部源码核对）

```
主线程 App 组件 mount (packages/tui/src/app.tsx:397)
  → props.pluginHost.start({...})              // createLegacyTuiPluginHost()
  → runtime.ts: init(input) → load(input)
  → PluginLoader.loadExternal({ kind: "tui" })  // 读 config.plugin_origins
  → 逐个 activatePluginEntry(...)
  → 调 tui(api, options, meta)                  // ← 我们的插件体在这里跑
```

关键点（`packages/tui/src/app.tsx:354-409`）：

```tsx
function App(props: { pluginHost: TuiPluginHost }) {
  // ...
  const [ready, setReady] = createSignal(false)
  props.pluginHost
    .start({ api, config: tuiConfig, runtime: pluginRuntime, dispose: ... })
    .catch(...)
    .finally(() => setReady(true))
```

**`pluginHost.start` 在 App 组件 body 顶层调用 = App 一挂载就跑**。它：
- **不经过 InstanceStore / bootstrap / plugin.init**；
- **不需要任何 session**（home 页、空 session 都一样）；
- 在**主线程**（TUI 渲染线程），不是 worker。

→ 这正是用户要的「opencode 完成启动、进入主界面后就运行」。

### 2.2 TUI 插件能拿到和 server 插件**完全相同**的递送 API

这是反转 D19 §二的决定性证据。`packages/plugin/src/tui.ts`：

```ts
export type TuiPluginApi = {
  client: OpencodeClient          // ← 与 server 插件 input.client 同一个类型！
  ui: { toast, Slot, Prompt, ... }
  keymap: TuiKeymap               // 含 dispatchCommand
  lifecycle: TuiLifecycle         // onDispose(fn) — 框架驱动清理
  // ...
}
export type TuiPlugin = (api: TuiPluginApi, options, meta) => Promise<void>
export type TuiPluginModule = { id?: string; tui: TuiPlugin; server?: never }
```

`api.client` 就是 `OpencodeClient`，即 `api.client.tui.appendPrompt / submitPrompt / clearPrompt / showToast` **全部可用**。

> **D19 §二的反转理由「server 插件能用 `client.tui.*` 控制输入框」对 TUI 插件同样成立**——TUI 插件的 `api.client` 就是同一个 client。server 插件在这一点上**没有任何独占优势**。

### 2.3 反过来，TUI 插件有 server 插件没有的三个优势（针对启动需求）

| 需求 | server 插件 | TUI 插件 |
|------|------------|----------|
| **启动时机**（注册名字/开 socket） | instance bootstrap gating → 首次访问 instance 路由才起（=「发 hi 才起」） | **App mount 即起**（主界面就绪） |
| **显示名字**（在 TUI 上画 `● Bravo`） | server 插件在 **worker 线程**，要跨线程把名字送到主线程 TUI 去渲染 | TUI 插件**就在主线程**，`api.ui.Slot`/`api.ui.toast` 直接画 |
| **清理**（关 socket / 删注册文件） | 仅 `process.once("exit")` 兜底 | `api.lifecycle.onDispose`（框架驱动，对称 aibp-pi 的 `session_shutdown`） |

## 三、与你的启动需求逐一匹配

你明确的三件启动事，TUI 插件全部天然契合：

| # | 启动事 | TUI 插件怎么做 | 与 aibp-pi 对照 |
|---|--------|---------------|----------------|
| 1 | 向 aibp 注册名字（`ai-Bravo.json`） | `tui(api)` 函数体里：`loadNamePool` → `allocateName` → `writeFileSync(regFile)`（**逐字复制 aibp-pi**） | `session_start` 里同样这套 |
| 2 | 在 opencode 界面显示名字 | `api.ui.Slot name="home_footer"`/`"home_prompt_right"`（持久）或 `api.ui.toast`（一次性，见 §五） | `ctx.ui.setStatus("aibp", "● Bravo")` |
| 3 | 开 socket 监听 | `tui(api)` 里 `net.createServer(connectionHandler).listen(socketPath)`（主线程，**逐字复制 aibp-pi**） | `session_start` 里同样这套 |

> 「收到消息才往 LLM 塞」**完全不需要在启动时做**——递送逻辑挂在 socket 的 `data` handler 里（和 aibp-pi 完全一样），与启动时机无关。这正是你说的「没收到消息前不向 LLM 发任何东西」。

## 四、为什么 D19 §二的反转不成立（必须翻回来）

D19 §二「选型重估」用四条理由把调研的 TUI 结论翻成 server：

| D19 §二理由 | 本次核对的真相 |
|------------|--------------|
| ① 「server 插件能用 `client.tui.*`，与 aibp-pi 同构」 | **TUI 插件的 `api.client` 就是同一个 `OpencodeClient`**，`client.tui.*` 一样可用。server 无独占优势。❌ |
| ② 「server 插件更简单稳（TUI 插件 3 个命脉级待验证）」 | 那 3 个（调研 §8 #3/#4/#5）是针对「用 `TuiPromptRef.set+submit` 做递送」的旧 TUI 方案。**我们的递送走 `api.client.tui.*`（与 server 插件同款），不碰 ref/session 定位**，那 3 个待验证项**对当前需求不适用**。❌ |
| ③ 「`client.tui.*` 在 TUI 模式进程内可达」 | 事实正确，但这条只说明 server 插件**够得着** TUI，不说明它**更适合**。TUI 插件本身就在 TUI 进程/线程。⚠️ 正确但不构成 server 优势 |
| ④ 「v1 不需要 TUI 插件独有的 widget/slash」 | 反过来说：v1 也不需要 server 插件独有的 messages.transform 等 hook。**需求是「启动即注册 + 显示 + socket」——这恰恰是 TUI 插件 eagr 加载 + 主线程 + Slot 的强项**。❌ |

**净结论**：D19 §二的四条理由在本次启动需求面前**全部失效或反向**。翻回 TUI 插件。

## 五、显示名字的两种姿态（按复杂度，二选一）

aibp-pi 用 `ctx.ui.setStatus("aibp", "● Alpha")` 在 footer 持久显示。opencode TUI 没有完全等价的 setStatus，但有：

### 方案 A：`api.ui.toast`（零 JSX，最简）
```ts
await api.ui.toast({ message: "aibp 已就绪 ● Bravo", variant: "info" })
```
启动时弹一次。零 JSX，与 aibp-pi 同构度最高（单文件无 JSX）。代价：不持久。

### 方案 B：`api.ui.Slot`（持久显示，需一点 JSX）
```tsx
// 在 tui(api) 里注册一个挂到 home_footer / home_prompt_right 的 SolidJS 文本
api.slots.register({
  id: "aibp-status",
  render: () => <Slot name="home_footer">● Bravo</Slot>,
})
```
持久显示在 footer/输入框右侧，最贴近 aibp-pi 的 `setStatus` 语义。代价：引入 `@opentui/solid` 的 JSX（一个极小组件），偏离「单文件纯 TS」。

**推荐**：v1 用 **方案 A（toast）**——和 aibp-pi 同构度最高、零 JSX；名字的「持久可见性」其实由注册表 `ai-Bravo.json` + microNeo 端 `:check-agent` 提供，用户需要时能查到。若日后想要持久 footer，再升方案 B。

## 六、`index.ts` 骨架（TUI 形态，相对 D19 §五的改动）

**唯一结构差异**（相对 server 形态）：

| 位置 | D19 server 形态 | TUI 形态（推荐） |
|------|----------------|------------------|
| 导出 | `export default { id, server: fn }` | `export default { id, tui: fn }` |
| 函数签名 | `async (input: PluginInput) => Hooks` | `async (api: TuiPluginApi, options, meta) => Promise<void>` |
| 拿 client | `input.client` | `api.client`（同一个类型） |
| 拿 directory | `input.directory` | `api.state.path.directory` |
| 显示名字 | （需跨线程） | `api.ui.toast(...)` 或 `api.slots.register(...)` |
| 清理 | 返回 `{ dispose }` | `api.lifecycle.onDispose(cleanup)` |
| 启动时机 | instance bootstrap gating | **App mount（主界面就绪）** |

骨架（其余函数——registryDir/normalizeNames/loadNamePool/allocateName/formatText/handleLine——**逐字复制 aibp-pi**，仅 notify→toast）：

```ts
import type { TuiPluginApi, TuiPlugin } from "@opencode-ai/plugin"
import * as net from "node:net"
// ... node:fs / path / os / url ...

const PROTOCOL = /* 同 D19，读 package.json */
const DEFAULT_NAMES_STR = "Alpha Bravo ..."

export default {
  id: "aibp-opencode",
  tui: (async (api: TuiPluginApi) => {
    const client = api.client                       // ← 与 server 形态的 input.client 同类型
    const directory = api.state.path.directory

    const names = loadNamePool(api)                  // notify→api.ui.toast
    if (!names) return

    const connectionHandler = (conn) => { /* 复制 aibp-pi 分帧 */ }

    const got = await allocateName(names, connectionHandler, (s) => { server = s })
    if (!got) { await toast(api, "⚠ 名字池已满", "warning"); return }

    name = got.name; socketPath = got.socketPath
    regFile = path.join(registryDir(), `ai-${name}.json`)
    fs.writeFileSync(regFile, /* 同 aibp-pi 注册结构 */)

    await toast(api, `aibp 已就绪 ● ${name}`, "info")    // ← 启动显示名字（方案 A）
    api.lifecycle.onDispose(cleanup)                     // ← 框架驱动清理（对称 session_shutdown）

    function handleLine(line: string) {                  // ← 递送仍用 api.client.tui.*
      let env: any
      try { env = JSON.parse(line) } catch { return }
      if (env.v !== PROTOCOL_MAJOR || env.type !== "context") return
      void onMessage(env, api)
    }
  }) as TuiPlugin,
}

async function onMessage(env: any, api: TuiPluginApi) {
  const p = env.payload
  const text = formatText(p)                              // ← 复制 aibp-pi
  try {
    if (p.message) {
      await api.client.tui.clearPrompt()
      await api.client.tui.appendPrompt({ body: { text } })
      await api.client.tui.submitPrompt()
      await toast(api, "microNeo: 已发起对话", "info")
    } else {
      await api.client.tui.clearPrompt()
      await api.client.tui.appendPrompt({ body: { text } })
      await toast(api, "📌 已放入输入框，可编辑后发送", "info")
    }
  } catch (e) { await toast(api, `⚠ 递送失败: ${(e as Error).message}`, "warning") }
}
```

**关键**：递送段（`clearPrompt+appendPrompt+submitPrompt`）和 D19 server 形态**逐字相同**——因为 `api.client` 就是 server 形态的 `input.client`。协议层、名字池、formatText、分帧、版本校验——**全部不变**。

## 七、加载方式（与 D19 §6.2 一致，无变化）

TUI 插件读的是**同一个** `config.plugin_origins`（`opencode.json` 的 `"plugin"` 数组），同一个 `PluginLoader`，同一套 resolve 逻辑。区别只是导出形态 `{tui}` vs `{server}`，loader 用 `kind:"tui"` 加载（`shared.ts:readV1Plugin` 对 tui 形态校验 `mod.default.tui`）。

- npm：`opencode plugin aibp-opencode -g`（写全局 `opencode.json`）
- 文件式：`~/.config/opencode/plugins/aibp-opencode.ts`，**必须带 `id`**（`shared.ts:resolvePluginId`：文件式无 id 抛错——与 D19-aibp-opencode-源码核对.md §2.2 一致）

`package.json` 不变（`main:./index.ts`、`aibp.protocol`、`engines.opencode`）。

## 附：决策一句话

> 用户的启动需求 = 「注册名字 + 显示名字 + 开 socket，启动即做，不碰 LLM」。
> server 插件的启动时机受 instance bootstrap gating（=「发 hi 才起」的根因）；
> **TUI 插件在 App mount 时立即加载**，且 `api.client` 与 server 插件同款、`api.lifecycle.onDispose` 对称 aibp-pi 的 session_shutdown、显示名字零跨线程。
> 所以翻回 TUI 插件——这正是调研报告最初的结论，D19 §二的反转理由在本次需求面前全部失效。
