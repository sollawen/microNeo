# D1.2 三分屏方案简化调研

> 基于 `splits.go` 源码的深度追踪，对比 D1（原始方案）和 D1.1（实施方案），确认 D1.1 中高度计算与 split 调整过度复杂，提出简化建议。

---

## 一、调研背景

D1.1 实施方案引入了 `layoutPanes()` 函数，包含：
- 精确计算 `topHeight = curLine + 1`
- 两步 `ResizeSplit`（先调 top，再调 input）
- `maxBottomReserve` 常量和空间不足时的压缩逻辑
- `origTotalHeight` 字段用于手动计算总高度

直觉上这些过于复杂。于是深入 `splits.go` 源码，逐一追踪 `vHSplit`、`applyNewSize`、`vResizeSplit`、`markResize`、`Resize` 的完整链路，确认 micro 的原生机制到底能做到什么程度。

## 二、关键源码发现

### 2.1 `vResizeSplit` 只调整相邻一对兄弟

```go
// splits.go:143
func (n *Node) vResizeSplit(i int, size int) bool {
    var c1, c2 *Node
    if i == len(n.children)-1 {
        c1, c2 = n.children[i-1], n.children[i]   // 最后一个 → 和前一个配对
    } else {
        c1, c2 = n.children[i], n.children[i+1]   // 否则和后一个配对
    }
    toth := c1.H + c2.H
    if size >= toth {
        return false        // 内置保护：不允许把兄弟挤没
    }
    c2.Y = c1.Y + size
    c1.Resize(c1.W, size)
    c2.Resize(c2.W, toth-size)   // 只动这对，其他兄弟纹丝不动
    n.markSizes()                 // 更新 propH，后续 resize 按新比例走
    n.alignSizes(n.W, n.H)
    return true
}
```

**结论**：对于三兄弟 `[top, input, bottom]`，`ResizeSplit(input)` 只调整 input 和 bottom 之间的分配，**top 完全不受影响**。这意味着不需要两步调整。

### 2.2 `applyNewSize` 在 HSplit 时做均分

```go
// splits.go:357 (vHSplit 非叶子分支)
height := (n.H - nonrh) / (numr + 1)   // 所有可 resize 子节点均分
n.applyNewSize(height, true)            // 每个子节点都设为 height
```

两次 HSplit 后，三个子节点高度确实是 `H/3`，但 `markResize()` 会更新 `propH`，之后所有的 `Resize`（如终端 resize）都按比例分配——**这是免费的**。

### 2.3 `markSizes` → `Resize` 自动维护比例

```go
// splits.go:249
func (n *Node) markSizes() {
    for _, c := range n.children {
        c.propH = float64(c.H) / float64(n.H)
        c.markSizes()
    }
}

// splits.go:209
func (n *Node) Resize(w, h int) {
    for _, c := range n.children {
        cH := int(float64(h) * c.propH)   // 按 propH 比例分配
        c.Resize(cW, cH)
    }
}
```

`ResizeSplit` 调用后会触发 `markSizes`，自动更新 `propH`。之后终端 resize 时 `Node.Resize` 会按新比例重排。**不需要手动计算任何高度**。

### 2.4 `ResizeSplit` 自带溢出保护

```go
if size >= toth {
    return false   // 拒绝把兄弟挤没
}
```

不需要 `maxBottomReserve` 常量或手动空间检查——micro 已经内置了这个保护。

## 三、D1.1 的过度设计 vs 实际需要

| D1.1 多出的设计 | 是否必要 | 原因 |
|---|---|---|
| `layoutPanes()` 函数 | ❌ 删除 | 不需要精确计算三层高度 |
| `topHeight = curLine + 1` 精确计算 | ❌ 删除 | top 不需要精确停到光标行，均分即可 |
| 两步 `ResizeSplit` | ❌ 简化为一部 | 只需调 input 高度，top/bottom 自动处理 |
| `origTotalHeight` 字段 | ❌ 删除 | 不需要手动算总高度，`node.H` 就是 |
| `maxBottomReserve` 常量 | ❌ 删除 | `ResizeSplit` 自带溢出保护 |
| `adjustInputHeight` 中读 `topNode.H` 算可用空间 | ❌ 简化 | 直接从 input 和 bottom 的总高中分配 |

## 四、简化后的 open() 流程

```go
func (ii *InlineInput) open(bp *BufPane) {
    // 0. 快照
    ii.curLine = lowestBoundary(bp)
    ii.origStart = bp.GetView().StartLine

    // 1. 第一次 HSplit: bp → [bp, inputPane]
    inputBuf := buffer.NewBufferFromString("", "", buffer.BTDefault)
    ii.inputPane = bp.HSplitIndex(inputBuf, true)

    // 2. 第二次 HSplit: inputPane → [inputPane, bottomPane]（仅当不在末行）
    if ii.curLine < bp.Buf.LinesNum()-1 {
        ii.bottomPane = ii.inputPane.HSplitIndex(bp.Buf, true)
    }
    ii.topPane = bp

    // 3. 调整 input 高度为 1 行（一次 ResizeSplit 搞定）
    inputNode := ii.inputPane.tab.GetNode(ii.inputPane.splitID)
    if inputNode != nil {
        inputNode.ResizeSplit(1)
        ii.inputPane.tab.Resize()
    }

    // 4. 设置视口
    tv := ii.topPane.GetView()
    tv.StartLine = display.SLoc{Line: 0, Row: 0}
    ii.topPane.SetView(tv)
    ii.reStamp()

    // 5. 激活焦点锁
    ii.active = true
    InlineInputInstance = ii
}
```

**对比**：删除了 `layoutPanes()`、`origTotalHeight`、两步 ResizeSplit、空间压缩逻辑。open() 从约 40 行简化到约 25 行。

## 五、简化后的 adjustInputHeight()

```go
func (ii *InlineInput) adjustInputHeight() {
    if !ii.active || ii.inputPane == nil {
        return
    }

    desired := ii.inputPane.Buf.LinesNum()
    if desired < 1 {
        desired = 1
    }
    if desired > ii.maxLines {
        desired = ii.maxLines
    }

    inputNode := ii.inputPane.tab.GetNode(ii.inputPane.splitID)
    if inputNode != nil && inputNode.H != desired {
        if inputNode.ResizeSplit(desired) {
            ii.inputPane.tab.Resize()
            ii.reStamp()
        }
    }
}
```

不需要读 `topNode.H`，不需要算 `availForInput`，不需要 `effectiveMax`。`ResizeSplit` 自带的 `size >= toth` 保护足够了。

## 六、close() 的简化建议

D1.1 的 close() 做了大量防御性检查。micro 的 `Node.Unsplit()` 已经自带了从 children 中移除、重新均分、flatten 的完整逻辑。简化为：

```go
func (ii *InlineInput) close() {
    content := ii.inputPane.Buf.String()
    if ii.onClose != nil {
        ii.onClose(content)
    }

    tab := ii.inputPane.tab

    // 移除 bottomPane
    if ii.bottomPane != nil {
        node := tab.GetNode(ii.bottomPane.splitID)
        if node != nil {
            node.Unsplit()
        }
        removePaneByPtr(tab, ii.bottomPane)
    }

    // 移除 inputPane
    node := tab.GetNode(ii.inputPane.splitID)
    if node != nil {
        node.Unsplit()
    }
    removePaneByPtr(tab, ii.inputPane)

    // 还原焦点和视口
    setActiveByPane(tab, ii.origPane)
    v := ii.origPane.GetView()
    v.StartLine = ii.origStart
    ii.origPane.SetView(v)

    ii.active = false
    InlineInputInstance = nil
}
```

防御性的 node-nil 检查保留（成本为零），但去掉了复杂的"origPane 是否仍在 tab 中"的多步验证。

## 七、结论

D1 原始方案的核心判断完全正确：

1. **"resize 完全免费"** — `Node.Resize` 按 `propH` 比例自动重排，确认成立
2. **"动态高度调一次 `ResizeSplit` 就行"** — `vResizeSplit` 只动相邻一对兄弟，确认成立
3. **"不需要手动维护坐标"** — `markSizes` + `Resize` 自动维护，确认成立

D1.1 在这些正确判断的基础上引入了不必要的复杂度（`layoutPanes`、两步调整、精确高度计算、空间压缩），应该回退到 D1 的简洁风格。**核心简化**：删除 `layoutPanes()`，所有高度调整只通过一次 `ResizeSplit` 完成。
