# microNeo 国内分发渠道调研

## 问题

当前 microNeo 只通过 GitHub Releases 分发（`install.sh` 从 `github.com/sollawen/microNeo/releases/download/...` 拉二进制）。
国内不挂 VPN 几乎无法访问 GitHub Releases 与 raw.githubusercontent.com，导致安装入口断掉。

## 当前发布链路

| 渠道 | 入口 | 国内可用 | 备注 |
|---|---|---|---|
| GitHub Releases | `install.sh` | ❌ | 官方唯一发布渠道 |
| Snapcraft (`snapcraft.yaml`) | Snap Store | ❌ | snapcraft.io 在国外，源在国外 |
| 源码 `go install` / `make build` | GitHub clone | ❌ | 同上 |

`tools/cross-compile.sh` + `.github/workflows/release.yaml` 仍然只在 GitHub 流水线产出二进制。

---

## 方案对比

### 方案 A：发到 npm（`npm i -g microneo`）

**国内可用性：**
- npm registry 主站 `registry.npmjs.org` 在国外，但**国内开发者几乎都配置了 npmmirror（淘宝镜像）**：`registry.npmmirror.com`
- 若发布时让包默认走 `npmmirror`，或文档提示用户设置镜像，安装包这一步**极快**
- **但！npm 包只是"壳"，最终还是要去下载二进制**
  - 如果二进制还是放 GitHub Releases → npm 安装时仍要拉 GitHub → 还是卡
  - 解决办法：把二进制也放到 npmmirror / 国内 OSS（见方案 C）

**实施成本：**
- 写一个 `package.json`（含 `bin: { microneo: "..." }`）
- 写一个 `install.js`（postinstall 脚本检测 OS/Arch → 下载二进制 → chmod）
- 二进制托管点可选：GitHub Releases / npmmirror / 自有 OSS
- 在 README 加一行安装说明

**优点：**
- 开发者熟悉，体验顺滑
- 可走 npmmirror，安装包过程秒级

**缺点：**
- 需要 Node.js（国内基本都有，但相比 curl 一键安装多一个前提）
- 不解决二进制下载的墙问题（除非二进制托管点换掉）
- 对非开发者用户（只用终端编辑器的）不友好

---

### 方案 B：发到 Homebrew

**国内可用性：**
- Homebrew 主站 `brew.sh` / `github.com/Homebrew/homebrew-core` 都在国外
- 装 tap（`brew tap sollawen/microNeo`）本身要访问 GitHub
- **macOS 用户**：体验好，但国内拉 Formula 慢
- **Linux 用户**：Linuxbrew 较小众，国内占比更低

**结论：基本不解决国内问题**，只是 macOS 国际用户的另一个选项。

---

### 方案 C：让 install.sh 自动识别国内环境，走国内镜像 ⭐推荐

**思路：** 不换分发渠道（仍以 GitHub 为权威），只让 `install.sh` 在检测到国内环境时，切换下载源。

**实施：**
1. 在国内 CDN/对象存储 同步发布产物：
   - **Gitee Releases**（免费、自动同步）→ `gitee.com/sollawen/microNeo/releases`
   - 或 **npmmirror binary 镜像**（npmmirror 接受任意二进制 tarball 同步）
   - 或 自建阿里云 OSS / 腾讯云 COS / 七牛（最灵活但要钱 + 域名备案）
2. `install.sh` 开头加检测逻辑：
   ```sh
   # 检测国内环境：locale / IP / 或让用户显式指定
   if [ "$MICRONEO_CN_MIRROR" = "1" ] || curl -s --max-time 2 ipinfo.io | grep -q '"country": "CN"'; then
       BASE="https://gitee.com/sollawen/microNeo/releases/download/v${VERSION}"
   else
       BASE="https://github.com/sollawen/microNeo/releases/download/v${VERSION}"
   fi
   ```
3. CI 增加一步 `sync-to-gitee`（GitHub Actions 跑完 release 后用 `git push --tags` + Gitee API 同步 release 资产）

**优点：**
- ✅ 一行安装命令不变：`curl ... | sh`，但国内用户自动走 Gitee，速度飞快
- ✅ 维护成本低（一次同步配置）
- ✅ 不引入 Node.js 等额外依赖
- ✅ 非开发者也友好
- ✅ 仍以 GitHub 为单一权威，避免多源不一致

**缺点：**
- 需要配置 Gitee 账号 + token（CI secret）
- 国内镜像可能有 1~5 分钟延迟（与 GitHub release 几乎同时）
- Gitee 政策风险（Gitee 对大文件/商业项目偶尔调整）

---

### 方案 D：Gitee 完整镜像（最省事但风险大）

直接 `gitee.com/sollawen/microNeo` 镜像整个仓库，国内用户从 Gitee clone / 下载 release。
- 优点：所有资产都搬到国内
- 缺点：双仓库维护（git remote 配两个）；Gitee 对个人项目的政策不稳定；社区/issue 分散

**建议：不推荐作为主方案，作为方案 C 的兜底即可。**

---

### 方案 E（补充）：文档 + 镜像提示

不改 install.sh，只在 README 加一段：

```bash
# 国内用户推荐：设置 npm 镜像或使用 Gitee 加速
export MICRONEO_CN_MIRROR=1
curl -fsSL https://raw.githubusercontent.com/sollawen/microNeo/master/install.sh | sh
```

配合 Gitee raw 镜像：
- `https://gitee.com/sollawen/microNeo/raw/master/install.sh` （Gitee 也支持 raw）
- 但 Gitee raw 国内可访问

可以做个 `install-cn.sh` 脚本从 Gitee 拉，但最终仍要下载 GitHub Release → 仍慢。

## 我的建议：方案 C 为主，A 为辅

| 渠道 | 角色 | 国内覆盖 |
|---|---|---|
| GitHub Releases | 权威源 | ❌（国际用户） |
| Gitee Releases | 自动同步镜像 | ✅（国内主入口） |
| install.sh 国内检测 | 自动切换 | ✅ 一行命令搞定 |
| npm（可选第二阶段） | 开发者可选 | ⚠️ 需配 npmmirror 或挂二进制到 npmmirror |
| Homebrew（可选） | macOS 友好 | ❌（不解决国内） |

### 短期可做（半天工作量）

1. 注册 Gitee 账号，建 `sollawen/microNeo` 镜像仓库
2. `.github/workflows/release.yaml` 末尾加 step：用 `git push --mirror` 或 Gitee OpenAPI 同步 tag + release assets
3. `install.sh` 加国内检测（locale/IP/env var），失败回退到 GitHub
4. README 增加一句 "国内用户" 提示

### 中期（如果想做得更全）

5. 发 npm 包：`microneo`，包内只放 `install.js` + 占位 README，二进制从 npmmirror 或 GitHub Release 拉
6. 发 Homebrew tap：`sollawen/homebrew-microneo`，Formula 指 GitHub Release URL

---

## 关键决策点

1. **是否愿意维护 Gitee 镜像？** → 决定了方案 C 是否可行
2. **是否接受 install.sh 增加几行检测逻辑？** → 决定是否国内自动切换
3. **是否要 npm 包？** → 取决于目标用户是否偏开发者
4. **是否要 Homebrew？** → 看是否在意 macOS 国际用户体验
