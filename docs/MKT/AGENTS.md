## 基本纪律

- **不要在策略文档里"加内容"**。如果瓶颈是 shipped 太少，再多策略只是"我在做营销"的错觉。先 ship，再调策略。
- **plan 文档不是 KPI 表演场**。每加一节先问：这节会让哪一篇更快 shipped？答不上来就别加。
- **写作素材必须绑真实文档**（`docs/` 下的架构/方案/总结/PRD）。不空想选题。选题池里每一行都得能指向某个真实 .md 文件。
- **修改 .md / .txt 之外的文件必须先征得用户许可**（pi PLAN 模式硬规则）。
- **涉及平台 UI/流程的 instructions，必须当下查证，不凭记忆**。SaaS 网站 UI 经常改版（dev.to 2024-2026 从"Settings 输入框"改成了"tag 页 Follow 按钮"）。凭记忆给错 instructions 比不给更坏——用户浪费 10 分钟找不到不存在的输入框。能用 chrome_inspect 查就查，不能查就明说"我不确定，请你去验证"。

## dev diary 文风（来自 #1 写作过程，2026-06-19 收敛）

这些规则是写 `dev-diary-01.md` 时挨个挨个挨打挨出来的。每一条对应一次具体错误。**不是抽象原则，是教训。**

### 结构

- **现象先行，结论后置**。开头写时代/习惯变化，让需求自己浮现。不要用 "I wanted to build X" / "我想要做 X" 开头。
- **时代是第一主体**，作者被推着走。例："vibe coding 来了之后，我发现..."——时代先动，作者第二。
- **延迟揭名**。产品名 microNeo 不在开头出现。开头主角是 Micro（被改造的素材），microNeo 是产物，§5 才揭晓。开头喊产品名 = 广告贴。
- **单篇不做 series 暗示**。读者从 dev.to/掘金单篇路径进来，不知道是 series。cliffhanger、"第一篇"序数词、"欲知后事如何" 都不要。series 认知靠读者日后读过几篇自然形成。
- **Closure 优先于 cliffhanger**。第一篇尤其。读者要的是"这篇文章值了"的满足感，不是"文章没写完"的吊胃口。
- **一次只预告下一篇，不押多张牌**。不预告 4 篇。过度承诺 + 万一某篇写不动就全盘崩。
- **术语押韵**。开头术语（line editor）↔ 结尾术语（rich-format editor）形成对仗。这是叙事的语言锚，不能丢。

### 句子

- **短句、断句、节奏**。"忍了几个月。我忍不了了。" 不膨胀成 "After several months of frustration..."。中文短，英文也要短。
- **诚实写失败/撞墙**。HN 最吃这个。"一开始我想...然后撞上一堵墙" 比 "我设计了优雅的架构" 强 10 倍。
- **保留作者口语感**。工程师冷幽默 > 学术腔。"dumb and single-purpose" / "drawn out of thin air" 这类词保留，不要改成 "single-responsibility" / "synthesized"。

### 中文 → 英文翻译（最高优先级）

- **中文是 source of truth**。英文里任何中文没有的词都是嫌疑——可能是擅自添加。
- **不擅自加营销词**。"independent fork"、产品名、卖点词——中文不提，英文绝不提。如果觉得"加上更好"，**先问用户**，不擅自加。
- **不逐字译中文虚词**。"在 X 的基础上" → "based on X"，**不是** "taking X"。"出来" / "一下" / "起来" 这类虚词没有逐字英文，必须按功能等价译。
- **不把作者口语/人格化译成产品术语**。"新的 micro" → "the new baby"（保留亲昵），**不是** "new version of Micro"（PM 腔）。揭名句、关键转折句这种情感权重高的位置尤其要守住。
- **格式也属擅自添加**。中文用引号的地方英文用引号，中文不昭不加粗。格式是中文的一部分，不是翻译时可以重新设计的。
- **保留中文的句法节奏**。中文短句翻英文不能因为英文有从句就拉长。

### 失败模板（避免）

- ❌ 议论文式立论（"为什么 X 这么难"）—— 要铺垫、慢、容易卡
- ❌ 开头就喊产品名 —— 等于广告，HN 一秒折叠
- ❌ 每篇都预告 N 篇 —— 过度承诺
- ❌ cliffhanger 当留人钩子 —— 在单篇发现路径下是负面体验
- ❌ 工程师 spec 腔（"所以 / 那意味着 / 因此"）—— 把叙事翻成论文
- ❌ PM 腔擅自"优化"作者决定 —— "这个词有市场价值，加上吧" 是最危险的"优化"

## 渠道策略现状（截至 2026-06-19）

- **HN / Reddit 主帖被卡**（账号门槛）。后台攒 karma，不阻塞主路径。
- **主战场是无门槛渠道**：dev.to（英文长文）/ Twitter / V2EX / 掘金。
- **dev diary = karma 弹药库**。每篇可拆解成 HN/Reddit 高质量评论的现成模板。
- **install.sh 不动**。走 binary 下载，装机数看 GitHub Releases API（~41 终身）。

详见 `MKT-plan.md`。本文件只记录写作约束，不重复渠道数据。
