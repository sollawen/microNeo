# 文件管理器独立成包 —— 设计方案

**动机**：fileManager 后续会扩展删除 / 新建 / 复制 / 移动 / 重命名，长成完整的文件导航器。在那之前先定清楚它的代码归属与对外契约，避免子系统膨胀时把 `internal/action/` 撑成杂物间，也避免事后再做伤筋动骨的搬迁。

---

## 1. 目标

- 把fileManager，fileSelector，file_git，这些程序，全部移到 internal/filemanager/目录下，独立成包
- 把 fileManager 当成一个**「数据进、数据出」的模态组件**——它的「函数体」恰好是一段交互式模态会话。
- 所有有状态资产（BufPane / buffer / tab）归调用者（action），fileManager 只拥有自己那一次短暂的会话。
---

## 2. 几个原则

1. **自洽模态**：调用者启动它，它独占整个会话的交互（esc / ctrl-q / 导航 / 挑选 / 未来的删改都在内部处理），它自己关掉自己。
2. **输入侧走纯参数**：启动所需的一切由调用者以**纯值**传进 `Open()`
	- 当前bufPane的rect、当前文件的全路径，或者是当前目录+noName空文件
	- **不传 `*BufPane`、不要它满足任何接口、不做全局注入。**
3. **输出侧走单一结果**：关闭时回传**一个结果结构**，承载整次会话的产出
	- 关闭时的cwd, 
	- selected filename(如果是按回车退出的话，就有filename) 
	- selected filename 这个文件的状态，分成 nomal, deleted, rename之类的。这个需要讨论
	- 关闭的原因，分成 esc, enter, resize, quit(用户按ctrl-q时)
	- 调用者拿到结果后后自己解释、并做出针对性的处理，比如open file, quit 之类的
4. **零状态、零反向依赖**：fileManager 无包级可变状态、零 `action` 引用，用纯值 + stub Host 即可单测。
5. fileManager提供几个公共接口给调用者。调用者一定是某个bufPane
	- Open()
	- isOpen()，供调用者判断是不是open状态
	- 每帧标准接口，这两个接口的命名要和所有的pane一致
		- HandleEvent(event), 由调用者把键盘鼠标事件通过这个接口传给fileManager
		- Display()，由调用者在每一帧render的时候，调用这个接口来绘制fileManager
	- 其它需要的接口，需要讨论
6. fileManager、fileSelector不再使用和依赖floatFrame，和floatFrame断绝关系

# 定稿方案（讨论收敛后）

上面是讨论起点。下面是经过代码核对 + 双方辩论后收敛出的可执行方案。两件事分开写：**finder** 是独立成包的模态组件本身；**fileops** 是留在 action 的 owner 侧接线层。它们是两回事情，契约各管各的。

## 命名决定

| 现在 | 改成 | 理由 |
|---|---|---|
| `FileSelector`（类型） | `FileManager` | 产品定位文档第一句"fileManager = 终端里的 Finder"；FileSelector 只是它 v1 的 browse-only 阶段名，给 delete/move/copy/rename 留位 |
| 新包 `internal/filemanager/` | `internal/finder/` | 贴产品词汇；避免 `filemanager.FileManager` 的重复。最终形态 `finder.FileManager`（或 `finder.Manager`/`finder.Session`，类型名落地前再定） |
| `filemanager.go`（action 内） | `fileops.go` | 这文件是 owner 侧胶水，不是 finder 的一部分；改名消除和新包名的撞名，也避免误导后人以为"fileManager 的实现在这" |

搬移范围：`fileselector.go` + `fileselector_git.go` + 对应测试 → `internal/finder/`。`filemanager.go`（→`fileops.go`）**留在 action**，因为它调 `pane.OpenCmd`/`pane.Quit`、读 `pane.Buf`，搬走会形成反向依赖。

---

## A. finder 包（FileManager 的对外契约）

finder 是一个**数据进、数据出的自洽模态组件**。它独占一次会话的交互，自己关掉自己，关掉时如实上报发生了什么。**它不知道、也不关心 owner 为什么开它**——这是本章最重要的原则。

### A.1 边界与归属

- finder 拥有：自己那一次会话的状态（当前目录、条目、光标、git 标志、显示模式）
- finder 不拥有：BufPane、buffer、tab、任何进程级可变状态
- finder 不关心调用者是谁，也不知道调用者的一切
- finder 不引用：`action` 包、`*BufPane`、`TheFloatFrame`、`InfoBar`
- 调用者永远是某个 BufPane（通过 fileops.go 中介）

### A.2 输入侧：纯值

```go
func (fm *FileManager) Open(
    rect     Rect,       // owner pane 的绝对屏幕矩形(对角两点坐标)
    cwd      string,     // 起始目录;noName 时由 owner 自决定(见 B.3)
    file     string,     // 当前文件名(不含目录);noName 传 ""
    isQuit   bool,       // 是否为 quit 触发(Ctrl-q 传 true);finder 不读,close 时原样回填进 Result.IsQuit
    onClose  func(Result),
)
```

**不传 `*BufPane`、不要 owner 满足任何接口、不做全局注入、不传 Host。** finder 直接 import `screen` + `config`，画自己时调 `screen.Screen.SetContent` / `config.GetColor`——和 floatFrame、NotePane 画自己的方式完全一致。screen 和 config 都是比 action 更底层的包，`finder → screen` / `finder → config` 这两条 import 边合法、不循环。

finder 唯一够不到的 action 资产是 `InfoBar.Message`（现在拒开时由 `computeLayout` 直接调，见 `fileselector.go:365/369`）。搬包后这个调用挪到 fileops.go：finder 照常通过 `onClose` 上报 `TooSmall`，fileops 在回调里判到 `TooSmall` 就自己调 InfoBar 报 "pane too narrow/short"——提示本就是讲 owner 的 pane 太窄，由 owner 自己说最自然。**Open 不需要返回值**：size 拒开和 Esc/Quit 一样走 callback，对调用方完全透明（每次 Open 必然对应一次 onClose，无论 finder 有没有真显示出来）。

当前 `FileSelector` 对 `*BufPane` 的全部依赖只有两处只读：`pane.Buf.AbsPath`（取当前文件名做光标定位 + 推起始目录）和 `pane.BWindow.GetView()`（取 rect 算 layout）——拆成 `cwd`（起始目录）+ `file`（光标定位）两个纯值传进来。noName pane 的 cwd 由 owner pane 用 `OpenFinder` 内的 `AbsPath=="" → cwd` fallback 自动派生（见 B.3），finder 无感。

### A.3 输出侧：单一结果结构

```go
type Result struct {
    Reason CloseReason   // 会话结束原因(五种,见下)
    Cwd    string        // 关闭时所在目录(始终填,不带文件名)
    File   string        // Reason==Picked 时:选中的文件名(不含路径)
    IsQuit bool          // 原样回显 Open 入参 isQuit;finder 不读,仅让 onClose 成为 Result 的纯函数
}

// CloseReason:扁平枚举,Picked 与其余四个同级(不再分 Kind)
//   Picked   — Enter 选中文件
//   Esc      — 用户按 Esc
//   Quit     — 用户按 Ctrl-q(或 welcome 态的 q)
//   Resize   — 运行中终端 resize,已显示后被打断
//   TooSmall — 打开时窗口过小,finder 从未显示
```

字段说明：

- `Reason`：finder 如实上报"会话怎么结束的"，五种取一。**没有 Kind 分级了**——Picked 和 Esc/Quit/Resize/TooSmall 是同级的原因。原来 Kind=Picked|Closed 的两级分类是过度结构，已坍。
- `Cwd`：**始终填 finder 关闭时的 `state.currentDir`**（纯目录、不带文件名）。哪怕用户 Esc 取消，他在会话里可能已经导航到了别处——owner 侧的 shell 同步（见 B.8）需要这个值。这是这次讨论新增的字段，原方案里没有，理由见 B.8。
- `File`：仅 `Reason==Picked` 时填——选中的文件名（**不含路径**，目录在 `Cwd` 里）。其余四种为空字符串。owner 用 `filepath.Join(r.Cwd, r.File)` 还原绝对路径交给 `OpenCmd`。
- `IsQuit`：原样回显 Open 入参 `isQuit`，finder 全程不消费它。存在的唯一理由：让 owner 的 onClose 成为 Result 的纯函数——不用闭包记忆、不用 owner 侧状态，TooSmall 分支直接 `if r.IsQuit` 即可（见 B.5/B.6）。finder 构造 Result 集中在一处（一个 `close(reason)`），IsQuit 在那里统一填，5 条路径只换 Reason，不会漏填。
- **预留 `FileStatus`（normal/deleted/renamed）字段位**：等 delete/rename 功能落地再加。现阶段不建，避免空头结构。结构前向兼容。

### A.4 公共 API

finder 提供给 owner（实为 fileops.go）的接口：

- `Open(...)` — 启动一次会话
- `IsOpen() bool` — owner 判断 finder 是否在开
- `HandleEvent(event)` — owner 每帧把键盘/鼠标事件转发进来（命名对齐全 pane 约定）
- `Display()` — owner 每帧 render 时调用绘制 finder

后两个接口的命名与所有 pane 一致（BufPane / NotePane 都有同名方法），让 finder 在 owner 眼里就是一个"临时 pane"。

**每帧由 owner 转发调用——这是 fileops.go 的硬契约**：

```go
func (h *BufPane) HandleEvent(event tcell.Event) {
    // 位置约束：modal 屏蔽必须在 reload check 之前，否则 InfoBar.YNPrompt 抢焦点/Esc
    if h.finder.IsOpen() {
        h.finder.HandleEvent(event)
        return
    }

    if h.Buf.ExternallyModified() && !h.Buf.ReloadDisabled {
        reload := h.getReloadSetting()
        if reload == "prompt" {
            InfoBar.YNPrompt("The file on disk has changed. Reload file? (y,n,esc)", ...)
        } else if reload == "auto" {
            h.ReOpen()
        } else if reload == "disabled" {
            h.Buf.DisableReload()
        }
    }

    // ... owner 自己原本的事件分发 (switch event.(type)) ...
}
```

- `owner.HandleEvent(event)` 头部第一件事判 `finder.IsOpen()`：开了就把 event 转给 `finder.HandleEvent(event)`、立刻 `return`；没开就跑 owner 自己的事件处理。finder 内部怎么消化（吃 / 丢 / 内部副作用）是它自己的事——**modal 期间 owner 整段按键 / 鼠标 / 分屏调整处理全部不跑**，不存在"finder 没消化、owner 接着处理"的二阶段分流。屏蔽点是 owner.HandleEvent 头部的这个 `if return`，finder.HandleEvent 是被动接收方——这条 modal 屏蔽就是 §B.3「onClose 也不防御 nil」论据里"键路由全被屏蔽"在升格后的等价表述（升格前是 FloatFrame 屏蔽，升格后是 owner 头部 return 屏蔽，不变式不变）。
- `owner.Display()` 末尾追加：`if finder.IsOpen() { finder.Display() }`。画在 owner pane rect 内，finder 是 pane-local 钉左上角，**不会被兄弟 pane / divider / InfoBar / NotePane 的 Display() 覆盖**——它们的 Display 画的是自己的 rect，rect 不同就没有覆盖（详见 §D.1 收敛）。

finder 自己不挂 timer、不注册主循环回调、不订阅 resize——它就是一个被 owner 调用的对象，和 BufPane / NotePane 同构。open 由 owner 调 Open 发起、finder 内部触发 close，整个生命周期闭环在「owner ↔ finder」两边之间，主循环不介入。

**没有 `Close()`，也没有 resize 通知接口。** owner 不主动关 finder——会话的终结（用户 Enter/Esc/Quit、运行中 resize）全由 finder 内部判定后回调 onClose，owner 只负责 Open 和收结果。其中 resize 根本不是单独的通知：它是 `tcell.EventResize`，和按键走同一通道进 `HandleEvent`，finder 在里面拦截后自己 Close。

### A.5 与 floatFrame 断绝（硬约束 + 设计选择）

现状 finder 经由 `TheFloatFrame` 中转：把 `display`/`handleEvent` 作为函数值塞进 `FloatOpenSpec`，由 FloatFrame 代为画框 + 路由事件。finder 搬进自己的包之后**必须停止**用 floatFrame——这既是 Go 的硬约束，也是我们的设计选择，两条理由各自独立成立。先看硬约束：

```
fileops.go (action 包) → import finder（调 Open）
finder                → import action（用 TheFloatFrame）  ← 这条成环，编译报错
```

floatFrame 在 action 包里，finder 一搬走就物理上够不到它——上面的图就是这条：成环编译报错，非断不可。但**就算没有这条 import 限制**，设计上我们也要让 finder 升格成自洽的 pane 级 overlay，自己画边框、自己接管事件路由（像 floatFrame / NotePane 一样直接 `screen.Screen.SetContent`），不再寄生在一个住在 action 包里的共享 host 上。"未来要不要嵌套 input" 是另一个动机，目前未触发，跟这次断绝无关。

finder 升格为 pane 级 overlay，自己实现 `Display()`/`HandleEvent()`，**画法照搬 floatFrame**：直接 `screen.Screen.SetContent` 画边框/内容/clear、HideCursor、resize 拦截。这三块(边框+title 嵌入 + resize 拦截 + 尺寸预检)约 60–80 行，全是机械代码，从 `floatframe.go` 搬进 finder 即可，无新设计。边框代码就是这么直白：4 个角 `┌┐└┘` + 上下 `─` + 左右 `│` + 上边框嵌 `──title──...─`，全是 dumb SetContent。

FloatFrame 的去留与 finder 无关：FloatFrame 继续服务 SelectPane 等简单一次性 widget（N0a/N0b 的 per-owner 演进照常进行）。两者对 FloatFrame 的处置不同，必须分开，不要把 N0 和 N0b 搅成一坨。

### A.6 intent-agnostic 原则（关键）

**finder 的行为永远不因 owner 为什么开它而改变。** owner 可能是为了 browse（Ctrl-o）、为了 quit（Ctrl-q）、因为是 noName 新 pane（birth）——这些 intent 全部住在 owner 侧（fileops.go，见 B.2/B.4）。

finder 对 intent 有一条硬线：**可以原样转运，绝不据此分支。** Open 收一个 `isQuit` 入参，finder 全程不读它做任何决定，只在 close 时原样填进 `Result.IsQuit` 回传——纯转运。finder 自己的显示、按键响应、关闭判定一概与 isQuit 无关。

finder 的职责只有一条：**如实上报会话怎么结束的**（Result.Reason），外加把 owner 寄存的 isQuit 原样带回。怎么解释 Reason、要不要据此 Quit/OpenCmd，全是 owner 的家事。

这条原则要求的是"finder 的**行为**对'为什么被调'完全透明"——是行为透明，不是"连一个 bool 都不许过手"。让 isQuit 以数据形式进、以数据形式出，正是兑现第 1 节"数据进、数据出"的北极星：owner 的 onClose 成为一个对 Result 的纯函数，不靠闭包记忆、不靠 owner 侧可变状态。代价是 finder 多转运一个它自己用不上的 bit，收益是 owner 侧零状态。这笔交易值得。

**硬约束**：finder 内部禁止出现任何 `if isQuit` / `switch isQuit`。一旦出现，说明 finder 真的要按 intent 表现不同——那时应当升级成正式的 mode 参数、重新设计，而不是偷偷分支。

### A.7 单测边界（诚实评估）

- **已可纯函数单测**：git 解析（`parsePorcelain`/`getGitStatus`，现已有测试）、字符串工具（`truncateNameKeepExt`/`humanSize`/`fitMeta`/`truncateLeftPath`，现已有测试）。
- **display / handleEvent 不做 stub 单测**：它们直接调 `screen.Screen.SetContent` / `config.GetColor` 这些包级单例（和 floatFrame、NotePane 一样），**没有 Host 抽象**，出问题靠手测。这是有意取舍——finder 里真正难、易回归的逻辑（git 解析、字符串截断）都是纯函数、已覆盖；为 display 加抽象层的成本远大于手测收益，等真觉得痛了再说。

## B. fileops.go（owner 侧胶水与 onClose）

fileops.go 是 BufPane 和 finder 之间的接线层。它干三件事：**接 binding target 的委托调 finder（browse / quit）**、**接 BufPane 构造完成的自动触发（birth）**、**在 finder 关闭后用 onClose 解释 result 并执行 owner 侧动作**。它留在 `internal/action/`，因为它要调 `pane.OpenCmd`/`pane.Quit`、读 `pane.Buf`、写 `lastWorkingDir`——这些都是 BufPane 的家事，搬进 finder 会形成反向依赖。

**fileops.go 是 finder 包唯一的调用方**——所有调用（无论 binding 触发的 browse / quit、还是 BufPane 自动的 birth）都通过 fileops.go 的 `OpenFinder` 进 finder。意图按触发方式分布：browse / quit 的意图住在 binding target 函数体里（哪个 keymap 就走哪个 intent）；birth 干脆不归 binding 管——它是 `BufPane` 这一类型的语义行为（"noName pane 出生就开 finder"），与 keymap / 命令 / spawn wrapper 全解耦。

### B.1 角色

```
binding 事件                                BufPane 构造完成
(Ctrl-o / Ctrl-q / :file)                  (tab.Resize → pane.Resize → finishInitialize)
       ↓                                              ↓
binding target 函数                         if buf.AbsPath == "":
(FileCmd / QuitNeo)                         3 行 hook 自动调 (见 B.4)
       ↓                                              ↓
h.OpenFinder(bool)                           h.OpenFinder(false)
       ↓                                              ↓
       └────────────────┬─────────────────────────────┘
                        ↓
              fileops.go: OpenFinder
        (唯一对外接口，算 rect/cwd/file，调 finder.Open)
                        ↓
                  finder 会话（自洽模态）
                        ↓
                  finder 关闭，回传 Result
                        ↓
              fileops.go: 共享 onClose
                (按 Result.Reason 选动作并执行)
                        ↓
                    BufPane 动作
              (OpenCmd / Quit / 写 cwd tmp)
```

fileops.go 不持有 finder 的会话状态。它只在两个时刻起作用——"开"和"收"。"开"时一份代码搞定所有触发方式（无论来自 binding 还是创建钩子）；"收"时一份共享 onClose 解释 Result。

#### 每帧路由：两条转发线全在 owner 上

owner 持有一个 finder 引用（fileops.go 新增 `finder *finder.FileManager` 字段，由 `OpenFinder` 注入）。事件流和渲染流的转发点都钉在 owner 这一个对象上，不进主循环、不跨对象：

```
        主循环                            owner (BufPane)
          |                                     |
          |  event                              |
          |   ↓                                 |
          |  owner.HandleEvent(e)               |
          |      ├─ if finder.IsOpen():         |
          |      │     finder.HandleEvent(e)    |
          |      │     return  ← modal 整段屏蔽  |
          |      └─ owner 自己的事件处理(finder 关时) |
          |                                     |
          |  ...                                  |
          |                                     |
          |  owner.Display()                    |
          |      ├─ ... 画 pane 内容 ...         |
          |      └─ if finder.IsOpen():         |
          |            finder.Display()         |
          |                                     |
          |  兄弟 pane.Display / divider / InfoBar.Display / NotePane.Display
          |  ← 全不覆盖 finder 区域（pane-local,各自 rect 不同）
```

owner.HandleEvent 头部的「接管 return」是 modal 语义的关键——finder 会话期间,owner 自己其余的键位 / 鼠标 / 分屏调整处理全部不跑。owner.Display 末尾的追加是同构的 modal overlay,画在 owner pane rect 内,不被兄弟 pane 覆盖(详见 §D.1 收敛)。

**两端点都在 owner,不发散**——这条设计直接复用 micro 原生 BufPane 的事件/渲染通道,主循环 0 改动、tab / tabList 0 改动、InfoBar / NotePane 0 改动。N1a ADR-7 那种「事件路由走主循环 resize 分支、渲染走主循环 Render 末尾」的方案是给 SelectPane 越界用的,finder 不越界,不走那条路——两套策略并存、对应两种 widget 的视觉属性(见 §D.1)。

### B.2 三个 trigger：两条入口线

三个 trigger 现在分布在两条物理入口线，各走各的、最终汇入同一个 `h.OpenFinder(bool)`：

| trigger | 入口 | 触发条件 | 行为 |
|---|---|---|---|
| **browse** | binding target 函数 | Ctrl-o / `:file` | `FileCmd` → `h.OpenFinder(false)` |
| **quit** | binding target 函数 | Ctrl-q on noName pane | `QuitNeo` 判 `h.isNoName` → `h.OpenFinder(true)`；file-born pane 直接 `h.Quit()`，不进 finder |
| **birth** | `BufPane.finishInitialize` | noName pane（`Buf.AbsPath==""`）刚构造完成 | 见 B.4 的 3 行 hook → `h.OpenFinder(false)` |

**binding 侧完全不知道 birth 的存在**。6 个 spawn wrapper（`neoAddTabAction` / `neoVSplitAction` / `neoHSplitAction` / `neoNewTabCmd` / `neoVSplitCmd` / `neoHSplitCmd`）连同 `cmd/micro/micro.go` 启动段的全套 `OpenBirthSelector` 调用——加起来 7 处——在 `finishInitialize` 那一行全自动覆盖。

**binding 只剩两个用途**：`FileCmd`（browse 出口）和 `QuitNeo`（quit 出口 + file-born 直通）。`OpenFinder` 的 7 个原 call site 塌成 1 个 method、2 个 binding 调用 + 1 个 finishInitialize 调用。

**唯一可携带的语义差异仍是 `isQuit`**：browse / birth 传 `false`，quit 传 `true`。`OpenFinder` 是 finder 包唯一的调用方——bindings 不直接 import / 调 finder。

`isQuit` 经 finder 原样回填进 `Result.IsQuit`（finder 全程不读它，见 A.6），**只在 onClose 的 TooSmall 分支被读到**（见 B.6），其余分支按 reason 走、不碰它。现状三个 `openXxxSelector` 各写一遍重复回调的"乱"，塌成"一个共享 `OpenFinder` + 一个共享 onClose"。

### B.3 OpenFinder：唯一对外接口

```go
// fileops.go (action 包) —— BufPane 与 finder 之间的接线层
// 三个 caller(FileCmd / QuitNeo / finishInitialize hook)全部经 OpenFinder 进 finder;
// finder 关闭后经 onFinderClose 解释 Result 并执行 owner 侧动作

// OpenFinder —— 启动 finder 会话(三个 caller 的唯一对外接口)
// caller 契约: h.Buf != nil 且 h.BWindow 已拿到真尺寸(见 B.4 hook 时序图)
func (h *BufPane) OpenFinder(isQuit bool) {
    var dir string
    file := h.Buf.AbsPath
    if file != "" {
        dir = filepath.Dir(file)
    } else {
        dir, _ = os.Getwd() // noName / 启动段 fallback 到 cwd
    }
    finder.Open(h.BWindow.GetView(), dir, file, isQuit, h.onFinderClose)
}

// onFinderClose —— 共享 onClose:按 Result.Reason 分发(见 B.6 表格)
func (h *BufPane) onFinderClose(r finder.Result) {
    switch r.Reason {
    case finder.Picked:
        if filepath.Join(r.Cwd, r.File) == h.Buf.AbsPath {
            return // 同文件 no-op
        }
        h.OpenCmd([]string{filepath.Join(r.Cwd, r.File)})
    case finder.Quit:
        h.doQuit(r)
    case finder.TooSmall:
        if r.IsQuit {
            h.doQuit(r) // quit 入口窄窗也要能逃出去
        }
    case finder.Resize, finder.Esc:
        // no-op
    }
}

// doQuit —— 先暂存 r.Cwd 再调原生 h.Quit();顺序不能反(Quit 后 pane 没了)
func (h *BufPane) doQuit(r finder.Result) {
    LastFinderCwd = r.Cwd
    h.Quit()
}

// LastFinderCwd —— 包级暂存变量(导出);doQuit 写 / lastWorkingDir 在 exit 时读
var LastFinderCwd string
```

**为什么 onFinderClose 是 method，不是包级函数**：回调体要调 `h.OpenCmd` / `h.Quit`，而三个 caller 的 `h` 是不同的 `*BufPane` 实例（browse/quit 是焦点 pane，birth 是新 pane）。包级无状态函数拿不到 `h`——照写编不过（`h` 在包级函数作用域里 undefined）。把 `onFinderClose` 做成 `(*BufPane)` 的 method，`OpenFinder` 里传 `h.onFinderClose`（Go 的绑定方法值，bound method value），当次的 `h` 就被烤进函数值里、随回调带走。共享的是 method 的代码（定义一次），不是某个包级 var；回调值本身是 per-call 的，每次 `OpenFinder` 绑当次的 `h`——这正是它该有的样子，因为 `h` 本来就 per-call。别把它"优化"回包级函数，会重新引入 `h` 未定义；同理 `doQuit` 也是 method（它内部调 `h.Quit()`）。

**为什么 onClose 也不防御 nil**：现状 `filemanager.go` 在三处 onClose 闭包里都写了 `if pane.Buf == nil { return }`（R7 防御，see F1-架构设计方案.md §3.2/§14）。但全库扫过，`h.Buf = nil` 没有任何赋值点——`BufPane.Close()` 只调 `Buf.Close()` 不置 nil、`Buffer.Close()` 调 `Fini()` 清 OpenBuffers 但不清零 `b`、`Tab.RemovePane` 把 `t.Panes[i] = nil` 但 `pane.Buf` 仍指向已 Fini 的 Buffer（不是 nil）。加上 finder 打开后 `FloatFrame.HandleEvent` 拦截所有键事件，主循环/binding 全被屏蔽，外部没有任何路径在 finder 会话期间改 pane.Buf。这层 R7 防御**触发条件为空集**——是历史包袱/防御未来代码，不解决任何现实 bug。

新设计里把 R7 防御整段塌掉，连同现状 `filemanager.go:29/67/85/104` 的 nil 检查全部删除——OpenFinder 入口一处都不防。注释只在 `OpenFinder` 函数体上保留"caller 契约 + h.Buf 永不为 nil 是事实"一句，提醒后人不要把 R7 加回来。

若未来有真路径引入 `h.Buf = nil`（比如 Lua plugin、async 回调、跨 pane 引用），届时再补防御——但那是新场景、新代码、新 R8，不是历史 R7 的延续。

**入参 `isQuit`**：三 trigger 唯一差异。browse / birth 传 `false`，quit 传 `true`。`finder` 全程不消费它（见 A.6），只在 close 时原样回填进 `Result.IsQuit`，供 onClose 的 TooSmall 分支读（见 B.6）。

**入参 `h`（method receiver）**：

- browse / quit：`h` 是当前焦点 pane（`FileCmd` / `QuitNeo` 的 receiver）。
- birth：`h` 是**正在 finishInitialize 的新 pane**——hook（见 B.4）把它自己当 receiver 调 OpenFinder。

**`dir` / `file` 的派生**：

- `AbsPath != ""`：`dir = filepath.Dir(AbsPath)`、`file = AbsPath`——browse / quit（pane 有文件）的标准行为。
- `AbsPath == ""`：`dir = cwd`、`file = ""`——birth（新 pane 还没文件）/ 启动段（micro 刚起来第一个 pane 是 noName）的 fallback。

`startDirOf` 和 `birthDir` 这两个原 helper 都删了，逻辑全部吸进这一处。

**行为差异（简化代价）**：现状 `OpenBirthSelector(pane, dir)` 显式收 spawn 时捕获的父目录，HSplit 在 `foo.md` 旁边时新 pane 起始目录 = `dir(foo.md)`。新方案 birth 不传 dir，新 pane 没文件 → fallback cwd。**仅当 micro 的 cwd ≠ dir(当前文件) 时才有差**——典型场景：`:cd ~/other` 后再 HSplit，新 pane finder 起始目录会从 `dir(当前文件)` 退到 cwd。绝大多数使用下 cwd == dir(当前文件)，用户进 finder 也能跳转，所以是简化换来的微小回归。

`file` 在 browse / quit 场景下被 finder 用作光标定位（locate 到当前文件那一行），birth / 启动态 `file = ""` 时 finder 没有可定位目标、行为正确。

**入口不写 nil 防御、onClose 也不写**：`h.Buf` 字段类型是 `*buffer.Buffer`（nullable，bufpane.go:210），但整库没有任何代码路径会让 `h.Buf` 变 nil。OpenFinder 入口：caller 链三层全覆盖（`FileCmd` / `QuitNeo` 走焦点 pane 必非 nil；`finishInitialize` hook 在 if 分支内调已先验）。onClose 闭包：闭包持有的 pane 引用是 open 时绑定的，会话期间外部键路由全被 FloatFrame 拦截，没人能从外部改 `pane.Buf`；全库 `h.Buf = nil` 也不存在赋值点。两处都直接 `h.Buf.AbsPath` / `h.OpenCmd(...)`，**契约写进注释**，不写防御性 if。现状 `filemanager.go:29/67/85/104` 那些 nil 检查是历史防御性编程（R7），触发条件为空集、跟未来真场景（如 Lua plugin / async）押错宝，新设计里整段塌掉。

回调是 `(*BufPane).onFinderClose` 这个 method——共享的是 method 代码（定义一次），不是包级 var。`OpenFinder` 里传 `h.onFinderClose`（Go 绑定方法值），当次的 `h` 随回调带走。2 个 binding 调用 + 1 个 hook 调用共三个 caller，每次各绑各的 `h`（browse/quit 绑焦点 pane，birth 绑新 pane）。详见上面「为什么 onFinderClose 是 method」。

### B.4 birth trigger：`finishInitialize` 自动 hook（与 binding 解耦）

**触发点**：`bufpane.go` 的 `BufPane.finishInitialize()`（bufpane.go:297）末尾，3 行：

```go
func (h *BufPane) finishInitialize() {
    h.initialRelocate()
    h.initialized = true

    err := config.RunPluginFn("onBufPaneOpen", luar.New(ulua.L, h))
    if err != nil {
        screen.TermMessage(err)
    }

    // MicroNeo: birth trigger —— noName 三条件(沿用 isNoNameBuf 语义)全满足
    // 时置 isNoName=true 并开 finder;否则显式置 false。三条件确保 Help/Raw 等
    // 特殊类型和 Lua 预填内容不会误触发。isNoName 是 sticky 出身标记,一次性写入。
    if h.Buf != nil
        && h.Buf.AbsPath == ""
        && h.Buf.Type == buffer.BTDefault
        && h.Buf.Size() == 0 {
        h.isNoName = true
        h.OpenFinder(false)
    } else {
        h.isNoName = false
    }
}
```

#### 为什么是这个时点

pane 何时拿到真实 rect？`newBufPane` 末尾 BWindow 是 `(0, 0, 0, 0)` 占位（`display.NewBufWindow(0,0,0,0,buf)`，bufwindow.go:55），真实尺寸是 split tree 算出来的。完整时序：

```
display.NewBufWindow(0, 0, 0, 0, buf)   ← 构造时显式占位
       ↓
newBufPane(buf, win, tab)               ← h.BWindow 还是 (0,0,0,0)
       ↓ 进 split tree
h.tab.Resize() {                        ← tab.go:380 遍历
    for pane in t.Panes:
        n := t.GetNode(p.ID())
        pv := p.GetView()
        pv.X = n.X+offset, pv.Y = n.Y     ← 真坐标
        p.SetView(pv)
        p.Resize(n.W-offset, n.H)         ← 真尺寸
              ↓
              h.BWindow.Resize(w, h)      ← BufWindow.X/Y/W/H 全是真值
              if !h.initialized {         ← 守卫：只在首次 Resize 触发
                  h.finishInitialize()
              }
}
                                  ↑
                              hook 加在这里
```

这一行满足"pane 真出生"的所有条件：

- ✓ `h.BWindow` 的 X / Y / W / H 全是真值（已由 `h.BWindow.Resize(w, h)` 设置）
- ✓ `h.initialRelocate()` 已调用——cursor 校准到位
- ✓ `if !h.initialized` 守卫天然保证**只在首次触发**，pane 后续 Resize 不会重复触发 OpenFinder
- ✓ 与既有 plugin 回调 `onBufPaneOpen` **平级**——Lua plugin 现有的"pane ready"语义点直接复用

#### 为什么不是其他候选点

| 候选 | 问题 |
|---|---|
| `newBufPane` 末尾 | BWindow 还是 `(0,0,0,0)` 占位，finder 要么被自身 preCheck 拒开、要么画错位置 |
| `Tab.AddPane` 末尾 | `tab.Resize()` 还没跑，pane 还没拿到真尺寸；且所有 caller 都得自己紧接着调 Resize——漏抽象 |
| `BufPane.Resize` 末尾 | 缩进进 Resize 函数体字段耦合更紧，且无独立守卫（每次 resize 都会进来）；放 finishInitialize 正利用 `!h.initialized` 已有守卫 |
| `BufPane.Relocate` 末尾 | 触发频率过高（initialRelocate + 每次 resize/ScrollAdjust 都调），不是"首次 birth"语义 |

`finishInitialize` 是当前 micro 公认的"pane 已 ready"语义事件（`onBufPaneOpen` 就是在这一行 fire 的），OpenFinder hook 借势接管，不引入新概念。

#### 不需要外部门控：三条件在 hook 内联判断

file-born pane（`:tab foo.md` / `:vsplit foo.md` / `:hsplit foo.md`）走的也是同一 `finishInitialize`，但它们的 buffer 通过 `NewBufferFromFile` 构建、`AbsPath` 已有路径，**天然不满足 hook 的 noName 三条件**（`AbsPath==""` 这条首先就不满足）——`isNoNameBuf` helper 整函数可删，原 helper 的"门控"职责塌进 hook 的 if 表达式。

带文件参数的 command 路径也一致：原生 `NewTabCmd` / `VSplitCmd` / `HSplitCmd` 带参时分发到 `NewBufferFromFile(a)`，新 buffer `AbsPath` 非空，自动跳过 hook 的 true 分支。跟现状 `OpenBirthSelector` 内部 `isNoNameBuf` 跳过的行为完全一致。

#### `isNoName` 字段保留，赋值点从 `OpenBirthSelector` 挪到 hook

`isNoName bool`（bufpane.go:260）是 sticky 出身字段，赋值一次终身不变——它记录的是 **pane 出生时是不是 noName**，不是 "pane 当前 buffer 是否无路径"。这两件事不能混：

- 出生 noName → 用户在 birth selector 里 **Enter 选文件** → buffer 换成 `foo.md`（`AbsPath != ""`）→ `isNoName` **仍为 true**（sticky）
- 用户再按 Ctrl-q → QuitNeo 看 `isNoName == true` → 开 quit selector 让用户重新选

如果改用 `h.Buf.AbsPath == ""` 替 `h.isNoName`，用户挑完文件后再 Ctrl-q 就不再开 selector（直接走原生 Quit）——这是**可见的行为变更**，不该走。所以字段必须保留。

新方案里赋值点从 `filemanager.go:61`（`OpenBirthSelector` 入口）挪到 hook（见上面 `finishInitialize` 末 if-else 块）。`OpenBirthSelector` 整函数塌掉，赋值点跟着走。但有两点跟原行为不同，必须讲清：

1. **赋值从「满足时赋 true」变成「if-else 两路都赋值」**：原 `OpenBirthSelector` 只在三条件满足时置 `isNoName=true`，不满足时直接 `return` 不动字段——隐含依赖 BufPane 刚 newBufPane 出来 `isNoName` 字段是零值 `false` 这个偶然事实。新 hook 改成 if/else 两路显式赋值（满足→true，否则→false），保证 hook 跑完 pane 的出身状态完全确定，不依赖零值。**最终状态与原行为等价**（bail 时 `isNoName=false`），只是显式胜过隐式。
2. **hook 三条件判定从单条件扩到三条件**：如果 hook 只判 `AbsPath==""`(丢两条件)，会让 Help/Log/Raw/Scratch/Info 等特殊类型(Type!=BTDefault)和 Lua 预填内容的 noName(Size()>0)错误触发 birth selector。新版 hook 三条件齐全，与原 `isNoNameBuf` 语义 1:1 对齐。

字段本身、字段的 sticky 语义、`QuitNeo` 的判逻辑（`command_neo.go:100` 的 `if !h.isNoName`）**全部保留**。

清理：

| 文件 | 行 / 函数 | 操作 |
|---|---|---|
| `bufpane.go:260` | `isNoName bool` 字段定义 | **保留** |
| `bufpane.go:finishInitialize` 末尾 | （新增）if-else 两路：`isNoName=true`（满足三条件 + 开 finder）/ `isNoName=false`（不满足 + 不开） | 替换原 `if h.Buf.AbsPath == ""` 单条件版 |

| `filemanager.go:20` | `isNoNameBuf` helper | 删（三条件在 hook 内联判断，门控职责塌进 hook 的 if 表达式） |

净效果：`isNoName` 字段和 QuitNeo 的 `isNoName` 分支**保留**，赋值点从 `OpenBirthSelector` 挪到 hook；`isNoNameBuf` helper 和 `OpenBirthSelector` setter 整函数消失。用户行为与现状**完全一致**——零行为变更。

### B.5 intent 只活在 TooSmall 这一格

三个 trigger 能共用一个无状态 onClose，是因为 finder 关闭时的 `Result.Reason` 已经把用户意图表达清楚了——Enter（Picked）、Esc、Ctrl-q（Quit）、resize 打断（Resize），都是**用户在会话里亲手做的决定**，onClose 直接读 reason 就够。

唯一例外是 **TooSmall**：finder 一次都没开起来，用户没机会在会话里按键，TooSmall 这个 reason 本身**不携带任何意图**。这时 onClose 没法只凭 reason 推断"该 no-op 还是该 Quit"，只能读 `Result.IsQuit`——这个值是 open 时 owner 传进来的 `isQuit`，被 finder 原样带回（finder 全程没用它，见 A.6）。

所以 isQuit 不是贯穿全程、影响 finder 行为的状态，而是**一个只在 TooSmall 这种"用户失语"情况下才被查阅的兜底数据**。它以"数据进、数据出"的形式流转（Open 入参 → finder 暂存 → Result 字段），owner 的 onClose 因此成为 Result 的纯函数，无需闭包记忆、无需 owner 侧可变状态。

### B.6 onClose 决策表

onClose 是 `Result` 的纯函数，按 `Result.Reason` 一维分发（`r.IsQuit` 只在 TooSmall 那格登场）：

| Reason | onClose 动作 | Result.Cwd 取值 |
|---|---|---|
| Picked | 同文件（`filepath.Join(Cwd,File) == 当前 buffer 路径`）→ no-op；不同 → `OpenCmd(Join(Cwd,File))`，OpenCmd 自带 modified 检查 | 用户最终停留的目录 |
| Esc | no-op | 用户最终停留的目录(可能已导航) |
| Resize | no-op（环境事件，非用户决定；`r.IsQuit` 也不强退） | 用户最终停留的目录 |
| Quit | `doQuit(r)` | 用户最终停留的目录 |
| TooSmall | 读 `r.IsQuit`：true → `doQuit(r)`；false → no-op | startDir(Open 入参的 cwd，用户没机会导航) |

`doQuit(r)` = 先把 `r.Cwd` 暂存到包级变量（供 `lastWorkingDir` 在 exit 时读，见 B.8），再调 `h.Quit()`。Quit 和 TooSmall 且 `r.IsQuit` 两条路径共享它。

**`doQuit` 是 BufPane 的新 method（内部 helper），不是原生 Quit**——原生 `BufPane.Quit()` 在 `actions.go:1927`（自带 modified 检查 + 关 pane）；`doQuit` 只做"先暂存 cwd 再转交原生 Quit"，不重新实现 modified 检查、不预关闭。fileops.go 新增的所有符号清单：

```go
// fileops.go (action 包)

func (h *BufPane) OpenFinder(isQuit bool)           // 唯一对外接口,启动 finder 会话
func (h *BufPane) onFinderClose(r finder.Result)     // 共享 onClose:按 Result.Reason 分发
func (h *BufPane) doQuit(r finder.Result)            // 暂存 r.Cwd + 调原生 h.Quit()
var LastFinderCwd string                            // 包级暂存变量,doQuit 写/lastWorkingDir 读
```

原生的 `h.Quit()` 在 `actions.go:1927`（自己带 modified 检查 + 关 pane）——`doQuit` 只是「先做 owner 侧家务（暂存 cwd）再转交原生 Quit」，不重新实现 modified 检查，不预关闭。

**Picked 同文件跳过**是这次新增：用户在 finder 里挑了正编辑的那个文件，直接 no-op，避免无谓的 OpenCmd。noName pane（`AbsPath=""`）天然不命中，无需特例。

**TooSmall × quit 这一格**是"窄窗口死锁逃路"：noName pane 在极窄窗口按 Ctrl-q，finder 拒开，若 no-op 用户就卡死（再按 Ctrl-q 还是没反应）。靠 `r.IsQuit` 把它救出去。browse/birth 传 false，TooSmall → no-op，回编辑态，没有"必须出去"的压力。

### B.7 TooSmall vs Resize 的语义提炼

两个 resize-family reason 在 onClose 眼里语义不同，不要混：

- **TooSmall = "finder 一次都没能开起来"**（打开时窗口过小，预检拒开）→ owner 回退到自己开 finder **之前**想做的事。browse/birth 之前没事干→no-op；quit 之前想退→靠 `r.IsQuit` 触发 Quit。
- **Resize = "finder 开起来过、中途被打断"**（运行中终端 resize）→ 当成用户没明确决定 → 取消（no-op），**即使 `r.IsQuit` 也不强退**。理由：resize 是环境事件，不是用户决定，据此踢用户出去会意外。

这条规则解释了为什么 onClose 的 TooSmall 分支必须查 `r.IsQuit`，而 Resize 分支根本不查、直接 no-op（见 B.5）。

### B.8 cwd 缺口与修法（m() / lastWorkingDir）

这是这次讨论挖出的**真实缺口**，这也是 Result 加 `Cwd` 字段的具体理由。

现状链路：最后一个 pane Quit → microNeo 退出 → `exit()` → `lastWorkingDir()` 读 `MainTab().CurPane().Buf.AbsPath` 的父目录 → 写进 `--cwd-file` → shell 的 `m()` 读它 cd。

问题：`lastWorkingDir` 读的是 pane 当前 buffer 的路径，**完全不知道用户在 finder 里导航到哪了**。

塌陷场景：`m` 在 A 目录启 microNeo → noName pane → Ctrl-q 开 quit finder → 导航到 B 目录 → 再 Ctrl-q（Quit）→ `h.Quit()` → exit → `lastWorkingDir` 读 noName pane 的 `AbsPath=""` → 返回 `""` → 什么都不写 → m() 留在 A。用户明明在 finder 里走到了 B，shell 却不跟过去。

修法形状：

1. `Result` 加 `Cwd`（A.3 已含），finder 关闭时填 `state.currentDir`
2. onClose 的 `doQuit(r)` 把 `r.Cwd` 暂存到一个 owner 侧包级变量（不能挂在 pane 上——Quit 关掉最后一个 pane 之后、exit 里 `lastWorkingDir` 才读它，那时 pane 已经没了）
3. `lastWorkingDir()` 优先读这个值，fallback 才读 buffer 路径

这一步把"finder 会话的导航成果"接进了 shell 同步链路，是 Result.Cwd 存在的唯一理由——务必在代码注释里讲清这个 why，否则后人会当它过度设计。

### B.9 OpenCmd / Quit：owner 家事，不进 finder 契约

拿到 result 之后 owner 怎么执行动作，是 fileops.go 的实现细节，不属于 finder 的对外契约：

- `Picked → OpenCmd(filepath.Join(Cwd, File))`：不要真的"先 close 当前文件再 open"。`OpenCmd` 是原地换 buffer、自带 modified 检查（y=存/n=丢/esc=取消换入），效果等于换了文件，机制是单步 swap。保持现状。同文件命中时直接跳过这步（见 B.6）。
- `Quit / TooSmall(r.IsQuit) → doQuit(r)`：暂存 `r.Cwd` 后调一次 `h.Quit()`，它内部做 modified 检查 + 关 pane。别预关闭。

这两条只在 fileops.go 的 onClose 注释里留，不写进 finder 的接口文档。

## D. 落地前需澄清（开干前定）

这几条本次讨论没收敛，落地前必须拍板：

1. ~~overlay 渲染归属。~~ **已收敛：owner-local，不是主循环。** finder 由某个 owner（BufPane）打开，事件路由挂在 owner.HandleEvent 头部、渲染路由挂在 owner.Display 末尾（每帧两条转发线都在 owner 上，详见 §A.4 / §B.1 后半）。理由：finder 是 pane-local 钉左上角（AutoExpand=false），视觉区域完全在 owner pane rect 内，**不会被兄弟 pane / divider / InfoBar / NotePane 的 Display() 覆盖**——它们的 Display 画的是自己 rect，rect 不相交就没有覆盖问题。

    **与 N1a §五 ADR-7 的差异**：ADR-7 选主循环末尾统一 overlay 是因为 SelectPane 用 AutoExpand=true 可能越界（`:theme` 锚点 `(0, -1)` 紧贴 InfoBar 上方、NotePane receiver 锚点会跨 owner pane 边），必须在主循环画最末才能不被覆盖。**finder 不越界，所以不走那条路**——反过来走 owner-local 更轻（modal 视觉范围等于 owner pane rect，谁也不越界谁、谁来画都行，但挂在 owner 自己最直观）。两套策略并存，对应两种 widget 的视觉属性，不冲突。

    注意：仍要把"lifecycle owner（开/收 result）= BufPane"和"per-frame renderer（画屏）= BufPane"区分开——这两件事**都是同一个 BufPane**，不要和 N1a 那种「lifecycle owner 和 per-frame renderer 是不同对象」的情况混。N1a 把这两件事拆开是因为 SelectPane receiver 可以跨 owner；finder 不存在这个 receiver,两件事自然都是 owner 自己。
2. **finder 类型名**。`FileManager` / `Manager` / `Session`。包名已定 `finder`。
3. **与 N0b 的执行顺序**。N0b 在动 overlay 路由（per-owner FloatFrame），本方案也在动 overlay 路由（finder 离开 FloatFrame）。两件事同时做会互相踩，必须错开成两个 PR，且定先后。**推荐：N0b 先走其第 2、3 步(per-owner FloatFrame 改造)，finder 再走本方案第 3 步(离开 FloatFrame)。理由：finder 还挂在 FloatFrame 上时先让 N0b 改 FloatFrame 逻辑，finder 再离开不会冲突；若 finder 先离开，其直接调 screen.Screen.SetContent 的路径可能与 N0b 后续改动不兼容。**
