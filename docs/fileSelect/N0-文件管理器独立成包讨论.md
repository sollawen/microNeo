# 文件管理器独立成包 —— 设计方案

**状态**：核心原则已达成共识；拆包范围与命名待最终拍板。本文记录原则、事实依据、目标架构与设计取舍，定稿后另开 PLAN 文档（N0a）列迁移步骤。
**动机**：fileManager 后续会扩展删除 / 新建 / 复制 / 移动 / 重命名，长成完整的文件导航器。在那之前先定清楚它的代码归属与对外契约，避免子系统膨胀时把 `internal/action/` 撑成杂物间，也避免事后再做伤筋动骨的搬迁。

---

## 1. 一句话原则

把 fileManager 当成一个**「数据进、数据出」的模态组件**——它的「函数体」恰好是一段交互式模态会话。所有有状态资产（BufPane / buffer / tab）归调用者（action），fileManager 只拥有自己那一次短暂的会话。

---

## 2. 五条细则

1. **自洽模态**：调用者启动它，它独占整个会话的交互（esc / ctrl-q / 导航 / 挑选 / 未来的删改都在内部处理），它自己关掉自己。
2. **输入侧走纯参数**：启动所需的一切由调用者以**纯值**传进 `Open()`——rect、当前路径、起始目录，外加一个 `Host`（用来渲染浮窗）。**不传 `*BufPane`、不要它满足任何接口、不做全局注入。**
3. **输出侧走单一结果**：关闭时回传**一个结果结构**，承载整次会话的产出（选中路径、停留目录、执行过的 fs 操作…）。调用者拿到后自己解释、自己改自己的状态。**会中不做即时回调。**
4. **零状态、零反向依赖**：fileManager 无包级可变状态、零 `action` 引用，用纯值 + stub Host 即可单测。
5. **依赖单向**：action → file，永不反向。

---

## 3. 事实依据（已验证）

### 3.1 当前耦合比直觉窄得多

5 个相关文件（共 1908 行，全在 `package action`）对 `action` 包的依赖：

| 文件 | 行数 | 对 action 的依赖 |
|---|---|---|
| `fileselector.go` | 1172 | 仅两处碰 `*BufPane`（见下）；外加 `TheFloatFrame`、`Pos/Size/FloatOpenSpec` |
| `filemanager.go` | 116 | 在 `*BufPane` 上**定义方法**；调 `MainTab`/`OpenCmd`/`NewFileSelector` |
| `fileselector_git.go` | 176 | **零依赖**（只 `config` + `os/exec`） |
| `fileselector_test.go` | 199 | 测试（截断等纯函数） |
| `fileselector_git_test.go` | 245 | 测试（porcelain 解析） |

`fileselector.go` 全文搜对 `*BufPane` 的访问，**只有两个点**，且都是只读：

```
177:  if pane.Buf != nil && pane.Buf.AbsPath != "" {   // 读当前文件名（高亮用）
344:  view := pane.BWindow.GetView()                    // 读rect {X,Y,Width,Height}（算浮窗锚点）
```

它**不操作 buffer 内容、不调 OpenCmd、不动光标**。从 pane 那里只取两样：当前文件名、四元rect。

**循环依赖的来源**：`command_neo.go`（在 action）反过来调用 `filemanager.go` 里的 `OpenBirthSelector` / `birthDir`（行 251-302）。若 `filemanager.go` + `fileselector.go` 都搬走，会形成 action ↔ file 双向 import。

**硬卡点**：`filemanager.go` 在 `*BufPane` 上定义了方法 `openBrowseSelector`（行 82）/ `openQuitSelector`（行 101），被 `command_neo.go` 以 `h.openQuitSelector()`（行 103）/ `h.openBrowseSelector()`（行 113）调用。Go 不允许跨包定义方法，这两个**必须留在 action**。

### 3.2 resize 即关 → rect是静态快照（关键验证）

入口 `FloatFrame.HandleEvent`（`floatframe.go:219`）：

```
事件进来
  ├─ 是 EventResize → 存 onResize → f.Close()（关浮窗）→ 调 onResize → return
  │                   （resize 不转发给 FileSelector.handleEvent）
  └─ 否            → 正常转发
```

`fileselector.go:380` 注册的 `OnResize` 会 `fs.finish(SelectResult{Kind: Closed, Reason: ReasonResize})`。也就是说：**resize 不是「重新布局」，而是「直接关掉、整窗拆掉」**，FileSelector 的 `handleEvent` 永远收不到 resize（`fileselector.go:730` 注释确认）。

**这验证了「rect是 Open 瞬间的静态快照」**：rect在 `computeLayout` 里只在 Open 时读一次；会话中一旦 resize，浮窗在到达 FileSelector 之前就被关掉，根本不存在「需要重新读rect」的时刻。

因此「Open 时传一个 `Rect` 参数」与「持有 pane 实时读 `GetView()`」**行为完全等价**——live 引用整次会话只被读一次，之后要么没 resize（rect没变），要么 resize（直接 teardown）。**第 2 条「输入侧走纯参数」零行为损失。**

## 4. 成长方向

### 4.1 fs 操作本体：纯、零依赖

导航器要加的功能，核心全是标准库 fs 操作，对 action 零依赖：

| 功能 | 核心调用 | 对 action 的依赖 |
|---|---|---|
| 删除文件 | `os.Remove` / `os.RemoveAll` | 无 |
| 新建文件 / 目录 | `os.Create` / `os.MkdirAll` | 无 |
| 复制 | `io.Copy` + `os.Open` + `os.Create` | 无 |
| 移动 / 重命名 | `os.Rename` | 无 |
| 多选 | UI 状态扩展 | 仅经入参rect |

### 4.2 波及 open buffer 的写回（未来，现在未实现）

fs 操作完成后，可能要影响「打开它的那个 buf」。两个已预见场景：

| 场景 | 触发 | 期望效果 |
|---|---|---|
| 删掉当前正在编辑的文件 | `os.Remove(path)`，且 `path == buf.AbsPath` | 把该 buf 清成 noName |
| noName 跟随浏览目录 | 当前 buf 是 noName，用户切了目录后 ESC | 该 noName 归属目录更新为最后浏览的目录 |

**按第 3 条，这两个都不做会中即时回调，统一在关闭时塞进结果结构回传**：

- 删文件 → 会中只往结果的 `Ops` 列表追加一条；关闭时调用者遍历 `Ops`，把 AbsPath 命中的 buf 清成 noName。会中 buf 内容暂不动也无妨——用户正盯着 selector。
- noName 跟随目录 → 关闭时结果带 `FinalDir` 回传，调用者若是 noName pane 就更新归属。

file 全程不持 `*BufPane`、不调 buf 方法，只「报事实」；「碰 buf」的策略全在调用者侧（和 F1 的 isNoName 出身模型同源维护）。

## 5. 目标架构

### 5.1 依赖与文件构成

```
action ──imports──▶ file     （单向；file 不 import action）
```

```
internal/file/
├── manager.go      ← FileSelector（UI + 交互）+ 未来 Delete/Create/Copy/Move/Rename
├── entries.go      ← 目录列举、truncateNameKeepExt
├── git.go          ← porcelain（现 fileselector_git.go，已零依赖）
├── result.go       ← ManagerResult 等结果类型
└── host.go         ← Host 接口（仅渲染浮窗用）+ Rect/Spec 小结构体

internal/action/
├── bufpane.go          ← 不变
└── filebridge.go       ← 保留 *BufPane 薄方法（openBrowseSelector 等），提取参数后调 file.Open
```

### 5.2 输入侧：纯参数

`Open` 的形状（示意）：

```
type Rect struct{ X, Y, W, H int }

func (m *FileManager) Open(
    host        Host,         // 渲染浮窗：Open(spec)/Close()
    rect        Rect,         // 锚点 rect（pane.BWindow.GetView() 的四元组）
    currentPath string,        // 当前文件名（pane.Buf.AbsPath，高亮用）
    startDir    string,
    onSelect    func(ManagerResult),
)
```

调用者（`filebridge.go`）负责从 `*BufPane` 提取这两个值再传入。file 看到的只是 `int` + `string` + `Host`，编译期零 action 引用。

### 5.3 输出侧：ManagerResult

```
type ManagerResult struct {
    Kind     ResultKind   // Picked | Closed（含 esc / ctrl-q / resize）
    Path     string       // Picked 时：选中文件
    FinalDir string       // 浏览结束时停留的目录（noName 跟随用它）
    Ops      []FileOp     // 会中执行过的 fs 操作：{Kind: Delete|Move|..., Paths: ...}
    Reason   CloseReason  // 已有
}
```

回调时机与现在 `onSelect` 完全一致（关闭时一次），只是结构从「选中信息」扩成「整次会话产出」。现在（只读）`Ops` 为空、`FinalDir` 不填，但抽屉先打出来。

### 5.4 Host 也走参数（不做全局注入）

`Host`（对应现 `TheFloatFrame`）作为 `Open` 的入参传入，**不搞 `file.SetHost(...)` 全局注入**。调用者调 `Open` 时手上一定有 `TheFloatFrame`，天然无注入顺序问题——file 零包级可变状态。

## 6. 设计取舍记录（为什么不那么做）

把这几条写下来，避免日后被「优化」成反方向：

1. **读侧不引入 `PaneView` 接口。** 输入是 Open 瞬间的静态快照（§3.2 已证），传纯值比传接口更直接、更可测、调用点数据流更显式。接口是为「live 读取」这种不存在需求买的单。
2. **Host 不全局注入。** 全局注入会引入 init-order 风险（调用前忘注入即 nil panic）。走参数消灭这整类风险。
3. **不在 `*BufPane` 上加 `ResetToNoName()` / `SetBirthDir()` 之类写方法，也不在 file 定义对应接口。** 那会把 action 的内部状态变更泄漏进 file、让接口随功能膨胀。写侧统一走关闭时的结果结构，策略集中在调用者。
4. **会中不做即时回调（onEffect）。** 「用户折腾期间和调用者无关」——所有 fs 操作只往结果累积，关闭时一次性回传。一种载体、一个时机，心智最简。

## 7. 假设与风险

1. **假设：resize 即关**（§3.2 已验证为当前行为）。这是「输入传静态 Rect」的背书。若将来产品改成 resize 自适应重布局，只需把入参从 `Rect` 升级成 `func() Rect`（rect 提供者回调），或让 `Host` 驱动重布局——**原则不动**，仍是数据进数据出、零 action 引用，只是「数据」从值升级成一个小回调。改 resize 行为时应自动触发对本输入模型的复核。
2. **`*BufPane` 方法接收者必须留 action**。`openBrowseSelector` / `openQuitSelector` 保留为 action 内薄方法，内部委托 `file.Open`。失去一点「挂在 pane 上」的写法，换依赖干净。
3. **InfoBar 耦合**。`fileselector.go` 多处 `InfoBar.Message`（「pane too narrow」等）。file 内部只返回 `error` / 状态码，由调用者翻译成 `InfoBar.Message`——把「展示」留在 action。
4. **测试迁移**。两个 `_test.go` 从 `package action` 改 `package file`。截断等纯函数直接跟走；selector 测试用纯值 + stub Host，比造真 `BufPane` 更容易。
5. **可渐进执行**。先搬零依赖的 git + 截断建立 `internal/file/` 骨架；再做 `Open` 改纯参数 + 迁 selector；最后迁 manager。每步独立编译通过。
