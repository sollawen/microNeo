# 文件管理器独立成包 —— 讨论文档

**状态**：讨论中，未决策。本文只摆事实和方案对比，不包含实施步骤。
**动机**：fileManager 后续会扩展删除 / 新建 / 复制 / 移动 / 重命名等操作，长成一个完整的文件导航器。在那之前先想清楚它的代码归属——是留在 `internal/action/`，还是拆出独立包——避免子系统膨胀时把 action/ 撑成杂物间，也避免事后再做伤筋动骨的搬迁。

---

## 1. 问题陈述

当前与文件选择 / 管理相关的代码全部位于 `internal/action/`（`package action`），共 5 个文件、1908 行：

| 文件 | 行数 | 角色 |
|---|---|---|
| `filemanager.go` | 116 | 执行者：`OpenBirthSelector` / `openBrowseSelector` / `openQuitSelector` / `isNoNameBuf` / `birthDir` |
| `fileselector.go` | 1172 | `FileSelector` UI：State/View/Controller 三层 + 布局 |
| `fileselector_git.go` | 176 | porcelain 解析、git 状态聚合 |
| `fileselector_test.go` | 199 | 测试（文件名截断等） |
| `fileselector_git_test.go` | 245 | 测试（porcelain 解析） |

疑问：**要不要把这一坨移到 `internal/file/`（或类似新包）？** 直觉上「功能内聚的代码聚到一个包」听起来更清晰，但能不能搬、值不值得搬、搬的代价多大，取决于它们和 action 的真实耦合度。下面先把耦合量清楚。

---

## 2. 依赖精查 —— 耦合比直觉窄得多

这是本次讨论最关键的事实。逐文件列出对 `action` 包内符号的引用：

### 2.1 `fileselector.go` —— 只有两处碰 `BufPane`

全文搜 `pane.` / 对 `BufPane` 实例的访问，只有两个点：

```
177:  if pane.Buf != nil && pane.Buf.AbsPath != "" {       // 读当前文件名（高亮用）
344:  view := pane.BWindow.GetView()                        // 读几何 {X,Y,Width,Height}（算浮窗锚点）
```

也就是说，`FileSelector` **完全不操作 buffer 内容、不调 OpenCmd、不动光标**。它从 pane 那里只取两样东西：当前文件名、四元几何。其余的 `view.X/Y/Width/Height` 都是 `GetView()` 返回值上的字段访问，不是直接依赖 `BufPane`。

**注意：这是「当前代码」的现状，不是接口设计的上限。** 未来导航器会增加对 buf 的**写回**需求（如删掉当前打开的文件后把 buf 清成 noName、noName 跟随浏览目录改变归属），见 §3.2。接口要为这些写回预留通道，但不必现在实现——只把边界画对。

对 `action` 的其余依赖是**基础设施型**：

| 符号 | 用途 | 出处 |
|---|---|---|
| `TheFloatFrame` | `.Open(spec)` 挂浮窗、`.Close()` 关浮窗 | `floatframe.go` 单例 |
| `Pos` / `Size` / `FloatOpenSpec` | 布局原语 / 浮窗规格 | `floatframe.go` 类型定义 |
| `SelectPane` | 仅注释提及，无代码引用 | — |

其余外部依赖（`config` / `screen` / `tcell` / `go-runewidth`）都不属于 action，新包照常 import，无障碍。

### 2.2 `filemanager.go` —— 有一处硬卡点

```go
func (h *BufPane) openBrowseSelector() { ... }   // 行 82
func (h *BufPane) openQuitSelector()   { ... }   // 行 101
```

**Go 不允许在其它包里给一个类型定义方法。** 只要这两个还是 `*BufPane` 的方法接收者，`filemanager.go` 就钉死在 action 包内。它们被 `command_neo.go` 以 `h.openQuitSelector()` / `h.openBrowseSelector()` 调用（行 103 / 113）。

其余依赖：`MainTab()`、`buffer.Buffer` / `buffer.BTDefault`、`NewFileSelector`、`pane.OpenCmd`、`SelectResult`。

### 2.3 `fileselector_git.go` —— 零 action 依赖

只 import `config` 和标准库 `os/exec`。是 5 个文件里**已经天然可独立成包**的。

### 2.4 依赖方向小结

```
filemanager.go ──uses──▶ BufPane(方法接收者)、NewFileSelector、OpenCmd、MainTab
fileselector.go ──uses──▶ *BufPane(仅2处)、TheFloatFrame、Pos/Size/FloatOpenSpec
command_neo.go ──uses──▶ OpenBirthSelector、birthDir、startDirOf  (反向！)
```

注意第三行：`command_neo.go`（在 action）反过来调用了 `filemanager.go` 里的 `OpenBirthSelector` / `birthDir`。这正是循环依赖的来源——如果 `filemanager.go` 和 `fileselector.go` 都搬走，会形成 **action ↔ file 双向 import**，编译不过。

## 3. 成长方向分析 —— 新增功能几乎都是纯 fs 操作

这是支持「应该拆」的决定性论据。

### 3.1 fs 操作本体：纯、零依赖

把导航器要加的功能列出来，看它们各自的依赖：

| 功能 | 核心调用 | 对 action 的依赖 |
|---|---|---|
| 删除文件 | `os.Remove` / `os.RemoveAll` | 无 |
| 新建文件 / 目录 | `os.Create` / `os.MkdirAll` | 无 |
| 复制 | `io.Copy` + `os.Open` + `os.Create` | 无 |
| 移动 / 重命名 | `os.Rename` | 无 |
| 多选 | UI 状态扩展 | 仅经 `BufPane` 几何接口 |
| 删除当前打开的文件 | 删除 + 刷新 buffer 列表 | 一个「影响 open buffer」的回调钩子 |

**结论**：长大后的导航器，其主体（所有文件系统操作）对 action 的依赖是零。fs 操作本身干净；会碰到 action 的只有两类边界副作用——消息上报（§6.4）和波及 open buffer 的写回（§3.2）。后者是接口设计必须**预留空间**的地方。

### 3.2 波及 open buffer 的写回场景（未来，现在未实现）

导航器做完 fs 操作后，可能需要让**打开它的那个 buf** 发生变化。两个已预见的场景：

| 场景 | 触发 | 期望效果 |
|---|---|---|
| 删掉了当前正在编辑的文件 | file 内 `os.Remove(path)`，且 `path == 该 buf 的 AbsPath` | 把该 buf 清成 noName（出身重置） |
| noName buf 跟随浏览目录 | 当前 buf 是 noName，用户在 selector 里切了目录后 ESC | 该 noName 的归属目录更新为最后浏览的目录（下次 birth selector 从这开） |

**关键约束**：这两个效果都作用于「打开 selector 的那个 buf」，而 buf 是 action 的资产。file **不应**为了实现它们而 import action、或持有一个 `*action.BufPane` 来反向调用其方法——那会重新引入循环依赖，把第 4、5 节的断环成果抵消掉。

**正确做法**：file 只负责**宣告发生了什么**（「我删了 path X」「用户最终停留在目录 D」），由 action 侧（拥有 BufPane、知道哪些 buf 的 AbsPath 命中 X、知道当前 pane 是不是 noName）**自己决定并执行**清空 / 改归属。即 file 发出**意图 / 效果（effect）**，action 解释落地。file 始终单向依赖，零 action 引用。具体载体见 §5.2。

也就是说：**成长方向是干净的，越长大，独立成包的收益越高——只要写侧通道一次画对。**

## 4. 三个方案

按「动手范围 / 收益」排成三档。

### 方案 A：最小抽取（保守）

只搬**已经零依赖**的纯逻辑到 `internal/file/`：

- `fileselector_git.go`（porcelain 解析）
- `truncateNameKeepExt` 等截断函数（当前在 fileselector.go）
- 目录列举、排序逻辑（如有可剥离的）

`FileSelector` UI、`filemanager.go` 执行者全部留在 action。

**依赖**：file → config（无环）。
**收益**：低。UI 和执行者仍在 action 里膨胀，治标不治本。
**代价**：小。但要忍受「纯逻辑在一个包、UI 在另一个包」的割裂感。

### 方案 B：子系统整体成包（推荐）

建立 `internal/file/`，把 `FileSelector` + manager + git + entries 都放进去，靠一个窄接口做依赖反转打破循环：

```
internal/file/
├── host.go         ← PaneView + Host 两个接口（依赖反转）
├── selector.go     ← FileSelector（UI + 交互）
├── manager.go      ← Delete/Create/Copy/Move/Rename（未来 fs 操作都在这长）
├── entries.go      ← 目录列举、truncateNameKeepExt
└── git.go          ← porcelain（现 fileselector_git.go）

internal/action/
├── bufpane.go      ← BufPane 实现 file.PaneView
└── filebridge.go   ← 保留 *BufPane 方法（openBrowseSelector 等），薄调 file.OpenSelector
```

**依赖**：file 只依赖 `config`/`screen`/`tcell` + 自己定义的两个**接口**；action 依赖 file。**无环。**

**打破循环的关键**（第 5 节展开）：在 file 里定义接口而非直接引用 `*action.BufPane`，由 action 侧注入实现。第 2.1 节已经证明这个接口可以很窄。

**收益**：高。子系统真正自成一体，后续所有导航器功能都在 file 里长，不污染 action；纯 fs 部分可独立单测。
**代价**：中。需要引入接口、做一次性注入改造、迁移 1908 行代码 + 测试。

### 方案 C：连浮窗基础设施一起抽（彻底）

进一步把 `floatframe.go` / `selectpane.go` 里的 `Pos` / `Size` / `FloatOpenSpec` / `TheFloatFrame` 抽成 `internal/float/`，让 file 和 action 都依赖它：

```
internal/float/   ← FloatFrame、Pos/Size/FloatOpenSpec、SelectPane
internal/file/    ← Selector + manager（依赖 float）
internal/action/  ← BufPane、NotePane、InfoPane（依赖 float + file）
```

**收益**：概念最干净——「浮窗基础设施」是独立于「文件子系统」的一层，note/info 等浮窗也受益。
**代价**：大。牵动 `notepane.go` / `infopane.go` 等所有浮窗消费者，是一次跨子系统的重构，与当前「先把文件管理器理顺」的目标偏离。

## 5. 打破循环依赖的机制（方案 B 核心）

第 2.1 节已证明 `FileSelector` 对 `BufPane` 的全部需求是「当前文件名 + 四元几何」。据此在 `internal/file/host.go` 定义两个窄接口：

```
type PaneView interface {
    Geom() (x, y, w, h int)   // 对应 pane.BWindow.GetView() 的四元组
    CurrentPath() string       // 对应 pane.Buf.AbsPath
}

type Host interface {
    Open(spec Spec) bool       // 对应 TheFloatFrame.Open
    Close()                    // 对应 TheFloatFrame.Close
}
```

- `action.BufPane` 写两个方法即可满足 `PaneView`（薄包装现有 `BWindow.GetView()` 和 `Buf.AbsPath`）。
- `TheFloatFrame` 满足 `Host`（或包一层 adapter）。
- action 在初始化时把 `TheFloatFrame` 注入 file（如 `file.SetHost(TheFloatFrame)`）。

file 包内 `FileSelector.Open(view PaneView, host Host, dir string, onSelect ...)` 全程只见接口，编译期零 action 引用。**循环依赖不复存在。**

### 5.1 读侧（PaneView）保持窄

`PaneView` 只放 file **渲染所必需的只读信息**（几何 + 当前路径）。刻意不放 `ResetToNoName()` / `SetBirthDir()` 之类的写方法——那会把 action 的内部状态变更泄漏进 file，且让接口随功能膨胀。读侧稳定，写侧另走通道。

### 5.2 写侧（效果通道）—— 为 §3.2 预留的空间

file 需要影响 buf 时，不调 buf 的方法，而是把**发生了什么**抛回 action，由 action（拥有 BufPane、知道哪些 buf 命中、知道当前 pane 是不是 noName）**自己决定并执行**清空 / 改归属。file 发出**意图 / 效果（effect）**，action 解释落地——file 始终单向依赖，零 action 引用。

§3.2 的两个场景天然落在两种载体上：

- **会中 fs 操作**（删掉当前文件）→ 操作完成的瞬间抛 `Effect{Kind: FileDeleted, Path: ...}`，经一个 `onEffect` 回调回 action。action 比对该 path 与各 open buf 的 AbsPath，命中者清成 noName。
- **关闭时的目录信息**（noName 跟随浏览目录）→ 随现有的 `SelectResult`（已有 `Closed` / `Reason`）带一个 `FinalDir` 字段回 action，由 onSelect handler 处理。

两种载体都让 file 只「报事实」、不「碰 buf」。新增写回场景（如「复制后把新文件开进新 pane」）只需扩 `EffectKind` 或 `SelectResult` 字段 + 在 action 侧加 handler 分支，不动 file↔action 的依赖边界。

现在（selector 只读、不写回）`onEffect` 可空、`FinalDir` 可不填，但接口形状先定下——这就是「预留空间」的具体形态：边界一次画对，后续只填业务，不再动依赖结构。

`Pos` / `Size` / `FloatOpenSpec` 三个 floatframe 原语：方案 B 下最省事的做法是 file 包内重定义同构小结构体（`file.Pos{X,Y}` / `file.Size{W,H}`），在 action 的 adapter 边界做一次转换，避免把 floatframe 概念拖进 file。若觉得双份定义别扭，就走方案 C 把它们提升到 `internal/float/`。

## 6. 风险与注意事项

1. **`*BufPane` 方法接收者必须留在 action**。`openBrowseSelector` / `openQuitSelector` 改成自由函数 `file.OpenBrowse(h)` 也可，但会改调用方（`command_neo.go` 两处），且失去「`h.xxx()` 这种挂在 pane 上的自然写法」。建议保留为 action 内薄方法，内部委托给 `file`。

2. **注入时机**。`TheFloatFrame` 是 action 包级单例，file 用它得靠运行时注入。要确保 file 的任何公开 API 被调之前 host 已注入（main 初始化序列里靠前的位置 `file.SetHost(TheFloatFrame)`），否则 nil panic。可用「未注入即 panic + 明确错误信息」守护。

3. **测试迁移**。两个 `_test.go` 现在是 `package action`，搬走后改 `package file`。`fileselector_test.go` 测的是纯函数（截断），直接跟走；若将来 selector 的测试需要 mock PaneView，接口反而让测试更好写（不必造真 `BufPane`）。

4. **InfoBar 耦合**。`fileselector.go` 多处用 `InfoBar.Message` 上报（「pane too narrow」等）。这是 action 的全局消息条。方案 B 下要么把 `InfoBar` 也经接口注入（多一个接口），要么 file 内部只返回 `error` / 状态码、由 action 侧翻译成 `InfoBar.Message`。后者更干净，把「展示」留在 action。

5. **一次到位 vs 渐进**。方案 B 可以渐进执行：先把零依赖的 git + 截断搬过去建立 `internal/file/` 骨架，再做接口注入迁移 selector，最后迁 manager。每步可独立编译通过。

## 7. 待决策

| 项 | 选项 |
|---|---|
| 是否拆包 | A 最小 / **B 推荐** / C 彻底 / 不拆 |
| `Pos`/`Size`/`FloatOpenSpec` 归属 | file 内重定义+边界转换（B）/ 提升到 `internal/float/`（C） |
| `*BufPane` 方法去留 | 留 action 薄委托（推荐）/ 改自由函数 |
| 消息上报（InfoBar） | 经接口注入 / file 返回状态码由 action 翻译（推荐后者） |
| 新包名 | `internal/file/` / `internal/filemgr/` / `internal/navigator/`（若定位是导航器） |

倾向结论：**方案 B + 留 action 薄委托 + file 返回状态码 + 包名 `internal/file/`**。理由是第 3 节证明的成长方向干净，第 2 节证明的耦合窄到值得做一次依赖反转，而方案 C 的范围超出当前目标。
