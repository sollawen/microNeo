# 文件管理器独立成包 —— 设计方案

**动机**：fileManager 后续会扩展删除 / 新建 / 复制 / 移动 / 重命名，长成完整的文件导航器。在那之前先定清楚它的代码归属与对外契约，避免子系统膨胀时把 `internal/action/` 撑成杂物间，也避免事后再做伤筋动骨的搬迁。

---

## 1. 目标

- 把fileManager，fileSelector，file_git，这些程序，全部移到 internal/filemanager/目录下，独立成包
- 把 fileManager 当成一个**「数据进、数据出」的模态组件**——它的「函数体」恰好是一段交互式模态会话。
- 所有有状态资产（BufPane / buffer / tab）归调用者（action），fileManager 只拥有自己那一次短暂的会话。
---

## 2. 几个原则

1. **自洽模态**：调用者启动它，它独占整个会话的交互（esc / ctrl-q / 导航 / 挑选 / 未来的删改都在内部处理），它自己关掉自己。
2. **输入侧走纯参数**：启动所需的一切由调用者以**纯值**传进 `Open()`
	- 当前bufPane的rect、当前文件的全路径，或者是当前目录+noName空文件
	- **不传 `*BufPane`、不要它满足任何接口、不做全局注入。**
3. **输出侧走单一结果**：关闭时回传**一个结果结构**，承载整次会话的产出
	- 关闭时的cwd, 
	- selected filename(如果是按回车退出的话，就有filename) 
	- selected filename 这个文件的状态，分成 nomal, deleted, rename之类的。这个需要讨论
	- 关闭的原因，分成 esc, enter, resize, quit(用户按ctrl-q时)
	- 调用者拿到结果后后自己解释、并做出针对性的处理，比如open file, quit 之类的
4. **零状态、零反向依赖**：fileManager 无包级可变状态、零 `action` 引用，用纯值 + stub Host 即可单测。
5. fileManager提供几个公共接口给调用者。调用者一定是某个bufPane
	- Open()
	- isOpen()，供调用者判断是不是open状态
	- 每帧标准接口，这两个接口的命名要和所有的pane一致
		- HandleEvent(event), 由调用者把键盘鼠标事件通过这个接口传给fileManager
		- Display()，由调用者在每一帧render的时候，调用这个接口来绘制fileManager
	- 其它需要的接口，需要讨论
6. fileManager、fileSelector不再使用和依赖floatFrame，和floatFrame断绝关系

# 定稿方案（讨论收敛后）

上面是讨论起点。下面是经过代码核对 + 双方辩论后收敛出的可执行方案。两件事分开写：**finder** 是独立成包的模态组件本身；**fileops** 是留在 action 的 owner 侧接线层。它们是两回事情，契约各管各的。

## 命名决定

| 现在 | 改成 | 理由 |
|---|---|---|
| `FileSelector`（类型） | `FileManager` | 产品定位文档第一句"fileManager = 终端里的 Finder"；FileSelector 只是它 v1 的 browse-only 阶段名，给 delete/move/copy/rename 留位 |
| 新包 `internal/filemanager/` | `internal/finder/` | 贴产品词汇；避免 `filemanager.FileManager` 的重复。最终形态 `finder.FileManager`（或 `finder.Manager`/`finder.Session`，类型名落地前再定） |
| `filemanager.go`（action 内） | `fileops.go` | 这文件是 owner 侧胶水，不是 finder 的一部分；改名消除和新包名的撞名，也避免误导后人以为"fileManager 的实现在这" |

搬移范围：`fileselector.go` + `fileselector_git.go` + 对应测试 → `internal/finder/`。`filemanager.go`（→`fileops.go`）**留在 action**，因为它调 `pane.OpenCmd`/`pane.Quit`、读 `pane.Buf`，搬走会形成反向依赖。

---

## A. finder 包（FileManager 的对外契约）

finder 是一个**数据进、数据出的自洽模态组件**。它独占一次会话的交互，自己关掉自己，关掉时如实上报发生了什么。**它不知道、也不关心 owner 为什么开它**——这是本章最重要的原则。

### A.1 边界与归属

- finder 拥有：自己那一次会话的状态（当前目录、条目、光标、git 标志、显示模式）
- finder 不拥有：BufPane、buffer、tab、任何进程级可变状态
- finder 不关心调用者是谁，也不知道调用者的一切
- finder 不引用：`action` 包、`*BufPane`、`TheFloatFrame`、`InfoBar`
- 调用者永远是某个 BufPane（通过 fileops.go 中介）

### A.2 输入侧：纯值

```go
func (fm *FileManager) Open(
    rect     Rect,       // owner pane 的绝对屏幕矩形（对角两点坐标）
    cwd      string,     // 起始目录全路径；noName pane 传父 pane 的文件目录（spawn 时捕获）
    file string,         // 当前文件名（不含目录）；noName 传 ""
    onClose  func(Result),
)
```

**不传 `*BufPane`、不要 owner 满足任何接口、不做全局注入、不传 Host。** finder 直接 import `screen` + `config`，画自己时调 `screen.Screen.SetContent` / `config.GetColor`——和 floatFrame、NotePane 画自己的方式完全一致。screen 和 config 都是比 action 更底层的包，`finder → screen` / `finder → config` 这两条 import 边合法、不循环。

finder 唯一够不到的 action 资产是 `InfoBar.Message`（现在拒开时由 `computeLayout` 直接调，见 `fileselector.go:365/369`）。搬包后这个调用挪到 fileops.go：finder 照常通过 `onClose` 上报 `TooSmall`，fileops 在回调里判到 `TooSmall` 就自己调 InfoBar 报 "pane too narrow/short"——提示本就是讲 owner 的 pane 太窄，由 owner 自己说最自然。**Open 不需要返回值**：size 拒开和 Esc/Quit 一样走 callback，对调用方完全透明（每次 Open 必然对应一次 onClose，无论 finder 有没有真显示出来）。

当前 `FileSelector` 对 `*BufPane` 的全部依赖只有两处只读：`pane.Buf.AbsPath`（取当前文件名做光标定位 + 推起始目录）和 `pane.BWindow.GetView()`（取 rect 算 layout）——拆成 `cwd`（起始目录）+ `file`（光标定位）两个纯值传进来。noName pane 的 cwd 由 owner 用现有 `birthDir` 逻辑算（父 pane 文件目录，spawn 时捕获），finder 无感。

### A.3 输出侧：单一结果结构

```go
type Result struct {
    Reason CloseReason   // 会话怎么结束的（五种，见下）
    Cwd    string        // finder 关闭时所在目录的全路径（始终填，不带文件名）
    File   string        // Reason==Picked 时：选中的文件名（不含路径）
}

// CloseReason：扁平枚举，Picked 与其余四个同级（不再分 Kind）
//   Picked   — Enter 选中文件
//   Esc      — 用户按 Esc
//   Quit     — 用户按 Ctrl-q（或 welcome 态的 q）
//   Resize   — 运行中终端 resize，已显示后被打断
//   TooSmall — 打开时窗口过小，finder 从未显示出来
```

字段说明：

- `Reason`：finder 如实上报"会话怎么结束的"，五种取一。**没有 Kind 分级了**——Picked 和 Esc/Quit/Resize/TooSmall 是同级的原因。原来 Kind=Picked|Closed 的两级分类是过度结构，已坍。
- `Cwd`：**始终填 finder 关闭时的 `state.currentDir`**（纯目录、不带文件名）。哪怕用户 Esc 取消，他在会话里可能已经导航到了别处——owner 侧的 shell 同步（见 B.6）需要这个值。这是这次讨论新增的字段，原方案里没有，理由见 B.6。
- `File`：仅 `Reason==Picked` 时填——选中的文件名（**不含路径**，目录在 `Cwd` 里）。其余四种为空字符串。owner 用 `filepath.Join(r.Cwd, r.File)` 还原绝对路径交给 `OpenCmd`。
- **预留 `FileStatus`（normal/deleted/renamed）字段位**：等 delete/rename 功能落地再加。现阶段不建，避免空头结构。结构前向兼容。

### A.4 公共 API

finder 提供给 owner（实为 fileops.go）的接口：

- `Open(...)` — 启动一次会话
- `IsOpen() bool` — owner 判断 finder 是否在开
- `HandleEvent(event)` — owner 每帧把键盘/鼠标事件转发进来（命名对齐全 pane 约定）
- `Display()` — owner 每帧 render 时调用绘制 finder

后两个接口的命名与所有 pane 一致（BufPane / NotePane 都有同名方法），让 finder 在 owner 眼里就是一个"临时 pane"。

**没有 `Close()`，也没有 resize 通知接口。** owner 不主动关 finder——会话的终结（用户 Enter/Esc/Quit、运行中 resize）全由 finder 内部判定后回调 onClose，owner 只负责 Open 和收结果。其中 resize 根本不是单独的通知：它是 `tcell.EventResize`，和按键走同一通道进 `HandleEvent`，finder 在里面拦截后自己 Close。

### A.5 与 floatFrame 断绝（硬约束 + 设计选择）

现状 finder 经由 `TheFloatFrame` 中转：把 `display`/`handleEvent` 作为函数值塞进 `FloatOpenSpec`，由 FloatFrame 代为画框 + 路由事件。finder 搬进自己的包之后**必须停止**用 floatFrame——这既是 Go 的硬约束，也是我们的设计选择，两条理由各自独立成立。先看硬约束：

```
fileops.go (action 包) → import finder（调 Open）
finder                → import action（用 TheFloatFrame）  ← 这条成环，编译报错
```

floatFrame 在 action 包里，finder 一搬走就物理上够不到它——上面的图就是这条：成环编译报错，非断不可。但**就算没有这条 import 限制**，设计上我们也要让 finder 升格成自洽的 pane 级 overlay，自己画边框、自己接管事件路由（像 floatFrame / NotePane 一样直接 `screen.Screen.SetContent`），不再寄生在一个住在 action 包里的共享 host 上。"未来要不要嵌套 input" 是另一个动机，目前未触发，跟这次断绝无关。

finder 升格为 pane 级 overlay，自己实现 `Display()`/`HandleEvent()`，**画法照搬 floatFrame**：直接 `screen.Screen.SetContent` 画边框/内容/clear、HideCursor、resize 拦截。这三块（边框 + title 嵌入 ~30 行 + resize 拦截 ~8 行 + 尺寸预检 ~5 行）全是机械代码，从 `floatframe.go` 搬进 finder 即可，无新设计。边框代码就是这么直白：4 个角 `┌┐└┘` + 上下 `─` + 左右 `│` + 上边框嵌 `──title──...─`，全是 dumb SetContent。

FloatFrame 的去留与 finder 无关：FloatFrame 继续服务 SelectPane 等简单一次性 widget（N0a/N0b 的 per-owner 演进照常进行）。两者对 FloatFrame 的处置不同，必须分开，不要把 N0 和 N0b 搅成一坨。

### A.6 intent-agnostic 原则（关键）

**finder 永远不知道 owner 为什么开它。** owner 可能是为了 browse（Ctrl-o）、为了 quit（Ctrl-q）、因为是 noName 新 pane（birth）——这些 intent 全部住在 owner 侧（fileops.go 的回调里，见 B.3），finder 不接收 intent 参数、不根据 intent 改变行为。

finder 的职责只有一条：**如实上报会话怎么结束的**（Result.Reason）。怎么解释这个 Reason、要不要据此 Quit/OpenCmd，全是 owner 的家事。

这条原则比"输入走纯参数"更强：它要求 finder 的行为对"为什么被调"完全透明。好处是 finder 单测时不用构造 intent 上下文，且未来加新 intent（如 delete-mode）时 finder 零改动。

### A.7 单测边界（诚实评估）

- **已可纯函数单测**：git 解析（`parsePorcelain`/`getGitStatus`，现已有测试）、字符串工具（`truncateNameKeepExt`/`humanSize`/`fitMeta`/`truncateLeftPath`，现已有测试）。
- **display / handleEvent 不做 stub 单测**：它们直接调 `screen.Screen.SetContent` / `config.GetColor` 这些包级单例（和 floatFrame、NotePane 一样），**没有 Host 抽象**，出问题靠手测。这是有意取舍——finder 里真正难、易回归的逻辑（git 解析、字符串截断）都是纯函数、已覆盖；为 display 加抽象层的成本远大于手测收益，等真觉得痛了再说。

## B. fileops.go（owner 侧胶水与 dispatcher）

fileops.go 是 BufPane 和 finder 之间的接线层。它干两件事：**在合适的时机开 finder**，**在 finder 关闭后解释 result 并执行 owner 侧动作**。它留在 `internal/action/`，因为它要调 `pane.OpenCmd`/`pane.Quit`、读 `pane.Buf`、写 `lastWorkingDir`——这些都是 BufPane 的家事，搬进 finder 会形成反向依赖。

### B.1 角色

```
BufPane 事件（spawn/Ctrl-q/Ctrl-o）
      ↓
  fileops.go: trigger（一行 finder.Open + 选回调）
      ↓
  finder 会话（自洽模态）
      ↓ finder 关闭，回传 Result
  fileops.go: dispatcher（解释 result → 执行 owner 动作）
      ↓
  BufPane 动作（OpenCmd / Quit / 写 cwd tmp）
```

fileops.go 不持有 finder 的会话状态，它只在"开"和"收"两个时刻起作用。

### B.2 三个 trigger 点（各一行）

现状三个 `openXxxSelector` 各写一遍重复回调，是"乱"的源头。塌掉重复之后，每个 trigger 就剩一行 `finder.Open(...)` 加一个回调工厂：

- **birth trigger**：noName pane 刚诞生时调（spawn 钩子，6 处包内 + micro.go 启动）。回调走 browse 语义（见 B.4）。
- **browse trigger**：`:file` / Ctrl-o 调。回调走 browse 语义。
- **quit trigger**：Ctrl-q 调（`QuitNeo`）。回调走 quit 语义。

三个 trigger 的事件源物理上不同（spawn 时、按键、命令），**没法合成一个 call site**——但每个点就一行，无逻辑。真正被统一的是回调里的 dispatch 逻辑。

### B.3 intent 住在回调里，不在 finder

owner 在开 finder 时"记住"自己的 intent——这个记住不是运行时 flag，而是**结构性地体现在选了哪个回调**：

- browse/birth trigger 选 browse 回调
- quit trigger 选 quit 回调

finder.Open 签名**不带 intent 参数**。intent 是 owner 的内部状态，finder 无感。

### B.4 dispatcher 决策表

owner 侧把 `(intent, Result)` 组合成动作。把现状三个回调的行为摊开，决策表是这样：

| intent \\ reason | Picked | Esc | Resize | TooSmall | Quit |
|---|---|---|---|---|---|
| browse / birth | OpenCmd(filepath.Join(Cwd,File)) | no-op | no-op | no-op | Quit |
| quit | OpenCmd(filepath.Join(Cwd,File)) | no-op | no-op | **Quit** | Quit |

关键发现：**intent 只影响一个格子**——`TooSmall × quit`。其余所有格子两种 intent 完全一致。

这一格就是"窄窗口死锁逃路"：noName pane 在极窄窗口按 Ctrl-q，finder 拒开（TooSmall），如果不退用户就卡死。browse/birth 不需要它（取消就是回编辑态，没有"必须出去"的压力）。

### B.5 TooSmall vs Resize 的语义提炼

两个 resize-family reason 在 owner 眼里语义不同，不要混：

- **TooSmall = "finder 一次都没能开起来"**（打开时窗口过小，预检拒开）→ owner 回退到自己开 finder **之前**想做的事。browse 之前没事干→no-op；quit 之前想退→Quit。
- **Resize = "finder 开起来过、中途被打断"**（运行中终端 resize）→ 当成用户没明确决定 → 取消（no-op），**即使 intent=quit 也不强退**。理由：resize 是环境事件，不是用户决定，据此踢用户出去会意外。

这条规则让 dispatcher 不用查二维表：`TooSmall` 一律触发"回退到 pre-open intent"，`Resize` 一律 no-op。

### B.6 cwd 缺口与修法（m() / lastWorkingDir）

这是这次讨论挖出的**真实缺口**，这也是 Result 加 `Cwd` 字段的具体理由。

现状链路：最后一个 pane Quit → microNeo 退出 → `exit()` → `lastWorkingDir()` 读 `MainTab().CurPane().Buf.AbsPath` 的父目录 → 写进 `--cwd-file` → shell 的 `m()` 读它 cd。

问题：`lastWorkingDir` 读的是 pane 当前 buffer 的路径，**完全不知道用户在 finder 里导航到哪了**。

塌陷场景：`m` 在 A 目录启 microNeo → noName pane → Ctrl-q 开 quit finder → 导航到 B 目录 → 再 Ctrl-q（Quit）→ `h.Quit()` → exit → `lastWorkingDir` 读 noName pane 的 `AbsPath=""` → 返回 `""` → 什么都不写 → m() 留在 A。用户明明在 finder 里走到了 B，shell 却不跟过去。

修法形状：

1. `Result` 加 `Cwd`（A.3 已含），finder 关闭时填 `state.currentDir`
2. fileops.go 的 dispatcher 在（至少）quit 路径上把 `r.Cwd` 暂存到一个 owner 侧变量（包级或挂在 pane 上）
3. `lastWorkingDir()` 优先读这个值，fallback 才读 buffer 路径

这一步把"finder 会话的导航成果"接进了 shell 同步链路，是 Result.Cwd 存在的唯一理由——务必在代码注释里讲清这个 why，否则后人会当它过度设计。

### B.7 OpenCmd / Quit：owner 家事，不进 finder 契约

拿到 result 之后 owner 怎么执行动作，是 fileops.go 的实现细节，不属于 finder 的对外契约：

- `Picked → OpenCmd(filepath.Join(Cwd, File))`：不要真的"先 close 当前文件再 open"。`OpenCmd` 是原地换 buffer、自带 modified 检查（y=存/n=丢/esc=取消换入），效果等于换了文件，机制是单步 swap。保持现状。
- `Quit / TooSmall(quit) → h.Quit()`：直接调一次 `Quit()`，它内部做 modified 检查 + 关 pane。别预关闭。

这两条只在 fileops.go 的 dispatcher 注释里留，不写进 finder 的接口文档。

## D. 落地前需澄清（开干前定）

这几条本次讨论没收敛，落地前必须拍板：

1. **overlay 渲染归属**。finder 的 `Display()` 由谁每帧调用？
    - 若由 `BufPane.Display()` 末尾调用——会画在兄弟 pane / NotePane **之前**，作为模态会被后画的盖掉（渲染顺序 bug）
    - 若由主循环末尾统一调用 overlay——和现状 FloatFrame 的画法一致，但"调用者"就不再是 BufPane 了
    - N0b §五.1/§12.1 为同一问题纠结过，结论是 overlay 渲染回到主循环。本方案需对齐。
    - 注意：这里要把"lifecycle owner（开/收 result）= BufPane"和"per-frame renderer（画屏）= 主循环"分开，不要捏成一个"调用者"。
2. **finder 类型名**。`FileManager` / `Manager` / `Session`。包名已定 `finder`。
3. **与 N0b 的执行顺序**。N0b 在动 overlay 路由（per-owner FloatFrame），本方案也在动 overlay 路由（finder 离开 FloatFrame）。两件事同时做会互相踩，必须错开成两个 PR，且定先后。

## E. 建议的落地顺序（由低风险到高风险）

1. **平移**：`fileselector_git.go` + 测试 + 纯字符串工具搬进 `internal/finder/`，零行为变化。立刻减重、立刻可跑。
2. **断 BufPane 耦合**：`Open` 改纯值入参（rect + cwd + file），`filemanager.go` 改名 `fileops.go` 留在 action 当 adapter。对应 N0b §12.3 的变换。
3. **升格 pane 级 overlay + 统一 dispatcher**（A.4/A.5 + B 全章）：从 `floatframe.go` 搬边框/resize/预检代码进 finder，改渲染/事件路由到主循环 overlay 位（D.1 定），三个 trigger 塌成一个共享 dispatcher。最大一步，单独一个 PR。
4. **接 cwd 链路**（B.6）：Result 加 Cwd，dispatcher 暂存，lastWorkingDir 优先读。可与第 3 步合并或紧跟。

第 3、4 步要和 N0b 的 per-owner FloatFrame 改造明确错开（D.3）。
