# FileSelector 位置与尺寸设计

**文档定位**：FileSelector 浮窗的几何（位置、尺寸、锚定、面包屑）决策与原理。是 `:file` 几何行为的**权威来源**。

**阅读对象**：要实现 FileSelector 渲染/布局、或要调整其位置尺寸的开发者。

**关联**：
- [file命令设计.md](file命令设计.md)（FileSelector 整体架构）
- [产品分析.md](产品分析.md)（用户场景：portrait 终端、split 习惯）
- [弹窗框架设计.md](../弹窗机制/弹窗框架设计.md)（FloatFrame 几何模型）

## 1. 核心原则：pane-local

FileSelector 只影响**发起 `:file` 的那一个 buffer**，对其它 bufWindow 零影响。由此推出本设计的根本原则：**FileSelector 的几何从"当前活跃 bufWindow"派生，不从全局终端派生。**

它视觉上"属于"即将被改动的那个 pane——位置、尺寸、边界都锚在这个 pane 上。这与 `:theme` 形成对照：`:theme` 无 pane 归属（全局选择），故用全局 sentinel（`Y:-1`，锚在全局 InfoBar 上方）；FileSelector 有明确 pane 归属，故用 pane 派生的绝对锚点。

**为什么 pane-local 是对的**：打开文件是 buffer 级操作。picker 视觉上"长在"它要修改的 pane 上，强化因果（用户看到：这个 picker → 那个 pane → 文件开进去）。覆盖范围也限制在该 pane 内，最小化对其它内容的遮挡。

## 2. 产品前提

来自 [产品分析.md](产品分析.md)，决定了几何的可行区间：

- **终端形态以 portrait（竖长）为主**：宽度有限（~80 列）、高度充裕。→ 宽度取分数（30%）即可放下文件名；高度取分数（50%）仍留足内容。
- **split 以上下（水平分割）为主**，左右分割少见；多文件更常用 tab。→ 几何只需为"上下 split + 单 pane"优化，左右 split 退化可接受。
- **编辑的是长文件**：保留内容上下文重要。→ picker 只占 pane 的一角，不遮满。

## 3. 布局模型：VSCode sidebar + Windows 开始菜单

FileSelector 借用两个最被熟知的 UI 模式组合：

- **横向 = VSCode 资源管理器侧栏**：贴 pane 左缘、占宽度的一个分数（30%），向右不超 pane。
- **纵向 = Windows 开始菜单**：从 pane 底部（statusLine 上方）向上展开，高度**封顶**在 pane 的 50%，有最小高度——不按条目数自适应，避免大 pane + 少文件时的空旷感。

两个模式合一个心智模型：**从活跃 pane 的左下角弹出一个面板，占其一角**。

```
┌─ 活跃 bufWindow（上 pane）──────────────────────┐
│ 文件内容…                                        │
│                                                  │
│ ┌──────────────┐                                │
│ │ microNeo/docs│ ← 面包屑（内容区第 1 行）       │
│ │ ▸ docs/      │                                │
│ │ ▸ src/       │ ← FileSelector                 │
│ │   file命令…  │   左下角 · 30% 宽 · 50% 高      │
│ │   产品分析…  │                                │
│ │ ▼            │                                │
│ └──────────────┘                                │
├──────────────────────────────────────────────────┤ ← 上 pane 的 statusLine（让出）
│ 文件内容（下 pane）…                             │
├──────────────────────────────────────────────────┤ ← 下 pane 的 statusLine
│ :file                                            │ ← 全局 InfoBar
└──────────────────────────────────────────────────┘
```

上 pane 活跃时，picker 出现在上 pane 左下角、向上右扩展；下 pane、全局 InfoBar 完全不动。下 pane 活跃时，picker 锚在其左下角（位置同全局底部）；无 split 时两者重合。

## 4. 几何规则

设 `pv` = 活跃 pane 的 `GetView()`（屏幕绝对坐标 `X/Y/Width/Height`，由 `Tab.Resize` 从 views.Node 树写入）。`avail = pv.Height - 1`（让出该 pane 自己的 statusLine）。

| 维度 | 规则 |
|------|------|
| 宽 `W` | `clamp(0.3 * pv.Width, minWidth, pv.Width)` |
| 高 `H` | 分段（见下） |
| anchor | `(pv.X, pv.Y + pv.Height - 2)` —— pane 左缘、statusLine 上方一行 |
| contentSize | `(W - 2, H - 2)`（减 FloatFrame 边框） |

**高度分段（开始菜单模型）**：

```
if 0.5*avail >= minHeight:   H = 0.5*avail      // 高 pane：取一半，优雅不空旷
elif avail >= minHeight:     H = minHeight      // 中 pane：守住最小可用高度
else:                        H = avail           // 矮 pane：塞满整个内容区
```

三段语义：高 pane 给半高（不贪屏）、中 pane 保最小可用、矮 pane 全填（退化但不溢出）。**H 与目录内条目数无关**——这是第 7 节架构红利的前提。

## 5. 映射到 FloatFrame：零框架改动

这是本设计最关键的一条：**pane-local 几何不需要扩展 FloatFrame，它早就支持。**

FloatFrame.`Open(anchor, contentSize, ...)` 接受**任意绝对锚点**。`Y:-1` 只是个便利 sentinel（解释成"全局 InfoBar 上方"），`Y≥0` 原样传入 `expandAnchor`。所以 FileSelector 自己算出 pane 派生的绝对 anchor，直接传进去即可——FloatFrame 继续当无 pane 知识的薄容器，**C5 守住**。

`expandAnchor`（`floatframe.go:230`）从锚点的行为天然给出"左下角钉住、向上右扩展"：

- **x 方向**：锚点在 `pv.X`（pane 左缘）→ `rightSpace`≈整屏宽 → 必定向右 → `fx = ax`，**左边贴 pane 左缘**。
- **y 方向**：锚点在 statusLine 上方 → `upSpace` 是上方整片 → 必定向上 → `fy = ay - outerH + 1`，**底边贴 statusLine 上方**。

合起来：frame 钉在 `(pv.X, statusLine行-1)`，向上向右扩展到 `(W, H)`。**正是第 3 节的布局**，无需任何框架改动。

**面包屑自绘、不用 FloatFrame 的 title 参数**：因为 `outerW = max(contentSize.W+2, len(title)+6)`——title 一长会把 frame 撑过 30% 宽。FileSelector 把面包屑画在内容区第 1 行（自己控制长度与截断），FloatFrame 的 title 留空或放静态 "Files"。这样连"导航换目录后改 title"都不需要任何 FloatFrame API。

## 6. 面包屑：当前目录显示

**两级**：`父目录/当前目录/`（如 `microNeo.master/docs/`）。不显示完整绝对路径。

**为什么两级够**：左右键导航**本身就是完整路径的体感**——用户一路按右键进来，路径在脑子里。面包屑只负责**确认"我现在在哪"**，不承担完整路径职责。两级（父 + 当前）足以确认位置。

**自绘于内容区第 1 行**，按内容宽度截断，**从左截、保留 `当前/` 完整**：当前目录才是用户正挑东西的地方，最相关；父目录可牺牲。两级也可能超宽（两个长目录名），从左截保证最重要的信息总在。

**边界**：根目录显示 `/`；浅层（如 `/Users`）显示存在的部分。

**备选（未来）**：若想要"空间够就多显几级、不够 `…` 截左"（VSCode/helix 做法），更灵活但更复杂。v1 用两级，简单可预测。

## 7. 架构红利：定尺寸消除导航重开

W 与 H 都是 **pane 几何的固定分数，与条目数无关**。而模态会话期间（C3 锁事件）pane 几何不变。于是 **frame 尺寸在整个会话里恒定**——它是个固定视窗，只有流过它的数据变。

由此连锁简化（详见 [file命令设计.md](file命令设计.md)）：

- **导航（← / →）= 重读目录 + 重绘**。frame 原地不动，**无需 Close+Open**。→ **ADR-2（导航触发同实例重开）整体作废**，导航对 C4（同步驱动）的依赖随之消失。
- **视图变更（`.` 切隐藏、排序）= 仅重绘**（本来就如此，现在更彻底）。
- **会话内唯一还要重开的，是终端 Resize**：pane 几何变了、要重算 W/H，而 FloatFrame 无 `Resize` → 只能 Close+Open。这是 ADR-3 唯一存留的触发点。
- **`maxVisible` 问题消解**：可见行数 = 定高能放多少就多少，多了滚动。不再是个待定参数。

这是一个产品决策（开始菜单封顶高度）与架构简洁性**自然对齐**的例子：选了"不按内容自适应尺寸"，就免掉了"尺寸变化要重接容器"的全部复杂度。

## 8. 边界与退化

| 场景 | 行为 | 评价 |
|------|------|------|
| 上 pane 活跃、但 pane 较矮（H > 上 pane 可用高） | expandAnchor 让 frame **向下溢出**跨过 split 边界、进入下 pane；有 clamp 兜底不崩 | 可接受退化，少见 |
| 左右 split（少见） | pane 半宽（~40 列），30%≈12 列 < minWidth → minWidth 兜底，局促但可用 | 不为罕见情况优化 |
| 极矮 pane（avail < minHeight） | H = avail，frame 塞满 pane 内容区 | 优雅退化 |
| 24 列宽放不下多列元数据 | v1 **只显 标记 + 文件名**（目录加 `/` 或 `▸`），砍 size/mtime/git 列 | 见下 |
| 会话内终端 Resize | 重算 W/H + Close+Open（唯一会话内重开） | 同步，C4 轻度适用 |

**关于元数据**：portrait 30%≈24 列，放不下"名 + 大小 + mtime + git"。v1 走 VSCode 侧栏风格 = 标记 + 文件名。连带 Model 简化：不必为显示不出的字段去 Stat 每个条目，`os.ReadDir`（name + isDir）足矣；size/mtime/git 推迟到将来确有宽屏/多列模式时再加。

## 9. 参数

| 参数 | v1 默认 | 理由 |
|------|--------|------|
| `widthFraction` | 0.3 | VSCode 侧栏感；portrait 下留 70% 内容可见 |
| `heightFraction` | 0.5 | 开始菜单感；留上半屏内容可见 |
| `minWidth` | 20 | portrait 24 列为底线；再窄文件名不可读 |
| `minHeight` | 5 | 面包屑 1 行 + 约 3 条目 + 滚动指示 |

默认值可经设置项调整（键名待定）。`maxVisible` 不再是参数（由定高派生）。
