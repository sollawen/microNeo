我们先从产品设计的角度来讨论这个问题。

# 几个常见的用户使用场景：

1，首先要确认一个定位：microNeo主要是用来编辑AI生成的方案的，或者是程序文件的。而这些文件通常都比较长。因此，
	- 用户通常应该是在一个竖长方形的终端窗口里使用microNeo，
	- 比较少的是接近正方形
	- 基本不太会在一个橫着的长方形里面使用

2，用户有可能会使用split同时打开两个以上的文件
	- 我觉得大部份用户应该是把竖长方形的窗口split成为上下两个窗口，而不是左右两个窗口
	- 或是用tab标签来打开两个文件
	- 再多文件的可能性就比较小了


# 我的想法
FileSelector是和当前活跃的那个bufWindow紧密相关的，因为它只影响这个buf里面去open一个文件，对其它bufWindow是没有影响的。

所以，我觉得fileSelector应该是
- 上下充满当前活跃的bufWindow，下边让出来这个bufWindow的statusLine
- 左边贴边开始，右边到这个bufWindow的30%。当然，可以设置一个最小宽度，不能小于这个最小宽度
- 其实这个设计，就和VScode差不多

当然，如果窗口上下很大，但是当前目录里文件不多，会在fileSelector里面显得很空
- 我们也可以下边永远从当前bufWindow的statusLine上一行开始
- 上边就以当前bufWindow的50%，但是有一个最小高度。如果bufWindow的高度不够了，就以bufWindow的高度为准
- 这个设计，就和windows的开始菜单差不多

关于当前目录可能很长
- 我觉得不要显示当前目录的全路径了。嵌套很深的时候，根本就显示不完
- 就显示两级：父目录/当前目录/
