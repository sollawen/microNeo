# fileManager 的产品定位与边界

## 这份文档是什么

`floatFrame的讨论.md` 是一次架构讨论。那场讨论结束的时候，结论已经超出"floatframe 能不能用"的范围了——它其实是说清楚了一件事：**我们到底要给什么人，做什么样的 fileManager**。这份文档把那个答案凝下来，作为以后所有 UI 决策和重设计时的裁决依据。

如果未来在某个细节上拿不准，回来读这一页。`floatFrame的讨论.md` 是过程；这里是结果。

---

## 我们服务的人

明确不服务：
- vim / emacs 的资深用户——他们有自己的生态和肌肉记忆，不会来，也不会被我们撬动
- GUI IDE 的重度用户——他们根本不想来终端，不要为"挽留他们"做妥协

我们要服务的人，具体长这样：
- 之前是 GUI 党，对 shell / 终端不陌生但基本不用
- 这两年因为 AI agent 全是 CLI，开始正经用终端，学了基础命令、知道了 ghostty、了解了 ssh
- 顺藤摸瓜用上了 vim / nano：
  - vim 太陡，**学不动**
  - nano 太弱，**不够用**
- 卡在中间，找到了 micro

**一句话画像：GUI 撤退到终端的 AI 时代用户。**

他们的真实特征，不是技术意义上的"新手"，而是**期望意义上的 GUI 原住民**——他们眼睛里习惯看 UI，不习惯看键位；对破坏性命令本能犹豫；期待所有 GUI 世界的 affordance：列标题、状态栏、鼠标、预览、确认、撤销。

---

## 我们要做的事

**microNeo 的 fileManager = 终端里的 Finder。**

不是一个 ranger 的克隆，不是一个 nerdtree 的简化版，也不是一个弹出来选个文件就走的小工具。它要做的是：

> 让这个用户在终端里管理文件，像在 Finder / Explorer 里一样自然——看得见列、点得动鼠标、看得见预览、按错能撤销、每个动作都"在视觉上"成立。

这件事背后有一个更硬的承诺：**all-in-one**。yazi 跟 micro 是能配合工作的，但要让这批用户去找第二个工具、装上它、设好环境变量、把它和 micro 配在一起用——他们会烦。更甚，**连在 micro 里装一个自带的 lua 插件**——要去查、要搜、要装——**对这群人也是负担**。他们想要的是：装好 microNeo，按一个键，文件树就出来了。他们的参照系是 VS Code：一个程序里完成所有事。文件管理是编辑动作的一部分，不该被外包给另一个进程。

具体包含：
- 浏览目录、切换目录
- 在文件列表上做 rename / move / copy / delete / mkdir / new file
- 看文本文件的内容预览
- 看到 git 状态（被改过 / 新增 / 未跟踪）
- 看到这个文件有多大、什么时候改的、谁能读写
- 在出错之前有提醒，错了之后能反悔

---

## 我们不做的事

为了让上面这件事做得扎实、有明确边界，**有几件事我们明确不沾**：

- 不做 IDE 风格的常驻 sidebar——窄终端是大量用户的现实，没有宽度不要硬塞
- 不做图片 preview、视频 preview、PDF preview——增加复杂度，对核心人群价值极低
- 不追求和 yazi / ranger 的快捷键对齐（`dd` 切 `p` 粘那套）——那是给 vim 难民的设计，对我们的用户是诅咒
- 不为 power user 加隐藏快捷路径——他们不是我们的人；为发现性设计不为炫耀设计
- 不让 fileManager 自己变成半个 editor——它是管文件的，不编辑内容；文件内容预览 ≠ 编辑
- 不把文件管理外包给外部工具（yazi / ranger）——文件操作和编辑在同一个进程，**all-in-one 是承诺不是口号**
- 不做成可选插件——fileManager 必须 build-in，连"装个插件"这一步对这人都是摩擦，装好 microNeo 就应该能用

## 和替代方案的关系

市场上至少有四类现成答案，我们都不是它们：

| 替代方案 | 它做对了什么 | 我们为什么不当它 |
|---|---|---|
| 原版 micro 的 filemanager 插件 | 简单、能用 | 信息密度低、没有 size/mtime/git、没有预览、rename 设计反 GUI |
| vim 的 NERDTree / nvim-tree | 功能齐全 | 是常驻 sidebar，要宽度；vim 生态，用户不会来 |
| vim 的 oil.nvim / emacs dired | "目录当文本编辑"的极简 | 模型太奇特，教育成本高于价值 |
| yazi / ranger 作为外部工具 | ranger 体验完整 | 要装、要设 `$EDITOR`、学两个工具之间来回切换——这批用户会被劝退 |
| Helix / kakoune 的内置 picker | 召之即来开完即关 | 不做文件管理，只做"打开"，人手差一半 |

我们的位置是：**召之即来的壳 + 浏览器级的功能 + 给 GUI 难民的视觉和交互**。这条路上没有现成的成熟方案，意味着没有现成答案可抄——既要敢自己定，也要敢迭代。

## 设计原则（每个 UI 决策过这条）

所有具体设计拿不准时，回到下面六条：

1. **Finder / Explorer 在这里会怎么呈现？** —— 列标题、列分隔、右下角信息栏、状态行，是默认要有
2. **这个动作可逆吗？不可逆的必须有确认。** —— 删文件必弹 y/n；不确定的时候宁可多问一次
3. **每个键在底部要让用户看到自己是干嘛的。** —— 用户的眼睛要先看到动作才动手指
4. **鼠标永远是合法的一等交互。** —— 这个用户不会为"少配两个键绑"沾沾自喜，他要点能点能拖
5. **F2 / n / d / Enter / Esc / 方向键，撑起 90% 的场景。** —— 别逼用户学超过这些
6. **预览是一档公民。** —— GUI 用户对"看一眼"是天然期待的，文本预览是底线

## 我们是怎么走到这里的（保留）

未来某天有人翻这一页，可能比现在更久以后，要看懂为什么选了这条线。简要留个档：

**最初的问题**：做一个 fileManager，需要输入文件名做 rename / 新建，但浮层容器的契约是不允许嵌套——架构上行不行？

**走错又收回的几个判断**：
- 一开始以为 yazi 的卖点是"边看列表边编辑"——不是。它的卖点是会话持久（FM 进程不死）和右边预览窗格。"边看边编辑"是 IDE sidebar 模型，跟我们无关
- 一开始以为需要给 fileManager 加"召回到上次的目录和位置"——已经有。打开时就是当前文件的目录，光标就在当前文件上，这条路自动满足
- 一度担心"召唤式浮层"和"想看列表"冲突——结论是不会，因为这群用户进 fileManager 就是为了看一眼列表、做完操作即走，本就不是同时阅读两个东西的人

**留下的方向**：用一个召唤即来、用完即走的轻量浮窗做容器，在里面把浏览器级功能做齐（rename / move / copy / delete / new file / mkdir / 文本预览 / git 状态）。所有交互按 GUI 难民的视觉和心理模型来做。

**关于"未来会不会变成 sidebar"**：会，但只有在终端宽度允许时。这是真选项，目前被环境条件按住了——不要为了"早晚要变"提前付架构的代价。

---

## 留作观察的节点

这些是未来可能打破当前结论的条件。一旦命中，重新开讨论：

- 有证据表明用户**真的开始频繁想要"列表常驻"**，且终端宽度普遍允许
- fileManager 的功能膨胀到六种以上交互，召唤式浮窗开始装不下
- microNeo 的目标用户画像漂移（比如开始往"硬核终端用户"扩展）

没命中之前，**别动**。这条线的成本是反复推倒重来，回报是微小的"看起来更完整"。
