# F1 · FileSelector 子系统架构总览

**状态**：已实现并测试通过。本文是 microNeo FileSelector 子系统的**架构指导文档**——给读者（含 AI）快速建立全貌心智模型：整体分层、核心概念、行为契约、代码约束。**具体实现细节（函数签名、行布局坐标、porcelain 解析、状态码映射等）一律读代码**，本文不重复——代码是单一事实来源，本文只讲「为什么这样设计、哪些是必须遵守的约束」。

产品交互细节以 F0 为准。

---

## 0. 一句话

microNeo 给每个 pane 引入「出身」概念（`isNoName`）：以无名空 buffer 诞生的 pane，诞生时弹文件选择器让你挑文件（birth selector），按 `Ctrl-q` 弹选择器让你挑「换入文件 or 退出」（quit selector）；`Ctrl-o` 同样用选择器替代原生 `:open` 文本输入（browse selector）。三个入口的选中换入全部复用 micro 原生 `OpenCmd`，modified 检查零自写。

## 1. 整体分层

```
┌──────────────────────────────────────────────────────────────┐
│  调用者层        command_neo.go                               │
│   FileCmd / QuitNeo（binding target）                         │
│   职责：识别用户意图 → 委托 OpenFinder(isQuit)                 │
├──────────────────────────────────────────────────────────────┤
│  owner 接线层    fileops.go                                   │
│   BufPane.OpenFinder(isQuit) —— 唯一对外接口                   │
│   onFinderClose(Result) —— 统一结果处理                        │
│   职责：算 rect/cwd/file，调 finder.Open，解释 Result         │
├──────────────────────────────────────────────────────────────┤
│  finder 包       internal/finder/                            │
│   Session：State/View/Controller                             │
│   Open(rect, cwd, file, isQuit, onClose) —— 纯值入参         │
│   HandleEvent / Display —— 事件与渲染                         │
│   职责：自洽模态会话，"数据进、数据出"                         │
├──────────────────────────────────────────────────────────────┤
│  owner 转发层    BufPane                                      │
│   HandleEvent 头部转发（modal 屏蔽）                           │
│   Display 末尾调用 finder.Display()                            │
│   职责：每帧把事件/渲染机会转给 finder                         │
├──────────────────────────────────────────────────────────────┤
│  底座            micro 主循环（cmd/micro/micro.go，零改动）    │
└──────────────────────────────────────────────────────────────┘
```

一次完整「打开 → 选中 → 开文件」数据流：

```
用户按 Ctrl-o（或 Ctrl-q / birth）
  → handler：FileCmd / QuitNeo
       └ 委托 → h.OpenFinder(isQuit)
            ├ 算 rect（从 BWindow.GetView()）
            ├ 算 cwd/file（从 Buf.AbsPath，noName 则 cwd）
            └ finder.Open(rect, cwd, file, isQuit, onFinderClose)
                 ├ 预检放不下 → Open 返回 false，owner 处理（quit 退、browse no-op）
                 ├ 成功 → State 初始化（readDirEntries 读全部条目，μs 级同步）
                 └ 首帧渲染（无 git 列）+ 后台 go fetchGit(dir)
[运行期] 事件流：主循环 → BufPane.HandleEvent → finder.HandleEvent
           渲染流：主循环 → BufPane.Display → finder.Display
resize → finder 拦截 EventResize → close(Resize) → onFinderClose
失焦 → Tab 鼠标路由 → SetActive(false) → onOwnerBlur → finder.close(Esc)
选中文件 → close(Picked) → onFinderClose → OpenCmd（复用原生 :open）
```

**关键设计**：finder 是 pane 级 overlay（owner-local modal），不是主循环级 modal。事件和渲染路由都钉在 owner 上，主循环零改动。

## 2. isNoName 出身模型

### 2.1 isNoName 是 sticky 出身，不是缓存

`BufPane.isNoName`（bool）记录 pane **怎么诞生的**，赋值一次终身不变：

- `true` = 以无名空 buffer 诞生（如 `Ctrl-t` 开新 tab）
- `false` = 以文件诞生（如 `:vsplit foo.md`）

用户在 birth selector 选文件换入后，`isNoName` **保持 true 不清**——它是「出身」语义，不是「当前是否无名」。精确的一句话定义：**isNoName = pane 是否用 quit selector 替代原生 Quit**。它**只管 Ctrl-q**：`true` 的 pane 按 Ctrl-q 先弹 quit selector，`false` 的 pane 直接走原生 Quit。Ctrl-o 对所有 pane 一视同仁，不看 isNoName。

### 2.2 isNoName 三条件判定

在 `BufPane.Display()` 的 `pendingBirth` 机制中判定：

```go
h.Buf != nil &&
h.Buf.AbsPath == "" &&
h.Buf.Type == buffer.BTDefault &&
h.Buf.Size() == 0
```

- `AbsPath==""`：无路径（非文件 buffer）
- `Type==BTDefault`：排除 Help/Log/Raw/Info/Scratch
- `Size()==0`：排除管道内容（管道有内容 → Size>0 → 非 noName）

`buffer.Size()` 逐行字节求和、末行不加分隔符。**勿改成 `len` 判空**（语义不同）。

## 3. 三个入口统一为 OpenFinder

### 3.1 三入口总表

| 入口 | 触发 | isQuit 参数 | 起始目录 |
|---|---|---|---|
| **birth** | pane 出生（pendingBirth 机制） | `false` | cwd（fallback） |
| **browse** | `Ctrl-o` / `:file`（FileCmd） | `false` | 当前文件所在目录，或 cwd |
| **quit** | `Ctrl-q`（QuitNeo，noName pane） | `true` | 当前文件所在目录，或 cwd |

三入口统一成 `h.OpenFinder(isQuit)`，唯一差异是 `isQuit` 参数——**这个参数只在 TooSmall 场景下被查阅**（见 §3.4）。

### 3.2 OpenFinder：唯一对外接口

```go
// fileops.go —— owner 与 finder 之间的接线层

func (h *BufPane) OpenFinder(isQuit bool) {
    var dir, file string
    if abs := h.Buf.AbsPath; abs != "" {
        dir  = filepath.Dir(abs)   // 当前文件所在目录
        file = filepath.Base(abs)  // 纯文件名（不含路径）
    } else {
        dir, _ = os.Getwd() // noName / 启动段：fallback 到 cwd
    }
    v := h.BWindow.GetView()
    ok := h.finder.Open(finder.Rect{X: v.X, Y: v.Y, W: v.Width, H: v.Height},
                       dir, file, isQuit, h.onFinderClose)
    if !ok {
        // 预检放不下：finder 没开会话、没触发 onClose
        // quit 入口窄窗直接退（破死锁），browse 入口 no-op
        if isQuit {
            h.Quit()
        }
        return
    }
}
```

**关键设计**：`finder.Open` 纯值入参，不传 `*BufPane`、不满足任何接口、不做全局注入。finder 直接 import `screen` / `config` 画自己，与 floatFrame / NotePane 同构。

### 3.3 onFinderClose：统一结果处理

```go
func (h *BufPane) onFinderClose(r finder.Result) {
    switch r.Reason {
    case finder.Picked:
        if filepath.Join(r.Cwd, r.File) == h.Buf.AbsPath {
            return // 同文件 no-op
        }
        h.OpenCmd([]string{filepath.Join(r.Cwd, r.File)})
    case finder.Quit:
        h.doQuit(r)
    case finder.TooSmall:
        // 读 r.IsQuit：quit 入口窄窗也要能逃出去
        if r.IsQuit {
            h.doQuit(r)
        }
    case finder.Resize, finder.Esc:
        // no-op
    }
}

func (h *BufPane) doQuit(r finder.Result) {
    LastFinderCwd = r.Cwd  // 先暂存
    h.Quit()              // 后调原生 Quit（顺序不能反）
}
```

`onFinderClose` 是 `(*BufPane)` 的 method（不是包级函数），因为回调体要调 `h.OpenCmd` / `h.Quit`，而不同 caller 的 `h` 是不同实例。Go 绑定方法值把当次的 `h` 烤进回调里。

### 3.4 TooSmall 破死锁（窄窗 quit 入口的逃路）

若 quit 入口在窄窗下也 no-op，会死循环：

```
Ctrl-q → QuitNeo → isNoName=true → OpenFinder(true)
      → finder.Open 预检失败（pane 太窄）→ 返回 false
      → 若 onFinderClose 里 TooSmall 分支 no-op
      → 回编辑 → 再按 Ctrl-q → 又开不出……无限循环
```

化解：TooSmall 分支查 `r.IsQuit`，true → `doQuit`。**isQuit 是 owner 传进来、finder 原样带回的数据**，finder 全程不读它、不据此改变行为（intent-agnostic 原则）。它只在「用户失语」（TooSmall，用户根本没机会按键）这种场景下，作为兜底让 quit 入口能逃出去。

browse / birth 入口传 `isQuit=false`，TooSmall → no-op，回编辑态——没有"必须出去"的压力。

### 3.5 Result 与 CloseReason

```go
// finder.Result：会话结束时回传给 owner 的唯一结果
type Result struct {
    Reason CloseReason  // 会话怎么结束的（5 种）
    Cwd    string       // 关闭时所在目录（始终填，纯目录）
    File   string       // Reason==Picked 时：选中的文件名（不含路径）
    IsQuit bool         // 原样回显 Open 入参 isQuit；finder 全程不读
}

// CloseReason：扁平枚举，5 种同级原因
type CloseReason uint8
const (
    Picked   CloseReason = iota  // Enter 选中文件
    Esc                          // 用户按 Esc
    Quit                         // 用户按 Ctrl-q（或 q）
    Resize                       // 运行中终端 resize，已显示后被打断
    TooSmall                     // 打开时窗口过小，从未显示
)
```

**语义说明**：

- `Picked`：用户在 finder 里选了文件 → `OpenCmd` 换入（自带 modified 检查）
- `Esc`：用户取消 → no-op
- `Quit`：用户在 finder 里按 Ctrl-q → 退出 pane（`doQuit`）
- `Resize`：运行中终端 resize → 取消（不退，这是环境事件）
- `TooSmall`：打开时窗口过小 → quit 入口退（破死锁），browse 入口 no-op

`Cwd` 的存在意义：让 shell 的 `m()` 能跟到用户在 finder 里导航到的目录（见 §8）。

## 4. finder 包：数据进、数据出的自洽模态组件

### 4.1 对外契约

finder 包只暴露这些符号：

```go
package finder

type Rect struct{ X, Y, W, H int }  // owner pane 绝对屏幕矩形

type CloseReason uint8              // 见 §3.5
const (Picked, Esc, Quit, Resize, TooSmall)

type Result struct{ Reason, Cwd, File, IsQuit }  // 见 §3.5

type Session struct{ ... }  // 不导出字段

func NewSession() *Session
func (fm *Session) Open(rect Rect, cwd, file string, isQuit bool, onClose func(Result)) bool
      // 返回 false = 预检放不下，不开会话、不触发 onClose

func (fm *Session) IsOpen() bool
func (fm *Session) HandleEvent(event tcell.Event)
      // 内部拦截 EventResize → close(Resize)，其余转 handleKey

func (fm *Session) NotifyBlur()
      // owner 失焦时调用，等价 Esc 取消（非 tcell 事件，走方法）

func (fm *Session) Display()
      // 自己画边框 + 内容
```

**关键设计**：

- 不引用 `action` 包、`*BufPane`、`TheFloatFrame`、`InfoBar`
- 只 import `screen` / `config` / `tcell` / `runewidth`（都比 action 底层，不构成环）
- Open 收纯值，回调收单一 Result——「数据进、数据出」
- 自己画边框、自己拦截 resize、自己做尺寸预检（照搬 floatframe.go 机械代码）

### 4.2 intent-agnostic 原则

**finder 的行为永远不因 owner 为什么开它而改变**。`isQuit` 入参只是数据进、数据出，finder 全程不读它做任何决定，只在 close 时原样填进 `Result.IsQuit`。

**硬约束**：finder 内部禁止出现任何 `if isQuit` / `switch isQuit`。一旦出现，说明 finder 真的要按 intent 表现不同——那时应当升级成正式的 mode 参数、重新设计，而不是偷偷分支。

这条原则让 `onFinderClose` 成为 Result 的纯函数，无需闭包记忆、无需 owner 侧可变状态。

### 4.3 升格为 pane 级 overlay

finder 不再寄生 `FloatFrame`（主循环级 modal），而是自己画边框、自己接管事件路由：

```go
// Display：自己画框（照搬 floatframe.go 机械代码）
func (fm *Session) Display() {
    if !fm.isOpen { return }
    screen.Screen.HideCursor()
    fm.drawBorder()  // 4 角 + 边 + clear + title 嵌入
    fm.drawContent() // 内缩 1 的内容区
}

// HandleEvent：头部拦 resize，其余转 handleKey
func (fm *Session) HandleEvent(event tcell.Event) {
    if !fm.isOpen { return }
    if _, ok := event.(*tcell.EventResize); ok {
        fm.close(Resize) // 拦截 resize，自己关
        return
    }
    fm.handleKey(event)
}
```

边框代码就是 dumb 的 `screen.Screen.SetContent`：4 个角 `┌┐└┘` + 上下 `─` + 左右 `│` + 上边框嵌 `──title──...─`。约 60 行，全是机械代码。

### 4.4 State / View / Controller 三层

与旧 FileSelector 同构，职责分离：

| 层 | 职责 | 不做什么 |
|----|------|---------|
| **State** | 目录、条目、视图、光标/视口、显隐与排序模式、git 标志 | 不碰屏幕、不读键 |
| **View** `display(area)` | 读 State 画面包屑 + 文件列表 + 末行元数据；截断、Reverse 光标、配色 | 不改 State |
| **Controller** `handleKey(ev)` | 读键 → 改 State 或触发选中/取消 | 不直接画屏 |

**键位映射**：

| 键 | 行为 |
|----|------|
| `↑`/`↓` | 移动光标（不循环，clamp `[0,len]`），滚动跟随 |
| `Enter`（面包屑/目录） | 回父级 / 进入目录 |
| `Enter`（文件） | `close(Picked)` |
| `←` | 回父级目录 |
| `→` | 光标在目录 → 进入 |
| `.` | 切换 dotfile 显隐 |
| `Esc` | `close(Esc)` |
| `Ctrl-q` / `q` | `close(Quit)` |
| 其它键 | 吞掉（modal） |

进入子目录光标落首条目；回父级落在「刚离开的目录」上（ranger/lf 约定）。

### 4.5 数据模型：读一次 / 过滤分离

每个条目 `entry` 存全量 `os.FileInfo`（lstat）+ `gitChar`（git 状态字符，0=干净）。State 维护两份条目切片：

- `allEntries`：当前目录**全部**条目（含 hidden），chdir 时读一次、排序
- `showEntries`：`allEntries` 按 `showHidden` 过滤后的**指针子集**

三个函数分工：`readDirEntries`（chdir 时调，读全部 + 排序 → allEntries）、`rebuildShow`（过滤 → showEntries）、`sortEntries`（原地排 allEntries）。

### 4.6 显示/排序模式字段（已留好，切换功能以后做）

State 有 `rightMode`（右列 size/time/none，当前恒 size）、`sortMode`（排序键 name/size/time，当前恒 name）、`sortDesc`（升降序，当前恒 false）。**字段已就位，循环切换的按键处理尚未实现**——以后接切换只加按键，数据结构不动。

## 5. owner 转发层：BufPane 的事件与渲染路由

### 5.1 两条转发线

owner（BufPane）持有一个 finder 引用，每帧做两件事：

```go
// bufpane.go

type BufPane struct {
    display.BWindow
    finder *finder.Session
    // ...
}

// HandleEvent：头部转发（modal 屏蔽）
func (h *BufPane) HandleEvent(event tcell.Event) {
    // finder modal 转发：会话开着时所有事件先交给 finder，整段 owner 逻辑全不跑
    if h.finder.IsOpen() {
        h.finder.HandleEvent(event)
        return  // modal：owner 整段全不跑
    }

    // 原 ExternallyModified reload 检查 + switch ...（不动）
}

// Display：shadow BWindow.Display()，末尾追加 finder
func (h *BufPane) Display() {
    // birth trigger：消费一次性触发许可（见 §6）
    if h.pendingBirth {
        h.pendingBirth = false
        h.isNoName = h.Buf != nil &&
            h.Buf.AbsPath == "" &&
            h.Buf.Type == buffer.BTDefault &&
            h.Buf.Size() == 0
        if h.isNoName {
            h.OpenFinder(false)
        }
    }

    h.BWindow.Display()  // 先画编辑内容
    if h.finder.IsOpen() {
        h.finder.Display()  // 后叠加 finder overlay
    }
}
```

**关键设计**：finder 是 pane-local overlay，画在 owner pane rect 内，**不会被兄弟 pane / divider / InfoBar / NotePane 的 Display() 覆盖**——它们的 Display 画的是自己的 rect，rect 不同就没有覆盖。

### 5.2 失焦关闭

finder 升格为 owner-local 后，鼠标路由变了：

- 旧 FloatFrame 是主循环级 modal，鼠标在主循环被全局拦下、`Tabs.HandleEvent` 根本不跑
- 新 finder 是 owner-local，`Tab.HandleEvent` 的鼠标路由（点 pane 切焦点 / 拖分割条 / 滚轮）在 owner 的 modal `return` **之上**，点别的 pane 会让 owner `SetActive(false)`、finder 孤儿化

化解：finder 暴露 `NotifyBlur()` 方法（走方法不走 `HandleEvent` 管道——失焦不是 tcell 终端事件），owner 失焦时调它：

```go
// bufpane.go

func (h *BufPane) SetActive(b bool) {
    if h.IsActive() == b { return }
    h.BWindow.SetActive(b)
    if b {
        // gutter + onSetActive 插件钩子（不动）
    } else {
        // 失焦时取消本 pane 的 finder 会话（若开着）
        h.onOwnerBlur()
    }
}

// fileops.go

func (h *BufPane) onOwnerBlur() {
    if h.finder != nil && h.finder.IsOpen() {
        h.finder.NotifyBlur()  // finder 内部 close(Esc)
    }
}
```

**语义等价「点框外取消」**，比「鼠标被锁死」更自然。失焦后 finder 按 Esc 自关，onFinderClose 的 Esc 分支恒 no-op，不会在 `Tab.SetActive` 的 pane 循环里引发突变。

## 6. birth 自动化：pendingBirth 机制

### 6.1 为什么不用 finishInitialize hook

文档 N0/N0a/N0b 计划在 `finishInitialize` 加 hook，但实际实施用了 `pendingBirth` 机制在 `Display()` 中触发。原因：

- `Resize` 后才拿到真实 rect，但 `Resize` 本身可能触发 `EventResize` → finder 的 resize 自关机制
- 若在 `Resize` 末尾（`finishInitialize`）开 finder，可能被随后传来的 `EventResize` 秒杀
- `Display` 在主循环的 `DoEvent` 完后才跑，此时所有 `EventResize` 已处理完

### 6.2 pendingBirth 机制

每个 pane 出生时（`newBufPane`）被授予一次「检查要不要开 birth finder」的许可：

```go
// bufpane.go

type BufPane struct {
    ...
    pendingBirth bool  // 一次性触发许可
}

func newBufPane(...) *BufPane {
    h := &BufPane{...}
    h.pendingBirth = true  // 出生时授予许可
    return h
}

func (h *BufPane) Display() {
    // 消费一次性触发许可
    if h.pendingBirth {
        h.pendingBirth = false  // 立即清（不重入）
        h.isNoName = h.Buf != nil &&
            h.Buf.AbsPath == "" &&
            h.Buf.Type == buffer.BTDefault &&
            h.Buf.Size() == 0
        if h.isNoName {
            h.OpenFinder(false)
        }
    }
    // ... BWindow.Display() + finder.Display()
}
```

**关键时序**：`Display` 在 `DoEvent`（处理 `EventResize`）之后才跑，此时 `OpenFinder` 不会被 finder 的 resize 自关机制秒杀。

### 6.3 birth 起始目录

birth 的 `OpenFinder(false)` 在 noName pane 上调，`h.Buf.AbsPath == ""`，fallback 到 `os.Getwd()`（micro 的 cwd）。

现状 `OpenBirthSelector(pane, dir)` 显式收 spawn 时捕获的父目录；`:cd ~/other` 后 HSplit，新 pane finder 起始 = `dir(当前文件)`。新方案 birth 不传 dir、新 pane 无文件 → fallback cwd。**仅当 micro 的 cwd ≠ dir(当前文件) 时有差**——绝大多数使用下 cwd == dir(当前文件)，用户进 finder 也能跳转，简化换来的微小回归。

## 7. cwd 同步到 shell

### 7.1 问题场景

现状链路：最后一个 pane Quit → microNeo 退出 → `lastWorkingDir()` 读 `MainTab().CurPane().Buf.AbsPath` 的父目录 → 写进 `--cwd-file` → shell 的 `m()` 读它 cd。

**塌陷场景**：`m` 在 A 目录启 microNeo → noName pane → Ctrl-q 开 quit finder → 导航到 B 目录 → 再 Ctrl-q（Quit）→ `h.Quit()` → exit → `lastWorkingDir` 读 noName pane 的 `AbsPath=""` → 返回 `""` → 什么都不写 → m() 留在 A。用户明明在 finder 里走到了 B，shell 却不跟过去。

### 7.2 修法：Result.Cwd → LastFinderCwd → lastWorkingDir

```go
// fileops.go

var LastFinderCwd string  // 包级暂存变量

func (h *BufPane) doQuit(r finder.Result) {
    LastFinderCwd = r.Cwd  // 先暂存
    h.Quit()                // 后调原生 Quit
}

// cmd/micro/micro.go

func lastWorkingDir() string {
    if action.LastFinderCwd != "" {  // finder 会话的导航成果优先
        return action.LastFinderCwd
    }
    // fallback：读当前 pane buffer 的路径
    if t := action.MainTab(); t != nil {
        if pane := t.CurPane(); pane != nil && pane.Buf != nil {
            if ap := pane.Buf.AbsPath; ap != "" {
                return filepath.Dir(ap)
            }
        }
    }
    return ""
}
```

**Result.Cwd 的唯一理由**：把"finder 会话的导航成果"接进 shell 同步链路。务必在代码注释里讲清这个 why，否则后人会当它过度设计。

## 8. FloatFrame：继续服务 SelectPane

FloatFrame 去留与 finder 无关。finder 升格为 pane 级 overlay、自己画框后，FloatFrame 继续服务 SelectPane（`:theme` 用）：

- FloatFrame 是主循环级 modal（`micro.go:585` `else if TheFloatFrame.IsOpen()`）
- SelectPane 用 `AutoExpand=true` 可能越界（`:theme` 锚点 `(0, -1)` 紧贴 InfoBar 上方），必须在主循环画最末才不被覆盖
- finder 用 `AutoExpand=false`、pane 派生 rect，不越界，所以走 owner-local 路由

**两套策略并存**，对应两种 widget 的视觉属性，不冲突。finder 不走 FloatFrame，**不是 FloatFrame 废弃的理由**。

## 10. git 状态子模块

finder 包内唯一一项「外来」能力——micro 全仓库无任何 git 调用经验。封装为独立、可降级的子模块（`git.go`），**任何环节失败都静默降级为「不显示状态列」，绝不阻塞浏览**。

### 10.1 定位与降级哲学

自由函数 `getGitStatus(dir) (isRepo bool, branch string, chars map[string]rune)`：isRepo=false 时 branch=""、chars=nil，调用方降级隐藏 git 列。**无缓存、无接口注入**——每次 selector 打开/chdir 都重新 fork 系统 git（工作树可能被别的进程改，缓存会陈旧；后台几十 ms 不卡首次渲染）。

显示 git 需同时满足：`diffgutter=true` + 在 git 仓库内 + 2s 内返回，任一不满足整列隐藏。

### 10.2 数据流

`Open()/chdirTo` 同步读全部条目（`readDirEntries`，gitChar 全 0）后立即首帧渲染（列表可见、无 git 列，用户可立即浏览）；随后 `go fetchGit(dir)` 后台跑 `getGitStatus`，回来持 Lock 检查 `currentDir==dir`（防 TOCTOU）后把 chars 按文件名合并进各 `allEntries[i].gitChar`，再 `screen.Redraw()` 补 git 列。

降级 ≠ 干等 2s：超时指 goroutine 没回来，列表早显示、git 列始终没填（静默隐藏）。`fetchGit` 回来若已导航离开则丢弃，不污染新 entries。

### 10.3 子目录状态透传（一句话原理）

git status 恒输出**仓库根相对**路径，需用 `rev-parse --show-prefix` 得到的「当前目录相对仓库根」前缀换算成相对当前目录，再**取顶层分量**——子目录内任意深度有变更时，当前列表里对应的顶层条目亮状态。跨平台只切 `/`（git porcelain 在所有平台都发 `/`，勿用 `\`）。

### 10.4 并发模型

`allEntries`（含各 `gitChar`）、`showEntries`（指针）、`isRepo`/`gitBranch` 都是 fetchGit（后台写）与 display（UI 读）的共享结构，`mu`（RWMutex）全保护。display 在 RLock 内读 `isRepo` + 拷出可见 entry 值副本（含 gitChar），**RUnlock 后才画**（画的过程不持锁、不嵌套 RLock，避开 Go RWMutex 不可重入的死锁）。

## 11. 渲染（View，display）

内容区自上而下三段：

1. **行 0 面包屑**：左截断全路径、恒保留「当前目录/」；根目录显 `/`；光标在此行时该行 Reverse。截断在路径段边界，截断处用 `…/` 标记。
2. **行 1..H-2 文件列表**：视口 `[topIdx, topIdx+visibleH)`；每行列结构 `[marker] [name…] [sep?] [right?] [scroll] [git?]`。marker 目录 `▸`/文件空格；name 目录带 `/` 后缀，超长中间截断保扩展名（rune-safe）；right 随 `rightMode`（当前恒 size=5 列 humanSize 右对齐）；scroll 溢出时首/末可见行覆盖 `▲`/`▼`；git 仅仓库内预留 1 列。选中行整行 Reverse。
3. **末行元数据**：权限（`fi.Mode().String()`）+ size（`humanSize(fi.Size())`）+ mtime（`formatMtime(fi.ModTime())`）。窄屏按「权限→size」依次砍、mtime 保底。

配色：目录名 colorscheme `type` 色；文件名 `default`；git 状态按 `diff-modified`/`diff-added`/`diff-deleted`；边框零值 → `config.DefStyle`。写屏一律按 CJK 显示列宽推进列坐标（`runeWidth`/`stringWidth`）。

## 13. 关键不变式（勿违反）

1. **isNoName 是 sticky 出身，不是缓存**。Picked 换入后不清。QuitNeo 用 isNoName（出身）判分流。只管 Ctrl-q，不看 Ctrl-o。
2. **isNoName 三条件用 AND，含 Size()==0**。勿改 `len` 判空。
3. **intent-agnostic**：finder 内部禁止出现任何 `if isQuit` / `switch isQuit`。
4. **finder 不能 import action**：依赖关系是 `action → finder`，反向成环编译报错。
5. **modified 检查走原生，不自写**。Picked → `OpenCmd`、Quit → `h.Quit()`、Esc → 不检查。
6. **display 锁边界**：git 共享结构由 `mu` 全保护；display 必须「RLock 内拷值 → RUnlock 后画」，画的过程不持锁（RWMutex 不可重入）。
7. **目录只读一次**：`readDirEntries` 只在 chdir 调；toggleHidden/sort 切换只 `rebuildShow`/`sortEntries`，不重读目录、不重取 git。
