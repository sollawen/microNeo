# F1 · FileSelector 架构设计方案

**状态**：v1 草案。本文是 FileSelector（`:file`）的**架构**方案，承接 F0（产品）/ F0a（框架改造），指导后续详细设计与实现。
**范围**：只讲架构——模块划分、职责边界、关键契约、数据流、侵入点、风险。**不讲**逐函数实现、不贴大段代码。
**F1** = Feature 1，FileSelector 的技术架构。F0 = 产品方案，F0a = FloatFrame 重构（Open 签名 / layout 开关 / resize 即关）。

## 3. 架构总览

### 3.1 分层

```
┌─────────────────────────────────────────────────────────┐
│  命令层   :file → FileCmd（command_neo.go 注册）          │
│           · modified 保护（YNPrompt）                    │
│           · 取当前 pane、算 layout、算起始目录               │
├─────────────────────────────────────────────────────────┤
│  具体浮窗  FileSelector（新）  ← 三层分离                 │
│   ┌──────────────┬─────────────┬──────────────────┐     │
│   │ State        │ View        │ Controller       │     │
│   │ 目录/条目/   │ display()   │ handleEvent()    │     │
│   │ 光标/滚动/   │ 面包屑/列表/│ 键位→改 State /   │     │
│   │ showHidden   │ hint 三段式 │ Enter 上下文     │     │
│   └──────────────┴─────────────┴──────────────────┘     │
│   附属：layout 计算（pane-local） · git 状态子模块           │
├─────────────────────────────────────────────────────────┤
│  容器层   FloatFrame（现有，经 F0a 重构）                 │
│           模态事件路由 · 画框 · 后写胜出 · resize 即关    │
│           · onCancel 钩子（F0a）                         │
├─────────────────────────────────────────────────────────┤
│  底座    micro 主循环（micro.go，零改动）                 │
└─────────────────────────────────────────────────────────┘
```

### 3.2 数据流（一次完整的"打开→选中→开文件"）

```
用户键入 :file
  → FileCmd(h)
       ├─ h.Buf.Modified()? → YNPrompt：y=保存后继续 / n=取消（F0 §8）
       ├─ pane = 发起 :file 的那个 BufPane（captured）
       ├─ 起始目录 = filepath.Dir(h.Buf.AbsPath)（无路径 fallback cwd）
       ├─ layout 预检（W<minWidth / avail<minHeight → InfoBar 提示并返回）
       └─ NewFileSelector().Open(pane, startDir, onSelect=打开文件)
            ├─ State 初始化（列目录、排序、定位当前文件、dotfile 自动显；gitStatus=空）
            ├─ 算 anchor+contentSize（§7）→ TheFloatFrame.Open(FloatOpenSpec{
            │      Anchor, ContentSize, Title:"Open File", Display, HandleEvent,
            │      OnCancel, AutoExpand:false})
            ├─ Open 返回 false → onSelect(nil) 透明返回（picker 没开，不查 git）
            └─ Open 成功 → 后台启 git 查询（§10.7，渐进显示）：
                  go func() { m,ok := gitCache.statusFor(dir)        // 带 2s ctx（§10.4）
                              fs.state.setGitStatus(m,ok); screen.Redraw() }
                  // 首次 Display 时 gitStatus 已空 → 列表可见但无 git 标志；
                  //   git 回来 → Redraw → 列表补 git 标志（光标/滚动在 State，不受影响）

[运行期] 主循环 micro.go:544-549 把键事件交给 FloatFrame.HandleEvent
  → FloatFrame.handleEvent（FileSelector 的 Controller）
       ├─ ↑↓ → 改 State.cursor / 滚动 → Redraw
       ├─ ←/→/Enter(面包屑)/Enter(目录)/Enter(文件) → 导航或选中
       ├─ . → 翻转 showHidden → 重列条目 → Redraw
       └─ Esc → Close + onSelect(nil)
  → FloatFrame.Display → FileSelector.display（View，读 State 画三段）

resize 到达 → FloatFrame 拦截（F0a）→ Close + onCancel → onSelect(nil)

选中文件：
  Enter(文件行) → Close + onSelect(&path)
    → onSelect 闭包：NewBufferFromFile → capturedPane.OpenBuffer(b)
       （与 :open 同路径，文件开进发起 :file 的那个 pane）
```

## 4. 核心架构决策

### D1 · FileSelector 作为 FloatFrame 浮窗（复用 modal host）

**决策**：FileSelector 是一个 FloatFrame 具体浮窗，与 SelectPane 同构——把 `display` / `handleEvent`（及 F0a 后的 `onCancel`）以函数值塞给 `TheFloatFrame.Open`。

**否决的方案**：
- 另起 overlay 体系（像 NotePane 那样自管）——需要在 `micro.go` 的事件路由和绘制顺序里加新分支，**侵入主循环**，违背"原生零侵入"原则。
- 新建独立 frame 容器——重复 FloatFrame 已有能力（模态、单例、画框、后写胜出），且 F0a 已明确把 FileSelector 列为 FloatFrame 的未来浮窗。

**收益**：事件路由、绘制顺序、单浮窗互斥、resize 即关（F0a 后）**全部白拿**，`micro.go` 零改动。

**代价/张力**：FloatFrame 现有 layout 是"锚点自适应展开"（服务于 SelectPane 那种"贴光标弹小窗"），而 FileSelector 要"pane 派生的确定性矩形"。→ 见 D2 化解。

### D2 · layout 由 FileSelector 自算（pane-local），传 `AutoExpand=false` 钉死左上角

**决策**：FileSelector 自己从 pane 读 `View`、套用 F0 §4 公式算出 picker 的屏幕矩形（左上角 + 内容尺寸），通过 `FloatOpenSpec` 传给 FloatFrame，并显式设 **`AutoExpand=false`**。FloatFrame 据此**跳过 `expandAnchor`、直接采用 `Anchor` 作外矩形左上角**——不做任何二次 layout 决策。

**为什么不用默认的 `AutoExpand=true`**：`expandAnchor`（`floatframe.go:228`）是"给一个参考点、自适应上下左右展开"的模型，服务 SelectPane 那种"贴光标弹小窗"。FileSelector 要的是 pane 派生的确定性矩形，让 FloatFrame 展开 = 让它二次决策一个我们已经定死的 layout，既无意义又脆弱——旧版 D2 靠"传一组恰好让 expandAnchor 变 no-op 的输入"实现，意图隐式，改宽度公式就可能翻车。`AutoExpand=false` 把意图变显式，根除该脆弱性（原 R1 风险据此删除）。

**职责划分（干净）**：
- FloatFrame：失败前置检查（outer 是否放得下屏）+ 画框 + 生命周期 + 事件路由。**不碰 pane 边界**（它的 `bottomLimit` 是屏幕级的，不懂 pane）。
- FileSelector：算 pane 派生的左上角 + 尺寸 + pane-local 预检。**"不溢出到别的 pane" 100% 由 FileSelector 把 `outerH` 钉成 pane 的 `avail`、`outerW` 钉成 `pickerW ≤ paneW` 保证**——与 FloatFrame 无关。哪怕屏幕下方还有大把空间（如 hsplit 上半 pane），picker 也只长到 `avail` 高。

**预检（拒绝打开条件，F0 §4.1/§4.2/§9）**：
- `pane.W < minWidth(20)` → 提示 pane 太窄，拒开。
- `avail = pane.Height - 1 < minHeight(10)` → 提示 pane 太矮，拒开。
- 通过后才进 `Open`。

### D3 · State/View/Controller 三层分离（为 v1.1 fuzzy 铺路，对齐 F0 §10.2）

**决策**：FileSelector 内部明确三层（不强制三文件，但职责清晰分离）：

| 层 | 职责 | 不做什么 |
|----|------|---------|
| **State** | 当前目录、条目切片（已排序/已过滤）、cursor、topIdx（滚动）、showHidden、picker 宽度（用于截断） | 不碰屏幕、不读键 |
| **View** `display(area)` | 读 State，画面包屑（行0）+ 文件列表（视口）+ hint（末行）；左截断、Reverse 光标、颜色 | 不改 State |
| **Controller** `handleEvent(ev)` | 读键 → 改 State（移动/导航/翻转显隐）或触发选中/取消 | 不直接画屏（只 `Redraw`） |

State 就是 FileSelector 的"当前状态/数据"——读它就知道该画哪些条目、光标停在第几个。不沿用"Model"一词，避免被读成"模型"。v1.1 fuzzy 只需在 State 上加 `filter string` + 过滤后的条目视图，View 几乎不动，Controller 加"可打印字符 → 追加 filter"。这是 F0 §10.2 "架构已铺路"的兑现。

### D4 · git 状态作为独立、可降级的子模块

**决策**：文件级 git 状态（F0 §7.5）封装为独立子模块，向 FileSelector 提供一个简单查询接口；任何环节失败（非仓库、超时、`diffgutter` 关闭）都**静默降级为"不显示状态列"**，绝不阻塞浏览。

**理由**：
- git 状态的成本模型（fork 进程 + 全仓索引比对 + 未追踪扫描）和读目录本质不同，需独立生命周期（异步/超时/降级）——详见 §10。
- 正常仓库几十 ms（fork 系统 git + pathspec 限定，和 eza 同档）；2s 是病理安全网，非预期耗时。
- micro 全仓库无 git 调用经验，是最"外来"的能力，隔离以便单独测试/回退。

详见 §10。

### D5 · 前置依赖：F0a（FloatFrame 重构）

**决策**：F0a 是 FileSelector 的**硬前置**，一次过交付三项 FileSelector 都要用的能力：
- `FloatOpenSpec`（options 入参）——FileSelector 的调用点。
- `AutoExpand=false`（layout 开关）——见 D2。
- `onCancel` + resize 即关——FileSelector 不自写 resize 分支，完全继承 FloatFrame 的统一关闭。

**顺序**：F0a 作为独立重构先行合入（SelectPane 是现成的回归验证对象）；FileSelector 在 F0a 之后实现。

**契约对齐**：FileSelector 的 `OnCancel` = “调 `onSelect(nil)`”（等同 Esc/取消），与 SelectPane 一致。`OnCancel` 内部不得再调 `TheFloatFrame.Close()`（F0a §4.3 重入约束）。

### D6 · 零侵入命令注册与配置注入

**决策**：
- `:file` 命令经 `InitNeoCommands()` → 原生 `MakeCommand` 注册（与 `:theme` 同通道），不改原生 `commands` map。
- 新设置 `fileselectwidth`（默认 0.4）直接加进原生 `settings.go` 的 `defaultCommonSettings` + `runtime/settings.json`（沿用 `status-separator` 先例，各一行）。microNeo 的设置项统一走这条路、一视同仁，不为单字段单建 `settings_neo.go`。
- `diffgutter` 直接复用原生设置，不新增。

**理由**：`status-separator` 已是此范式（`settings.go:102` + `settings.json:11`，均为 neo 改动）；neo 后续还会加各种参数，统一一处、降低碎片化，比为每类参数单建文件清爽。代价是对原生 `settings.go` 的一行 map 注入——极小侵入，可接受。

详见 §11、§12。

## 6. 关键接口契约

### 6.1 FloatFrame.Open（经 F0a 重构为 `FloatOpenSpec`）

```
Open(spec FloatOpenSpec) bool
```

`FloatOpenSpec` 字段（详见 F0a §3）：`Anchor` / `ContentSize` / `Title` / `FrameColor` / `Display` / `HandleEvent` / `OnCancel` / `AutoExpand`。

FileSelector 的填法：
- `Anchor = pane 左上角`（`AutoExpand=false` 时 = 外矩形左上角，含边框）。
- `ContentSize = (pickerW-2, avail-2)`。
- `Title = "Open File"`。
- `OnCancel = func(){ onSelect(nil) }`——FloatFrame 自身被关（resize）时回调，清理业务回调。
- `AutoExpand = false`——见 D2。
- 返回 `false`（已被占 / 屏幕放不下）时，FileSelector 不得留状态，直接 `onSelect(nil)`。

### 6.2 FileSelector.Open（对调用方）

```
Open(pane *BufPane, startDir string, onSelect func(path *string))
```
- `pane`：发起 `:file` 的 BufPane（layout 来源 + 文件开进去的目标）。
- `startDir`：起始目录（由 `FileCmd` 算好：`filepath.Dir(AbsPath)`，无路径用 cwd）。
- `onSelect`：选中→`onSelect(&path)`；Esc / resize / 拒开→`onSelect(nil)`。契约：**每次 Open 恰好一次 onSelect 回调**（与 SelectPane 一致）。
- 内部：建 State → layout 预检 → 调 `TheFloatFrame.Open(...)`。

### 6.3 display / handleEvent（塞给 FloatFrame 的函数值）

- `display(area Rect)`：area 为 FloatFrame 给的纯内容区（不含边框）。View 在此区内画三段。
- `handleEvent(ev tcell.Event)`：只处理键事件；resize 不会到达（FloatFrame 已拦截）。

### 6.4 git 子模块查询接口（建议）

```
type gitStatusCache interface {
    statusFor(dir string) (map[string]statusKind, bool)
    // false = 不可用（非仓库/超时/被关），调用方降级为不显示
}
```
- 实现可缓存"最近一次目录"的结果；导航切目录时重算。
- **调用契约（渐进显示）**：`statusFor` 本身是**阻塞**查询（内部 fork git、最多等 2s ctx）。调用方**不得在首次渲染路径上同步调它**——必须在 `Open` 成功后用 `go func()` 后台调、完成后 `screen.Redraw()` 通知重绘。首次 Display 读到的是空 gitStatus（列表照常显示），git 回来后 Redraw 把标志补上（见 §3.2、§10.7）。
- 降级：拿到 `false`（非仓库/超时/失败）就当无 git，整列静默不显示，不报错。

## 7. Layout 计算

### 7.1 输入

从发起 pane 读（参照 `notepane.go` 的 `reposition()` 取 `pane.BWindow.GetView()` 的范式）：

```
view := pane.BWindow.GetView()       // *View{X,Y,Width,Height}，含 gutter 与 statusLine 行
paneX, paneY   := view.X, view.Y
paneW          := view.Width
avail          := view.Height - 1    // 让出 statusLine（statusLine 占 BufWindow 末行）
```

**F0 §4.1 "盖住 gutter"**：picker 左缘 = `paneX`（pane 绝对左缘，含 gutter 区），不是 `BufView().X`（内容区）。对全屏 pane `paneX==0`，即 F0 文中的 "x=0"。

### 7.2 宽度（F0 §4.2）

```
widthFrac := config.GetGlobalOption("fileselectwidth").(float64)   // 默认 0.4
pickerW   := max(int(widthFrac * paneW), minWidth=20)
// 拒开条件：paneW < minWidth
// 不变式：pickerW ≤ paneW（由 widthFrac<1 + min 兜底保证）
```

### 7.3 高度（F0 §4.1 全高模型）

```
pickerH(outer) := avail              // 恒等，不自适应条目数
// 拒开条件：avail < minHeight=10
```

### 7.4 换算成 FloatFrame 入参

```
// 组装 FloatOpenSpec（字段语义见 F0a §3）
spec := FloatOpenSpec{
    Anchor:      Pos{X: paneX, Y: paneY},   // AutoExpand=false 时 = 外矩形左上角(含边框)
    ContentSize: Size{W: pickerW - 2, H: avail - 2},  // 减去 FloatFrame 左右/上下边框
    Title:       "Open File",
    Display:     fs.display,
    HandleEvent: fs.handleEvent,
    OnCancel:    func() { if fs.onSelect != nil { fs.onSelect(nil) } },
    AutoExpand:  false,                     // 钉死 Anchor, 跳过 expandAnchor（D2）
}
// → outerW = pickerW, outerH = avail（title 不撑宽：9+6=15 < minWidth(20) ≤ pickerW）
```

### 7.5 内容区三段布局（View 用）

内容区高 = `avail - 2`，自上而下：
- 行 0：面包屑（F0 §6）。
- 行 1 .. N-2：文件列表视口（可滚动）。
- 末行：hint（F0 §7）。

**无退化分支**：`minHeight=10`（§7.3）保证外高 avail ≥ 10，内容区恒 ≥ 8 行（面包屑 + ≥6 行文件列表 + hint），三段布局始终放得下。View 内不写"放不下三段"的退化。

## 8. 生命周期与事件流

### 8.1 打开（`FileCmd`）

1. modified 保护（F0 §8）：`h.Buf.Modified() && !h.Buf.Shared()` → `InfoBar.YNPrompt`；**y=保存后继续，n=取消**（注意：`:open` 的 n 是"不保存也继续"，`:file` 的 n 是"取消"——两者不同，详见 F0 §8）。
2. 算起始目录、读 pane layout、§7 预检。
3. `NewFileSelector().Open(pane, startDir, onSelect)`。

### 8.2 运行期（Controller）

键位映射（F0 §5）落到 `handleEvent` 的 switch：

| 键 | State 变化 | 副作用 |
|----|----------|--------|
| ↑/↓ | cursor ±1（不循环，clamp 到 [0, 末]） | 滚动跟随 + Redraw |
| Enter（面包屑行） | currentDir ← 父目录 | 重列 + 重定位 + Redraw |
| Enter（目录条目） | currentDir ← 该目录 | 同上 |
| Enter（文件条目） | — | Close + `onSelect(&path)` |
| ← | currentDir ← 父目录（不移光标） | 重列 + Redraw |
| → | 若光标在目录 → 进入 | 重列 + Redraw |
| . | showHidden 翻转 | 重列（光标 clamp）+ Redraw |
| Esc | — | Close + `onSelect(nil)` |
| 其它键 | 吞掉（modal） | — |

hint 文案随光标位置 + showHidden 翻转（F0 §7.1–7.3）。

### 8.3 关闭

- 选中文件 / Esc：Controller 内走 "存 `onSelect` → `TheFloatFrame.Close()` → 调 `onSelect`"（与 SelectPane 同序）。
- resize：FloatFrame 拦截 → `Close()` → 调 `onCancel` → `onSelect(nil)`（F0a）。
- `Close()` 后下一帧主循环回到无浮窗态，主编辑器 pane 自然重画（`micro.go` 既有行为）。

### 8.4 光标起始（F0 §5.3）

开 State 时：在条目里找当前 buffer 文件名；找到→停其上；当前文件是 dotfile 且默认隐藏→`showHidden=true` 后停其上；找不到/无路径→首条目。

## 9. 渲染（View）

`display(area)` 内三段：

1. **面包屑行**（`area.Y`）：左截断全路径，恒保留 `当前目录/`；根目录显 `/`（F0 §6.2）。光标在此行时该行 Reverse（与文件行选中态一致）。
2. **文件列表**（`area.Y+1 .. `）：视口 `[topIdx, topIdx+visibleH)`；每行格式 `[标记] 名[/] [git]`：
   - 标记：目录 `▸`、文件空格（预留 1 字符位）。
   - 名：目录带 `/` 后缀；超长**左截断保留扩展名**（F0 §4.2）。
   - git 状态：靠右、单字符、按 F0 §7.5 颜色；无 git 时整列隐藏。
   - 选中行 Reverse；溢出时右侧画 `▲`/`▼`（对齐 SelectPane 的滚动指示范式）。
3. **hint 行**（末行）：上下文文案（F0 §7.3）。

边框 / title：
- title 固定 `"Open File"`，由 FloatFrame 嵌入上边框（`────Open File────...─`）。
- 与面包屑不冲突：title 在上边框那行，面包屑在内容区行 0，是相邻两行、各管各的（F0 §6.1 的"面包屑不在 title 上"仍成立）。
- 边框色：传 `tcell.Style{}`（零值）→ FloatFrame 用 `config.DefStyle`。
- 内容区配色：目录名用 colorscheme `type`；文件名用 `default`；git 状态按 §7.5 表。

实现细节（截断保留扩展名、双宽字符对齐、Reverse 连续性等）留详细设计；架构层只锁定"三段 + 颜色来源 + 复用 SelectPane 的滚动指示范式"。

## 10. git 状态子模块

git 状态是 FileSelector 里**唯一一项"外来"能力**——micro 全仓库没有任何 git 调用经验。它的成本模型和其它能力（读目录、画 UI、算 layout）**本质不同**，所以单独成章：既要让它能用、好用，又要把风险完全隔离，不让它拖垮浏览体验。

### 10.1 成本模型：git 状态和"读目录"不是一回事

State 初始化时读目录用的是 `os.ReadDir`——一次系统调用、只看当前目录、纯内存判断类型（`d_type`），**μs 级**，同步塞进 `Open()` 毫无压力。

git 状态（我们用 `git status --porcelain -- .`，**带 pathspec**）成本模型如下：

| | `os.ReadDir`（目录列表） | `git status -- .`（git 状态，我们的用法） |
|---|---|---|
| 怎么拿 | 1 次系统调用 | **fork 一个 git 进程** |
| 范围 | 当前这一个目录 | 读 `.git/index`（一个文件，全仓追踪清单，**不可避免**）+ 实际比对/扫描**只在当前目录** |
| 读不读文件内容 | 不读 | 仅当前目录里 **stat 变了的**文件才哈希（git 内部确认改动） |
| 成本 ∝ | 当前目录条目数（几十） | index 大小 + 当前目录文件数 + 当前目录改动量 |
| 典型耗时 | 5–20 μs | 几十 ms（正常）/ 秒级（巨型 index 冷加载 或 当前目录本身巨多文件） |

git 在当前目录里干这些：① 加载 `.git/index`（全仓追踪清单，**这一个文件绕不开**，大仓几 MB，但 OS 缓存后更快）；② 对当前目录的已追踪文件做 `mtime/size` 比对——没改过的走 stat 快路径秒过，**stat 变了的**才重新哈希整文件确认；③ 扫当前目录的未追踪文件 + 用 ignore 规则匹配。**pathspec 把 ②③ 钉死在当前目录**，不波及仓库其它部分。

**为什么 stat 变了要哈希？** git 先用便宜的 stat（mtime/size/mode）和 index 比对：一致 → 认定没改，不哈希（快路径，绝大多数文件走这条）；不一致 → 文件**可能**改了，但 stat 不可靠（`cp -p` 保 mtime、内容变了 size 可能没变、`touch` 刷 mtime 但内容没变），所以 git **重新哈希整文件内容**、和 index 存的 blob hash 比，才能 100% 确定改没改——宁可多哈希一次，也不漏报改动。这是 git 内部的正确性机制，不是我们能选的；我们只是跑 `git status`，哈希谁由 git 决定。

**正是这套不同的成本模型，决定了 git 状态必须有独立的生命周期**（异步、超时、降级），不能像读目录那样同步塞进 State init。

### 10.2 机制：fork 系统 git + pathspec 限定到当前目录

```
git -C <currentDir> status --porcelain=v1 -z -- .
```

- **`-- .`（pathspec）**：把"工作"也限定到当前目录——不只过滤输出，未追踪扫描也只走这一棵子树。这正是 eza 快的做法：只关心列出来的那些文件。
- **`-C <dir>`**：在目标目录下跑，省去自己 chdir。
- **`--porcelain=v1 -z`**：机器可解析、`NUL` 分隔（文件名含空格/换行也安全）。
- **用系统装的 git**：零新依赖、版本永远跟用户一致、自动尊重用户的 `config` / `.gitignore` / submodule / hook。

**职责边界（重要）**：上面成本模型里那些（index/stat/哈希/扫描）**全是 git 进程内部干的，我们的代码一行都不碰**。FileSelector 对 git 状态的全部工作就是：`os/exec` 起 git → 读 stdout → 解析两字符状态码 → 显示。**我们从不读文件内容、不算哈希、不碰 `.git/index`**——git 对我们是黑盒，只消费它的输出文本，和 `ls` 给文件名列表没本质区别。之所以关心 git 的内部成本，只因为它决定我们要**等多久**（所以有 2s 超时），不是我们要去实现它。

**状态码映射**（porcelain 两字符 `XY` → 显示用一个标记）：

| porcelain | 含义 | 显示 |
|---|---|---|
| `??` | 未追踪 | **U** |
| 含 `M`（` M` / `M ` / `MM`） | 已修改 | **M** |
| `A ` | 已暂存新增 | **A** |
| `D ` / ` D` | 删除 | **D** |
| `R ` | 重命名 | **R** |
| `!!` | 被 ignore（需 `--ignored`） | **I**（见 §10.5） |
| 全空格 | 干净 | 无标记 |

### 10.3 被否决的方案：进程内（libgit2 / go-git）

"既然 eza 进程内用 libgit2 更快，microNeo 为什么不也进程内？"——评估过，否决：

- **libgit2（经 cgo + git2go）**：编译产物 ~1MB，但要给纯 Go 项目引入 **cgo 构建**（交叉编译、CI 都变复杂）、绑定 libgit2 版本。
- **go-git（纯 Go 重写）**：体积**几 MB**，还得自己追 git 格式变化、处理用户 config/submodule 的怪癖。

换来的收益：每次 `:file` 省 **1–5ms** 的进程启动——而 **eza 自己每次也是冷启动一个新进程**，证明这点开销人眼无感。**为无感的收益引入有感的依赖维护成本，不划算。** fork 系统 git 是更干净的选择。本决策记录在此，不再重复评估。

### 10.4 性能与降级：2s 是安全网，不是预期耗时

**正常情况就是快的**——几百文件的仓库、缓存热（一直在编辑、没什么大批量改动）：几十到一两百 ms，和 eza 同档，人眼"瞬间"。这一点不要被"git 状态可能很慢"的直觉带偏。

**2s 超时是病理情况的安全网**：跑到秒级通常因为 ① 巨型仓库的 `.git/index` **冷加载**（几万文件 → index 几 MB~几十 MB、冷盘慢），或 ② **当前目录本身**有成千上万文件（如在 `node_modules` / 构建产物里开 `:file`）。NFS / 网络盘会放大两者。那种情况下 eza 一样会卡。

降级链（显示 git 状态需**同时**满足，F0 §7.5）：
- `config.GetGlobalOption("diffgutter")` 为 true；
- 当前目录在 git 仓库内；
- `git status` 在 **2s** 内返回。

任一不满足 → **整列静默隐藏**（不报错、不阻塞浏览）。超时降级后，**本次会话内该目录不再重试**（避免每次导航都卡 2s）。实现用 `exec.CommandContext` + 2s ctx。

### 10.5 I（Ignored）的延后

`--porcelain` 默认**不列 ignored 文件**，I 状态需要额外 `--ignored`——它会**额外扫一遍被忽略的文件**，开销更大，且 dotfile 默认隐藏时 I 触发面很小。

→ v1 可先只做 **M/U/A/D/R**，I 视成本延后（仅 dotfile 显隐时启用 `--ignored`）。详见 §13 R4。

### 10.6 缓存与解耦

- **缓存**：以 `currentDir` 为 key 缓存最近一次结果；导航切目录即失效重算。**不做文件系统 watch**（YAGNI——浏览会话短，用户不会开着 `:file` 等外部改动）。
- **解耦**：子模块经 §6.4 的接口注入 FileSelector，便于 mock 掉 git 调用做单测、将来替换实现（如换成 go-git 或加 fs watch），都不波及 FileSelector 主体。

### 10.7 异步时序与渐进显示

git 查询**绝不阻塞首次渲染**——这是 §6.4 调用契约的落脚点。时序：

```
1. Open() 同步：State init（os.ReadDir，μs 级）→ gitStatus = 空
2. FloatFrame.Open 成功 → 首次 Display：列表已可见、无 git 标志（用户可立即浏览）
3. go func() 后台：fork git status（带 2s ctx，§10.4）
4. git 返回 → state.setGitStatus(...) → screen.Redraw()
5. 主循环下一帧 Display：列表补上 git 标志（光标/滚动在 State，重画不打断浏览）
```

关键点：
- **首次渲染不等 git**——列表用空 gitStatus 先画出来，用户立刻能上下选。
- **git 回来靠 `screen.Redraw()` 触发补画**——micro 现成机制（`internal/screen/screen.go:52`，线程安全；先例 `internal/buffer/buffer.go:1027-1034` 的 `go func(){ …; screen.Redraw() }`）。
- **降级 ≠ 干等 2s 才显示**：§10.4 的"超时降级"指第 3 步的 goroutine 到 2s 没回来，列表在第 2 步**早已显示**、git 列始终没填上（静默隐藏）——不是让用户对着空白等 2s。
- **导航切目录**：每次切目录按此时序重来（新目录 git 后台重查、老结果作废，见 §10.6）。
