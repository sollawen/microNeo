# Z0 · 多文件操作架构总览

**状态**：已实现并测试通过。本文是 microNeo 多文件操作（pane ↔ tab 互转、分屏比例控制）的**架构指导文档**——给读者（含 AI）快速建立全貌心智模型：整体约束、核心概念、行为契约、代码约束。具体实现细节（函数签名、算法细节、边界条件等）一律读代码，本文只讲「为什么这样设计、哪些是必须遵守的约束」。

---

## 0. 一句话

microNeo 把 micro 的分屏/标签从「任意多 pane」收束到「每 tab 最多 2 pane 的对照模式」，用 `:big` / `:small` 两个命令实现 pane ↔ tab 互转，用 `Alt-=` / `Alt--` 在三档比例（25:75、50:50、75:25）之间步进，到达边界后自然溢出到 pane ↔ tab 转换。

---

## 1. 整体约束与设计哲学

### 1.1 每 tab 最多 2 pane

**不变量**：无论从键位、命令、help、log、启动期进入，一个 tab 永远只允许 ≤ 2 个 pane。

**实现位置**：`VSplitBuf` / `HSplitBuf`（`internal/action/bufpane.go:743 / 755`）顶部主卡口：

```go
func (h *BufPane) VSplitBuf(buf *buffer.Buffer) *BufPane {
    if h.tab != nil && len(h.tab.Panes) >= 2 {
        InfoBar.Message("already 2 panes in this tab")
        return h  // 返回当前 pane，调用方不会 NPE
    }
    return h.VSplitIndex(buf, h.splitRight)
}
```

**为什么一处卡口就能覆盖全部入口**：所有「开一个 pane」的入口（键位 Ctrl-t、`:vsplit`/`:hsplit`、`:help` 多主题、`:openlog`、启动期 `InitTabs`、Lua 插件）最终都汇聚到 `VSplitBuf`/`HSplitBuf`。

### 1.2 对照模式 vs 切碎屏幕

**产品定位**：microNeo 的分屏 = 「对照模式」，一个 tab 只允许两个 buffer 并排看/改；要同时开更多文件 → 开更多 tab（快速切换），要并排不同程序 → 交给 tmux/zellij。这和 FileSelector、birth selector、quit selector 一脉相承，都是「专注一个 buffer + 快速切换」，而非「把屏幕切碎」。

**推论**：
- 三档比例（25:75、50:50、75:25）只在 2 pane 时有语义
- 一旦允许 3 pane，「当前 pane 的兄弟」就不止一个，档位语义崩塌
- 所以「砍到 2」和「Alt-=/Alt--」是配套的，不是两个独立特性

### 1.3 micro 原子 + microNeo 编排

pane ↔ tab 互转不是新增底层原语，而是把 micro 已有的原子（`NewTabFromPane`、`AddTab`、`RemoveTab`、`Unsplit`、`SetTab`、`SetID` 等）按特定顺序编排。micro 缺的是「跨 TabList ↔ Tab 内 split tree 边界」的上层动作，不是缺原子。

## 2. 三档比例控制

### 2.1 比例集合与步进

**比例集合**：`S = {0.25, 0.5, 0.75}`（当前 pane 自身占父容器的份额）。

**步进键位**：
- `Alt-=`：扩大当前 pane（向下一档更大的比例）
- `Alt--`：缩小当前 pane（向下一档更小的比例）

**边界行为**（见 §4）：
- 到 75% 再按 `Alt-=` → 溢出到 `:big`（promote 到独立全屏 tab）
- 到 25% 再按 `Alt--` → 溢出到 `:small`（demote 到新 tab 或 absorb）

### 2.2 像素比较 vs 比例比较

pane 尺寸是整数像素，`ResizeSplit` 取整后实际落点会偏离名义档位（例如 total=82、目标 75%，下 pane 实际是 62/82 ≈ 0.756）。若拿比例去比 `0.756 > 0.75`，缩小键会把目标又指回 75%，而它已经在 75% → 卡死。

**所以档位要换算成「像素」，且必须照搬 `ResizeSplit` 的取整方式**：

```go
stepPixel := func(r float64) int {
    if isFirst {
        return int(r * float64(total))        // 上/左 pane：ResizeSplit 直接截断
    } else {
        return total - int((1-r)*float64(total))  // 下/右 pane：取补
    }
}
```

两个分支不同的原因：`ResizeSplit` 把 `size` 永远装给 `children[0]`（上/左 pane），所以 children[0] 的尺寸是 `int(r*total)`（截断），children[1] 是 `total - int((1-r)*total)`；两者都 ≠ `round(r*total)`，且不能同时等于 round（两个 round 之和可能 ≠ total）。必须按当前 pane 是第几个分别算，否则差 1 像素就会卡住。

### 2.3 实现位置

- `stepPaneRatio(grow bool) (int, paneLimit)`：计算下一档位或边界
- `GrowPane()` / `ShrinkPane()`：`Alt-=` / `Alt--` 的执行体
- 代码位置：`internal/action/command_neo.go:239-327`

## 3. :big / :small 命令总览

### 3.1 两个命令，四个场景

| 命令 | 场景 | 条件 | 行为 | 焦点去向 |
|---|---|---|---|---|
| **:big** | 1 | 当前 tab ≥ 2 pane | 把活动 pane 拆出去，包成新 tab 占满 | 新 tab |
| **:big** | 2/3/4 | 当前 tab = 1 pane | no-op + 提示 "pane already at max" | — |
| **:small** | 1 | 当前 tab ≥ 2 pane | 把活动 pane 拆出去，包成新 tab，原 tab 仍 active | 原 tab |
| **:small** | 2 | 当前 tab = 1 pane + 有其他 tab（单 pane） | 从另一个 tab 吞一个 pane 进来（HSplit），源 tab 删除 | 原 tab |
| **:small** | 3 | 当前 tab = 1 pane + 有其他 tab（2 pane） | 从另一个 tab 吞一个 pane 进来（HSplit），源 tab 剩 1 pane | 原 tab |
| **:small** | 4 | 当前 tab = 1 pane + 无其他 tab | HSplit 出新空 pane（等价原生 `:hsplit`） | 原 tab |

### 3.2 核心语义

`:big` / `:small` 描述的是**活动 pane 的「显隐状态」**，不是字面尺寸：

- `:big` = pane 被 promote 到 active tab（可见 = 大）
- `:small` = pane 被 demote 到非 active tab（被遮挡 = 小）

场景 1 里两个命令产出**完全一样的 TabList 拓扑**（paneA 在单 pane 新 tab），唯一区别在 active tab。`:small` 的场景 2/3/4 才是「凑成 2-pane 对照模式」的语义。

## 4. 边界溢出：Alt-=/Alt-- 的自然延伸

### 4.1 设计思想

`Alt-=` 是「放大」，放到极限（75%）再按就「放成全屏 tab」；`Alt--` 是「缩小」，缩到极限（25%）再按就「缩走」。连续按键从档位 stepping 自然过渡到 pane ↔ tab 转换，无需新键位、无需记新命令。

### 4.2 paneLimit 类型

`stepPaneRatio` 返回 `(int, paneLimit)`，`limit` 直接标明边界状态：

```go
type paneLimit int

const (
    limitNone paneLimit = iota  // 正常档位，用 size resize
    limitMax                    // 已到放大极限（75% 或单 pane 全屏）
    limitMin                    // 已到缩小极限（25% 或单 pane）
)
```

### 4.3 溢出逻辑

```go
func (h *BufPane) GrowPane() bool {
    size, lim := h.stepPaneRatio(true)
    if lim == limitMax {
        return h.BigPane()    // 溢出到 :big
    }
    h.ResizePane(size)
    return true
}

func (h *BufPane) ShrinkPane() bool {
    size, lim := h.stepPaneRatio(false)
    if lim == limitMin {
        return h.SmallPane()   // 溢出到 :small
    }
    h.ResizePane(size)
    return true
}
```

### 4.4 边界语义验证

| 当前状态 | 按键 | 行为 |
|---|---|---|
| 75% 档位 | `Alt-=` | `limitMax` → `:big` → promote 到全屏 tab，焦点跟随 |
| 25% 档位 | `Alt--` | `limitMin` → `:small` → demote 到新 tab，焦点留下 |
| 单 pane | `Alt-=` | `limitMax` → `:big` → 单 pane no-op（"pane already at max"） |
| 单 pane | `Alt--` | `limitMin` → `:small` → 吞 pane 或 HSplit |

## 5. 场景 1：extractPaneToNewTab（:big / :small 共用）

### 5.1 动作拆解

场景 1 里 `:big` 和 `:small` 共享 `extractPaneToNewTab(activate)`，仅 `activate` 参数不同：

```
1) capture  ：在原 tab 记 pane idx 和 split tree node id（纯读取，无副作用）
2) reparent：NewTabFromPane(...) 把 h 归属改到新 tab（内部调 SetTab+SetID）
3) remove   ：Tab.RemovePane(idx) + Node.Unsplit() 从原 tab 摘 h
4) append   ：Tabs.AddTab(新 tab)
5) resize   ：newTab.Resize() + tab.Resize()
```

### 5.2 时序约束（关键）

- **step 3（摘）必须在 step 4（AddTab）之前**：`NewTabFromPane` 调 `SetID(t.ID())` 把 h.splitID 改成 newTab root id，若先 AddTab，其内部 `TabList.Resize` 遍历到原 tab 时 `原tab.GetNode(h.ID())` 返回 nil（原 tab split tree 里没有 newTab root id）→ nil deref panic。
- **step 3 顺序**：先 `RemovePane` 再 `Unsplit`，让 `Tab.Panes` 始终不出现「指向已删除 node」的脏状态。
- **step 5**：`Tabs.AddTab` 内部已调 `t.Resize()`，但补一次 `tab.Resize()` 是幂等保险（microNeo 已知 pattern）。

### 5.3 实现位置

- `extractPaneToNewTab(tab, h, activate bool)`：`internal/action/command_neo.go:141-171`
- `BigPane()` / `SmallPane()`：`internal/action/command_neo.go:105-132`

## 6. 场景 2/3：absorbPaneIntoTab + pickAbsorbTarget

### 6.1 动作拆解

```
1) capture src    ：记录 srcIdx / srcSplitID
2) reparent       ：当前 tab split tree HSplit 出下叶子，srcPane 挂过去（SetTab+SetID）
3) append         ：Tab.AddPane(srcPane, ...) 插到 h 之后
4) remove src tab ：场景 2 删除整 tab（RemoveTab），场景 3 摘 srcPane（RemovePane+Unsplit）
5) resize         ：h.tab.Resize() + Tabs.Resize()
```

### 6.2 关键时序

**场景 2（src tab 单 pane）**：
- 必须在 `RemovePane` **之前** `RemoveTab`：`RemoveTab` 靠 `Panes[0].ID()` 定位（`tab.go:59-77`），若先 `RemovePane` 则 `Panes` 空、RemoveTab 的 `len==0 continue` 跳过，删不掉空 tab。
- `SetID(newNodeID)` 后 `srcPane.ID()` 即 `newNodeID`，直接 `RemoveTab(srcPane.ID())` 能匹配。

**场景 3（src tab 2 pane）**：
- `srcSplitID` 是非 root 叶子（`parent != nil`），`Unsplit()` 正常拆掉、root flatten 成单叶自洽。
- `Unsplit()` 要求 `n.parent != nil`：场景 2 单 pane 时 srcSplitID 是 root（`parent == nil`），`Unsplit()` 返回 false 是 no-op——但场景 2 走 `RemoveTab` 整 tab 删除，split tree 随 tab 一起 GC。

### 6.3 pickAbsorbTarget：选源 pane

**简化版**：取 `Tabs.List` 末位非当前 tab 的 tab，再取其 `Panes` 末位 pane（最近打开）。

```go
func pickAbsorbTarget() (*BufPane, *Tab, bool) {
    for i := len(Tabs.List) - 1; i >= 0; i-- {
        t := Tabs.List[i]
        if t == MainTab() { continue }
        if len(t.Panes) == 0 { continue }
        bp, ok := t.Panes[len(t.Panes)-1].(*BufPane)
        if !ok { continue }
        return bp, t, true
    }
    return nil, nil, false
}
```

**理由**：`HSplitIndex` 总 append 到末尾，末位 pane = 最近打开。完美方案是给 `BufPane` 加 `lastActive` 字段，`SetActive(true)` 时更新（见 §8 边界）。

### 6.4 实现位置

- `absorbPaneIntoTab(srcPane, srcTab)`：`internal/action/command_neo.go:178-207`
- `pickAbsorbTarget()`：`internal/action/command_neo.go:212-227`

## 8. 关键不变量与边界

### 8.1 不变量

1. **每 tab ≤ 2 pane**（不可绕过）：`VSplitBuf`/`HSplitBuf` 主卡口覆盖全部入口。
2. **单 pane 时 `GetNode(splitID)` 返回 root 叶子（`parent == nil`）**：`GrowPane`/`ShrinkPane` 必须先判 `n.Parent() == nil` 提示后 return，否则 `ResizeSplit` 会 nil deref。
3. **跨 TabList 写后补 `Resize()`**：`Tabs.AddTab` / `RemoveTab` / 拆叶子后补一次 `Tabs.Resize()` 是幂等保险（microNeo 已知 pattern）。
4. **MovePane 不复制 pane**：搬的是 `BufPane` 实例，不是 buffer。src tab 的 pane 被整个搬到 dst tab，buffer 跟着 pane 走。

### 8.2 边界

1. **单 pane `Alt-=`**：`limitMax` → `BigPane` → 单 pane no-op（"pane already at max"，已全屏）。
2. **单 pane `Alt--`**：`limitMin` → `SmallPane` → 吞 pane 或 HSplit（场景 2/3/4）。
3. **2 pane `Alt-=` 到 75%**：`limitMax` → `BigPane` → promote 后再按 `Alt-=` → 又是单 pane no-op。
4. **2 pane `Alt--` 到 25%**：`limitMin` → `SmallPane` → demote 后再按 `Alt--` → 新 2-pane 继续 stepping。
5. **pickAbsorbTarget 的简化**：用 `tab.Panes` 末位 = 最近打开，若用户先开 paneB、再开 paneC 但实际工作一直在 paneB，会吞 paneC，与「最近修改」不符。这是已知简化。
6. **场景 4 复用 `:hsplit` 包装**：`return h.neoHSplitAction()`（HSplit + OpenBirthSelector），不是裸 `HSplitAction`，否则新 pane 不开 birth selector、退出时走不到 quit selector。
7. **`SetID` 不动 split tree**：`SetID` 只改 pane 字段、不动 split tree，`GetNode(oldSplitID)` 仍按树遍历找到原叶子，`Unsplit()` 才能拆掉。场景 2 单 pane 的旧叶子是 root（`parent == nil`），`Unsplit()` 返回 false 是 no-op——但场景 2 走 `RemoveTab` 整 tab 删除，split tree 随 tab 一起 GC。

## 11. 测试验证

### 11.1 2 pane 约束

- 1 pane 按 `Ctrl-t` → 2 pane 50:50
- 2 pane 再按 `Ctrl-t` → 提示 "already 2 panes in this tab"，仍是 2 pane

### 11.2 三档比例

- 2 pane `Alt-=` 连按 → 50:50 → 75:25 → 提示 "pane already at max"（promote 到全屏 tab）
- 2 pane `Alt--` 连按 → 50:50 → 25:75 → demote 到新 tab

### 11.3 :big / :small

- 场景 1（2 pane）→ `:big` → paneA 变全屏新 tab，焦点跟随
- 场景 1（2 pane）→ `:small` → paneA 变新 tab，焦点留原 tab 的 paneB
- 场景 2（2 个单 pane tab）→ `:small` → 当前 tab 有 a 上 b 下，b tab 删除
- 场景 3（单 pane + 2 pane tab）→ `:small` → 当前 tab 有 a 上 c 下（c 最近打开），b tab 剩 1 pane
- 场景 4（单 pane）→ `:small` → 当前 tab HSplit 出新空 pane
- 单 pane 按 `:big` → 提示 "pane already at max"

### 11.4 边界溢出

- 75% 按 `Alt-=` → promote 到全屏 tab，焦点跟随
- 25% 按 `Alt--` → demote 到新 tab，焦点留下
- 单 pane `Alt-=` → no-op
- 单 pane `Alt--` → 吞 pane 或 HSplit

## 12. 与 F1 的关系

| 文档 | 范围 |
|---|---|
| F1（架构设计） | FileSelector 子系统（finder 包、owner 接线层、事件路由） |
| Z0（多文件操作） | pane ↔ tab 互转、分屏比例控制、边界溢出 |

两者无代码交集，是 microNeo 的两个独立特性子系统：
- F1 聚焦「选文件 + 打开文件」
- Z0 聚焦「已开文件的 pane/tab 布局」

但共享同一设计哲学：**专注一个 buffer + 快速切换**，而非把屏幕切碎。
