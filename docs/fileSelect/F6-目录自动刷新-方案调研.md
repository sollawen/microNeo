# F6 · finder 目录自动刷新：三条路径调研

**性质**：调研 + 路径摸排，不做实施承诺。三条路径并列，权衡列清楚，决策在 §6。

**用户场景**：用户在另一个终端改了当前 finder 所在目录的文件 / 新增 / 删除条目；切回 microNeo 后，希望 finder 自动反映这些变化，不必手动重开。

---

## 1. 现状盘点（事件源 + timer 基础设施）

| 能力 | 现状 | 依据 |
|---|---|---|
| tcell focus 事件 | **不存在** | `micro-editor/tcell v2.0.13` 全包 grep `focus` 仅 Windows console 一处无关注释；无 `EventFocus` / `EnableFocus` |
| 主循环 `timerChan` | 一次性，不适合周期 | `cmd/micro/micro.go:48,557`；只支持投递单个 func 一次性触发 |
| `time.AfterFunc` 范式 | 已用 | `internal/buffer/buffer.go:1419`（diff 防抖） |
| 后台 goroutine + 锁范式 | 已用 | `internal/finder/session.go:198`（`fetchGit`） |
| 失焦链路（NotifyBlur） | **已完整** | 详见 §2 |
| 得焦链路（NotifyFocus） | **不存在** | grep 全包零结果 |
| 目录刷新能力 | `chdirTo` 已有 | `session.go:758-790`：读目录 + 重建视图 + 定位光标 + 异步 git |

**关键结论**：
- 「真 focus 事件」走不通（tcell 不支持，要自己塞 raw escape，侵入大）
- 想做只能走 **timer 轮询** 或 **复用失焦链路的镜像**
- 这两条路径的能力（stat + 条件刷新）已经齐全，只需挂触发器

## 2. 现有失焦链路（NotifyBlur）

完整链路（5 跳，全是已实现代码）：

```
鼠标点别的 pane / 拖分割条
    ↓
BufPane.SetActive(false)              internal/action/bufpane.go:793-798
    ↓ h.onOwnerBlur()
BufPane.onOwnerBlur()                 internal/action/fileops.go:79-83
    ↓ if h.finder != nil && h.finder.IsOpen()
Session.NotifyBlur()                  internal/finder/session.go:324-329
    ↓ if fm.isOpen
Session.close(Esc)                    internal/finder/session.go:285-300
    ↓
BufPane.onFinderClose(r)              internal/action/fileops.go:48-62
    → Esc 分支：no-op
```

**核心代码**（3 处）：

`internal/finder/session.go:324-329`
```go
func (fm *Session) NotifyBlur() {
    if fm.isOpen {
        fm.close(Esc)
    }
}
```

`internal/action/fileops.go:79-83`
```go
func (h *BufPane) onOwnerBlur() {
    if h.finder != nil && h.finder.IsOpen() {
        h.finder.NotifyBlur()
    }
}
```

`internal/action/bufpane.go:793-798`（SetActive 的 else 分支）
```go
} else {
    // 失焦时取消本 pane 的 finder 会话（若开着）。
    h.onOwnerBlur()
}
```

**语义**：owner 失焦 → finder 按 Esc 自关（"点框外 = 取消"）。

## 3. 路径 A：纯 tick 轮询（mtime）

### 3.1 原理

```
Open() ─► 启动 refreshTimer (time.AfterFunc, 2s 周期)
            ↓ 到期
            refresh():
              os.Stat(currentDir)
                ├─ err ─────────► return（目录没了，no-op）
                ├─ mtime 未变 ──► return（一次 syscall，无 UI 副作用）
                └─ mtime 变了 ──► readDirEntries + rebuildShow
                                 + 保留光标（按 name 重定位）
                                 + Redraw
                                 + go fetchGit（重查 git 状态）
              ↓ 不管刷没刷
              scheduleRefresh()（重排下一个 tick）
close() ─► refreshTimer.Stop()
```

**触发器是时钟，不是事件**。`Session` 自带定时器，零外部依赖。

### 3.2 改动点（仅 session.go）

| 位置 | 改动 |
|---|---|
| `Session` 结构 | 新增字段 `refreshTimer *time.Timer`、`dirMtime time.Time` |
| `Open()` 末尾 | 调 `fm.scheduleRefresh()`（异步启 tick，不阻塞打开） |
| `chdirTo()` 末尾 | 更新 `fm.dirMtime`（手动切目录后 dirMtime 要刷新，否则下一个 tick 立刻误判为"变了"） |
| `close()` 开头 | `fm.refreshTimer.Stop()`（防止 timer 在 reset 后还在飞） |
| 新增方法 | `refresh()`：stat + diff + 复用 chdirTo 核心逻辑（保留光标按 name）；`scheduleRefresh()`：AfterFunc + 自重排 |

### 3.3 开销

- **每 tick 一次 `os.Stat(dir)`**：~1–10 μs（单次 `stat(2)` syscall，只读 inode）
- **空闲时（无外部改动）**：每 tick ~5 μs → 2.5 μs/s ≈ **0.00025% CPU**
- **有改动时**：stat + readdir + 一次 git 后台 = 一次性成本，与手动切目录等同

### 3.4 延迟

- 外部改动 → 下次 tick 检测到 → 刷新。**最坏 ≤2 s**
- 用户切回窗口但不按键：等下次 tick 即可

### 3.5 局限性

**只盯目录 mtime** 抓不到"文件内容被外部修改但条目没变"的场景（dir mtime 只在增删/重命名时变）。要补这条需要再 stat 光标所在文件（见 §5）。

## 4. 路径 B：复用现有失焦链路镜像（NotifyFocus）

### 4.1 原理

失焦链路已有完整 5 跳基础设施。镜像得焦链路只需在 2 个分支各加 1 行调用 + 1 个新方法。

```
鼠标点回 finder 所在 pane
    ↓
BufPane.SetActive(true)               internal/action/bufpane.go:768-792（现有 if b 分支）
    ↓ 新增：h.onOwnerFocus()
BufPane.onOwnerFocus()                internal/action/fileops.go（新增，与 onOwnerBlur 镜像）
    ↓ if h.finder != nil && h.finder.IsOpen()
Session.NotifyFocus()                 internal/finder/session.go（新增，与 NotifyBlur 镜像）
    ↓ if fm.isOpen
Session.refresh()                     internal/finder/session.go（新增，stat + 条件刷新）
```

### 4.2 改动点（4 处，全是新增）

| # | 文件 | 改动 | 体量 |
|---|---|---|---|
| 1 | `session.go` | 新增 `NotifyFocus()` 方法（~5 行） | 5 行 |
| 2 | `session.go` | 新增 `refresh()` 方法（~30 行，复用 chdirTo 核心段） | 30 行 |
| 3 | `fileops.go` | 新增 `onOwnerFocus()`（镜像 onOwnerBlur） | 5 行 |
| 4 | `bufpane.go` | `SetActive(true)` 的 `if b` 分支末尾加 `h.onOwnerFocus()` | 1 行 |

合计 **~40 行**，4 处全是新增，零现有逻辑修改。

### 4.3 链路对照表（失焦 vs 得焦）

| 跳数 | 失焦（已有） | 得焦（要加的镜像） |
|---|---|---|
| 1 | `BufPane.SetActive(false)` | `BufPane.SetActive(true)` |
| 2 | `h.onOwnerBlur()` | `h.onOwnerFocus()` ← 新 |
| 3 | `Session.NotifyBlur()` | `Session.NotifyFocus()` ← 新 |
| 4 | `close(Esc)` | `refresh()` ← 新 |
| 5 | `onFinderClose`（no-op Esc） | （无需回调，纯内部动作） |

第 5 跳不对称：失焦要回调 owner（关掉 finder 后续动作），得焦不需要（refresh 全在 Session 内部完成）。

### 4.4 开销

- **零定时器**、零后台 goroutine、零轮询
- 纯事件驱动：只在 `SetActive(true)` 那一刻 stat 一次
- mtime 没变时也是一次 syscall，无 UI 副作用

### 4.5 延迟

- **即时**：点回 finder 立刻 stat，立刻刷新（如果 mtime 变了）

### 4.6 局限性

依赖 `BufPane.SetActive(true)` 真的会被调用。看现状：
- Tab 切换 pane：调 `SetActive`（`bufpane.go:720, 734`）
- 但**鼠标点击切 pane** 是否调 `SetActive(true)`？这点需要进一步验证（grep 现有调用点，看 mouse routing 是否最终走到这里）

## 5. 路径 A+B 组合（推荐）

两条路径**不互斥**，可以叠加：

```
路径 A（tick 轮询）: 兜底，覆盖所有没被 focus 触发的情况
    +
路径 B（NotifyFocus 镜像）: 即时触发，缩短路径 A 的 ≤2s 延迟
```

**协同设计**：
- `refresh()` 是两条路径**共用**的核心（stat + diff + 条件重读）
- 路径 A 的 tick 调用 `refresh()` 完后自重排
- 路径 B 的 `NotifyFocus` 也调用 `refresh()`，但不重排定时器
- 两条路径**不会打架**：`refresh()` 内部持锁临界区，互斥
- 同一秒内 tick 和 focus 都触发 → 第二次进 `refresh()` 看到 mtime 已更新 → 早 return，不会双刷

**改动量 = 路径 A + 路径 B**：~70 行，5 处改动。

**额外增强（可选）**：`refresh()` 内部 stat 完目录后，若光标在文件上（而非目录/面包屑），再 stat 该文件：
- 抓"文件内容被外部改"的场景
- 多一次 syscall，仍是微秒级

## 6. 各种边界（与现有逻辑的交互）

| 场景 | 行为 |
|---|---|
| 目录被外部删除（stat err） | `refresh()` 内 stat 返 err → 直接 return，不动 state，不 Redraw |
| 用户在 finder 内手动 Enter 进子目录 | `chdirTo()` 更新 `dirMtime`（不更新就会让 tick 误判） |
| 用户在 finder 内按 `.` 切显隐 | `toggleHidden()` 只 rebuildShow 不重读 → 不更新 `dirMtime`（仍正确，目录没变） |
| 用户在 finder 内 Esc / 切窗口关掉 | `close()` 先 `refreshTimer.Stop()` 再 reset → 不留 timer 残留 |
| timer 回调飞的时候 close 了 | `close()` 调 `Stop()`；callback 内 `fm.isOpen` 双重防护 |
| timer 飞的时候 chdirTo 了 | `refresh()` 锁内比对 `currentDir`（同 `fetchGit` 的 TOCTOU 防御，session.go:205）→ 旧 callback 看到 dir 已变 → 丢弃 |
| finder 关闭后 timer 仍在飞 | callback 入口 `if !fm.isOpen { return }` 兜底 |
