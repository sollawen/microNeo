# F0 · FileSelector 产品设计方案（v1 定稿）

**状态**：v1 定稿。本文是 FileSelector（`:file` 命令）的权威产品方案，自包含。
**范围**：只讲产品与交互。架构与实现以工程文档为准。
**F0** = Feature 0，`:file` 文件选择浮窗。

---

## 1. 概述

`:file` 打开一个**模态文件选择浮窗（FileSelector）**，让用户在目录树里浏览、导航、选中文件并在当前 pane 打开。

**一句话定位**：贴在当前 pane 左缘、上下顶满的轻量文件侧栏——开即用、选即开、Esc 即关。

**取代 `:open`（长期目标）**：
`:file` 让用户**浏览挑选**文件，覆盖不确定名字、想看目录结构、挑同目录兄弟文件等场景。鉴于 §2.4 的工作流（一次只开一个文件、靠浏览定位），`:file` 稳定后 `:open` 可移除，简化命令面。
- **v1**：`:file` 与 `:open` 并存（`:file` 上线、`:open` 暂留作过渡）。
- **`:file` 稳定后**：移除 `:open`（见 §10.3）。

**目标用户**：在 portrait（竖长）终端里编辑长文档 / 代码的 microNeo 用户。

**不是什么**：
- 不是 fuzzy 跳转器（v1.1 再做）
- 不是全功能文件管理器（不删 / 不改名 / 不预览）
- 不是树状展开浏览器（平铺导航，见 §5）

---

## 2. 用户场景与设计前提

### 2.1 终端形态
- **portrait（竖长）为主**：宽度有限（~80 列）、高度充裕
- 接近正方形较少；横长方形基本不会
- → picker 走"窄而高"的侧栏形态最匹配

### 2.2 多文件使用习惯
- 多文件主要靠 **split** 或 **tab**
- 同时开 3 个以上文件少见
- → FileSelector 是 pane-local（§3.1）：单 pane、上下 split、左右 split、多 tab 四种布局下，picker 都从「当前活跃 bufPane」派生 layout、落在该 pane 内。**行为完全一致，无需为任何一种单独优化或退化处理。**

### 2.3 文件特征
- 编辑的是**长文件**（AI 生成的方案、程序代码）
- → 保留内容上下文重要；picker 只占 pane 一角，右侧仍可见内容

### 2.4 工作流（关键洞察）
- 用户**一次只开一个文件**：编辑 A 时想起要看 B → split / tab → 在新 pane 里 `:file`
- **不存在"批量开同目录多文件"的场景**——没人会在 open 时就计划好同时开多个
- → picker **不需要记忆上次目录**：起始目录 = 当前 buffer 目录，每次独立、可预测（见 §5.6）

## 3. 核心设计原则

### 3.1 pane-local：picker 长在发起它的 pane 上
- 位置、尺寸、边界都从"当前活跃 bufPane"派生
- **视觉因果清晰**：picker → 它要改的那个 pane → 文件开进去
- 遮挡范围限制在该 pane 内，其它 pane 零影响

### 3.2 模态：开 picker 期间锁事件
- picker 打开时键事件被 picker 接管，无法切 pane / 编辑
- 关闭触发：Esc / 选中文件打开 / 终端 resize
- → 用户注意力聚焦；目标 pane 不会中途易主

## 4. 布局与尺寸

### 4.1 形态：VSCode 侧栏（左缘、上下顶满）
- **左**：紧贴 pane 最左边（x=0），**盖住 gutter / 行号列**（picker 是 overlay，打开期间不需要看底下的行号）
- **上下**：顶到 pane 内容区顶部；底到 statusLine 上方（**让出 statusLine**）
- **高 H** = pane 可用高度 `avail = pane.Height - 1`（一旦打开，恒等于 avail；**打开前提是 avail ≥ minHeight(10)，否则拒开并提示 pane 太矮**）
  - 一旦打开，**绝不溢出到别的 pane**（pane-local 硬约束，见 §3.1）——宁可挤、不越界
  - **不按条目数自适应**——视窗稳定，导航不抖、不多不少正好填满 pane

### 4.2 宽度
- **默认 40%** of pane width（经 settings.json 的 `fileselectwidth` 配置）
- 文件名超长截断时**保留扩展名**（`very_long_fi…me.md`，而非 `very_long_file…`）
- 宽度公式：`width = max(widthFraction·W, minWidth)`——40% 优先，下限 `minWidth` 保可读。
- pane 宽 < `minWidth` 时，`:file` **拒绝打开**并提示 pane 太窄（太窄的 pane 本身也几乎没法编辑；与其开个看不清的 picker，不如拒开）。
- 因打开前提是 W ≥ minWidth，width 必 ≤ W，**天然不越界**（无需额外封顶，与高度同原则）。

分档示例（`widthFraction=0.4`、`minWidth=20`）：

| pane 宽 W | picker 宽 | 说明 |
|-----------|----------|------|
| W ≥ 50 | 40%·W | 40% 已 ≥ minWidth，按比例 |
| minWidth ≤ W < 50 | minWidth（20） | 40% 不足，取下限 |
| W < minWidth | — | 拒绝打开 |

### 4.3 参数与常量

**用户可配置（settings.json）**：

| 参数 | v1 默认 | 说明 |
|------|--------|------|
| `fileselectwidth` | 0.4 | picker 宽度占比 |
| `diffgutter` | true | 复用：控制编辑器 diff indicators **和** FileSelector git 状态显示（见 §7.5） |

**内部常量（代码写死，不暴露给用户）**：

| 常量 | 值 | 说明 |
|------|----|------|
| `minWidth` | 20 | picker 最小宽度：40%·W 不足此值取此值，pane 宽 < 此值拒开（见 §4.2） |
| `minHeight` | 10 | picker 最小高度：pane 可用高 avail < 此值拒开（外高 10 → 内容区 8 行 = 面包屑 + 6 行文件列表 + hint，保证可浏览） |

高度无额外参数——H 恒等于 pane 可用高度（全高模型，见 §4.1）；打开前提是 avail ≥ minHeight（见内部常量表）

### 4.4 布局示意

```
┌─ 活跃 bufPane ──────────────────────────────────┐
│             ┌─────────────────────┐                │
│  文件内容   │ …/proj/docs/        │ ← 行0 面包屑    │
│ （右侧 60% │ ▸ sub/              │ ← 行1 光标默认 │
│  仍可见）  │   file命令.md       │                │
│            │   产品分析.md       │                │
│            │   …                 │                │
│            │ Enter 打开 · . 显示隐藏 · Esc │ ← hint │
│             └─────────────────────┘                │
├───────────────────────────────────────────────────┤ ← pane statusLine（让出）
└───────────────────────────────────────────────────┘
```

picker 是**浮层 overlay**：覆盖左 40%，右侧 60% 文件内容仍可见（但因模态锁，不可交互，仅供视觉参考）。

## 5. 交互设计

### 5.1 键位

| 键 | 行为 |
|----|------|
| `↑` / `↓` | 移动光标（**不循环**：顶到面包屑行为止，底到最后条目为止） |
| `Enter` | **上下文敏感**（见 §5.2） |
| `←` | 回父级目录（快捷键，任意位置生效） |
| `→` | 进入光标所在子目录（快捷键） |
| `.` | 切换 dotfiles 显隐 |
| `Esc` | 关闭 picker，不打开任何文件 |

**`←` / `→` 是"隐藏的快捷键"**：Enter 已按上下文覆盖全部动作，`←` / `→` 仅服务于"不移光标就能回父 / 进子"的效率场景。**不进 hint、不广告**，留给有 ranger / lf 习惯的用户自然发现。

### 5.2 Enter 的上下文语义

Enter 的行为由**光标所在行**决定：

| 光标位置 | Enter 行为 | hint 文案 |
|---------|-----------|----------|
| 面包屑行（行 0） | 回父级目录 | `Enter 上一级` |
| 子目录条目 | 进入该目录 | `Enter 进入` |
| 文件条目 | 在当前 pane 打开 | `Enter 打开` |

**设计理由**：Enter = "激活当前选中项"是几乎所有文件管理器的共识（Finder / Explorer / ranger / lf / netrw / helix）。让 Enter 在不同行做不同事，是"万能激活"的最小心智模型；配合上下文 hint，零学习成本。

### 5.3 光标起始位置：当前文件

开 picker 时光标停在 **当前 buffer 对应的文件**上——给用户一个"你在这里"的锚点（对齐 VSCode / helix 在 tree 里高亮 active file 的行为）。

边界处理：

| 情况 | 光标落点 | 附带状态 |
|------|---------|---------|
| 当前文件在 currentDir、可见 | 停当前文件 | — |
| 当前文件是 dotfile 且默认隐藏 | 停当前文件 | **`showHidden` 自动置 true**（见 §5.4） |
| 当前 buffer 无路径（未保存 / 新文件） | 首条目 | 起始目录 = cwd |
| currentDir 里找不到（防御性） | 首条目 | — |

**不做"持续高亮 active file"**：光标起始锚点已足够定位；用户移走光标后不需要常驻标记追着。

### 5.4 隐藏文件（dotfiles）

- **默认隐藏**（对齐行业默认、首屏干净）
- `.` 切显隐
- **自动显示例外**：若当前 buffer 是 dotfile，开 picker 时自动置 `showHidden = true`，保证 §5.3 的"光标停当前文件"无例外成立
  - 该状态在整个 picker 会话持续（非闪烁式）
  - hint 文案随之翻转（显示 `. 隐藏文件`），给用户一个隐性信号
  - 用户可再按 `.` 藏回；此时若光标在已藏掉的 dotfile 上，光标 clamp 到最近可见条目

### 5.5 排序
- 目录在前、文件在后
- 各自大小写不敏感字母序

### 5.6 起始目录
- = **当前 buffer 所在目录**
- 无路径时（未保存 / 新文件）fallback 到 cwd
- **不做"记忆上次浏览目录"**：见 §2.4，该场景不存在；可预测优先

## 6. 面包屑（行 0）

### 6.1 双职责：路径显示 + 回父入口
- 画在内容区第 1 行（**不在边框 / title 上**）
- **可选、可 Enter**——光标移到这行 + Enter = 回父级（与 `←` 等价）
- **取代传统 `../` 入口**：一行同时承担"我在哪"+"能往回走"两个职责，省一行

### 6.2 显示策略：左截断全路径
- 空间够 → 显示完整路径
- 不够 → 左边 `…` 截断，**永远保留 `当前目录/`**
- 例：`proj/src/components/md/`（宽）→ `…/components/md/`（窄）
- 根目录显示 `/`

**为什么不固定两级**：左截断在窄屏下的退化形态恰好就是两级（`…/父/当前/`），等于"两级是左截断的特例"——一次做对，省一次重构。

### 6.3 不做分段跳转
- v1 整行 = "回父一级"，简单可预测
- VSCode 那种"点面包屑某段跳到该祖先"是 GUI 产物（需要面包屑内部左 / 右移光标选中段），TUI 罕见、复杂度高，**不学**

### 6.4 可发现性
面包屑无特殊视觉标记（就是个路径文本）。可发现性靠两条天然机制：
1. 用户按 ↑/↓ 探索时，光标会自然移到首行——发现"首行可停"
2. 底部 hint 行在光标位于面包屑时显示 `Enter 上一级`，明确告知该行作用

## 7. 底部 hint 行（末行）

### 7.1 上下文敏感
hint 的 Enter 文案随光标位置变化（见 §5.2 表）。这让 hint 永远描述"按下去会发生什么"，零歧义，同时自然教育用户"光标在哪、Enter 就干嘛"。

### 7.2 持久部分
- `.` 的文案随显隐状态翻转：隐藏时"显示隐藏"、显示时"隐藏文件"
- `Esc 取消` 常驻

### 7.3 完整文案示例

```
（光标在文件）    Enter 打开  ·  . 显示隐藏  ·  Esc 取消
（光标在子目录）  Enter 进入  ·  . 显示隐藏  ·  Esc 取消
（光标在面包屑）  Enter 上一级  ·  . 显示隐藏  ·  Esc 取消
（dotfile 已显）  Enter 打开  ·  . 隐藏文件  ·  Esc 取消
```

### 7.5 文件列表显示格式

**每行格式**：`[标记] 文件名 [状态]`

- **标记**：目录 `▸`，文件空白（**预留 1 字符位**——v1 用 `▸` 表示目录；未来可扩展为 Nerd Font 图标等）
- **文件名**：
  - 目录：带 `/` 后缀（如 `src/`）——一眼区分目录/文件
  - 文件：完整名称（如 `main.go`）
  - 超长时左截断（保留扩展名，见 §4.2）
- **状态**：git 状态，颜色区分，靠右显示（仅在有 git repo 时显示）

#### git 状态定义

| 状态标记 | 含义 | 颜色建议 |
|---------|------|----------|
| `M` | Modified（已修改，未暂存） | 黄色/橙色 |
| `U` | Untracked（未跟踪） | 蓝色/青色 |
| `I` | Ignored（.gitignore） | 灰色（dim） |
| `A` | Added（已暂存新增） | 绿色 |
| `D` | Deleted（已删除） | 红色 |
| `R` | Renamed（已重命名） | 紫色 |

**M/U/I 是核心三态**（最常遇到）。A/D/R 在文件选择器场景较少见，但支持。

#### 颜色区分

- **目录名**：使用 colorscheme 的 `type` 颜色（默认蓝色）——醒目、易识别
- **文件名**：使用 colorscheme 的 `default` 颜色（普通文本）——保持低调

#### 非仓库场景

- 非 git 仓库内：状态列隐藏（只显示 `[标记] 文件名`）
- 性能兜底：若 `git status` 超时（>2s）或用户关闭 `diffgutter` setting，git 状态自动禁用（v1 默认 `diffgutter=true`）

## 8. 打开文件的行为

**选中文件后**：
- 在**发起 `:file` 的那个 pane** 打开（与 `:open` 一致）
- picker 自动关闭

**发起 `:file` 时（当前 buffer 已修改的防御性保护）**：
- infoBar 提示 `当前文件已修改，保存？y/n`
- 用户输入 `y` → 保存当前文件 → 打开 FileSelector
- 用户输入 `n` → 取消，什么都不发生（不打开 picker）
- 对齐 `:open` 的行为

## 9. 边界与退化

| 场景 | 行为 | 评价 |
|------|------|------|
| 极矮 pane（avail < minHeight=10） | 拒绝打开并提示 pane 太矮（保证内容区 ≥ 8 行、文件列表 ≥ 6 行可滚动） | 与 minWidth 对称；太矮的 pane 本身没法用，拒开比开个废品好 |
| 左右 split（pane 半宽） | 半宽 pane 按 §4.2 宽度公式：≥50 走 40%、[minWidth,50) 取下限、<minWidth 拒开 | pane-local 正常，按宽度公式处理 |
| 终端 Resize | 关闭 picker（等同 Esc，无操作） | FloatFrame 统一：所有浮窗 resize 即关 |
| 24~32 列宽放不下多列元数据 | v1 只显"标记 + 文件名"（目录 `▸`、文件无标记），砍 size/mtime 列；git 状态改为行内显示（见 §7.5） | 见 §10.2 |

注：全高模型下 H 恒等于 pane 可用高度，**不会跨 split 边界溢出**——picker 永远恰好填满发起它的那个 pane。

## 10. v1 范围与 Roadmap

### 10.1 v1 包含
- 上述全部交互、布局、边界处理
- 浏览 / 导航 / 选中打开的完整闭环
- 上下文 hint、面包屑导航、当前文件定位、dotfile 自动显示

### 10.2 v1 不做（明确推迟）

| 能力 | 推迟原因 | 何时再做 |
|------|---------|---------|
| **fuzzy 过滤** | 架构已铺路（Model/View 分离），但 v1 先把基础浏览做扎实 | v1.1 优先项 |
| **元数据列**（size / mtime） | 窄宽放不下；多数场景"标记 + 文件名"够用；git 状态已改为行内显示（见 §7.5） | 出现宽屏 / 多列模式需求 |
| **持续高亮 active file** | 光标起始锚点已够 | 用户反馈需要常驻标记 |
| **面包屑分段跳转** | GUI 产物，TUI 复杂度高 | 几乎不会做 |
| **记忆上次目录** | 不存在该用户场景（§2.4） | 除非未来出现真实需求 |
| **树状展开 / 折叠** | GUI 产物，与 `←/→` 平铺导航冲突 | 不做 |
| **`:` 命令 / `:save-as` 等复用** | FileSelector 设计为可复用的路径选择器，但 v1 只接 `:file` | 出现第二个使用方 |

### 10.3 后续里程碑（v1 之后）

**移除 `:open`**：`:file` 稳定后执行。`:file` 只做浏览、**不接受路径参数**（设计决策，见 §11）；`:open` 删除后“打路径打开文件”的能力随之消失——用户靠浏览或（v1.1）fuzzy 定位。删除时需给出迁移说明：原 keybindings / 配置里写死的 `:open <file>` 需改用 `:file`。

## 11. 设计决策溯源（为何这样选）

简列关键取舍的理由，供未来 revisit：

| 决策 | 否决的替代方案 | 理由 |
|------|--------------|------|
| Enter 上下文敏感（进目录 / 打开 / 回父） | Enter 仅打开文件，目录上 no-op | "Enter = 激活"是跨工具共识；目录上 no-op 违反直觉 |
| 面包屑行取代 `../` | 保留独立 `../` 行 | 信息密度翻倍；可发现性靠 hint + 光标探索补足 |
| 上下文 hint 行 | 静态 hint + `◂` 视觉标记 | 动态 hint 信息密度更高、零额外占用 |
| 光标起始 = 当前文件 | 光标起始 = 首条目 | "你在这里"锚点，对齐 VSCode / helix |
| dotfile 默认隐藏 + 自动显示例外 | 默认显示 / 例外时退到首条目 | 对齐行业默认；自动显示保住"光标停当前文件"无例外契约 |
| 全高（H = avail） | 50% 封顶（开始菜单模型） | 多数目录文件够多；稀疏可接受；视窗稳定 |
| 起始目录 = buffer 目录 | 记忆上次目录 | 不存在批量开多文件场景；可预测优先 |
| 左截断面包屑 | 固定两级 | 两级是左截断的特例，一次做对 |
| `←` / `→` 保留但不广告 | 删掉，只靠 Enter | 无害的效率快捷键，留给有习惯的用户 |
| `:file` 纯浏览、不接路径参数 | `:file <path>` 作为 `:open` 超集 | 命令面单一心智；用户靠浏览定位；接受 keybindings 迁移代价 |
