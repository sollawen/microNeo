# 访问历史 · 实现方案（最简版）

## 一、目标

记录「最近访问过的目录」（不重复、有序、有上限），落盘到用户配置目录，供后续 fileSelector 做「最常去的目录，快速跳过去」。

本期只做**记录**，不做交互。

需求原点：每次打开 microNeo 都要在目录树里导航到目标目录，太烦。要解决的是"快速跳到常去的目录"，所以记录的是**目录**，不是文件。

---

## 二、关键设计决策：用"最近 N 个"，不用"热度榜"

先说明选型理由，因为这决定了整个数据结构。

最初考虑过「目录 + 访问计数」的热度榜方案，但它有个致命死穴：**新目录永远挤不进去**。

- 新去一个目录，它的计数 = 1，永远是最小的。
- 当 map 已满 20 条、又来个新目录时，要么拒收（新目录永远进不来），要么挤掉计数最小的（而最小的正好就是新目录自己）。
- 两个分支都导致：**没去过的目录，永远记录不下来**——而这恰恰是用户最需要的场景（去个新地方，下次想快速跳回去）。

改用「最近访问 N 个不重复目录」后：

- 每个去过的目录都立刻上榜，公平。
- 更贴合真实工作方式——人按项目阶段切换，这个月做 A、下个月做 B，「最近去过」比「累积最热」实用，后者会被历史包袱拖累。
- 常去的目录会反复回到队首，隐含了热度信息。

放弃的是「精确频率」信息，但对"快速跳转"目的够用。

---

## 三、现状调研（关键事实）

### 1. 所有「打开真实文件」都汇聚到唯一入口

`buffer.NewBufferFromFileWithCommand`（`internal/buffer/buffer.go:292`）是唯一的真实文件打开函数，下列入口全部最终调用它：

| 入口 | 调用位置 |
|---|---|
| 命令行参数启动 | `cmd/micro/micro.go:215` |
| `open` 命令 | `internal/action/command.go:307` |
| `tabopen` | `command.go:558` |
| `vsplit` | `command.go:516` |
| `hsplit` | `command.go:536` |
| Lua 插件 | `cmd/micro/initlua.go:139` |

而 `NewBufferFromString`（scratch/help/log/raw 等非文件 buffer）、stdin 管道 buffer **不经过**这个函数——天然就被排除在历史之外，语义正好干净。

**结论**：只要在这一个函数里挂钩，就能无遗漏地覆盖所有真实文件打开行为。

### 2. 配置目录与落盘惯例

- `config.ConfigDir` = `$XDG_CONFIG_HOME/microNeo`（`internal/config/config.go:28`），启动时由 `InitConfigDir` 保证已创建。
- 目录里已是扁平文件：`settings.json`、`bindings.json`。新增 `history.json` 完全符合现有惯例。
- JSON 是项目通用格式（settings 用 `json5`，纯 JSON 文件与其兼容）。

### 3. microNeo 现有代码不关心历史

microNeo 完全没有「文件/目录访问历史」功能：`recent` / `openhistory` / `filehistory` 等字眼全仓库零命中，`buffer.AbsPath` 从未被累积或落盘。现存的 `internal/info/history.go` 是 InfoBar 的**命令行历史**（gob 二进制、`map[string][]string`、受 `savehistory` 开关控制），数据模型和用途都不兼容，不可复用。因此这是在白纸上做最简实现。

## 四、最简洁方案

### 1. 数据结构

一个有序去重的字符串切片：

```go
[]string   // 队首 = 最近访问的目录，元素不重复，长度上限 N
```

N 暂定 **50**。理由：记录端存宽松点，展示端再决定显示几个；存了不一定全显示，但存少了后面想多显示就没数据了。50 条切片的读写开销可忽略。

落盘文件：`$ConfigDir/history.json`，形如

```json
[
    "/home/Solla/microNeo/internal/buffer",
    "/home/Solla/microNeo/cmd/micro",
    "/home/Solla/microNeo/docs/fileSelect"
]
```

纯数组，第一个就是最近访问的目录，人读直观。

### 2. 代码结构

放进现有 `internal/config/` 包，**不新建包**。理由：`RecordDirHistory` 唯一的外部依赖就是 `config.ConfigDir`，而 `config` 包本来就负责用户配置目录下的持久化文件（`settings.json`、`bindings.json` 都由它读写），`history.json` 是同性质数据，属于 config 的职责域。新建一个独立包反而要多一层 import 依赖，没有必然理由。

新增文件 `internal/config/history.go`（~40 行），包名仍为 `config`，暴露**一个函数**：

```go
package config

// RecordDirHistory 记录一次目录访问：传入被打开文件的绝对路径，
// 内部对其取 filepath.Dir，作为目录记录到最近访问列表。
// 同一目录再次访问时移到队首；列表去重、按时间倒序、长度上限 N。
// 首次调用时懒加载 history.json 到内存，之后每次更新并整体写回。
func RecordDirHistory(fileAbsPath string)
```

内部实现要点：

- 包级变量 `recentDirs []string` + `dirHistoryLoaded bool`
- `loadDirHistory()`：读 `ConfigDir/history.json`，`json.Unmarshal` 到 slice；文件不存在视为空 slice（首次运行）
- `RecordDirHistory`：若未 loaded 先 load；`dir := filepath.Dir(fileAbsPath)`；先从 slice 删除已有的 `dir`（去重），再 `prepend` 到队首；长度 > N 则截断到 N；`json.MarshalIndent` 整体写回
- 写回用 `os.WriteFile`（micro 单线程 UI，无并发竞争）

把 `filepath.Dir`、去重、截断逻辑都收在函数内，buffer 侧调用方不用关心任何细节，保持触发点改动最小。函数名加 `Dir` 前缀避免与 `info.SaveHistory` 等命名混淆。

**不需要**：退出钩子、定时刷盘、加锁、增量更新。每次打开就写最简单，且抗崩溃。

### 3. 触发点（唯一改动）

在 `NewBufferFromFileWithCommand`（`buffer.go:292`）拿到合法 `filename`（已 `ReplaceHome`、已确认是普通文件）之后、`return buf` 之前，加一行：

```go
if buf.AbsPath != "" {
    config.RecordDirHistory(buf.AbsPath)
}
```

只此一处改动，覆盖 6 个入口。`buffer` 本就 `import config`，**无需新增 import**。`AbsPath` 在 `NewBuffer` 里已 `ResolvePath` 成绝对路径，`RecordDirHistory` 内部再取 `Dir`，无需调用方算。

**noName 空 buffer 天然被排除**，无需额外处理。两层保障：

1. **挂钩点判断 `buf.AbsPath != ""` 直接拦住**：noName 的定义就是 `b.Path == ""`（见 `GetName()` `buffer.go:562`），此时 `AbsPath` 必为空，不进 `RecordDirHistory`。
2. **noName buffer 根本不经过挂钩点**：它的两条产生路径都绕开了 `NewBufferFromFileWithCommand`——
   - 启动无文件参数 → `micro.go` 的 `len(files)==0` 分支
   - `vsplit`/`hsplit` 无参数 → `command.go:516/536` 的 `len(args)==0` 分支

两层一致，语义干净：只有「真实从磁盘打开的文件」所在目录才会被记录。

## 五、取舍说明（CTO 视角）

### 为什么用"最近 N 个"、不用"热度榜"？

详见第二节。核心：热度榜的死穴是「新目录永远挤不进 top N」，而"最近 N 个"让每个去过的目录都立刻上榜。放弃的是精确频率信息，但对"快速跳转"够用，且常去目录会反复回队首、隐含热度。

### 为什么挂在 buffer 包，而不是 action 层？

- action 层入口分散（6 处），改动点多、易遗漏。
- buffer 包已依赖 `config`、`screen`、`md` 等包，并非纯数据层，再多调用一个 `config.RecordDirHistory` 不破坏现有架构边界。
- 选 `NewBufferFromFileWithCommand` 而非更底层的 `NewBuffer`：后者会被 scratch/help/log buffer 共用，会误记入历史；前者只在「打开真实文件」时触发，语义精准。

### 为什么每次都整体写回，不搞增量？

- 历史文件体量小，整体 `MarshalIndent + WriteFile` 的成本是微秒级。
- 增量更新要自己管文件偏移、删除条目，复杂度陡增，违背「最简」。
- 退出钩子方案在 micro 崩溃/被 kill 时会丢数据，反而不可靠。

### 为什么用目录绝对路径、且元素去重？

- 同一目录可能从不同 CWD 用相对路径打开其中的文件，绝对路径天然去重。
- `buf.AbsPath` 已由 `util.ResolvePath` 处理好，`filepath.Dir` 零成本复用。
- 不 follow symlink（保持最简；若后续有需求再考虑 `filepath.EvalSymlinks`）。
