# C0 光标滚动 —— 现状调研

> 关联背景文档：`docs/光标的位置.md`
> 本系列：
> - **C0 现状调研**（本文）：问题、根因、影响范围、已确认事实
> - **C3 最小修复**（`docs/C3-光标滚动-最小修复.md`）：阈值从"按 line 算"改成"查 `viewportRowBufLine`"

---

## 1. 问题复述

- 光标停在屏幕底边（statusLine 上方第 1~3 行）时按 ↓
- 期望：内容整体上滚 1 行，光标始终保持在 statusLine 上方 `scrollmargin`（默认 3）行之内
- 实际：光标直接下移出屏幕，没有触发上滚
- micro 原生路径在同样场景下表现正确

> click / 上下键的 buffer line 映射已工作正常，不在本文问题范围内——它们用的是另一套数据（见 §2.3）。

---

## 2. 根因定位

### 2.1 原生滚动模型的隐含假设

micro 的"视口滚动"完全建立在 `display.SLoc{Line, Row}` 上（`internal/display/softwrap.go:13`）。在 **非 softwrap** 模式下，下面这组函数都把 **buffer 行号** 直接当作 **屏幕行号** 用（1 buffer 行 = 1 屏幕行）：

| 函数 | 位置 | 非 softwrap 行为 |
|------|------|------------------|
| `SLocFromLoc(loc)` | `softwrap.go:313` | `return SLoc{loc.Y, 0}` |
| `Scroll(s, n)` | `softwrap.go:292` | `s.Line += n` |
| `Diff(s1, s2)` | `softwrap.go:304` | `s2.Line - s1.Line` |

`Relocate()`（`internal/display/bufwindow.go:237`）的底边判定正是基于这套 1:1 假设：

```go
// bufwindow.go:255-256
if c.GreaterThan(w.Scroll(w.StartLine, height-1-scrollmargin)) && c.LessEqual(w.Scroll(bEnd, -scrollmargin)) {
    w.StartLine = w.Scroll(c, -height+1+scrollmargin)  // 让光标落在底边 scrollmargin 行内
    ret = true
}
```

其中 `c = SLocFromLoc(cursor.Loc) = SLoc{cursor.Y, 0}`，
判定条件等价于：`cursor.Y > StartLine.Line + height-1-scrollmargin`。

### 2.2 MD 渲染打破了这个假设

MD 渲染会为一个 buffer 行额外产生**装饰行**（标题下划线、表格分隔线、代码块边框、HR 等）。例如：

- `internal/md/render_heading.go`：每个标题行 = 1 内容行 + 1 装饰横线
- 代码块、表格、列表同理

于是出现：
- 一个 buffer 行 → 多个屏幕行
- buffer 行号 `cursor.Y` 与它在屏幕上的实际行号**不再相等**

所以 `cursor.Y > StartLine.Line + height-1-scrollmargin` 这个判定**系统性低估**了光标的真实屏幕位置：当光标真实屏幕行已经在底边时，`cursor.Y` 这个数值可能还远小于 `StartLine.Line + height-1-scrollmargin`，于是判定为 false，**不触发上滚** → 光标跑出屏幕。

### 2.3 两套位置数据的真实用法 + 真正的断层

经代码核查，当前与"屏幕行 ↔ buffer 行"相关的数据有**三类**，用法如下：

**① `MDSegments`（碎片化，但它是渲染管线的生产者）** —— 全部引用仅 5 处：
```
buffer.go:93            字段定义
buffer.go:212, 1032     reHighlight 时重建
bufwindow_md.go:717     displayBufferMD 读取 → 过滤 + 渲染
```
即 `MDSegments` 现在**只服务于渲染管线本身**，没有任何"位置计算"在直接遍历它。

**② `viewportRowBufLine`（整屏模型，服务于位置查询）** —— 非测试消费者：
```
bufwindow_md.go:706-734   构建（displayBufferMD 渲染时扁平化写入）
bufwindow_md.go:833       screenOffsetToBufferLine   ← click 用
bufwindow_md.go:847       bufferLineToScreenOffset   ← 反查
bufwindow.go:295          LocFromVisual              ← click 坐标映射
```
所以 click、光标↔line 映射**已经走整屏模型**，不是碎片的。

**③ `SegmentMeta`（`md.go:31`）—— 死代码**：定义了"缓存到 BufWindow 上供 Scroll/Diff 使用"，但全代码库**零引用**。是当年规划好却没落地的半成品。

**真正的断层**：滚动这套函数（`Relocate`/`Scroll`/`Diff`/`SLocFromLoc`）既没用整屏的 `viewportRowBufLine`，也没用碎片的 `MDSegments`——它还活在 micro 原生那个"1 buffer 行 = 1 屏幕行"的旧世界里。这才是底边不滚动的根因。

| 消费者 | 现在用什么 | 状态 |
|--------|-----------|------|
| click / 光标↔line 映射 | `viewportRowBufLine`（整屏） | ✅ 已迁移 |
| 渲染管线内部 | `MDSegments`（碎片，但是**生产者**） | ✅ 合理 |
| **滚动**（`Relocate`/`Scroll`/`Diff`/`SLocFromLoc`） | **micro 原生的 1:1 假设** | ❌ 两套都没用 |

> 由此也澄清一个常见的误判："很多地方还在用碎片化的 MDSegment"——实际不成立，位置计算早就迁移到 `viewportRowBufLine` 了。未迁移的是**滚动模型**，且它两套都没用，而非用错了哪一套。这个区分决定了修复方向（见 `docs/C3-光标滚动-最小修复.md`）。

## 3. 影响范围（同一根因，不止 ↓ 键）

这套 1:1 假设被打破后，受影响的不只是 ↓ 键底边：

| 场景 | 现象 | 来源 |
|------|------|------|
| ↓ 键，光标到底边 | 不上滚（本次报告） | `Relocate` 底边分支 |
| ↑ 键，光标到顶边 | 对称问题，可能不下滚 | `Relocate` 顶边分支 `bufwindow.go:248-250` |
| PageUp/Down、HalfPageUp/Down | 按屏幕行翻页会偏差，越接近装饰密集区越明显 | `actions.go` 里的 `Scroll` 用法 |
| 滚轮 | 同上 | `actions.go` 滚轮处理 |
| 滚动条位置 / statusline 百分比 | 偏移，因为依赖 `Diff` | `statusline.go`、scrollbar |

> 优先级建议：先修 `Relocate`（覆盖 ↑↓ 键这个最高频、用户最敏感的场景），其余作为后续。

---

## 4. 关键事实（已确认 ✅）

> 以下三件事已通过代码核查确认，每条附证据（文件:行号 + 代码片段）。

### 4.1 事实 1：按键处理时序 ✅

**确认**：`CursorDown` 内先 `MoveCursorDown(1)`，紧接 `Relocate()`；`viewportRowBufLine` 仅在 `displayBufferMD` 中重建，而后者由外部 `Display()` 在 `CursorDown` 返回后才调用——故 **`Relocate` 执行时刻，`viewportRowBufLine` 仍是上一帧（光标移动前）的数据**。

证据 1 — `CursorDown` 调用顺序，`internal/action/actions.go:264-271`：
```go
func (h *BufPane) CursorDown() bool {
    selectionEndNewline := h.Cursor.HasSelection() && h.Cursor.CurSelection[1].X == 0
    h.Cursor.Deselect(false)
    if selectionEndNewline {
        h.Cursor.Start()
    } else {
        h.MoveCursorDown(1)
    }
    h.Relocate()   // ★ 此时 viewportRowBufLine 还是上一帧的
    return true
}
```

证据 2 — `viewportRowBufLine` 的唯一重建点，`internal/display/bufwindow_md.go:706-734`：
```go
if cap(w.viewportRowBufLine) < bufHeight {
    w.viewportRowBufLine = make([]int, bufHeight)
}
w.viewportRowBufLine = w.viewportRowBufLine[:bufHeight]
for i := range w.viewportRowBufLine {
    w.viewportRowBufLine[i] = -2     // ★ 重置
}
// ...
startVY := vY - len(rowBufLines)
copy(w.viewportRowBufLine[startVY:vY], rowBufLines)  // ★ 写入
```

证据 3 — `Display()` 在 `CursorDown` 返回后才被外部调用，`internal/display/bufwindow.go:927-936`：
```go
func (w *BufWindow) Display() {
    w.updateDisplayInfo()
    w.displayStatusLine()
    w.displayScrollBar()
    if !w.Buf.IsMD || !w.mdConfig.MDRender {
        w.displayBuffer()
    } else {
        w.displayBufferMD(w.editMode)   // ★ 这里才重建 viewportRowBufLine
    }
}
```

**推论（确认）**：`Relocate` 时刻 `viewportRowBufLine` 反映的是**移动前**的视口；光标已跳出旧视口时 `bufferLineToScreenOffset` 查不到，只能部分推断出界。这意味着**任何用 `viewportRowBufLine` 查"光标跳转新位置"的方案在跳转场景都会失效**——这是 `viewportRowBufLine` 的客观限制。但 C3 发现：本 bug 只查视口的阈值行、不查光标，故不受此约束（详见 `docs/C3-光标滚动-最小修复.md` §3）。

### 4.2 事实 2：`StartLine` 的粒度 ✅

**确认**：非 softwrap 模式下 `StartLine.Row` 恒为 0、`StartLine.Line` 只能落在 buffer 行；装饰行经 `firstContentBufLine`/`lastContentBufLine` 绑定到内容行，与 `filterSegmentsToVisible`/`VisibleStart` 自洽——**滚动目标只需精确到 buffer 行级别，装饰行会随内容行一并正确显示**。

证据 1 — 非 softwrap 下 `SLocFromLoc` 的 `Row` 恒为 0，`internal/display/softwrap.go:313-317`：
```go
func (w *BufWindow) SLocFromLoc(loc buffer.Loc) SLoc {
    if !w.Buf.Settings["softwrap\"].(bool) {
        return SLoc{loc.Y, 0}   // ★ Row 恒 0，Line = buffer 行
    }
    return w.getVLocFromLoc(loc).SLoc
}
```

证据 2 — 软切换时强制把 `StartLine.Row` 清 0，`internal/display/bufwindow.go:64-67`：
```go
if option == "softwrap" {
    if nativeValue.(bool) {
        w.StartCol = 0
    } else {
        w.StartLine.Row = 0     // ★ 非 softwrap 模式 Row 恒 0
    }
}
```

证据 3 — 装饰行绑定到内容行，`internal/display/bufwindow_md.go:97-136`：
```go
// ★ 预扫描：找到第一个内容行的 BufLine（绝对行号）
// 前置装饰行归属此行。若第一个内容行已滚出 viewport，前置装饰行也滚出。
firstContentBufLine := -1
for _, row := range rendered.Rows {
    if row.BufLine >= 0 {
        firstContentBufLine = row.BufLine
        break
    }
}
// ...
lastContentBufLine := -1 // 追踪最近的内容行，用于装饰行可见性判断
for _, row := range rendered.Rows {
    // ...
    effectiveLine := row.BufLine
    if effectiveLine < 0 {
        if lastContentBufLine >= 0 {
            effectiveLine = lastContentBufLine      // 装饰行复用最近内容行（后置）
        } else if firstContentBufLine >= 0 {
            effectiveLine = firstContentBufLine     // 前置装饰行归属第一个内容行
        }
    }
    // ...
    rowBufLines = append(rowBufLines, effectiveLine)  // ★ 装饰行的 effectiveLine = 绑定的内容行
}
```

证据 4 — `VisibleStart` 与 `StartLine.Line` 的对齐，`internal/display/bufwindow_md.go:750-775`：
```go
s.VisibleStart = s.BufStartLine
if s.VisibleStart < startY {
    s.VisibleStart = startY      // ★ startY 即 visibleStart = w.StartLine.Line
}
```
而 `displayBufferMD` 里 `visibleStart := w.StartLine.Line`。两者对齐 → 把 `StartLine.Line` 设到 buffer 行 N，buffer 行 N 的前置装饰行（`effectiveLine = firstContentBufLine = N`）会被纳入可见性判断并一并显示。

**推论（确认）**：滚动目标只需确定到 buffer 行。装饰行随内容行正确出现，不会出现"内容行显示但前置装饰行丢失"的割裂。→ 所有滚动方案只需把 `StartLine.Line` 算到正确的 buffer 行即可，无需也无法表达"装饰行偏移"。

### 4.3 事实 3：`scrollmargin` 默认值 ✅

**确认**：`scrollmargin` 默认 `float64(3)`，与报告中"光标与 statusLine 之间还有三行"一致。

证据 — `internal/config/settings.go:92`：
```go
"scrollmargin":    float64(3),
```

### 4.4 汇总 + 对修复的意义

| 事实 | 结论 | 关键证据 |
|------|------|---------|
| 1. 时序 | ✅ 确认 | `actions.go:264-271` + `bufwindow_md.go:706-734` + `bufwindow.go:927-936` |
| 2. `StartLine` 粒度 | ✅ 确认 | `softwrap.go:313` + `bufwindow_md.go:97-136` + `bufwindow_md.go:750-775` |
| 3. `scrollmargin` | ✅ 确认 | `config/settings.go:92` |

**对修复方案的关键意义**：

1. 纯光标移动（↑↓/PageUp/Down/滚轮）**不会**触发 `b.MDSegments` 重建——重建只发生在 `internal/buffer/buffer.go:212` 与 `:1032` 的 reHighlight，即**编辑**时。因此**导航期间段数据稳定**，正好覆盖本 bug 的高频场景（连续按 ↓ 键）。→ 修复方案可安全使用段级缓存。

2. 事实 1 的时序坑（`viewportRowBufLine` 滞后于 `Relocate`）对"查光标跳转"是限制，但**对本 bug 不构成约束**：本 bug 的判定只查**视口的两个固定阈值行**（`viewportRowBufLine[scrollmargin]` / `[height-1-scrollmargin]`），不查光标新位置；而按下键时视口（`StartLine`）未变，这两个阈值对判定是当前有效的。详见 `docs/C3-光标滚动-最小修复.md` §3。

---

## 5. 小结：根因一句话 + 引向修复方案

**根因**：`Relocate/Scroll/Diff/SLocFromLoc` 在非 softwrap 下把 buffer 行号当屏幕行号，而 MD 装饰行打破了这个 1:1 假设。`viewportRowBufLine` 虽是正确的整屏模型，但只服务于 click/上下键映射，且因时序坑（事实 1）无法直接喂给 `Relocate`。

修复方案见 `docs/C3-光标滚动-最小修复.md`：把 `Relocate` 的顶边/底边 margin 阈值从"按 line 算"改成"查 `viewportRowBufLine`"，约 +6 行，不引入新数据结构、不改时序、不动 `Relocate` 的动作语句。
