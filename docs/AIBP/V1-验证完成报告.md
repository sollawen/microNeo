# V1 · 验证完成报告：外部消息能否变成 prompt 触发 LLM

> **状态**: ✅ 验证完全成功
> **通过时间**: 2025-06-23(实际时间:2026-06-23 16:12 UTC+8)
> **验证方法**: 5 个版本的探针插件逐步排查,最终在 opencode TUI 真实环境验证

---

## 验证目标

从 opencode 进程外部,能不能把一段文本送进去,让它作为 prompt 触发 LLM 回复。

---

## 最终结论

✅ **完全可行。** 使用 opencode server 插件,通过 `client.tui.appendPrompt` + `client.tui.submitPrompt`,能把外部消息(经 unix socket 送进插件)变成真正的 prompt,**落进用户当前打开的 session**,触发 LLM 回复。

**关键事实**:
- 消息会自动落进**当前打开的 session**(不是新建)
- 用户体验自然,就像自己在 TUI 输入框打字 + 回车
- 无需手动指定 session ID

---

## 验证过程(5 个版本探针)

### v1 → v2:解决插件阻塞问题

**问题**: v1 在插件顶层 `await client.session.list()`,导致 opencode 启动卡死(界面进不去)。

**原因**: opencode 加载插件时,client 的 HTTP 通道还未就绪,插件顶层 await 会阻塞整个启动流程。

**解决(v2)**: 把所有 client 调用推迟到事件回调里,插件顶层只做同步操作(起 socket)。

**教训**: **插件顶层不能 await client API。**

### v3 → v4:解决 console.error 干扰问题

**问题**: v3 用 `console.error` 写日志,导致 opencode TUI 画面被破坏,看起来像卡死。

**原因**: opencode TUI 占用整个终端,插件往 `console.error` 输出会破坏渲染画面。

**解决(v4)**: 只写文件日志(`/tmp/aibp-probe.log`),不用 console。

**教训**: **插件不能用 console.error/log,只能写文件。**

### v4:双路径测试

同时测两条路径:
- 路径 A: `client.tui.appendPrompt` + `client.tui.submitPrompt` ✅✅ **成功**
- 路径 B: `client.session.promptAsync` ❌ 服务端报错(`UnknownError: Unexpected server error`)

**结果**: 路径 A 完全成功,路径 B 因 body 格式问题报错(放弃修复,直接用 A)。

### v5:验证 session 行为

**问题**: v4 虽然成功,但**新建了一个 session**(title "New session - 时间戳"),而不是落进用户当前打开的那个。

**疑问**: `appendPrompt` 操作的到底是哪个输入框?会不会每次都新建?

**验证(v5)**: 用户先打开一个已有 session(和他聊了两句),然后发消息。

**结果**:
```
提交前最新 session: ses_10c76de9effeVLwiS6aQ1BlTrQ (初次问候与自我介绍请求)
提交后最新 session: ses_10c76de9effeVLwiS6aQ1BlTrQ (同一个)
🟢 落进已有 session! ID=ses_10c76de9effeVLwiS6aQ1BlTrQ
```

**结论**: `appendPrompt` 操作的是 **"TUI 当前显示的那个输入框"**,当用户打开一个 session 时,消息会正确落进那个 session,**不会新建**。行为完全符合直觉。

---

## 最终技术方案

| 项目 | 决定 |
|------|------|
| **插件类型** | server 插件(`PluginModule.server`) |
| **递送路径** | 路径 A:`client.tui.appendPrompt` + `client.tui.submitPrompt` |
| **消息流向** | unix socket → 插件 → client.tui → 输入框 → 提交 → 当前 session → LLM |
| **session 行为** | 自动落进当前打开的 session(无需指定 ID) |
| **日志方式** | 只写文件,不用 console |
| **client 调用时机** | 仅在事件回调里,不在插件顶层 |

---

## 踩坑总结

1. **插件顶层不能 await client API** → 所有 client 调用必须在回调里
2. **插件不能用 console.error/log** → 只能写文件日志(否则破坏 TUI 画面)
3. **session.promptAsync 报错** → 改用 tui.appendPrompt 路径
4. **v4 新建 session 的误解** → v5 证明会正确落进当前 session,无需担心

## 2026-06-23 后续验证:Toast 与事件驱动

### 验证目标

验证 `client.tui.showToast()` 是否可用,以及事件驱动的正确写法。

### 验证结果

✅ **完全成功**

1. **`client.tui.showToast()` 可用**
   - 在 server 插件的 event 回调里调用成功
   - 弹出 toast 提示 `"✅ Toast 成功!"`
   - 触发了 `tui.toast.show` 事件(日志记录)

2. **事件驱动格式正确**
   ```typescript
   return {
     event: async ({ event }) => {
       if (event.type === "session.idle") {
         // 调用 API
       }
     }
   }
   ```

3. **`client.tui.appendPrompt` + `submitPrompt` 在事件回调里可用**
   - 验证可正常调用
   - 能将消息送进对话

### 发现的问题

❌ **`session.idle` 会不断触发**

- 每次进入 idle 状态都会触发
- 导致消息重复发送
- 不适合用于触发消息

### 正确的触发时机

对于 aibp-opencode,应该:

1. **外部触发** (推荐)
   - Unix socket 收到 AIBP 消息时才调用
   - 避免自动触发,只在有外部消息时发送

2. **一次性事件** (备选)
   - `server.connected` - 只在连接时触发一次
   - 需要添加防重复逻辑(标志位)

3. **其他事件** (备选)
   - 添加防重复标志,确保每个消息只发送一次
