## 一、结论先行（任务4 接口设计直接吃这张表）

| agent | update 等效命令 | 装在哪 | 退出码语义 |
|-------|-------------|--------|-----------|
| **pi** | `pi update npm:aibp-pi` | `~/.pi/agent/npm/node_modules/aibp-pi/` | 0=ok，非0=失败 |
| **opencode** | **`rm -rf cache + opencode plugin aibp-opencode -g`** | `~/.cache/opencode/packages/aibp-opencode@latest/node_modules/aibp-opencode/` | 0=ok，非0=失败 |

> ⚠️ **重要修正**：opencode 的 `--force / -f` **不**起强制重装作用——它只控制配置文件的字面替换，对 npm cache 覆盖无效。所以**真正的 update 命令是「清 cache + 重装」**，这和首次安装是同一个流程。
>
> 两个底层命令都是**幂等**的（已装最新就 noop，没装就装）。调用者只需要 `exec.Command` + 检查 exit code。

任务4 的 `UpdateAIBP()` 方法具体实现就是上面这两段命令。**对 opencode 来说，`UpdateAIBP()` 和 `InstallAIBP()` 的实现可以完全相同**（详见 §四）。

## 二、pi update 实测

### 2.1 命令清单（来自 `pi update --help`）

```
Usage:
  pi update [source|self|pi] [--self|--extensions|--all] [--extension <source>]

Short forms:
  pi update                Update pi only
  pi update --all          Update pi and all extensions
  pi update <source>       Update one package
  pi update pi             Update pi only (self works as alias to pi)
```

### 2.2 关键事实

- **单包升级**：`pi update npm:aibp-pi` —— 拉 latest 覆盖已装版本。
- **因为当前 `InstallAIBP()` 没 pin 版本**（`aibpPiSpec = "npm:aibp-pi"`，D5 决策），所以 update 拿到的就是 latest，不需要 `@x.y.z`。
- **批量升级**：`pi update --extensions` —— 升级所有 extension。粒度粗，但对「一键全部对齐」的场景有用。
- 用户已实测过这三组命令，均有效。

### 2.3 实测对 microNeo 端的影响

- 当前 `.pi/agent/npm/node_modules/aibp-pi/package.json` → `"version": "1.0.1"`，`"aibp": { "protocol": "aibp-1" }`。
- 升级后：源码被替换（路径不变），但 `aibp.protocol` 字段值由新版的 package.json 决定。
- `:check-agent` 协议体检读的是 `aibp.protocol`，**与 npm 包 version 是两个独立的字段**（详见 D23 §2.2 事实 1）。

## 三、opencode update 实测

### 3.1 命令清单（来自 `opencode plugin --help`）

```
opencode plugin <module>

install plugin and update config

Positionals:
  module  npm module name                                                        [string] [required]

Options:
  -h, --help        show help
  -v, --version     show version number
      --print-logs  print logs to stderr
      --log-level   ... [choices: "DEBUG", "INFO", "WARN", "ERROR"]
      --pure        run without external plugins
  -g, --global      install in global config                              [default: false]
  -f, --force       replace existing plugin version                       [default: false]
```

### 3.2 关键事实

- **没有原生 `update` 子命令**：`opencode plugin --help` 只有一个位置参数 `module`，没有 subcommand。所以 `opencode plugin add ...` 会因 `add` 被当成无效位置参数而**只打印 help、不执行**（实测现象，下面 §3.4）。
- **`-f / --force` 真实语义（源码层）**：在 `packages/opencode/src/plugin/install.ts` 里，`force` 只在 `patchPluginConfig()` 中使用，控制 **tui.json / opencode.json 中 plugin[] 列表项的字面替换策略**——当 spec 形式变化（如从 `pkg@1.0.1` 改为 `pkg@1.0.2`）时强制覆盖列表项。**它对「重新从 npm 拉新版」没有任何作用**。
- **真正能强制重装的，是「先 `rm -rf` cache 再装」**：因为 `packages/core/src/npm.ts` 的 `Npm.add()` 实现里有一段短路逻辑——
  ```ts
  if (yield* afs.existsSafe(path.join(dir, "node_modules", name))) {
    return resolveEntryPoint(name, path.join(dir, "node_modules", name))
    // ⭐ cache 存在就直接 return，不调 reify（不重新装）
  }
  ```
  只要 `~/.cache/opencode/packages/<spec>/node_modules/<name>` 存在，`Npm.add` 就直接返回，**根本不去查 registry**。所以 `-f` 救不了这种情况；唯一能强制重装的办法是事先 `rm -rf` 掉这个目录。
- **官方文档佐证**：「npm plugins are installed automatically using Bun at startup」（opencode 官方 plugins 文档）—— 但启动时也是走 `Npm.add`，所以**同样会短路**，不会主动查 registry 拿新版。这是个潜在的"启动也不自动更新"问题（详见 §五遗留事项）。

### 3.3 与当前 `OpencodeEnsurer.InstallAIBP()` 实现的关系

当前实现（D3 策略）—— **它就是对的**：
```go
cmd := exec.Command("opencode", "plugin", aibpOpencodeSpec, "-g")
// + 前置 os.RemoveAll(cacheDir) 保证 cache 不陈旧
```

**不需要简化**。源码证据显示 `rm -rf cache` 是必要的，`-f` 帮不上忙。D3 当时选择「rm -rf + plugin add -g」（而不是「plugin add -g -f」）是基于对 npm 缓存行为的正确理解。

> 此前我误以为「`plugin add -g -f` 等价于 rm -rf + plugin add -g」，是错的。本次工作记录就是来订正这个错误看法的。

### 3.4 实测结果（本次亲跑）

```bash
$ opencode plugin aibp-opencode -g -f

  Install plugin aibp-opencode
│
◇  Plugin package ready
◇  Detected tui target
◇  Plugin config updated
●  Already configured in /Users/sollawen/.config/opencode/tui.json
│
◆  Installed aibp-opencode
│
●  Scope: global (/Users/sollawen/.config/opencode)
│
└  Done

exit code: 0
```

观察（**已根据源码修正**）：

| 检查项 | 结果 | 解读 |
|--------|------|------|
| exit code | 0 | 命令本身成功 |
| `Plugin package ready` | opencode 走完 install 流程 | 但**只走了 `Npm.add` 的短路分支**（cache 已存在） |
| `Already configured in .../tui.json` | 配置已有，没改动 | 与 `-f` 无关——相同 spec 时本来就不会改 |
| `Installed aibp-opencode` | opencode 自己的总结日志 | **不代表重装了**——`Npm.add` 在 cache 存在时返回的是缓存结果 |
| `~/.cache/.../package.json` mtime | **未变** | 符合预期：`Npm.add` 走短路，根本没动 cache |
| tui.json 的 `plugin[]` | `["aibp-opencode"]`（仍 unpinned） | ✅ 没被改成 pinned 版本，update 链仍可走 |

> 如果想让 cache 真正刷新成新版（比如发 1.0.2 后再跑），需要**先 `rm -rf` 掉 `~/.cache/opencode/packages/aibp-opencode@latest/`**，然后再 `opencode plugin aibp-opencode -g`（不需要 `-f`）。

## 四、对任务4 接口设计的输入

任务4 要做的 `UpdateAIBP()` 方法（伪代码，最终代码以任务4 实施文档为准）：

```go
// pi：update 命令本身就是 update，无副作用区别
func (PiEnsurer) UpdateAIBP() error {
    cmd := exec.Command("pi", "update", aibpPiSpec) // "npm:aibp-pi"
    return cmd.Run()
}

// opencode：update 与 install 是同一套流程——清 cache + 重装
func (OpencodeEnsurer) UpdateAIBP() error {
    cacheDir := filepath.Join(opencodeCacheDir(), "packages", aibpOpencodeSpec+"@latest")
    _ = os.RemoveAll(cacheDir) // 同 InstallAIBP() 的 D3 策略
    cmd := exec.Command("opencode", "plugin", aibpOpencodeSpec, "-g")
    return cmd.Run()
}
```

> ⭐ **关键简化**：**`OpencodeEnsurer.UpdateAIBP()` 和 `InstallAIBP()` 的实现完全相同**。可以在代码里直接共用一段：
> ```go
> func (e OpencodeEnsurer) installOrUpdate() error {
>     cacheDir := filepath.Join(opencodeCacheDir(), "packages", aibpOpencodeSpec+"@latest")
>     _ = os.RemoveAll(cacheDir)
>     return exec.Command("opencode", "plugin", aibpOpencodeSpec, "-g").Run()
> }
> func (OpencodeEnsurer) InstallAIBP() error { return OpencodeEnsurer{}.installOrUpdate() }
> func (OpencodeEnsurer) UpdateAIBP()  error { return OpencodeEnsurer{}.installOrUpdate() }
> ```
> 这不是接口简化（两个方法语义仍不同，分别暴露给 `Ensure()` 编排逻辑），是实现复用。

**接口形态**（草案）：
```go
type AgentEnsurer interface {
    AgentName() string
    HasAgent() bool
    HasAIBP() bool
    AIBPVersion() (string, error)
    InstallAIBP() error   // 已有：首次安装
    UpdateAIBP() error    // 🆕：升级到 latest（今天摸清）
}
```

**与现有 `InstallAIBP()` 的语义区分**：

| 方法 | pi 的实现 | opencode 的实现 | 触发场景 |
|------|----------|----------------|---------|
| `InstallAIBP()` | `pi install npm:aibp-pi` | rm cache + `plugin add -g` | `:check-agent` 发现没装 |
| `UpdateAIBP()` | `pi update npm:aibp-pi` | rm cache + `plugin add -g`（**同上**） | `:check-agent` 发现 npm 有新版 |

> **「是否每次 check 都调 UpdateAIBP()」的取舍**：
> - **是（无条件）**：实现简单；副作用可控（已是 idempotent）。代价：每次 `:check-agent` 都执行一次 rm + 重装（联网、几秒延迟）。
> - **否（先查 latest，对比再决定）**：引入「查 registry」的逻辑（联网查 npm view / curl registry），失败处理要设计。更精确但更复杂。
>
> **倾向**：**先做无条件版本**。理由：(a) 命令本身 idempotent，副作用可控；(b) 实施简单，能尽快闭合任务4；(c) 用户体感：「`:check-agent` 后一切都是最新版」是直觉上的预期。如有性能/网络顾虑，再退回到查 latest 版本。**今天不决，留给任务4 实施时定**。
