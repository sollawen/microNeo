#### 2.2 notePaneOpen 重写（`notepane.go:133`）

```go
func notePaneOpen(h *BufPane) bool {
    n := TheNotePane
    if n == nil {
        return false
    }
    if n.isOpen {
        return true  // 已开，幂等
    }

    // 1. Discover
    receivers, err := aibp.Discover()
    if err != nil {
        InfoBar.Message("✗ discover error: " + err.Error())
        return false
    }
    if len(receivers) == 0 {
        InfoBar.Message("✗ no receiver found")
        return false
    }

    // 2. case 1：直接赋值（无需查缓存，只有一个）
    if len(receivers) == 1 {
        n.selectedReceiver = receivers[0]
        n.open()
        return true
    }

    // 3. case 2+：先查缓存命中（Socket 在 receivers 列表里）
    if n.selectedReceiver.Socket != "" {
        for _, r := range receivers {
            if r.Socket == n.selectedReceiver.Socket {
                n.open()  // 命中，复用缓存
                return true
            }
        }
    }

    // 4. case 2+ 未命中：弹 SelectPane
    names := make([]string, len(receivers))
    for i, r := range receivers {
        names[i] = r.Name
    }
    TheSelectPane.Open(names, "Receiver", func(s *string) {
        if s == nil {
            // Esc：清零缓存（走到此分支时缓存已失效，决策 14）
            n.selectedReceiver = aibp.RegFile{}
            InfoBar.Message("✗ 已取消")
            return
        }
        // Enter：找到 name 对应的 RegFile
        for _, r := range receivers {
            if r.Name == *s {
                n.selectedReceiver = r
                n.open()
                return
            }
        }
        // 理论上不会到这（SelectPane 的 items 来自 receivers），防御性
        InfoBar.Message("✗ internal: selected name not in receivers")
    })
    return true
}
```

## 四、风险与回滚

### R1：SelectPane callback 时序

**风险**：onSelect 回调里调用 `n.open()`，如果此时 SelectPane 还没 Close，两个浮层渲染冲突。

**验证**：读 `selectpane.go:100` `HandleEvent` 的 Enter 分支，确认执行顺序是：
```
s.Close()        // 先关
s.onSelect(&s.items[s.selected])  // 后回调
```
如果是先 callback 后 Close，要在 callback 内手动 `TheSelectPane.Close()` 开头加一行。

**回滚**：Phase 1 commit 单独 revert。

### R2：NotePaneSend 漏改 receiver 引用

**风险**：删 Discover 块后，函数体后面还有 `receiver.Socket` / `receiver.Name` 引用，但 `receiver` 变量没了 → 编译错。

**防御**：编译器会报错（Go 不允许未声明变量），不会静默通过。**改完必须 `make build`**。

### R3：micro 原生代码影响点

D12 改动的文件**全部是 microNeo 自己的文件**（`notepane.go` / `selectpane.go` / `bufpane.go` forwarding / `defaults_*.go`），**不动 micro 原生代码**。

唯一与 micro 原生耦合的是 `bufpane.go` 的两个 forwarding if——它们在 `BufPane.HandleEvent` 顶部和 `BufPane.Display` 末尾。micro 升级时如果重构了这两个方法，forwarding 要重新插入。**记一笔在 AGENTS.md 或本文档的"维护注意"**。

### R4：边框名字截断边界

**风险**：notePane 宽度很窄时（终端很窄），`nameCap-3` 可能 ≤ 0，截断切片 `name[:负数]` 会 panic。

**防御**：Phase 2 代码骨架加 `if nameCap < 6 { /* 太窄就不显示名字 */ return }` 守卫。

### 回滚策略

| 范围 | 操作 |
|------|------|
| 单 Phase 回滚 | `git revert <phase-N-commit>` |
| 全部回滚 | `git reset --hard d12-baseline`（Phase 0 打的 tag） |

每个 Phase 独立 commit，**不要 squash**——保留可回滚粒度。

## 七、维护注意（写完后留给未来）

1. **bufpane.go forwarding 是与 micro 原生代码的唯一耦合点**。micro 升级时如果 `BufPane.HandleEvent` 或 `BufPane.Display` 被重构，forwarding 的 2 个 if 要重新插入对应位置。
