# 方案 B 实施计划（合并版 001）

> 依据：`docs/光标滚动-方案B.md`（原始需求与设计）
> 现状：`docs/光标滚动-修改总结.md`（v1.0.5 已知限制）
> 范围：把光标滚动从「每帧 render + viewportRowmap shadow」重构为「displayToBuffer → screenBuffer（单一数据源）→ showBuffer blit」
> 编译：`make build`（或 `make build-quick` 跳过 generate）

---

## 1. 目标与根因

### 1.1 目标根 bug

以 `docs/sample.md` line55-59 表格为例：
- 表格段在原生编辑态占 5 行；光标移出后切回 MD 渲染态，凭空多出若干装饰行（frame/标题等）→ 占 9 行
- 光标在 line59 时按 ↓：程序基于「无装饰行的旧 viewportRowmap」计算 scroll up 1 行
- scroll 完成后表格段才切回 MD 展开装饰行 → 旧 map 失配 → 光标被挤出 viewport 底边之外，用户看不见

### 1.2 根因

`viewportRowmap` 是**上一帧** `displayBufferMD` 的产物。`Relocate()` 做滚动决策时，这张 map 反映的是「光标在旧位置、旧 editMode 段」时的 viewport 几何。光标一跨段，editMode 段切换（native ↔ MD），装饰行增删，map 立即失效。

### 1.3 方案 B 核心

把「渲染」和「显示」分离，引入离屏单一数据源 `screenBuffer`：

```
原生时序：  状态变更 → display(startLine) 渲染并直接写屏 → show
方案 B：    状态变更 → displayToBuffer(startLine) 渲染到 sb（不写屏）
                     → Relocate 基于 fresh sb 做滚动决策
                     → showBuffer(startLine) 把 sb blit 到真屏
```

**关键承诺**：Relocate 决策前永远有一份 **fresh** 的 screenBuffer，从根本上消灭「map 落后一帧」的时序断层。

### 1.4 度量目标

| 目标 | 度量 |
|------|------|
| 修复跨段切换时光标飞出 viewport | sample.md 表格场景连续按 ↓ 时光标始终可见 |
| Relocate 决策永远基于 fresh map | screenBuffer 在每次 Relocate 入口重渲，与当前 cursor 位置一致 |
| 主循环零改动 | `cmd/micro/micro.go` git diff = 0 |
| render 管线零改动 | `internal/md/render_*.go` git diff = 0 |
| 原生渲染输出不变 | 非 MD 文件**输出**字节级不变（sink 恒 realScreenSink 透传，逐字节等价）；共享 gutter 函数（drawGutter/drawDiffGutter/drawLineNum）函数体被改但行为等价，调用路径多一层 `setCell` 间接，性能可忽略 |

**非目标**：不重写原生 `displayBuffer()`；不改动 `softwrap.go`；不改动 `actions.go`（84 处 `h.Relocate()` 全不动）；不解决 micro 原生跨段渲染问题。

## 3. 核心设计

### 3.1 数据结构：screenRow 同源

```go
// internal/display/bufwindow_md.go 顶部新增

// mdCell 是 screenBuffer 里一个显示单元（rune + 合字 + 样式）。
type mdCell struct {
    r     rune
    combc []rune
    style tcell.Style
}

// screenRow 是 screenBuffer 的一行：line 元数据 + cells 显示数据【同源生成】。
// cells 覆盖整行宽度（gutterOffset + bufWidth），gutter 区也纳入，供 showBuffer 整行 blit。
type screenRow struct {
    line   int       // 对应 buffer 行（绝对）；-1=装饰行，-2=空白填充。供查询
    segRow int       // softwrap 段号（同 SLoc.Row）。供查询
    cells  []mdCell  // 显示数据；showBuffer blit 后置 nil 释放
}

// screenBuffer 是 MD 渲染的离屏单一数据源：既是渲染结果，又是 line↔vY 查询地图。
type screenBuffer struct {
    rows      []screenRow  // 长度动态，≤ 2*bufHeight
    startLine SLoc         // 本批 rows 渲染时的起点（case A/C 范围检查用）
    blitStart int          // 上次 showBuffer blit 的 rows 起始下标（点击映射用，见 §6.2）
    originX   int          // 渲染窗口原点 X（= w.X），绝对坐标→本地坐标转换
    originY   int          // 渲染窗口原点 Y（= w.Y）
    width     int          // 每行 cell 宽度（= gutterOffset + bufWidth）
    overflow  bool         // 2×bufHeight 兜底触发标记（→ fallback）
    cursorX   int          // 光标在 sb 坐标系的位置（showBuffer 同步用）
    cursorY   int
    cursorOK  bool         // 是否记录到有效光标
}
```

**BufWindow 字段调整**（`bufwindow.go:38-42`）：

```go
// 删除：viewportRowmap []SLoc
// 新增：
sb          *screenBuffer  // MD 离屏缓冲（单一数据源）
sink        cellSink       // 当前渲染目标；默认 realScreenSink，displayToBuffer 期间切到 sb
prevCursorY int            // 上一帧光标所在 **buffer 行号**（=activeCursor.Y，micro Cursor 嵌入 Loc，其 Y 本就是 buffer 行而非屏幕行）。与 segment.BufStartLine/BufEndLine 同坐标系，供 findSegmentContaining 查旧段。-1=未初始化
```

**设计要点**：
- `line`/`segRow` 与 `cells` **同源生成** —— displayToBuffer 一次渲染同时产出，永不失配（对比方案 A 的 viewportRowmap shadow 双份账）
- `cells` 逻辑弃、物理复用：showBuffer blit 后置 `row.cells = nil`（释放引用）；`reset`/displayToBuffer 在 cap/width 不变时复用底层 slice 仅覆盖写，不重新分配，避免每帧 make 的 GC 压力
- `line`/`segRow` 长期保留，供 Relocate 查询与点击映射
- `prevCursorY = -1` sentinel：首帧 `findSegmentContaining(-1)` 返回 nil → 停止条件 `oldSegDone=true`，不会过度渲染

### 3.2 retarget 机制：cellSink 接口

整个方案唯一有设计陷阱的点，单列一节。

#### 问题

`displayToBuffer` 期间，renderer 和 gutter 函数**都不能写真屏**（真屏由 showBuffer 统一 blit）。但现状它们都直接调 `screen.SetContent`（绝对坐标），且 `drawGutter/drawDiffGutter/drawLineNum` 是 native + MD 共享的 —— **直接给它们加 `buf` 参数会侵入原生路径**，违背项目原则。

#### 方案：runtime switchable sink

```go
// cellSink 是渲染目标抽象。SetContent 签名与 screen.SetContent 对齐（绝对坐标）。
type cellSink interface {
    SetContent(x, y int, mainc rune, combc []rune, style tcell.Style)
}

// realScreenSink 包装 screen.SetContent，原生路径行为字节级不变。
type realScreenSink struct{}
func (realScreenSink) SetContent(x, y int, mainc rune, combc []rune, style tcell.Style) {
    screen.SetContent(x, y, mainc, combc, style)
}

// screenBuffer 实现 cellSink：绝对坐标 → 本地 (row, cell)。
func (s *screenBuffer) SetContent(x, y int, mainc rune, combc []rune, style tcell.Style) {
    row := y - s.originY
    col := x - s.originX
    if row < 0 || row >= len(s.rows) || col < 0 || col >= s.width {
        return // 越界丢弃（screenBuffer 是 2x cap，渲染可能溢出尾部）
    }
    s.rows[row].cells[col] = mdCell{r: mainc, combc: combc, style: style}
}

// screenBuffer 记录光标位置（cells 坐标系）。
func (s *screenBuffer) ShowCursor(x, y int, main bool) {
    s.cursorX, s.cursorY, s.cursorOK = x, y, true
}

// BufWindow 统一入口：renderer / gutter 全部改调这个。
func (w *BufWindow) setCell(x, y int, mainc rune, combc []rune, style tcell.Style) {
    w.sink.SetContent(x, y, mainc, combc, style)
}
func (w *BufWindow) setShowCursor(x, y int, main bool) {
    if _, ok := w.sink.(*screenBuffer); ok {
        w.sb.ShowCursor(x, y, main)
    } else {
        w.showCursor(x, y, main)  // 原生路径直接写真屏
    }
}
```

#### retarget 改动点（机械替换，行为等价）

`screen.SetContent(...)` → `w.setCell(...)`，`w.showCursor(...)` → `w.setShowCursor(...)`：

| 文件 | 行号 | 函数 | 说明 |
|------|------|------|------|
| `bufwindow_md.go` | 195 | `renderSegmentMD` 内容区 | MD 渲染 |
| `bufwindow_md.go` | 520 | `renderSegmentNative` 行内 | MD 的 native 回退 |
| `bufwindow_md.go` | 667 | `renderSegmentNative` 行尾填充 | MD 的 native 回退 |
| `bufwindow_md.go` | 751 | `displayBufferMD` 末尾空白填充 | 末阶段随 displayBufferMD 改造 |
| `bufwindow_md.go` | 813 | `drawGutterAndLineNumMD` | 续行/空行行号位 |
| `bufwindow.go` | 328 | `drawGutter` | **共享**（native+MD） |
| `bufwindow.go` | 361,377,383,385,392 | `drawDiffGutter` / `drawLineNum` | **共享**（native+MD） |
| `bufwindow_md.go` | 525 | `renderSegmentNative` 内 `w.showCursor` | → `w.setShowCursor` |

**`bufwindow.go` 的 `displayBuffer()`（native，713/860）不改**：MD 路径不调 `displayBuffer`，native 路径 sink 恒 realScreenSink，保持最小侵入。

#### 关键安全性

native 路径（非 MD）**从不切换 sink**，`w.sink` 恒为 realScreenSink。`w.setCell` → `realScreenSink.SetContent` → `screen.SetContent`，**输出字节级不变**，仅多一层方法调用（性能可忽略）。需诚实说明：retarget 改动了 native + MD 共享的 `drawGutter`/`drawDiffGutter`/`drawLineNum` 函数体（bufwindow.go:328/361/377/383/385/392），但 native `displayBuffer`（713/860）的 SetContent 调用路径不变；改动是"函数体里多一层 setCell 间接"、行为等价，而非"零侵入"。验证手段：开非 MD 文件，输出应与改动前完全一致（见 §10.6）。

#### sink 切换时机

```go
// displayToBuffer 入口
w.sink = w.sb
... renderer 跑 ...
defer func() { w.sink = realScreenSink{} }()  // 复位（防御性）

// showBuffer：不切 sink（直接读 sb.rows blit，不经过 renderer）
```

### 3.3 时序：三种事件路径统一

完整时序（主循环 `cmd/micro/micro.go` 不动）：

```
BufWindow.Display (bufwindow.go:932, 微调)：
    if !w.Buf.IsMD { w.displayBuffer(); return }
    // MD 路径
    w.displayBufferMD(w.editMode)
        → displayToBuffer(w.StartLine, showCursor=true)   // 渲染到 sb
        → Relocate 已在上一轮算好 StartLine（或本轮 Display 前已调）
        → showBuffer(w.StartLine)                          // sb blit 到真屏
        → w.updatePrevCursor()

BufWindow.Relocate (bufwindow.go:249, 微调)：
    if w.Buf.IsMD {
        w.relocateVerticalMD(c, scrollmargin, height)      // 内部调 displayToBuffer（fresh）
    }
```

三种 case 在「Relocate 入口必渲染 fresh sb + showBuffer 从 sb blit」收敛下，**不需要 case 分发**：

| 场景 | 触发 | displayToBuffer 渲染起点 | showBuffer 起点 |
|------|------|------------------------|----------------|
| **case A** 编辑/光标移动（跨段） | MoveCursor → Relocate | 上一帧 `w.StartLine`（cursor 仅移动 1 行，大概率在 sb 2× 覆盖内） | Relocate 精算后的 StartLine（在 sb 范围内 → 复用） |
| **case B** 纯视口（scroll/page） | HandleEvent 改 StartLine → Display | 新 `w.StartLine`（case B 不关心 cursor 段） | 同一 `w.StartLine` |
| **case C** goto/search 大幅移动 | Relocate 估算 → 精算 | 估算 `c.Line - scrollmargin`（1:1） | Relocate 精算后（在 sb 2× 范围内 → 复用） |

**关键洞察**：
1. **Relocate 入口必渲染**：保证决策基于 fresh map。这是修正「startLine 相等就复用」stale bug 的根本 —— cursor 跨段时 StartLine 未变、editMode 段切换，cells 必须更新。
2. **showBuffer 从 sb blit，不再二次渲染**：Relocate 算出的 StartLine 落在 sb 2× 覆盖区间内 → 直接 blit，零闪烁。
3. **cells 是帧内临时资源**：showBuffer 完成后 `cells = nil`，rowmap（line/segRow）保留供下一帧 Relocate 决策。

## 4. 核心函数实现

### 4.1 displayToBuffer（渲染到 screenBuffer）

```go
// displayToBuffer 从 startLine 渲染到 screenBuffer，单一数据源。
// showCursor: 是否把 cursor 段走 native（按 call site 固定取值，见 §7）。
func (w *BufWindow) displayToBuffer(startLine SLoc, showCursor bool) {
    b := w.Buf
    if w.Height <= 0 || w.Width <= 0 { return }
    w.ensureMDConfigReady()

    bufWidth, bufHeight := w.bufWidth, w.bufHeight
    cap := 2 * bufHeight  // 2× 吸收跨段展开增量

    // 初始化 screenBuffer（2× 容量，行宽含 gutter）
    w.sb.reset(cap, w.gutterOffset+bufWidth, w.X, w.Y)
    w.sb.startLine = startLine
    w.sb.cursorOK = false

    // 切 sink（§3.2）
    w.sink = w.sb
    defer func() { w.sink = realScreenSink{} }()

    // 可见段（沿用现状 filter）
    visibleStart := startLine.Line
    visibleEnd := visibleStart + cap
    if visibleEnd >= b.LinesNum() { visibleEnd = b.LinesNum() - 1 }
    segments := filterSegmentsToVisible(b.MDSegments, visibleStart, visibleEnd)
    cursors := b.GetCursors()

    // 旧光标段（跨段切换时，旧段从 native 切回 MD，装饰行必须整段渲染）
    oldSeg := w.findSegmentContaining(w.prevCursorY)
    // oldSegDone 初值（M1 修复）：oldSeg 不在本次渲染窗口（整体在 startLine 上方，
    // 或整体在下方 +scrollmargin 余量外）时直接视为完成。否则循环永远遇不到 oldSeg
    // → 三目的凑不齐 → 一路渲染到 overflow。真实触发：PageDown 把 StartLine 推到
    // oldSeg 之下后按 ↓（case A），oldSeg 整体在 startLine 之上。
    oldSegDone := oldSeg == nil ||
        oldSeg.BufEndLine < startLine.Line ||
        oldSeg.BufStartLine > startLine.Line + bufHeight + scrollmargin

    scrollmargin := int(b.Settings["scrollmargin"].(float64))
    cursorShowed := false

    line := startLine.Line
    vY := 0
    for {
        // 兜底：2×bufHeight 强制退出
        if vY >= cap { w.sb.overflow = true; break }

        seg := w.findSegmentContaining(line)
        if seg == nil { break }  // buffer 末尾

        // ★ 同源写入 line/segRow + cells（renderer 内部完成，见 §4.4）
        if showCursor && w.editMode && hasCursorInside(*seg, cursors) {
            vY = w.renderSegmentNative(*seg, vY)  // cursorShowed 在 renderer 内置
            cursorShowed = true
        } else {
            vY = w.renderSegmentMD(*seg, vY)
        }
        if seg == oldSeg { oldSegDone = true }
        line = seg.BufEndLine + 1

        // 三目的停止条件（忠于需求文档伪代码）
        canBreak := true
        // 目的1：旧光标段必须完整渲染（跨段切换，旧段装饰行不能截断）
        if !oldSegDone { canBreak = false }
        // 目的2（showCursor=true）：cursor 段必须已渲染
        if showCursor && !cursorShowed { canBreak = false }
        // 目的3：必须渲染到 viewport 底边以下，留足 Relocate 微调余量
        if vY < bufHeight+scrollmargin { canBreak = false }
        if canBreak { break }
    }
    // 剩余不填满 —— screenBuffer 只渲染实际内容，showBuffer 负责尾部空白填充
}
```

**与方案 A preRender 的关系**：停止条件逻辑同源（目的驱动）。差异：方案 A dry-run 只写 rowmap；本方案正常渲染，同时写 rowmap（line/segRow）和 cells。

### 4.2 showBuffer（blit 到真屏）

```go
// showBuffer 把 screenBuffer 从 startLine 起的一段 blit 到真屏。
// 职责单一：copy + 边界处理，不做 dirty 判定。
func (w *BufWindow) showBuffer(startLine SLoc) {
    bufHeight, bufWidth := w.bufHeight, w.bufWidth

    // overflow 兜底：走原生 displayBuffer
    if w.sb.overflow {
        // ⚠️ 已知限制（见 §8.2）：displayBuffer 是原生渲染器，对 .md 会画出带
        // # / | 的原始字符，用户可见一帧"原始 markdown"。触发条件：单 segment
        // > 2×bufHeight（超长 codeblock 跨段）。下一帧 startLine 变化时自纠正。
        w.displayBuffer()
        return
    }

    // case B 范围检查：startLine 超出 sb 覆盖区间，或宽度变化（resize / 切 ruler 等改变
    // gutterOffset+bufWidth，旧 cells 会越界）→ 补渲染（showCursor=false：滚动场景）
    if !w.sb.covers(startLine) || w.sb.width != w.gutterOffset+w.bufWidth {
        w.displayToBuffer(startLine, showCursor=false)
    }

    // 找 startLine 在 sb.rows 中的起点 vY
    startVY, ok := w.sb.rowIndexOf(startLine)
    if !ok {
        // startLine 落装饰行/边界：向下找首个内容行（沿用现状 §修改总结3.3）
        startVY, ok = w.sb.rowIndexNearest(startLine)
        if !ok { return }
    }
    w.sb.blitStart = startVY  // 记录，供点击映射（§6.2）

    // blit，处理尾部不足
    endVY := startVY + bufHeight
    if endVY > len(w.sb.rows) { endVY = len(w.sb.rows) }

    for vY := 0; vY < bufHeight; vY++ {
        srcRow := startVY + vY
        if srcRow >= endVY {
            // 尾部空白填充（沿用 displayBufferMD 现状逻辑）
            w.drawGutterAndLineNumMD(vY, -1, false)
            for col := 0; col < bufWidth; col++ {
                screen.SetContent(w.X+w.gutterOffset+col, w.Y+vY, ' ', nil, config.DefStyle)
            }
            continue
        }
        row := &w.sb.rows[srcRow]
        // 整行 blit（gutter + content，§3.1 cells 覆盖全宽）
        // 防御性边界（M2）：cells 长度与 gutterOffset+bufWidth 不一致（resize/切 ruler
        // 未及时补渲染）时只 blit 实际存在的 cell，缺失列按空格补，绝不越界 panic
        blitW := w.gutterOffset + bufWidth
        for col := 0; col < blitW; col++ {
            if col >= len(row.cells) {
                screen.SetContent(w.X+col, w.Y+vY, ' ', nil, config.DefStyle)
                continue
            }
            c := row.cells[col]
            screen.SetContent(w.X+col, w.Y+vY, c.r, c.combc, c.style)
        }
    }

    // 同步光标位置到 terminal
    if w.sb.cursorOK {
        w.showCursor(w.X+w.sb.cursorX, w.Y+w.sb.cursorY, true)
    }

    // cells 用完即弃（内存优化），line/segRow 保留
    for i := range w.sb.rows {
        w.sb.rows[i].cells = nil
    }
}
```

### 4.3 relocateVerticalMD 改造（统一 case A/C）

```go
// relocateVerticalMD：入口 displayToBuffer（唯一一次渲染）+ 查询微调。
// case A（方向键）与 case C（goto/search/pageup/pagedown）同一逻辑，
// 唯一区别是渲染起点选择。
func (w *BufWindow) relocateVerticalMD(c SLoc, scrollmargin, height int) bool {
    // 1. 估算渲染起点（cheap 预判，渲染前）
    var displayStart SLoc
    if w.sb != nil && w.sb.coversLine(c.Line) {
        displayStart = w.StartLine              // case A：cursor 在上一帧 sb 覆盖内
    } else {
        displayStart = SLoc{Line: c.Line - scrollmargin, Row: 0}  // case C：跳远，1:1 估算
    }
    // ⚠️ coversLine 用上一帧 sb 判定【新】光标位置，存在误判：case C（远跳）被误判为
    // case A（用旧 StartLine 当 displayStart）→ 渲染后 cursor 不在 2× 内 → 下面
    // rowIndexOf(c) 失败 → relocateVerticalNativeFallback（1 帧不完美，下一帧自纠正）。
    // 这是降级不是 bug，实施时按此路径验收，勿当意外。

    // 2. 唯一一次渲染（showCursor=true：导航场景，希望看到 cursor 行）
    w.displayToBuffer(displayStart, showCursor=true)

    // 3. 查询微调（非渲染）：让 cursor 落在期望 margin
    if w.sb.overflow {
        return w.relocateVerticalNativeFallback(c, scrollmargin, height)  // 极端兜底
    }
    cursorRow, ok := w.sb.rowIndexOf(c)
    if !ok {
        // cursor 不在 2× 内（极端：单 segment > 2× viewport）→ native 兜底
        return w.relocateVerticalNativeFallback(c, scrollmargin, height)
    }
    botMarginRow := height - 1 - scrollmargin
    if cursorRow > botMarginRow {
        delta := cursorRow - botMarginRow
        loc, ok := w.sb.slocAt(delta)
        if !ok || loc.Line < 0 {
            // delta 落装饰行：向下找首个内容行（沿用现状）
            for d := delta + 1; d < len(w.sb.rows); d++ {
                if l, ok := w.sb.slocAt(d); ok && l.Line >= 0 { loc = l; break }
            }
            if loc.Line < 0 {
                return w.relocateVerticalNativeFallback(c, scrollmargin, height)
            }
        }
        w.StartLine = loc
        return true
    }
    if cursorRow < scrollmargin {
        w.StartLine = w.Scroll(c, -scrollmargin)  // 向上：段内 1:1（沿用现状）
        return true
    }
    return true
}
```

**case C 为什么一次就够**：估算时 cursor 在 `displayStart + scrollmargin`（1:1），渲染后 segment 展开让 cursor 屏行偏移 ±Δ，但 screenBuffer 是 2×，`scrollmargin < bufHeight`，Δ 被 2× 余量吸收 → cursor 必在 sb 内 → 查询微调即可，无需二次渲染。

### 4.4 displayBufferMD → 薄包装

```go
// bufwindow.go:940 分发点不变，displayBufferMD 改造为薄包装：
func (w *BufWindow) displayBufferMD(editMode bool) {
    if w.Height <= 0 || w.Width <= 0 { return }
    w.ensureMDConfigReady()
    // StartLine 已由上一轮 Relocate 算好；sb 失效则补渲染。
    // 失效条件：(1) 首帧 nil  (2) StartLine 不在 sb 覆盖区间  (3) overflow
    //          (4) 宽度变化（resize / 切 ruler·diffgutter·scrollbar 改变 gutterOffset+bufWidth）
    //              ——必须重渲，否则 showBuffer blit 循环用新宽度访问旧 cells → 越界 panic（M2）。
    //              softwrap/wordwrap/tabsize 切换：由 SetBuffer.OptionCallback 触发的 Relocate
    //              在 001 设计下无条件重渲 sb，已自纠正；如保守可在 sb 加 softwrap 镜像字段一并校验。
    if w.sb == nil || !w.sb.covers(w.StartLine) || w.sb.overflow ||
       w.sb.width != w.gutterOffset+w.bufWidth {
        w.displayToBuffer(w.StartLine, showCursor=true)
    }
    w.showBuffer(w.StartLine)
    w.updatePrevCursor()  // 供下一帧 displayToBuffer 的 oldSeg 停止条件
}

func (w *BufWindow) updatePrevCursor() {
    // ⚠️ 坐标系澄清：c 是 *buffer.Cursor，其 Y 是 buffer 行号（非屏幕行）。
    // micro 的 Cursor 嵌入 Loc{X,Y}，Y 直接作 buffer 行索引用（见 cursor.go: c.buf.LineBytes(c.Y)）。
    // 因此 prevCursorY 与 segment.BufStartLine/BufEndLine 同坐标系，findSegmentContaining 可直接匹配，无需 row↔line 转换。
    if c := w.Buf.GetActiveCursor(); c != nil {
        w.prevCursorY = c.Y
    }
}
```

## 5. viewportRowmap 迁移与删除

### 5.1 读取方迁移

| 现状函数 | 现状读法 | 迁移后 |
|---------|---------|--------|
| `LineToScreenRow(line, row)`（`bufwindow_md.go:855`） | O(bufHeight) 扫 viewportRowmap | O(len(sb.rows)) 扫 sb.rows 找 line+segRow，签名不变 |
| `ScreenRowToLine(offset)`（`bufwindow_md.go:842`） | viewportRowmap[offset] | sb.rows[sb.blitStart + offset].line |
| `LocFromVisual`（`bufwindow.go:292`，点击映射） | viewportRowmap[svloc.Y-w.Y] | sb.rows[sb.blitStart + (svloc.Y-w.Y)].line |

**复杂度备注（O2）**：`LineToScreenRow` 从 O(bufHeight) 变 O(2×bufHeight)（sb.rows 长度翻倍），但 ~100 次比较在 Relocate 热路径上仍可忽略，不优化。

### 5.2 ScreenRowToLine 的 offset 语义

`ScreenRowToLine(offset)` 现状 offset 是「相对 viewport 顶部的屏行」。方案 B 下 viewport 显示内容由 showBuffer 从 sb 的 `blitStart` blit，故 `offset ↔ sb.rows[blitStart + offset]`。

`blitStart` 由 showBuffer 一次赋值产生（§4.2），与 sb 同源，无第二份数据。

### 5.3 删除时机

`viewportRowmap` 字段在 **§8 阶段 6** 删除（所有读取方迁移完且验证通过后）。删除前用 git branch 保留可回退点。

## 6. showCursor 与 editMode 的职责分离

### 6.1 showCursor 按 call site 固定取值

不沿调用栈传操作类型，按 call site 取值：

| call site | showCursor | editMode gate | 说明 |
|-----------|-----------|---------------|------|
| `relocateVerticalMD` → `displayToBuffer` | **true** | `editMode && showCursor` → cursor 段 native | 导航：方向键/goto/search/pageup/pagedown |
| `displayBufferMD` → `displayToBuffer`（补渲染） | **true** | 同上 | Display 主路径，希望看到 cursor |
| `showBuffer` 自愈（case B 越界补渲染） | **false** | cursor 段也 MD（不关心） | 滚动：鼠标滚/ScrollUp·Down |

```go
// displayToBuffer 内部 native 分支
if showCursor && w.editMode && hasCursorInside(*seg, cursors) {
    // native（cursor 可见）
} else {
    // MD（cursor 段也走 MD 渲染）
}
```

editMode=false（纯阅读）时 `editMode && showCursor` 恒 false，全 MD，与 showCursor 无关。

### 6.2 光标同步

renderer 内部的 `w.showCursor` 调用（`renderSegmentMD`/`renderSegmentNative`，基于 cell 坐标 vloc.X/vloc.Y）retarget 为 `w.setShowCursor` → 记录到 `sb.cursorX/cursorY`。showBuffer blit 完成后，把 sb 光标坐标转屏坐标调 `screen.ShowCursor`（§4.2）。

**多 cursor**：`setShowCursor` 的 `main` 参数保留现状语义（`c.Num == 0` 主光标 → screen.ShowCursor，其余 → screen.ShowFakeCursorMulti）。

## 8. 边界场景与对策

### 8.1 跨段光标切换（v1.0.5 已知限制，核心修复目标）
**场景**：sample.md line55-59 表格，光标从 line59 按 ↓ 到 line60（脱离表格）。
**对策**（阶段 5 修复）：Relocate 入口 displayToBuffer 拿 fresh sb；三目的停止条件确保旧段（prevCursor 所在段）整段渲染 + cursor 段渲染 + viewport 填满；realStartLine 在 sb 2× 范围内 → showBuffer 复用 cells。
**回归**：连续按 ↓ 直到 viewport 底边，光标始终可见。

### 8.2 screenBuffer 容量不足（含 overflow 闪现）
**场景**：极大单 segment（长代码块 > 2×bufHeight）跨段。
**对策**：`sb.overflow = true`；showBuffer 走 `displayBuffer()` 原生兜底；Relocate 走 `relocateVerticalNativeFallback`。降级不崩，下一帧 startLine 变化时自纠正。
**已知视觉降级（C4）**：overflow 时 `displayBuffer()` 是原生渲染器，对 .md 会画出带 `#`/`|` 的原始字符，用户可见一帧"原始 markdown"。触发概率低（单段 > 2×viewport），列为已知限制，不为此加重 cap（阶段 7 性能验证后再决定是否 3×）。

### 8.3 showBuffer blit 边界
- `rowIndexOf` 落装饰行 → `rowIndexNearest` 向下找首个内容行（沿用修改总结 §3.3，已踩过坑）
- 尾部不足 → 空白填充
- softwrap 下 `startLine.Row ≠ 0` → SLoc 二元组精确匹配（沿用现状）

### 8.4 首帧渲染
BufWindow 刚创建，sb 为 nil：displayBufferMD/relocate 入口 `w.sb == nil` 检查触发 displayToBuffer，无 nil panic。`prevCursorY = -1` → `findSegmentContaining(-1)` 返回 nil → oldSegDone=true，不过度渲染。

### 8.5 editMode 切换（ESC/click）
observeEditModeToggle 切换 editMode 后，下一帧 displayToBuffer 的 `hasCursorInside` 分支重新判定（cursor 段 native ↔ MD），fresh 渲染保证正确。无特殊处理。

### 8.6 scrollbar / statusline / infobar
方案 B 不接管这些 UI 元素。`Display` 入口仍先调 `displayStatusLine()` + `displayScrollBar()`，再调 `displayBufferMD()`，保持现状不动。

### 8.7 多 BufWindow 内存
cells 用完即弃（showBuffer 完成立即 nil）。任意时刻仅 active window 持有 cells（~200KB），其余 sb 只占 line/segRow 索引（~1KB）。4 窗口同时打开峰值 ~800KB（仅 active 的 cells 活跃时）。

### 8.8 水平滚动（StartCol，O1 确认）
方案 B 不接管水平轴。`renderSegmentMD` 内部已有 `screenX` 可见窗口裁剪（bufwindow_md.go:188 前后），写入 cells 的就是 StartCol 偏移后的可见内容；`renderSegmentNative` 同理（其内部从 `w.StartCol` 取 startInfo）。showBuffer 从 cells 直接 blit，无需二次处理水平偏移。结论：水平滚动随每帧 displayToBuffer 自然重渲 sb，无额外逻辑。阶段 7 验收补一次水平滚动手测。

**预期**：目标行进 viewport 且在 scrollmargin 内；只渲染一次（无二次渲染闪烁）。

### 10.5 softwrap
窄窗口下长段落 softwrap，cursor 在 wrap 续行。
**预期**：segRow 元数据正确，垂直滚动 cursor 不丢。

### 10.6 非 MD 文件回归（侵入安全）
开 .go/.txt 等非 MD 文件。
**预期**：行为字节级与改动前一致（sink 恒 realScreenSink，displayBufferMD 不被调用）。

### 10.7 点击映射
点击装饰行、内容行、空白区。
**预期**：装饰行/空白回退原始 Scroll 逻辑；内容行定位到正确 buffer 行。

## 11. 风险与回退

### 11.1 主要风险点

| 风险 | 等级 | 触发 | 缓解 |
|------|------|------|------|
| retarget 漏改某处 SetContent | 高 | displayToBuffer 期间该处写真屏 | 阶段 2 全文 grep `screen.SetContent` 确认无遗漏（displayBuffer 内部除外）；displayToBuffer 期间可断言 `sink==sb` |
| cells 坐标系错位（vX/vY 偏移算错） | 高 | setCell 坐标转换 | 阶段 3 dump sb.rows 肉眼对比；与 v1.0.5 基线 diff |
| showBuffer blit 边界错（落装饰行） | 中 | startLine 恰好是表格 frame 行 | 沿用修改总结 §3.3「向下找首个内容行」（已踩坑） |
| case C 极端（单 segment > 2× viewport） | 中 | 超大 codeblock | relocate 内 cursorRow !ok → native fallback（§4.3） |
| sb.rows 2× cap 仍不够 | 低 | 极端 softwrap 一行展开很多 | 2× 是启发式；实测不够提到 3×（Goto 低频可放大） |
| prevCursorY 未及时更新 | 中 | Display 末尾漏调 updatePrevCursor | 阶段 4 验证跨段场景；阶段 5 专项回归 |
| 多 BufWindow cells 泄漏 | 低 | 引用未释放 | showBuffer 末尾 `cells = nil` 保证；单测覆盖 |
| oldSeg 在视口上方导致 overflow（M1） | 中 | PageDown 后光标滚出顶部，oldSeg 落在 startLine 之上，旧 oldSegDone 永不为 true → 一路渲染到 overflow → showBuffer 闪现原始 markdown | §4.1 displayToBuffer oldSegDone 初值加 "oldSeg 不与渲染窗口重叠即视为完成" 判定 |
| resize/切 ruler 改宽度未失效 sb 导致 panic（M2） | 高 | Display 内 updateDisplayInfo 改了 gutterOffset/bufWidth，但 sb 复用旧 cells；showBuffer blit 循环用新宽度访问旧 cells → index out of range | §4.4 displayBufferMD 守卫与 §4.2 showBuffer 自愈守卫都加 `sb.width != gutterOffset+bufWidth` 宽度校验；blit 循环加 `col >= len(row.cells)` 防御性边界 |

### 11.2 回退方案

调试期（阶段 5/6）保留 viewportRowmap 双写（renderer 同时写 viewportRowmap 和 sb），出问题可快速切回旧 relocate 逻辑。git branch 标记每阶段可回退点。

若实施中发现无法解决的问题（cells 释放闪烁 / overflow 频繁 / 内存压力）：
- **回退到方案 A**（`docs/方案A架构设计-很臭的方案.md`）：~105 行，不引入 cells，仅 viewportRowmap + preRender dryRun
- **回退到 v1.0.5**：git revert，接受"按 ↓ 到底边光标偶发消失一帧"作为已知限制

不引入长期 feature flag（维护两套路径成本高）。

## 12. 性能与内存评估

### 12.1 单次 displayToBuffer 成本

| 阶段 | 成本 |
|------|------|
| `findSegmentContaining`（while 循环 N 次） | O(N·n) ≈ 50-200μs（n = segment 数） |
| `renderSegmentMD` / `renderSegmentNative` | O(visible 行数) ≈ 1-3ms |
| cells 写入（同源） | O(bufHeight × bufWidth) < 0.5ms |
| **总计** | **~2-3ms**（单帧 16ms 预算内） |

**对比 v1.0.5**：displayToBuffer 比 displayBufferMD 略慢（多 cells 写入），但 Relocate 现在能拿到 fresh map，从根本上正确。

### 12.2 单次 showBuffer 成本

| 阶段 | 成本 |
|------|------|
| blit cells 到 screen | O(bufHeight × bufWidth) ≈ 2000-4000 次 SetContent ≈ 0.5ms |
| cells = nil | O(1) |
| **总计** | **~0.5ms** |

### 12.3 内存

| 组件 | 单 BufWindow | 4 BufWindow（仅 active 持 cells） |
|------|-------------|--------------------------------|
| `cells[2×bufHeight][gutterOffset+bufWidth]` | ~200 KB（活跃时） | ~200 KB（仅 1 个活跃） |
| `rows[].line/segRow`（cells 释放后） | ~1 KB | ~4 KB |
| `rowmap` 兼容（删除前） | ~800 B | ~3 KB |

**对比方案 A 的 viewportRowmap**：方案 A 单窗 ~800B；本方案单窗活跃 ~200KB。差距大但绝对值 < 500KB，可接受；cells 用完即弃，非活跃窗仅占索引。

## 13. 开放问题（实现时定）

1. **2× cap 是否够**：阶段 7 性能验证后定，不够则 3×（Goto 低频可放大）
2. **displayToBuffer 精确停止算式**：`vY < bufHeight+scrollmargin` 是否覆盖所有 case C 偏移，阶段 5 调试时打磨（保证 cursor 必在 sb 内）
3. **overflow 阈值**：2×bufHeight；若超大 codeblock 常见，考虑动态 cap（按 segment 最大 Rows）
4. **cursorShowed 判定位置**：放 renderer 内部（renderSegmentNative 返回时）还是 displayToBuffer 循环内（hasCursorInside 后），阶段 3 实现时定
