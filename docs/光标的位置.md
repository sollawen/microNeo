# 背景
microNeo在micro原生的基础上，增加了对md文件的渲染。
- 这导致了对md文件源代码的一行，可能在渲染后会显示成几行，出现了不影响Line number变化的装饰行
- 于是，在计算screen光标所对应的buffer.line的时候，需要计算。

开始我们设计了使用Segment--渲染片，里面来记录row与line的对应关系
- 但是因为许多个segment才组成screen，所以后来又增加了 BufWindow里面的 viewportRowBufLine[]

其实我一直有个担心
- mircoNeo最早的版本是只有MDSegments里面记录每片渲染片的row->line的对应关系
	- 所有的光标位置计算都是使用一个或多个渲染片的数据组合在一起计算
- 但是，其实一个screen是一个整体，应该有一个整体的screen row -> line 的数据。
	- 这样在需要计算光标位置与line的关系的时候就非常简单了，不用去扫描和遍历一堆的渲染片了
	- 所以在后期加入了 viewpoarRowBufLine
	- 但是这个新的数据引入比较晚，好多地方还在使用碎片化的MDSegment

现在的viewportRowBufLine有两个缺点
- 只记录了当前viewport，viewport之外的上面和下面，都没有数据
- 是relocate之后那刷新的

## 思路A
- 记录整个全文的row-line的对应关系
- 改成
	- 当buffer内容变化的时候，更新 row-line 的数据
	- 当屏幕宽度变化的时候，更新 row-line 的数据

这个思路有个缺点：
- 全文的row-line 关系，确实在巨大文件时计算更新比较慢

## 思路B

我觉得我们把问题想复杂了。

现状
- 按上键，向上移动引发滚动，非常好，没有任何问题
- 翻页键，非常好，没有问题
- click，非常好，没有问题
- 在viewport中间，上下键移动光标，非常好，没有任何问题

问题：
- 只是在viewport的底部，下键时没有判断出来边界，没有正确的向上滚动一行内容

所以，我不认为应该去构建一个全文的row-line数据。完全没有这个必要
- 我们需要做的，是在下键的时候，正确的根据row来判断是否需要滚动，而不是根据line来判断是否需要滚动。就足够了

我猜测micro原生的滚动，是根据line与viewport之间的关系来判断是否需要向上或向下滚动的
- 我们只要改成是根据row来判断是否需要向上或向下滚动，不就行了？


if cursorRown > height-1-scrollmargin {向下滚动}
if cursorRown < scrollmargin {向上滚动}

前提：cursor所在的row，一定不是在渲染状态下的，一定是在micro原生的编辑状态下的

所以，
- func 向上滚动() 是不是可以直接使用micro原生的处理逻辑
