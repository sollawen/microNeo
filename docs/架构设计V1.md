## 二、Micro 渲染架构分析

### 2.1 主循环

```
DoEvent() {
    1. screen.Screen.Fill(' ', defStyle)    // 清屏
    2. screen.Screen.HideCursor()           // 隐藏光标
    3. action.Tabs.Display()                // 显示标签栏
    4. 各个 pane.Display()                  // 显示每个编辑区
    5. action.InfoBar.Display()             // 显示底部信息栏
    6. screen.Screen.Show()                 // 刷到终端
    7. select { case event := <-screen.Events: ... }  // 等待事件
}
```

每个 pane 的 `Display()` 最终调用 `displayBuffer()`--核心渲染函数。

### 2.2 `displayBuffer()` 的本质

一个 **520+ 行**(`bufwindow.go:382-843`)的函数,逐行逐字符地把 buffer 内容写入 tcell screen:

```
for 每个可见的 buffer 行 {
    画 gutter + 行号
    for 该行的每个字符 {
        获取语法高亮颜色 → screen.SetContent(x, y, rune, style)
    }
}
```

**线性的逐行渲染,没有"块"或"段"的概念。**

但这 520 行里塞了不少东西,**绝对不只是"逐字符写入"那么简单**。至少有 5 层闭包逻辑:

1. **matchbrace 检测**(L404-417):遍历所有光标查找匹配括号,用于反显/下划线
2. **showchars 解析**(L475-495):解析 `key1=val1,key2=val2` 格式配置,决定 tab/space 怎么显示
3. **`getRuneStyle` 闭包**(L498-548):~50 行,处理 tab/空格的 showchars 替换、hltrailingws(尾随空格高亮)、hltaberrors(tab 错误高亮)
4. **`draw` 闭包**(L551-623):~70 行,处理高亮搜索、选区反显、cursorline 背景色、gutter message、colorcolumn、光标显示
5. **`wrap` 闭包**(L626-643):处理 softwrap 时 gutter/行号的续行
6. **wordwrap 逻辑**(L654-808):~150 行,glyph 收集 + 单词换行 + 宽字符处理

**这意味着理解这些闭包的作用是后续开发 `displayBufferMD()` 的基础。** 详见 §5.3 的特性去留清单。

### 2.3 滚动系统

核心是 `SLoc`(Scroll Location):`{Line, Row}`,表示窗口顶部对应的 buffer 位置。`Relocate()` 确保 cursor 在可见区域内。`Scroll(s, n)` 做上下滚动,`Diff(s1, s2)` 计算两个位置间的视觉行数差。

**这套系统假设:buffer 行 → 视觉行的映射只受 softwrap 影响。** 引入渲染片后(5 行 buffer → 8 行屏幕),这个假设失效。

#### ⚠️ 关键矛盾:Scroll 在显示前被调用

非 softwrap 模式下,`Scroll()` 极简(`softwrap.go:292`):

```go
if !w.Buf.Settings["softwrap"].(bool) {
    s.Line = util.Clamp(s.Line+n, 0, w.Buf.LinesNum()-1)
    return s
}
```

但渲染片模式下需要知道**每个 buffer 行在当前渲染片下占几个 screen row**--而渲染片信息在 `displayBufferMD()` 时才生成。

**调用时序冲突**:
- `Relocate()` → `Scroll()` → `Diff()` 都在 `displayBuffer()` **之前**调用(用于决定窗口从哪行开始显示)
- 但渲染片的"行高"信息只有 displayBuffer 渲染循环走到那行才计算

如果 Scroll/Diff 不知道渲染片信息,向上滚一屏(`PageUp`)可能滚到不正确的位置。这个时序冲突的解决方案见 §三。

### 2.4 事件处理

`BufPane.HandleEvent()` 接收 tcell 事件,分发到 key/mouse action 系统。所有编辑操作都通过 action 系统完成。

## 三、核心概念:渲染片

### 3.1 为什么需要渲染层

原版 Micro 里,buffer 有一行一行的原始文本,显示的就是这些文本,区别只在于每个字符的颜色。

我们要做 Markdown 的完整渲染,所以:
- buffer 里有些字符,渲染后不显示(如 `**`、`#`、`` ` ``)
- buffer 里有些字符,会变成别的字符显示(如 `---` 变成长横线)
- 有时需要显示 buffer 里根本不存在的行(如表格的外框线)

这就需要在 buffer 和 screen 之间加一层翻译。

### 3.2 为什么需要渲染片

逐行翻译能处理的事情(隐藏标记、链接只显示文本、`---` 变横线),不需要渲染片。

**渲染片存在的原因是:Markdown 里有些结构是跨行的,必须作为一个整体来渲染。** 比如表格,你不可能只看一行就知道列应该有多宽--必须看完整个表格才能决定每一列怎么对齐。

### 3.3 每一行都是一个渲染片

渲染片是渲染管线的统一单位。**每一行 buffer 都属于某个渲染片**,只是覆盖的行数不同:

- 表格:多行 buffer → 一个渲染片
- 代码块:多行 buffer → 一个渲染片
- 标题行:1 行 buffer → 一个渲染片
- 普通段落行:1 行 buffer → 一个渲染片

这样渲染管线完全统一,没有分支:

```
for 每个可见行 {
    找到这行所属的渲染片
    渲染片负责输出
}
```

表格渲染片、标题渲染片、段落渲染片,区别只是覆盖的 buffer 行数不同,以及 Segment 里挂的渲染函数不同。

### 3.4 渲染片里记录什么

- 对应 buffer 里的哪几行(`bufStartLine`、`bufEndLine`)
- 负责渲染这段行的函数(如 `RenderTable`、`RenderCodeBlock` 等)，是普通函数，不是接口/struct 方法
- 渲染后的每一行:
    - 对应 buffer 里的哪一行(非装饰行有对应,装饰行标记为 `-1`)
    - 可能多行对应同一个 buffer 行号(softwrap 或人为拆行)
- 每个字符:
    - 要显示的 rune
    - 颜色和字体样式(颜色从 Micro 语法高亮系统来,行内渲染器只负责加粗体/斜体)
    - 是真实字符还是装饰字符(影响点击定位)
    - 若是真实字符,则这个字符对应 buffer 对应 line 里的字符位置 `bufX`(**rune 偏移**,不是字节偏移,与 Micro 内部 `buffer.Loc.X` 保持一致)

### 3.5 每帧即弃

渲染片翻译的结果**不做持久化,每帧重新生成**。这和 Micro 原版的"每帧实时渲染"模式一致,避免了维护同步状态的复杂度。

**例外**：检测步骤的轻量元数据（片边界 + 行高）是唯一跨帧缓存的内容，供事件阶段的 `Scroll()`/`Relocate()` 使用。详见 §3.6。

### 3.6 检测与渲染分离

§2.3 指出了时序冲突:`Scroll()`/`Diff()` 在 `displayBufferMD()` 之前被调用,但它们需要知道每个渲染片占几个 screen row。

解决方案:把渲染片的"检测"和"渲染"拆成两步。

#### 第 1 步:检测(集中式扫描器)

`detect.go` 是唯一的检测入口。它逐行扫描 buffer,根据每一行的字符特征判断类型:

```
detect.go 扫描可见区域:
  读 line N
  ├── 以 ``` 或 ~~~ 开头? → 代码块开始,往下找配对的 ```,形成多行 segment → codeblockRenderer
  ├── 以 # 开头?        → 标题,单行 segment → headingRenderer
  ├── 以 > 开头?        → 引用块,连续 > 行合并 → blockquoteRenderer
  ├── 以 ---/***/___ ?  → 水平分割线,单行 → hrRenderer
  ├── 以 | 开头?        → 表格,连续 | 行合并 → tableRenderer
  ├── 以 -/* 加空格?    → 列表,连续列表项合并 → listRenderer
  └── 都不是?            → 兜底,单行 → paragraphRenderer
  输出: []Segment,每个 Segment = {BufStartLine, BufEndLine, Renderer}
```

Markdown 的行分类本质上是简单的模式匹配,每种类型的识别逻辑平均不到 10 行,总共不到 80 行。不需要为每种类型设计独立的 Detect 接口。

检测需要维护全局状态(如"我现在在代码块里面吗?"),集中在一个扫描器里做比分散到各模块更自然。扩展也简单:未来加 Mermaid,`detect.go` 加一个识别分支,新建 `mermaid.go` 写渲染逻辑。

#### 第 2 步:渲染(各 render 函数负责)

每个 render 文件导出一个普通函数,只负责一件事:**接收行内容,返回屏幕数据**。

```go
// render_table.go
func RenderTable(lines []string, width int, cfg MDConfig) *RenderedSegment { ... }

// render_codeblock.go
func RenderCodeBlock(lines []string, width int, cfg MDConfig) *RenderedSegment { ... }

// render_heading.go
func RenderHeading(lines []string, width int, cfg MDConfig) *RenderedSegment { ... }
```

Segment 直接持有函数引用:

```go
type Segment struct {
    BufStartLine int
    BufEndLine   int
    Render       func(lines []string, width int, cfg MDConfig) *RenderedSegment
}
```

detect.go 扫描时直接挂函数:

```go
segments = append(segments, Segment{
    BufStartLine: 15,
    BufEndLine:   20,
    Render:       RenderCodeBlock,
})
```

没有接口,没有 struct 方法,没有 `Renderer` 接口类型。Go 函数是一等公民,可以当值传递。render 函数不碰 tcell screen,不做检测,只做纯计算转换。

#### 检测的轻量元数据

检测只回答"片边界 + 片内行高",不计算具体字符--开销接近 O(1) 每行(详见 PRD 6.1 的性能约束)。渲染输出才是重活,逐字符写入 screen。

轻量元数据跨帧缓存,供事件阶段的 `Scroll()`/`Relocate()` 使用。

#### 检测在管线中的位置

只有 MD 文件才需要检测渲染片。检测和渲染都在 `displayBufferMD()` 内部完成,`BufWindow.Display()` 只需加一个 if/else 分支:

```
BufWindow.Display():
  ├── updateDisplayInfo()              ← 原样
  ├── displayStatusLine()              ← 原样
  ├── displayScrollBar()               ← 原样
  └── if w.IsMD {
        displayBufferMD()
          ├── 1. detect.DetectSegments(bufWidth)  ← 输出片边界+行高,缓存到 BufWindow
          └── 2. 渲染输出                          ← 逐 segment 调 renderer
      } else {
        displayBuffer()                 ← 原样
      }
```

**关键点**:BufPane 不需要 override `Display()`,所有改动都在 `display` 包内部。检测步骤以 `bufWidth`(`updateDisplayInfo()` 已算好)为输入。事件阶段的 `Scroll()`/`Relocate()` 查上一帧缓存的检测结果。

#### displayBufferMD() 内部流程

```
displayBufferMD():
  1. segments = detect.DetectSegments(bufWidth)  → 输出片边界 + 行高,缓存到 w
  2. for each segment:
      if segment contains cursor && editMode:
          原生逻辑渲染 segment     // copy 自 displayBuffer()
      else:
          segment.Render(lines, width, cfg)     // 直接调函数
```

逐片决策：editMode 为 true 时，光标所在渲染片走 copy 来的原生代码，其他渲染片走渲染输出。editMode 为 false 时，所有渲染片都走渲染输出。详见决策 4。

## 四、架构关键决策

本节列出整个架构的核心决策,按"先入口后细节"组织。

### 决策 1:IsMD 放 BufWindow,Buffer 层零改动

**MD 还是非 MD 是本项目的根本前提条件**:MicroNeo 只服务 `.md` / `.markdown` 文件,其他文件类型走 Micro 原版逻辑,MicroNeo 代码完全不介入(PRD 6.2)。

#### 为什么放 BufWindow

- `Display()` 在 BufWindow 上,`displayBufferMD()` 也在 BufWindow 上,`isMD` 判断就在这里--哪里用就放哪里
- BufPane 创建 BufWindow 时设一次 `win.IsMD = true`
- `display` 包保持不 import `action` 包,依赖方向 `action → display → buffer` 维持单向

#### 实现

```go
// internal/display/bufwindow.go
type BufWindow struct {
    // ... 现有字段
    IsMD bool  // MicroNeo: 创建时由 BufPane 设置
}
```

```go
// internal/action/bufpane.go - 创建 BufWindow 时
func NewBufPaneFromBuf(buf *buffer.Buffer, tab *Tab) *BufPane {
    w := display.NewBufWindow(...)
    w.IsMD = isMarkdownFile(buf.Path)
    ...
}
```

#### 判断逻辑

```go
func isMarkdownFile(path string) bool {
    ext := strings.ToLower(filepath.Ext(path))
    return ext == ".md" || ext == ".markdown"
}
```

#### 和决策 2 的关系

- `BufWindow.IsMD` 是**文件静态属性**(路径决定)→ 放 BufWindow,创建时算一次
- `BufPane.editMode` 是**渲染动态状态**(true=编辑,false=阅读)→ 放 BufPane
- Buffer 层零改动

### 决策 2:模式状态存 BufPane

模式状态涉及事件拦截、计时器管理、模式切换三个职责,都发生在 BufPane 层(`internal/action/`),所以放在 BufPane 上。

```go
// MicroNeo 新增到 BufPane
editMode       bool           // true=编辑模式, false=阅读模式
mdIdleTimer    *time.Timer
```

#### editMode 初始化策略（架构不变量）

- **非 MD 文件**：BufPane 构造时 `editMode = true`，永远不会变（因为没有渲染模式可回退）
- **MD 文件**：BufPane 构造时 `editMode = false`（默认渲染模式），用户按键/点击后切换为 `true`，10 秒计时器到期后切回 `false`

这是 §5.2 键拦截逻辑正确性的前提：`if !h.editMode` 只有 MD 文件在渲染模式下才会触发。如果初始化不正确，非 MD 文件的方向键会被错误拦截为滚动。

文档后续代码中 `h.editMode` 写法中,`h` 是 `*BufPane`。`w.IsMD` 写法中,`w` 是 `*BufWindow`。

editMode 只存在 BufPane 上,BufPane 自己的代码里直接读写。BufPane 内嵌了 `display.BWindow`。

### 决策 3:渲染片"检测"与"渲染"分离

详见 §3.6。决策 3 补充的实现要点:

- 渲染片检测结果可缓存到 `BufWindow`
- Scroll/Diff 先查缓存的检测结果,再做计算
- 缓存失效:buffer 内容变化时

### 决策 4:edit 模式 = 仅光标所在渲染片走原生逻辑

**原则**:进入 edit 模式后,**只有光标所在的渲染片走原生逻辑**,其他渲染片仍然走渲染逻辑。

**关键前提**:`displayBufferMD()` 包含了 `displayBuffer()` 的全部功能(是后者的超集)。它一个人负责整屏,内部自己决定每个渲染片走渲染逻辑还是原生逻辑。editMode 时的"回退原生"不是调另一个 `displayBuffer()`,而是 `displayBufferMD()` 里那段 copy 来的原生代码自然生效。

具体含义:
- 光标所在渲染片 → 走 copy 来的原生逻辑(逐字符渲染)
- 其他渲染片 → 走渲染逻辑(渲染片输出)
- 结果:用户看到的是「漂亮的渲染文档 + 中间一坨原生(可编辑)」

多光标的情况:
- 几个光标,就有几行(或几个渲染片)在原生编辑状态(PRD 3.2 原话)
- 同一渲染片内多个光标 → 整个片回退
- 多个渲染片各有光标 → 各片独立回退,互不影响

10 秒计时器到期的行为:
- 全部回渲染模式(光标所在片也恢复为渲染)
- 原生状态被拋弃,用户看到完整渲染效果

**这是 PRD 2.2 / 3.2 / 5.4 的原意**,本决策只是把「光标所在行的特殊处理」明确为渲染片级别的行为。

### 决策 5:10 秒计时器

用 `time.AfterFunc`,每次按键/鼠标点击时重置。计时器到期时通过 `screen.Redraw()` 触发重绘,不直接改模式(线程安全)。

### 决策 6:缓存策略

**初期每帧全量重算可见区域的渲染片。** 可见区域通常只有 40-60 行,性能不会是瓶颈。后续再引入增量更新。

注意:决策 3 的"检测"步骤仍然每帧做(即使不做渲染输出),否则 Scroll/Diff 拿不到行高信息。

**首帧缓存未命中**：BufPane 刚创建时(首帧)没有上一帧的检测结果。Scroll/Diff 按 buffer 行 == screen 行的零假设计算(即退化为原生行为),允许轻微错位,第二帧起使用真实检测结果。

## 五、具体实现点

本节列出所有需要单独讨论的实现细节,包括交互、模式行为、与 Micro 既有模块的协作。

### 5.1 Mouse Click 定位(已明确)

**数据基础**(见 §3.4「渲染片里记录什么」):每个渲染后的字符记录 `rune、样式、isDecorative`,真实字符还记录 `bufLine` 和 `bufX`(rune 偏移,与 `buffer.Loc.X` 一致)。

**Click 行为**:
- 点击装饰字符 → 忽略
- 点击真实字符 → 查表得到 `(bufLine, bufX)`,光标定位到该位置
- 点击渲染片**外**位置(装饰行之间、渲染片右边的空白等)→ 一并视为装饰,忽略

**边界情况:装饰行的点击定位**

渲染片输出的行数可能多于 buffer 行数(如表格外框行)。这些行没有对应的 buffer 行号,用户点了怎么响应?

- **方案 A("忽略",当前默认)**:装饰行上的所有点击都视为无效
  - 优点:实现简单,不会把光标定位到错误位置
  - 缺点:用户可能困惑--"为什么我点了一个有字的地方没反应"
- **方案 B("就近定位")**:装饰行的点击定位到最近的真实 buffer 行
  - 优点:用户至少能看到反馈(光标移动了)
  - 缺点:行为可能反直觉(点外框行结果光标跑到了内表格行)
- **方案 C("装饰行单独映射")**:装饰行记录自己"语义上对应"哪个 buffer 行
  - 优点:语义最准确
  - 缺点:实现复杂,需要为每种装饰(表格外框、代码块边框等)定义映射规则

**推荐方案 A 作为 v0.1 起步**,v0.2 根据用户反馈再迭代。

实现层:渲染管线输出时维护 `(x, y) → cell` 的查找表,`MousePress` 时直接查表(也可在现有 `LocFromVisual` 路径上做模式分支)。

### 5.2 ↑↓ 键拦截(已建议方案)

PRD 3.1 要求「渲染模式下 ↑↓ 滚动」,但 Micro 的键绑定是静态的(`internal/action/defaults_darwin.go:4-5`):

```
"Up"   → "CursorUp"     // 移动光标
"Down" → "CursorDown"
```

**建议方案**:在 `BufPane.HandleEvent()` 的 `case *tcell.EventKey:` 分支,`DoKeyEvent` **之前**加模式拦截。

```go
case *tcell.EventKey:
    ke := keyEvent(e)

    // MicroNeo: 阅读模式下拦截方向键
    if !h.editMode {
        switch ke.Key() {
        case tcell.KeyUp:
            h.ScrollUp(1)
            return  // 不再走 DoKeyEvent
        case tcell.KeyDown:
            h.ScrollDown(1)
            return
        case tcell.KeyPgUp:
            h.ScrollUp(h.BufView().Height)
            return
        case tcell.KeyPgDn:
            h.ScrollDown(h.BufView().Height)
            return
        case tcell.KeyHome:
            h.ScrollToTop()
            return
        case tcell.KeyEnd:
            h.ScrollToBottom()
            return
        }
        // 其他非方向键 → 切换到编辑模式,然后继续正常处理
        h.editMode = true
    }

    done := h.DoKeyEvent(ke)
    ...
```

**拦截点选 `HandleEvent` 的原因**:
- `DoKeyEvent` 走的是 `KeyTree` 绑定查找逻辑,在那里拦截需要先解析 key 再回头判断模式
- `HandleEvent` 拿到原始 `tcell.EventKey` 后第一时间判断,逻辑最直
- 拦截掉的方向键(↑↓/PgUp/PgDn/Home/End)调用现有的 `ScrollUp/ScrollDown` action,复用所有现有逻辑(包括 mousePressed、连续按键、relocate)

**其他滚动入口的对应**:
- `MouseWheelUp/Down`:已绑定到 `ScrollUp/ScrollDown` action(`defaults_darwin.go:97-98`),渲染模式下行为正确(都是滚动),**不需要额外拦截**
- `Ctrl+Home/End`:默认绑定到 `CursorStart/CursorEnd`(行内移动),渲染模式下需要拦截改为滚动到文件首/尾
- `Ctrl+PageUp/Down`:默认是 `PreviousTab/NextTab`(标签页切换),**不应该拦截**--这是 tab 级别的操作,不该被渲染模式改变
- `JumpToLocation` 命令:少有人用,且命令调用栈深,统一在 `HandleEvent` 入口加 mode 判断可能漏,统一在 `BufPane.Scroll*` 内部检查模式更稳妥
- scrollbar 拖动:直接改 StartLine,需要确认拖动时不会触发 `DoKeyEvent`

### 5.3 `displayBuffer()` 特性

现有 `displayBuffer()` 520+ 行,承载的特性复制到 displayBufferMD()之后
- 在editMode的时候,不变,全盘保留micro原生的逻辑和流程
- 在渲染模式下会有一些变化处理


**行号的处理决策(推翻 PRD 5.5)**:渲染片导致的一行 buffer → 多行 screen,**和 softwrap 的多行 screen 是同一类问题**。softwrap 现有逻辑就是「首行显示行号、wrap 出的额外行留空」,渲染片完全复用这套:
- 渲染片内某 buffer 行的首行 screen → 显示行号
- 同一个 buffer 行的后续 screen 行 → 行号位置留空
- 装饰行(表格外框等)→ 行号位置留空

这与 PRD 5.5 冲突,需 PRD 同步修订为「显示行号,复用 softwrap 的多行逻辑」。

**完整特性清单**:

| 特性 | 渲染模式下 | 状态 | 理由 |
|------|:---:|:---:|------|
| 行号 | 开 | 已定 | 复用 softwrap 多行逻辑(见上),block render 详见 §5.3b |
| diff gutter | 开 | 已定 | 跟随行号区一起显示,buffer 本地修改可见,对 AI workflow 有价值 |

| autocomplete suggestion | 关 | 已定 | 渲染模式不适用 |

| showchars (tab/space) | 关 | 已定 | 渲染后无 tab/space,渲染片输出的字符已经是最终显示内容 |
| colorcolumn | 关 | 已定 | 第 N 列画竖线参考线(提醒代码行别太长),Markdown 渲染后列号无意义 |

### 5.3b block render 中行号的显示

single line render 无特殊处理,和 softwrap 相同:buffer 行号显示在首行,续行留空。

block render(如表格)更复杂。以表格为例:

```
        |-----------|----------------------|
line5:  |  项目     |  说明                |
        |-----------|----------------------|
line6:  |microNeo   | 一个好玩的编辑器     |
        |-----------|----------------------|
line7:  |这个个名字 | 这次要换了           |
        |比较长的项 |                      |
        |目         |                      |
        |-----------|----------------------|
```

行号规则:
- **有对应 buffer 行的 screen 首行** → 显示 buffer 行号(如上例 line5、line6、line7)
- **续行**(同一 buffer 行因 cell 内容 wrap 产生的额外 screen 行)→ 行号位留空
- **装饰行**(表格外框 `|---|---|`、代码块边框等)→ 行号位留空

这与 softwrap 的处理模式完全一致:「首行显示、续行留空」。渲染片需要自己维护"哪些 screen 行对应哪个 buffer 行"的映射,行号区复用这套规则即可。

### 5.3c matchbrace 在渲染模式下的行为

matchbrace 的"查找"在 **buffer 层面**完成(`b.FindMatchingBrace` 遍历 buffer 原始文本),不在渲染层面。但"显示"(加下划线/反色)需要区分场景:

- **editMode**:光标所在字符 `a` 交给原生逻辑高亮。匹配到的字符 `b` 可能出现在其他渲染片中:
  - `b` 落在回退原生的渲染片内 → 原生逻辑直接高亮
  - `b` 落在生效的渲染片内 → 需要通过 buffer 坐标 `(bufLine, bufX)` 找到渲染片输出中的 screen 坐标,再叠加高亮
- **非 editMode**:不存在 matchbrace 检测

**v0.1 简化建议**:editMode 时 matchbrace 只高亮光标所在行(字符 `a`),字符 `b` 如果在渲染片内暂不高亮。Markdown 文件里括号配对场景本来就少,等基础功能稳定后再加跨渲染片高亮。

### 5.3d select 反显的两个场景

select 反显的行为取决于操作场景:

#### editMode 场景

用户在光标位置按 shift-方向键等方式选择连续文字。此时 select 区域**全部走原生逻辑**:
- select 只在当前 buffer line 内 → 简单,原生逻辑处理
- select 跨越多行(如 line10/11/12)→ 所有涉及的渲染片全部回退原生,和多光标机制一致:不管是 single line renders 还是 block renders,都不走 render

#### 非 editMode 场景

用户在渲染片区域里用鼠标拖动选择一片连续文字。此时由 **displayBufferMD()** 处理:
1. 鼠标按下时查渲染片输出表得到起始 buffer 位置
2. 鼠标拖动时查表得到终止 buffer 位置
3. 在渲染输出上对两个位置之间的所有字符叠加反色

这不依赖原生 `draw` 闭包的选区逻辑,是渲染片自身要实现的功能。

**额外讨论:cursorline 的替代方案**

关闭 cursorline 后,渲染模式下没有任何"当前行"视觉指示。考虑过两个方案:
- 关闭(当前决策):用户依靠光标闪烁知道当前位置--但渲染模式下光标被设计为不可见(PRD 3.1)
- **替代方案**:渲染模式下在最后操作过的 buffer 行加一个淡淡的标记(如左侧 gutter 的 1 像素高亮),作为"上次编辑位置"指示
- 留作 v0.2+ 优化,v0.1 先关闭 cursorline,让用户通过 10 秒规则自然感知"我现在在阅读"

### 5.4 softwrap 与渲染片的关系

渲染模式下,softwrap 和渲染片**根本不在同一条路径上**,不存在冲突问题:

| 场景 | 谁负责换行 | softwrap 是否参与 |
|------|----------|:---:|
| editMode 光标所在渲染片 | 原生逻辑(displayBufferMD 内 copy 的代码) | ✅ 原生自己处理 |
| 其他渲染片 | 各 renderer 自己 | ❌ 不参与 |

每个 renderer 在生成渲染片内容时,已经根据 screen 宽度计算好了换行--渲染片的输出就是 fit 在 screen 宽度里的,不存在超出宽度的可能性。所以不需要"强制关闭 softwrap"--渲染片路径上 softwrap 根本不会被调用,而 editMode 光标所在渲染片走原生路径时 softwrap 正常工作。

各 renderer 内部的换行策略由自己决定(表格 renderer 分配列宽并处理 cell 换行,代码块 renderer 按字符换行,段落 renderer 按单词换行等)。早期可以先简单实现(如表格 cell 超长截断),后续迭代智能分配。

**用户 FAQ**：如果 MD 文件设置了 `softwrap=true`，渲染模式下不会生效——渲染片路径不调 softwrap。只有 editMode 时光标所在渲染片按原生 softwrap 处理。这是设计如此，不是 bug。

### 5.5 BufPane 与 BufWindow 的协作模式

代码中 `BufPane` 内嵌 `display.BWindow` 接口:

```go
type BufPane struct {
    display.BWindow  // 接口,实际指向 *BufWindow
    Buf *buffer.Buffer
    ...
}
```

BufPane 内嵌了 `display.BWindow`(接口,实际指向 `*BufWindow`)。BufPane 可以直接调 BufWindow 的方法,也可以自己实现同名方法覆盖(Go 内嵌机制)。

分流在 BufWindow.Display() 内部做:`if w.IsMD` 决定调 `displayBufferMD()` 还是 `displayBuffer()`。BufPane 不需要 override `Display()`,BufWindow 保持改动最小(只在 `Display()` 加一个 if/else)。editMode 的分流在 `displayBufferMD()` 内部处理,BufWindow.Display() 不需要关心。

### 5.6 状态栏

**不需要修改。** 渲染模式和编辑模式的视觉差异本身就是最好的状态指示--渲染模式下看到漂亮的排版表格,编辑模式下看到原生 `# ** |` 符号。无需额外的文字提示。

`display` 包保持不 import `action` 包,依赖方向 `action → display → buffer` 维持单向。

### 5.7 配置项的实现位置

PRD 第七章列了 10 个配置项。当前 `internal/config/settings.go` 在 `init()` 的 `ds` map 中管理全局配置,需要:

```go
"mdrender":          true,
"mdrenderidle":      float64(10),  // 秒
"mdtablealign":      true,
"mdtableborder":     false,
"mdbolditalic":      true,
"mdcodeblock":       true,
"mdheading":         true,
"mdlist":            true,
"mdlink":            true,
```

注意几个配置项是**全局**(所有 buffer 生效)还是**buffer 本地**(只对当前 buffer 生效)。考虑:
- `mdrender` 全局(用户明确开启/关闭整个功能)
- `mdrenderidle` 全局(个人偏好)
- `mdtablealign`、`mdtableborder` 等单模块开关:buffer 本地(因为 MD 文件才相关,但和当前 buffer 的 syntax 设置一致更自然)

Micro 已有的 buffer 本地设置走 `Buf.Settings`(buffer 启动时从 syntax 配置中加载),全局走 `config.GetGlobalOption`。**建议**:
- 功能总开关(`mdrender`)放全局
- 单模块开关放 buffer 本地(和 `softwrap`、`ruler` 等一致)

实现前需要在 `internal/buffer/settings.go` 排查一下 buffer 本地设置的加载机制。

#### 配置项传递路径

配置项从 config/buffer 层传到 display 包渲染器的路径：

- **BufPane 构造时**：读取 `config.GetGlobalOption("mdrender")` 和 `Buf.Settings["mdtablealign"]` 等，把 MD 相关配置塞到 `BufWindow` 上（新增字段或 map）
- **BufWindow 持有配置**：渲染器和检测步骤直接从 BufWindow 读取，不反向依赖 config/buffer
- **配置变更时**：BufPane 在 setting change callback 中 push 更新到 BufWindow

这样 display 包不直接 import config 或 buffer 的配置系统，依赖方向 `action → display → buffer` 维持单向。具体字段设计留 Step 0 详细设计。

### 6.1 可行性

完全可行:

1. 显示层集中在 `internal/display/`,改一个目录就能影响所有渲染
2. buffer 层不需要改动
3. tcell 提供足够的终端能力
4. "每一行都是渲染片"的统一模型让管线代码简单无分支

### 6.2 工作量估算

PRD 附录 C 预估约 2000 行生产代码 + 1500 行测试。"每一行都是渲染片"的统一模型简化了管线代码,但**经过代码审查后,发现有 3 项被低估的复杂度**:

1. **`displayBufferMD()` 的开发成本**:整体 copy `displayBuffer()` 520+ 行,在此基础上添加渲染功能。不需要解构原有代码,所有闭包逻辑原样保留,工作量比预估小
2. **Scroll/Diff 系统改造**:需要修改 `softwrap.go` 中的核心函数(Scroll、Diff、SLocFromLoc、LocFromVLoc),这些函数被广泛调用,需要非常小心。加上"检测"与"渲染"分离的设计,工作量约 400-500 行
3. **鼠标定位完全重写**:`LocFromVisual()` 当前依赖 SLoc/VLoc 映射,渲染模式下需要查渲染片输出表。约 200-300 行

**修正估算:3000-3500 行生产代码 + 2500 行测试代码。**(第 1 项比原预估小,总数下调)

### 6.3 关于 Markdown 文件的行号

**结论:MD 文件显示行号**,复用 softwrap 的多行显示规则。PRD 5.5 需要同步修订。详见 §5.3。

理由:渲染片的 buffer 一行 → screen 多行,**和 softwrap 的 buffer 一行 → screen 多行 是同一类问题**。Micro 现有 softwrap 的处理就是「首行显示行号、wrap 出的额外行留空」,渲染片完全复用这套逻辑(装饰行也留空,因为没有对应 buffer 行号)。
