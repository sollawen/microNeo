# C3 光标滚动 —— 最小修复

> 关联：调研依据见 `docs/C0-光标滚动-现状调研.md`。
> 这是本 bug 的实际修复方案（§6 解释为何不需要全文映射）。

---

## 0. 重新聚焦问题边界

经过多轮讨论，问题范围比最初设想的小得多。**只有一类现象要修：**

| 场景 | 现状 | 是否在本次范围 |
|------|------|---------------|
| viewport 中间，↑↓ 移动光标 | ✅ 正常 | — |
| ↑ 键引发滚动 | ✅ 体验可接受（但**不精确**，见 §4，本次一并修） | ✅ 修 |
| 翻页（PageUp/Down、HalfPage） | ✅ 正常（buffer 行级翻页，不依赖屏幕行精度） | — |
| click | ✅ 正常（走 `viewportRowBufLine`） | — |
| **↓ 键在 viewport 底边，不触发上滚** | ❌ **本次 bug** | ✅ 修 |

**不需要构建全文 row→line 数据。** 只需让 `Relocate` 的边界判定从"按 line 算"改成"按 row 查"。

---

## 1. 根因（精确到表达式）

`Relocate`（`internal/display/bufwindow.go:248-260`）的顶边/底边 margin 判定：

```go
c := w.SLocFromLoc(activeC.Loc)   // 非 softwrap = SLoc{cursor.Y, 0}

// 顶边
if c.LessThan(w.Scroll(w.StartLine, scrollmargin))                    // (1) 顶边阈值
    && c.GreaterThan(w.Scroll(bStart, scrollmargin-1)) {
    w.StartLine = w.Scroll(c, -scrollmargin)                          // 动作
    ret = true
} else if c.LessThan(w.StartLine) {
    w.StartLine = c
    ret = true
}
// 底边
if c.GreaterThan(w.Scroll(w.StartLine, height-1-scrollmargin))        // (4) 底边阈值
    && c.LessEqual(w.Scroll(bEnd, -scrollmargin)) {
    w.StartLine = w.Scroll(c, -height+1+scrollmargin)                 // 动作
    ret = true
} else if c.GreaterThan(w.Scroll(bEnd, -scrollmargin))
    && c.GreaterThan(w.Scroll(w.StartLine, height-1)) {
    w.StartLine = w.Scroll(bEnd, -height+1)
    ret = true
}
```

非 softwrap 下，`Scroll(s, n)` 就是 `SLoc{s.Line + n, 0}`。所以两个阈值：

| 阈值 | 1:1 算出的值 | 实际想要的值 |
|------|------------|------------|
| (1) 顶边 margin | `StartLine.Line + scrollmargin` | "屏幕第 `scrollmargin` 行显示的那个 buffer 行" |
| (4) 底边 margin | `StartLine.Line + height-1-scrollmargin` | "屏幕第 `height-1-scrollmargin` 行显示的那个 buffer 行" |

MD 装饰行打破 1:1 → 算出的 buffer 行**偏大**（装饰行吃掉了屏幕行，真实位置对应的 buffer 行更小）。

而这个"实际想要的值"，`viewportRowBufLine[]` **已经有了**——它就是视口每个屏幕行实际显示的 buffer 行。

---

## 2. 修法：阈值从"算"改成"查"

**只改两个阈值表达式，动作（`w.Scroll(c, ...)`）一行不动。**

```go
// MD 模式下：顶边/底边阈值改为查 viewportRowBufLine（装饰行已计入）
topMargin := w.Scroll(w.StartLine, scrollmargin)                  // 原生 fallback
botMargin := w.Scroll(w.StartLine, height-1-scrollmargin)         // 原生 fallback
if w.Buf.IsMD && w.mdConfig.MDRender {
    if t := w.viewportRowBufLine[scrollmargin]; t >= 0 {           // -2 表示未填充，回退
        topMargin = SLoc{t, 0}
    }
    if t := w.viewportRowBufLine[height-1-scrollmargin]; t >= 0 {
        botMargin = SLoc{t, 0}
    }
}

if c.LessThan(topMargin) && c.GreaterThan(w.Scroll(bStart, scrollmargin-1)) {
    w.StartLine = w.Scroll(c, -scrollmargin)                       // 动作不变
    ret = true
} else if c.LessThan(w.StartLine) {
    w.StartLine = c
    ret = true
}
if c.GreaterThan(botMargin) && c.LessEqual(w.Scroll(bEnd, -scrollmargin)) {
    w.StartLine = w.Scroll(c, -height+1+scrollmargin)              // 动作不变
    ret = true
} else if c.GreaterThan(w.Scroll(bEnd, -scrollmargin)) && c.GreaterThan(w.Scroll(w.StartLine, height-1)) {
    w.StartLine = w.Scroll(bEnd, -height+1)
    ret = true
}
```

**为什么动作不用改**：动作是 buffer 行级的滚动（`StartLine.Line = cursor.Y ± ...`）。把 `StartLine.Line` 设到正确的 buffer 行后，该行的前置装饰行会随它一并正确显示（C0 §4.2 已论证）。滚动目标的精度只需到 buffer 行级——这点顶/底边一致。

## 3. C0 §4.1 的时序坑为何在这里不适用

这是我之前过度设计的根源，必须澄清。

时序坑说的是："光标跳出旧视口，`bufferLineToScreenOffset(cursor)` 查不到光标新位置。"

**但本修法根本不查光标**——查的是**视口的阈值行**：`viewportRowBufLine[scrollmargin]` 和 `viewportRowBufLine[height-1-scrollmargin]`，即"当前视口里，距顶/底边 scrollmargin 行的那两个屏幕行显示的是哪个 buffer 行"。

按下键 / 按上键时发生了什么：
- 视口内容（`StartLine`）**没变**，只有光标移动
- 所以"视口里哪个屏幕行显示哪个 buffer 行"这个映射**完全没变**
- → `viewportRowBufLine` 对阈值判定是**当前有效的**

时序坑只影响"查光标新位置"，不影响"查视口阈值行"。这正是能把问题缩小到这么小的关键，也是"全文映射 + 改时序"这类方案完全没必要的原因。

---

## 4. 为什么顶边也该修（之前说"良性"是症状轻，不是正确）

顶边的 1:1 误差方向与底边相反，但同样是错的：

- **底边**：阈值偏大 → 条件**该触发时不触发** → 光标飞出视口（灾难性，本次 bug）
- **顶边**：阈值偏大 → 条件**不该触发时触发了**（premature）→ 视口过早向上滚 → 光标被推离顶边比 scrollmargin 更远（轻度，用户难察觉，所以体验"可接受"）

具体例子：viewport 顶上是个含装饰的标题区，光标还在 margin 区下方（按屏幕行算没进 zone），1:1 阈值却已判它"进入 zone" → 触发一次多余的向上滚。用户感觉是"按上键时画面偶尔多跳一下"。

既然修法是**精确对称**的（顶/底边各加一个查表，同一个 `viewportRowBufLine` 数组、同一对索引），**没有理由只修底边**。一并修能让顶边滚动也变精确，消除那个偶发的多余滚动。

---

## 5. 落地步骤

1. **改 `Relocate`**（`bufwindow.go:248-260`）：按 §2 加 6 行（两个阈值的查表 + 一个 `if IsMD` 守卫）。原生非 MD 路径行为完全不变（fallback 到原 `Scroll` 表达式）。
2. **边界处理**：
   - `viewportRowBufLine[i] == -2`（未填充，短文件 / 视口没填满）→ 回退原生 `Scroll`（已含在 §2 的 `if t >= 0` 里）。
   - `height <= 2*scrollmargin`（极矮窗口）→ 索引越界风险，原生也有类似问题，可加 `if scrollmargin < height && height-1-scrollmargin >= 0` 守卫，或忽略（极端配置）。
3. **测试**：
   - 含大量标题/表格/代码块的 md，光标走到底边按 ↓ → 触发上滚，光标停在 statusLine 上方 scrollmargin 行内。
   - 对称验证 ↑ 顶边：光标走到顶边按 ↑ → 触发下滚，且不发生多余滚动。
   - 短文件（视口未填满）→ 回退原生，行为不退化。
   - 非 MD 文件 → 走 fallback，行为与原生完全一致。

**改动量**：`Relocate` 内约 +6 行，无新数据结构，无新函数，无时序调整，无渲染管线改动。

---

## 6. 为什么不采用"全文 row→line 映射"

讨论中曾考虑两类更重的方案——(a) 段级前缀和缓存、(b) buffer 级全文 `screenRow ↔ bufLine` 映射——但对本 bug 都过度设计：

- (a) 为"远距离跳转后查光标新位置"而设计，但按下键根本不查光标（§3），这套缓存用不上。
- (b) 为"viewport 外的 row→line 数据"而设计，但本修法只查 viewport 内的两个固定索引（§2），用不上 viewport 外的数据。

它们解决的是 C0 §4.1 时序坑的**泛化版本**，而本 bug 不踩那个坑。除非将来发现"远距离跳转（搜索命中 / 滚动条拖拽）后的滚动也有可感知问题"，否则不做全文映射——那时再恢复设计即可。

---

## 7. 一句话结论

`Relocate` 的顶边/底边 margin 阈值在 MD 下从"按 line 算"改成"查 `viewportRowBufLine`"，两个阈值对称改、动作不动、约 +6 行——即可同时修复 ↓ 底边不滚 + ↑ 顶边偶发多余滚动，且不引入任何新数据结构或时序改动。
