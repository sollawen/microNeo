## 一、产品定位

MicroNeo 是一个基于 [Micro](https://github.com/micro-editor/micro) 的终端文本编辑器，专注为 Markdown 文件提供**实时富渲染**能力。

**一句话定位**：**AI 时代的终端 Markdown 工作站——读起来是阅读器，用起来是编辑器。**

> 看起来像 Markdown 阅读器，用起来是 Micro 编辑器。

### 1.1 核心差异化

#### 时代背景 / 为什么是现在

**AI 的母语是 Markdown。** 每一次与 Claude Code / Cursor / Aider / Cline 的协作，产物都是 Markdown：spec、PRD、设计文档、agent 报告、CHANGELOG。开发者每天接触的 Markdown 字数，已经从"偶尔读个 README"暴涨到"读不完"。

与此同时，AI 工具链把一批原本只活在 GUI IDE 里的开发者推进了 terminal。他们对 terminal 工具有审美期待（用过 warp、bat、lazygit），但不愿意为了 Markdown 渲染去配置 nvim + Lua 插件生态。

但 terminal 里的 Markdown 工具栈，五年没更新。

#### MicroNeo 解决的两个真实痛点

terminal 里现有的 Markdown 工具分三类，各有缺口：

**① 纯查看类（glow / leaf / mdcat / bat）**
渲染漂亮，但只读。用户想在 AI 输出的报告里加一条注释、改一行字，必须退出查看器 → 打开 editor → 看到的又是原始 Markdown。从"漂亮的渲染"跌回"原始标记"的落差，每次都不爽。

**② CLI 内嵌输入（Claude Code / Cursor CLI / Aider 的小 input 框）**
一回车就发送。但凡要写三五句 spec、一段需求、一个修改意见，就痛苦不堪。毫无编辑体验可言。

**③ 重型方案（nvim + render-markdown.nvim / 切到 GUI 的 Obsidian / Typora）**
能做到渲染+编辑，但要么是 nvim 生态的高门槛，要么脱离 terminal 切到 GUI，工作流断裂。

**MicroNeo 占据的就是这三者之间的空白：留在 terminal、既能渲染又能编辑、零配置即用。**

#### MicroNeo 生在终端

不是退而求其次。Markdown 文档本来就生在 terminal——从 git 来、从 AI agent 的 stdout 来、从 CI log 来。它们不需要被搬运到 GUI 才能被阅读和修改。

#### 核心使用场景

1. **审阅 AI 输出**：打开 AI 生成的 PRD/方案/报告，看到富渲染效果。要标注、修改？光标点进去就是编辑模式，10 秒不动自动回到渲染视图。
2. **撰写 AI 输入**：告别 CLI 里那一行憋屈的 input 框。在真正的编辑器里写 spec、prompt、需求文档，写完复制粘贴（或未来直接 pipe 给 AI）。
3. **日常 Markdown 工作**：README、技术文档、笔记——任何在 terminal 里和 Markdown 打交道的场景。

#### 对比表

|                          | MicroNeo | Micro | glow/leaf/bat | nvim+插件 | Helix |
|--------------------------|:--------:|:-----:|:-------------:|:---------:|:-----:|
| 能编辑 Markdown          | ✅       | ✅    | ❌            | ✅        | ✅    |
| 富渲染（去标记符号）     | ✅       | ❌    | ✅            | ✅        | ❌    |
| 零配置即用               | ✅       | ✅    | ✅            | ❌        | ✅    |
| 不离开 terminal          | ✅       | ✅    | ✅            | ✅        | ✅    |
| 不学新交互范式           | ✅       | ✅    | ✅            | ❌        | ❌    |

注：glow/leaf/bat 是纯查看工具，列出是为了说明 MicroNeo 填补了"终端里既能渲染又能编辑 Markdown"这个空白。

### 1.2 目标用户

**核心用户（PMF 锚点）**
AI 工作流重度使用者——每天用 Claude Code / Cursor / Aider / Cline 等 AI 编程工具，频繁需要：
  - 审阅 AI 生成的 spec / PRD / 设计文档 / agent 报告
  - 撰写超过一行的话给 AI（需求、修改意见、context）
  - 在 terminal 里完成上述工作流，不愿意切到 GUI

**第一批辐射用户**
在 terminal 里写文档的开发者——README、技术博客、内部文档、设计文档。

**更广辐射用户**
terminal 重度 + Markdown 笔记党——坚持留在 terminal、不想为了 Markdown 笔记开 Obsidian/Typora 的人。

### 1.3 不做的事情

- ❌ 不做 LSP / tree-sitter / debug
- ❌ 不做通用代码编辑器的重度功能
- ❌ 不做 GUI
- ❌ 不做 Mermaid、LaTeX 等复杂渲染
- ❌ 不自己开发语法高亮引擎

## 二、核心理念

### 2.1 我们是渲染层

MicroNeo 对 Markdown 文件的渲染，不是"在 Micro 原生显示的基础上加点东西"，而是**我们自己就是渲染层**。

全文的渲染，由 MicroNeo 的渲染管线统一负责。

Micro 原生的显示代码，是我们的**兜底方（fallback）**。当我们自己的渲染模块还没有覆盖到某类内容时，我们把那一行委托给 Micro 原生代码来显示。

### 2.2 光标所在行的特殊处理

全文由我们渲染，但有一个例外：**光标所在行**。

当用户的光标在某一行上时，说明用户要编辑这行。这时这行（以及它所属的渲染片）脱离渲染模式，交由 Micro 原生代码显示和编辑。

### 2.3 10秒规则

如果用户 10 秒内没有任何操作（没有按键、没有点击鼠标），说明用户停下来了，在阅读。

这时，连光标所在行也交回给我们的渲染程序负责。用户看到的将是一份完整的、没有任何原始标记的渲染后文档。

**触发条件**：用户有按键或点击鼠标 → 重置计时器。鼠标移动不算操作。

### 2.4 渐进接管

早期，我们的渲染模块少，大部分行只能委托给 Micro 原生代码。用户看到的和 Micro 差不多，只是表格等少数地方变漂亮了。

随着我们一个模块一个模块地开发，我们自己渲染的行越来越多，委托给 Micro 的行越来越少。

最终，我们完全接管整个文档的渲染。Micro 原生显示代码不再被调用（但保留作为兜底）。

```
阶段1：  [我们的渲染] □□□■□□□□□□    只有表格
阶段2：  [我们的渲染] □□□■■□□□□□    + 粗体斜体
阶段3：  [我们的渲染] ■□□■■■□□□□    + 标题 + 代码块
阶段4：  [我们的渲染] ■■■■■■■■■■    全面接管

■ = 我们渲染   □ = 委托 Micro 原生
```

### 2.5 为什么不做坐标映射？

传统的富文本编辑器需要维护「屏幕坐标 ↔ buffer 坐标」的双向映射，复杂度极高。MicroNeo 选择回避这个问题：

- 渲染时，只管漂亮，不关心光标在哪里
- 编辑时，只管正确，回退到原生逻辑
- 两套模式互不干扰，代码各自独立

## 三、两种模式

MicroNeo 对 Markdown 文件有两个模式：**渲染模式**（默认）和**编辑模式**。这是整个产品交互的基石。

### 3.1 渲染模式（默认）

用户打开 Markdown 文件时的初始状态：

- 全文由渲染管线输出，无行号，无光标
- **方向键 ↑↓** = 上下滚动文档（类似 `less` 或 Leaf 的行为）
- **方向键 ←→** = 水平滚动
- 背后始终记录光标位置，只是不可见
  - 如果滚动导致记录的光标位置滚出可视区域，自动把光标移到窗口内（用户无感知）
- **其他任何键**（空格、回车、字母、数字……）= 在背后记录的光标位置进入编辑模式
- **鼠标点击** = 光标定位到点击的内容字符，进入编辑模式
- **鼠标拖选**（按下 + 移动 + 抬起）= 高亮选中渲染后的内容，松开鼠标时复制到剪贴板，状态栏闪现 `Copied`，**保持渲染模式**（不进入编辑模式）

### 3.2 编辑模式

- 光标所在行（及所属渲染片）回退 Micro 原生显示
- 方向键 = 移动光标（Micro 原生行为）
- 所有 Micro 编辑操作正常工作
- 多光标时：有几个光标，就有几行在编辑状态，这些行都交给 Micro 原生代码处理

### 3.3 模式切换

```
渲染模式 → 编辑模式：
  - 按任意非方向键
  - 鼠标点击

编辑模式 → 渲染模式：
  - 10 秒无操作（无按键、无鼠标点击。鼠标移动不算）
  - 超时时间可配置（mdrenderidle），0 = 从不自动进入渲染模式
```

### 3.4 状态指示

状态栏显示当前模式，让用户始终清楚自己在哪个状态：

- 渲染模式：状态栏显示 **`READING`**（或等效标识）
- 编辑模式：状态栏显示 **`EDITING`**（或等效标识）

此指示可通过配置项关闭。

### 3.5 视觉切换

模式切换时的跳变是**有意为之**的设计：

- TUI 环境下无法做平滑过渡
- 跳变本身就是视觉反馈："你现在在编辑模式" / "你现在在阅读模式"
- 10 秒规则让大部分阅读时间看到的是纯渲染效果

### 3.6 关键交互细节

#### 搜索（Ctrl+F）

在渲染模式下触发搜索：进入编辑模式，搜索行为与 Micro 原生一致，搜索的是原始 Markdown 文本。

#### 复制

- **编辑模式下**选中复制：沿用 Micro 原生机制，剪贴板内容为原始 Markdown 文本
- **渲染模式下**鼠标拖选复制：在富渲染后的内容上拖选一段，松开鼠标时自动复制到剪贴板，状态栏闪现 `Copied`，保持在渲染模式（不进入编辑模式）
  - 复制的具体内容（渲染后纯文本 vs 原始 Markdown）和装饰字符（如表格边框 `|`）的处理规则，待开发到该阶段时确定

#### 撤销/重做

模式切换不影响 undo 历史。渲染/编辑只是显示层的概念，buffer 层的操作记录不受影响。

#### 多文件与多视图（Tab / VSplit / HSplit）

Micro 支持三种多文件场景：Tab（标签页切换）、VSplit（垂直分屏）、HSplit（水平分屏）。

**模式状态按视图（pane）独立维护，不按文件，也不全局共享。**

规则：
- 每个 pane 自己记录当前是渲染模式还是编辑模式
- **只有 `.md` / `.markdown` 文件的 pane 才有渲染模式**，其他文件的 pane 永远是 Micro 原生显示，MicroNeo 完全不介入
- 同一个 Markdown 文件在多个 split 里打开时，各 pane 的模式状态独立——可以在左边编辑、右边渲染对照，互不影响
- 切换到某个 pane 时，看到的是该 pane 自己的状态
- 每个 pane 自己维护 10 秒计时器，互不影响

#### 鼠标点击定位

渲染模式下，鼠标的 click 和 drag 是两种不同的交互，需要明确区分：

- **Click**（按下后没有移动就抬起）= 定位光标，进入编辑模式
- **Drag**（按下后发生移动再抬起）= 拖选复制，保持渲染模式（见上文「复制」小节）

#### Click 定位规则

渲染模式下用户**点击**某个位置，该行（或渲染片）退出渲染、回退原生显示，光标出现在点击处。

渲染输出的每个字符分两类：
- **内容字符**：对应 buffer 中某个真实位置。如表格单元格里的文字、标题文本等。
- **装饰字符**：渲染模块为了美观加的，如表格框线 `|`、`-`、padding 空格等。用户点了也不算。

定位规则：
- 点击到**内容字符** → 光标定位到该字符对应的 buffer 位置
- 点击到**装饰字符** → 不处理，视为无效点击

## 四、功能需求

### 4.1 表格渲染（P0）

**已实现**：基本的表格列对齐（通过 `mdtablealign` 配置项控制）。

**需要改造为渲染片模式**：
- [ ] 当光标不在表格区域内时，整个表格以渲染片方式显示（无行号）
- [ ] 当光标进入表格区域时，回退到原生显示（有行号）
- [ ] 渲染片输出行数可以与 buffer 行数不同
- [ ] 支持用 ASCII 线条字符绘制更美观的表格边框

### 4.2 粗体和斜体渲染（P1）

**当前状态**：Micro 已有语法高亮，用不同颜色区分，但不去除标记符号。

**需求**：
- [ ] 渲染模式下，去除 `**` / `*` / `__` / `_` 等标记符号
- [ ] 用终端样式（颜色 + 加粗/斜体属性，如果终端支持）显示效果
- [ ] 光标所在行回退到原生显示（显示标记符号）
- [ ] 正确处理跨行的情况（保守处理：只要标记未闭合，该行就不渲染）
- [ ] 作为共用工具函数，供所有结构渲染器调用

### 4.3 代码块渲染（P1）

**需求**：
- [ ] 检测 fenced code block（` ``` ` / `~~~`）
- [ ] 渲染模式下，用特殊背景色或边框标识代码块区域
- [ ] 渲染片模式：整个代码块作为一个渲染片，光标进入时回退原生显示
- [ ] 代码块内容原样输出，不调用行内渲染器
- [ ] 语法高亮直接使用 Micro 已有的系统，不自己开发

### 4.4 标题渲染（P2）

- [ ] 渲染模式下，根据标题级别调整显示样式
- [ ] 可选：去除 `#` 符号，用颜色/缩进区分级别
- [ ] 光标所在行回退原生显示

### 4.5 列表渲染（P2）

- [ ] 渲染模式下，美化有序/无序列表的缩进和符号
- [ ] 光标所在行回退原生显示

### 4.6 链接渲染（P2）

- [ ] 渲染模式下，`[text](url)` 只显示 `text`，用特殊颜色标识
- [ ] 光标所在行回退原生显示

### 4.7 其他 Markdown 元素（P3）

- 引用块（`>`）
- 水平分割线（`---`）
- 图片标记（`![alt](url)`）
- 脚注

### 4.8 段落渲染器（P3）

普通段落文本。这是全面接管的最后一个模块——当所有其他渲染模块就位后，段落渲染器覆盖剩余的行，实现 100% 接管。

- [ ] 识别不属于任何其他渲染模块的普通文本行
- [ ] 可做简单的视觉优化（段落间距、去除多余空行等）
- [ ] 光标所在行回退原生显示

## 五、渲染架构（技术方案）

> 本章为技术设计参考，后续会独立为 TDD 文档。

### 5.1 三层分离

渲染由三个正交的层次组成，每层只管自己的事：

| 层 | 职责 | 输入 | 输出 | 谁来做 |
|----|------|------|------|--------|
| **结构渲染** | 这块内容在哪里、多大 | 一组 buffer 行 | 屏幕布局、边框、间距 | 各结构渲染器（表格、代码块、引用块……） |
| **行内渲染** | 这段文字长什么样 | 一行文本 | 去掉标记符号后的文本 + 终端属性（粗体/斜体） | 共用的行内渲染器 |
| **语法高亮** | 每个字符什么颜色 | 字符位置 | 颜色 | Micro 已有的高亮系统，我们不动 |

三层是**叠加**关系，不是替换关系：

```
每个字符的最终样式 = Micro 语法高亮的颜色 + 行内渲染器的终端属性（粗体/斜体）
```

### 5.2 行内渲染器是"胶水层"

行内渲染器（去除 `**`、`*`、`` ` ``、`[]()` 等标记，应用粗体/斜体等终端属性）不是和表格、代码块并列的一个渲染模块。它是**被所有结构渲染器调用的共用工具**。

```
表格渲染器
  → 画边框、定列宽
  → 每个单元格内容 → 调 行内渲染器 → 输出

标题渲染器
  → 去 # 号、定样式
  → 标题文本 → 调 行内渲染器 → 输出

段落渲染器
  → 没有特殊结构
  → 整行文本 → 调 行内渲染器 → 输出

引用块渲染器
  → 左侧标记、缩进
  → 引用文本 → 调 行内渲染器 → 输出

列表渲染器
  → 缩进、符号
  → 列表项文本 → 调 行内渲染器 → 输出
```

**唯一的例外：代码块渲染器不调用行内渲染器。** 代码块里的 `*` 不是斜体，`` ` `` 不是行内代码。代码块内容原样输出，语法高亮直接读 Micro 的。

### 5.3 渲染管线（Render Pipeline）

整个显示流程由渲染管线主导：

```
对 buffer 的每个可见行：

  光标在这行（或所属渲染片内）？
  ├─ 是 → Micro 原生显示（编辑模式）
  └─ 否 → 该行属于某个结构渲染器吗？
           ├─ 是 → 结构渲染器处理
           │      ├─ 内容部分 → 调用行内渲染器（代码块除外）
           │      └─ 叠加 Micro 语法高亮颜色
           ├─ 否 → 行内渲染器处理 + 叠加 Micro 语法高亮颜色
           └─ 都不行 → 委托 Micro 原生显示（fallback）
```

每个结构渲染模块只需要回答两个问题：
1. **这行我能处理吗？**（CanHandle）
2. **渲染输出是什么？**（Render）

回答不了第 1 个问题的，交给下一个模块。所有模块都回答不了的，交给 Micro 原生代码。

### 5.4 渲染片（Render Segment）

渲染片是多行结构渲染模块的基本单位。

**定义**：一个渲染片 = 原始 buffer 中连续的若干行 + 负责渲染这些行的模块。

**属性**：
- `lineStart`：在 buffer 中的起始行号
- `lineEnd`：在 buffer 中的结束行号（含）
- `renderer`：负责渲染的模块
- 渲染输出可以与原始行数不同（如 5 行 buffer → 8 行屏幕输出）

**关键规则**：
- 同一行只能属于一个渲染片（不重叠）
- 光标进入渲染片的任意一行 → 整个渲染片回退原生显示
- 多个光标时：只要任一光标在渲染片内，该片回退原生显示

**示例**（表格渲染片）：

```
buffer 行号    内容
─────────────────────────────────────────
line 1         # 借钱信
line 2         （空行）
line 3         你好：
line 4            我有件事情要求你
line 5         | 日期 | 金额 | 还款日期 |     ← 渲染片开始
line 6         |------|------|----------|
line 7         | 3/5  | 10   | 4/5      |
line 8         | 3/9  | 10   | 4/5      |     ← 渲染片结束
line 9         （空行）
line 10        你愿意借给我吗？
line 11        爱你的 Mike

当光标不在 line 5-8 时：
  line 1-4       → 委托 Micro 原生（早期没有渲染模块覆盖）
  line 5-8       → 表格渲染片输出（无行号，可能行数不同）
  line 9-11      → 委托 Micro 原生

当光标在 line 7 时：
  line 1-4       → 同上（委托原生）
  line 5-8       → 回退 Micro 原生显示（有行号，显示原始 markdown）
  line 9-11      → 同上（委托原生）
```

### 5.5 行号与 Gutter

MicroNeo 对 Markdown 文件**不显示行号**。

Markdown 是给人阅读和写作的文档格式，不是代码。用户在编辑 Markdown 时不需要行号（和 Leaf 等阅读器一样）。去掉行号还有一个好处：不需要 gutter 区域，内容区可以更宽。

**注意**：这只是对 Markdown 文件的规则。其他文件类型（`.go`、`.py` 等）不受影响，Micro 原生的行号照常显示。

### 5.6 渲染片内超宽内容的处理

分两阶段：

**早期**：渲染片内不换行，内容超宽时向右延展，用户通过方向键 ←→ 水平滚动。和 Micro 原生行为一致。

**终极目标**（后续版本实现）：
- 表格：单元格内容按单词换行（参考 Glow/Leaf 的做法），动态调整列宽
- 代码块：按字符换行（代码中空格有意义，不能按单词断）

## 六、非功能需求

### 6.1 性能

- 渲染片检测只针对可视区域，不全文件扫描
- 每帧渲染无明显延迟（目标 < 16ms）
- 渲染片列表有缓存机制，仅 buffer 内容变更时重算
- **架构约束**：必须支持 100% 行由渲染管线处理的场景。不能假设"大部分行走原生 fallback"。每行经过渲染管线的开销必须接近 O(1)

### 6.2 兼容性

- 必须兼容 Micro 原有的所有快捷键和操作
- 所有渲染功能可通过配置项独立开关
- **MicroNeo 的全部渲染代码只对 Markdown 文件（`.md`、`.markdown`）激活**。其他文件类型下，我们的代码完全不介入，Micro 原生行为不受任何影响

### 6.3 可回退

- `mdrender: false` 一键关闭所有渲染，完全回到 Micro 原生体验
- 单个渲染模块可独立关闭，关闭后该模块负责的行 fallback 到原生显示
- 渲染不影响文件保存的内容（buffer 数据永远不变）

### 6.4 终端兼容

- 基本功能必须在不支持特殊终端特性的环境下工作
- 高级渲染（如真粗体、真斜体）作为增强，终端不支持时优雅降级

## 九、风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| **Micro 上游合并冲突** | fork 版本与上游越拉越远，安全更新难以同步 | 改动集中在 `internal/display/` 层，保持其他模块尽量不动；定期同步上游 |
| **终端兼容性差异** | 不同终端对粗体/斜体/真彩色的支持不一致 | 定义最低兼容标准（基本功能可用），高级特性优雅降级；提前在主流终端测试（iTerm2、Terminal.app、Windows Terminal、Alacritty、kitty） |
| **模式切换带来用户困惑** | 用户不知道当前在渲染模式还是编辑模式 | 状态栏 READING / EDITING 指示；视觉跳变本身就是反馈 |
| **渲染管线性能** | 渲染计算拖慢显示，尤其大表格或复杂嵌套 | v0.1 阶段做 benchmark 基线测试；缓存机制避免每帧重算；只处理可视区域 |
| **鼠标定位精度** | 渲染输出的字符与 buffer 位置对应关系出错，点击定位不准 | 每个渲染模块在输出字符时同时记录映射关系；优先保证内容字符的定位准确，装饰字符降级处理 |
