## 帖子1：Ask HN: How are you preserving your skills while using AI?

**链接**: https://news.ycombinator.com/item?id=48463576

我的回复：

我也有同感。我现在的做法是：
- 在ai写代码前，绝对禁止它写代码。只允许它和我讨论方案，架构，主要流程，主要数据结构
- 一边讨论，一边形成详细的方案文档
- 最后文档我满意了，才让ai去动手写代码。

这样，虽然代码不是我写的，但是所有的结构都在我的脑子里。


**正文翻译**：

我是 [Big Company] 的高级工程师，AI 工具无处不在。虽然没有强制要求你使用，但它们太容易获得了，所以大多数人都用。

关于 AI 的社会层面有很多担忧，但在个人层面上，我开始慢慢觉得自己不如以前熟练了。我确实能做更多，但理解的更少。"提示-审查"循环的编码工具（Claude Code、Codex、Pi、OpenCode、Amp 等）Simply don't encourage mastery in the same way as shaping the code yourself. 当然，你可以说你在"更高层次抽象思考"。但当这个抽象失败时会发生什么？抽象经常失败。

这种技能侵蚀不是快速的过程。我不是一夜之间神奇地失去编码能力。但它感觉像是生锈。慢慢侵蚀支柱，直到倒塌。

这个工具（目前）需要一个熟练的手来正确引导。然而使用这个工具感觉会降解那只熟练的手。这个负反馈循环我真的很担心——无论是对于靠软件谋生的能力，还是对软件质量整体的冲击。

所以我问 HN。社区是如何保护他们的技能的？特别是当积极使用 AI 时。

## 帖子2：Show HN: OpenCode powered coding workspace for microcontroller development

**链接**: https://news.ycombinator.com/item?id=48471762

我的回复：

何止是“很多嵌入式工程师非常抗拒 AI”啊，我们公司里大部份软件工程师到现在还在从chatgpt复制粘贴代码呢。我劝了他们很多回都没用。他们就觉得自己亲自一个字一个字的敲到vscode里面特别心里踏实。

没办法。让人改变习惯，真的是一件很难的事情。

**正文翻译**：

大家好，我是 Razz。

我是个软件开发者。十几岁时，我沉迷于微控制器和 Arduino。最近，我想做一个简单的 ESP32 项目，然后意识到我基本上还在用 2015 年的工具和工作流程。微控制器开发的整体方法感觉没怎么变过。

我还注意到很多嵌入式工程师非常抗拒 AI。但当我问他们实际怎么用的时候，往往发现他们只是尝试从 ChatGPT 复制粘贴代码。这和软件开发人员现在每天使用的真正 AI 编码工具的工作流程完全不同。

所以我开始思考围绕这个想法做一个开源项目：使用 OpenCode（它有内置的免费模型），围绕它构建工具，让嵌入式开发者可以尝试真正的 AI 编码工具来进行微控制器开发。

我对项目方向非常灵活。我的主要目标是构建一个真正能帮助嵌入式开发者的 AI 驱动工具。

我很想听听你们的评论、批评和想法。什么样的 AI 工具对嵌入式开发真正有用？
