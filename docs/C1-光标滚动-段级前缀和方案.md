# C1 光标滚动 —— 段级前缀和方案（止血）

> 关联：调研依据见 `docs/C0-光标滚动-现状调研.md`（根因、影响范围、已确认事实）。
> 本文是**止血方案**：修复根因、覆盖全部受影响场景，但**不改变现有架构**（不收敛数据源、不清死代码）。与 `docs/C2-光标滚动-buffer级映射方案.md` 是"先跑通 vs 彻底做"的关系。

---

## 0. 定位

- **修什么**：让 `Scroll/Diff/SLocFromLoc` 在 MD 模式下走真实屏幕行，从而让 `Relocate` 的底边/顶边判定、PageUp/Down、滚轮、滚动条全部正确（覆盖 C0 §3 影响范围）。
- **怎么修**：在 `BufWindow` 上维护段级屏幕行数缓存（前缀和），在三个原生函数入口加 MD 分支。
- **不修什么**：不改 `Relocate` 主体、不改渲染管线、不收敛 `viewportRowBufLine`/`MDSegments`/`SegmentMeta` 三套并存的结构。这些是 C2 的范围。
- **代价**：会让代码里同时存在"整屏的 viewportRowBufLine"+"段级聚合的 segRowCount"两套并行数据，未来仍需 C2 收敛。

---

## 1. 思路

C0 §2.3 已确认：滚动函数既没用整屏的 `viewportRowBufLine`，也没用碎片的 `MDSegments`，而是活在 micro 原生的 1:1 假设里。

本方案落地的，是代码里**早已规划却没接线**的设计——`md.SegmentMeta`（`internal/md/md.go:31`，注释写明"缓存到 BufWindow 上供 Scroll/Diff 使用"，但全代码库零引用）。给它接上：

- 维护每个段渲染出的屏幕行数（含装饰行）
- 维护前缀和，使"任意 buffer 行 → 其屏幕行绝对位置"为 O(1)
- 让 `Scroll/Diff/SLocFromLoc` 在 MD 下查这套数据，而非走 1:1

---

## 2. 数据结构

在 `BufWindow`（`internal/display/bufwindow.go`）上新增（与 `b.MDSegments` 对齐的并行缓存）：

```go
// 每个段的屏幕行数（含装饰行）
segRowCount []int // segRowCount[i] = b.MDSegments[i] 渲染出的屏幕行数
segStartRow []int // 前缀和：segStartRow[i] = 第 i 段之前累计的屏幕行数
segCacheGen uint64 // 代际：b.MDSegments 引用变化 / bufWidth 变化时自增，令缓存失效
```

> 不复用 `SegmentMeta` 类型本身（它是死代码，留待 C2 决定是吸收还是删除），而是在 `BufWindow` 上直接开字段，保持改动局部。

---

## 3. 行数来源（零成本复用）

- `displayBufferMD`（`bufwindow_md.go:688`）渲染每个段时**已经**得到 `rowBufLines`，`len(rowBufLines)` 就是该段的屏幕行数 → 直接写入 `segRowCount[idx]`。**可见区的行数是免费的。**
- 不可见段（`Scroll/Diff` 跨视口时需要）按需懒算：调用该段的 `Render` 取 `len(rendered.Rows)` 并缓存。第一次远距离滚动会算一次，之后命中缓存。
- 缓存失效时机：`b.MDSegments` 重建（发生在 reHighlight，即编辑时，见 C0 §4.4）或 `bufWidth` 变化。纯光标移动不会重建，所以**导航期间缓存始终有效**——正是本 bug 的场景。

---

## 4. 三个核心函数（MD 分支，放在 `bufwindow_md.go`）

```
screenRowOfLine(line)   = segStartRow[segIdx(line)] + 段内偏移(段首到该 line 的屏幕行)
scrollMD(s, n)          = 从 s.Line 起，沿段累加 segRowCount 直到消耗 n 个屏幕行，
                         落在第 n 行所属的 buffer 行（装饰行归属相邻内容行，见 C0 §4.2）
diffMD(s1, s2)          = screenRowOfLine(s2.Line) - screenRowOfLine(s1.Line)
```

在 `Scroll` / `Diff` / `SLocFromLoc`（`internal/display/softwrap.go`）入口加 MD 分支：

```go
func (w *BufWindow) Scroll(s SLoc, n int) SLoc {
    if w.Buf.IsMD && w.mdConfig.MDRender && !softwrap {
        return w.scrollMD(s, n)
    }
    // 原生路径，一字不改
}
```

> 注意：当前 MD 渲染与 softwrap 是**互斥**的实际使用场景（MD 走非 softwrap 的 `renderSegmentMD` / `renderSegmentNative`）。MD 分支只需覆盖非 softwrap。

---

## 5. `Relocate` 的改动（最小）

**不需要改 `Relocate` 主体逻辑**。只要把它依赖的 `Scroll`/`SLocFromLoc` 换成屏幕行感知版本，底边/顶边判定就自动正确：

```go
c := w.SLocFromLoc(activeC.Loc)   // 现在 = 光标真实屏幕行
// 底边判定里所有 w.Scroll(...) 自动按屏幕行计算 → 装饰行被正确计入
```

即：**`Relocate` 本身可能一行都不用改**，只要它依赖的 `Scroll/SLocFromLoc/Diff` 在 MD 下变正确即可。这也最符合"少改 micro 原生代码"的原则——只需在 `Scroll/Diff/SLocFromLoc` 里各加一个 `if` 分支。

> 关键：这个方案之所以能绕开 C0 §4.1 的时序坑，是因为 `segRowCount/segStartRow` 缓存的是**全 buffer** 的段行数，与当前视口无关，在 `Relocate` 时刻就有效——不像 `viewportRowBufLine` 那样滞后于 `Relocate`。

---

## 6. 落地步骤

1. **加缓存字段**：在 `BufWindow` 增加 `segRowCount / segStartRow / segCacheGen`；在 `displayBufferMD` 渲染每段时写入 `segRowCount[idx]`。
2. **写工具函数**：`screenRowOfLine(line)`、`scrollMD(s, n)`、`diffMD(s1, s2)`（放在 `bufwindow_md.go`，纯新增）。
3. **加 MD 分支**：在 `softwrap.go` 的 `Scroll` / `Diff` / `SLocFromLoc` 非 softwrap 分支前各加一个 `if IsMD && MDRender` 早返回（每个函数改动 3~5 行）。
4. **缓存失效**：在 `b.MDSegments` 被重新赋值的两处（`buffer.go:212`、`buffer.go:1032`）以及 `bufWidth` 变化时，令 `segCacheGen` 失效——可通过现有的 `OptionCallback` / `Resize` 入口，或在 `displayBufferMD` 开头比对 `b.MDSegments` 指针自检失效（最低侵入）。
5. **测试**：
   - 构造一个含大量标题/表格/代码块的 md，光标走到底边按 ↓，验证触发上滚且光标停在 statusLine 上方 `scrollmargin` 行内。
   - 对称验证 ↑ 顶边。
   - 验证 PageDown / 滚轮 / 滚动条位置与原生观感一致。
   - 验证编辑后（段重建）缓存正确刷新。

---

## 7. 优缺点

**优点**
- 一次修好 C0 §3 列出的全部场景（↑↓、PageUp/Down、HalfPage、滚轮、滚动条、百分比）。
- `Relocate` 主体无需改动，原生侵入最小（仅 3 个函数各加一个 `if` 分支）。
- 导航期间缓存稳定，无额外渲染开销（可见区行数来自既有渲染）。
- 与"少改 micro 原生代码"的项目原则高度契合。

**缺点 / 风险**
- 首次远距离跳转需懒算若干段的行数（一次性，可接受；可加异步预热）。
- 需正确处理段边界、空 buffer、未闭合段（detect 已保证段连续覆盖全 buffer）。
- 缓存失效要与 `b.MDSegments` 重建点严格对齐，否则会用过期行数。
- **架构债**：又往碎片化方向加了一层（`segRowCount` 段级聚合），与 `viewportRowBufLine`（整屏）形成两套并行数据。修了 bug 但没收敛架构——这是 C2 要解决的问题。

---

## 8. 与 C2 的关系

本方案是 **C2 的过渡/子集**：

- C2 会把 `segRowCount/segStartRow`（段级）与 `viewportRowBufLine`（视口级）**合并**为同一个 buffer 级映射，`viewportRowBufLine` 降级为该映射的一个视图。
- C2 会吸收 `SegmentMeta` 死代码，统一成唯一真源。
- 因此本方案的所有工具函数（`screenRowOfLine`/`scrollMD`/`diffMD`）在 C2 落地后不会白写——它们只是换了底层数据源，签名与语义可保留。

可以先上 C1 止血、验证修复有效，再以 C2 做架构收敛；也可评估后直接做 C2（见 `docs/C2-光标滚动-buffer级映射方案.md`）。
