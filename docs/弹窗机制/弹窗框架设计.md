## 一、一句话架构

> **建一个轻量的 Float 容器作为弹窗框架；Selector 和 MessageBox 以兄弟关系嵌入它；事件路由在 micro 的 DoEvent 加一个集中分支；notePane 独立于本框架。**

---

## 二、设计目标

microNeo 需要弹窗（selectPane 已有，fileList / md 大纲 / theme 选择器 / 版本说明在规划中）。micro 在弹窗领域没有任何基础设施（见研究文档），所以框架得自建。但 microNeo 的整体定位是**轻量**，所以这个框架也必须轻——这是头号约束。

具体目标：
1. **消除重复**：把 selectPane 已写的"位置 + 边框 + title + 生命周期"逻辑抽出来，给所有未来浮窗共用
2. **边界清晰**：框架（容器）和应用（具体浮窗）职责分离，互不污染
3. **侵入收口**：对 micro 原生代码的改动集中在一处，不分散打洞
4. **简单可证**：v1 用单浮窗 + modal 约束，让"嵌套、z-order、焦点栈"等复杂问题根本不出现

---

## 三、硬约束（不可违反的不变量）

这些约束来自需求方的明确决策，是设计的地基，不是建议：

| # | 约束 | 含义 |
|---|---|---|
| C1 | **单浮窗** | 任意时刻全局最多一个 Float 打开 |
| C2 | **no-nesting** | Float 家族内部不允许嵌套（A 里开 B） |
| C3 | **modal** | 浮窗打开后，焦点锁在浮窗里，事件不流到底层 pane |
| C4 | **同步、用户驱动** | v1 不处理异步场景（进程、PTY、goroutine 驱动的关闭） |
| C5 | **框架要薄** | 容器只管位置/边框/title/生命周期/路由；业务能力（滚动、列表渲染等）下沉到具体浮窗 |

**这几个约束是相互成就的**：C1 + C2 让 z-order 和焦点栈问题不存在；C3 + C1 让事件路由可以做成单一分支；C4 让框架不必处理异步生命周期；C5 让框架本身代码量小、可独立演进。任何一条放松，复杂度都会非线性上升。

## 四、架构分层

```
┌──────────────────────────────────────────────────┐
│  弹窗框架（Float 容器）                          │  ← 唯一的"框架"
│  职责：位置 / 边框 / title / 生命周期 /          │
│        modal 事件路由 / 清屏                     │
│  侵入：micro.go 的 DoEvent（1 处事件 + 1 处绘制） │
└──────────────────────────────────────────────────┘
                 ▲                ▲
                 │嵌入            │嵌入
        ┌────────┴───────┐        └────────┐
        │                │                 │
┌───────────────┐  ┌──────────────┐
│   Selector    │  │  MessageBox  │   ← 框架的两个"使用者"，兄弟关系
│ 列表选择      │  │ 多行文本展示 │
│ items/选中/   │  │ lines/（未   │
│ onSelect 回调 │  │ 来的滚动）   │
└───────────────┘  └──────────────┘
   现有：selectPane     规划：版本说明
   规划：fileList
         md 大纲选择器
         theme 选择器
```

### 三层关系

| 层 | 是什么 | 数量 |
|---|---|---|
| **框架层（Float 容器）** | 共享的基础设施，抽象的、无业务的 | **1 个**（单数才是"框架"） |
| **应用层（具体浮窗）** | 跑在框架上的具体实例，有各自业务 | 兄弟关系，互相独立，可加可减 |

### 兄弟关系的精确含义

Selector 和 MessageBox 是兄弟，意味着三件事**同时**成立：
1. 同一个父亲：都嵌入 Float 容器
2. 互不引用：Selector 里没有 MessageBox，反之亦然
3. 互不继承：谁也不是谁的子类型

**实践后果**：两者可以分别开发、分别演进；加第三个兄弟成本极低（嵌入 Float + 写自己的业务）；Float 改一次全员受益，Float 不变时各兄弟完全独立。

## 五、关键决策记录（ADR）

每个决策给出：背景、选项、取舍。详细设计阶段不必再讨论这些，照此执行。

### ADR-1：单 Float 全局单例，no-nesting 为硬不变量

- **背景**：需求方明确要求"轻量"、"同一时间只有一个弹窗"、"不在弹窗里再弹弹窗"
- **决策**：全局只有一个 `TheFloat`（类比已有的 `TheNotePane` / `TheSelectPane`）。Open 时若已有 Float 打开，拒绝（具体处理：记日志 + 不 open，留给详细设计定）
- **代价**：不支持"在 selector 里再开一个 selector 筛选"这类场景
- **收益**：z-order、焦点栈、嵌套生命周期等复杂问题**根本不进入设计**

### ADR-2：Float 容器与具体浮窗分离，用 Go 嵌入表达 is-a

- **背景**：7 个场景（selectPane / fileList / outline / theme / 版本说明 / …）共享"位置 + 边框 + title + 生命周期"这部分能力；但每个场景的业务（列表选择 vs 文本展示）不同
- **决策**：抽出 Float 容器作为共享基底；具体浮窗嵌入它（范式参照 notePane 嵌入 `*BufPane`，详见研究文档 §3.3——白嫖实现，不是加入多态俱乐部）
- **取舍依据**：研究文档的核心原则"复用深度 = 能力重合度"。所有浮窗和 Float 容器的重合度都很高（位置/边框/title 大家都要），所以深度复用（嵌入）；而具体业务各不相同，所以不抽到容器里
- **反模式（已排除）**：让 Float 实现 micro 的 `Pane` 接口——Pane 接口是 tile 专属抽象（16 个方法里 13 个对浮窗无意义），详见研究文档 §9.4 的三条论证

### ADR-3：事件路由走 DoEvent 专门分支（方案 B），不走宿主转发（方案 A）

- **背景**：modal 语义要求"浮窗打开时事件全进浮窗，底层不收键"。selectPane 当前用的是宿主转发（在 `bufpane.go:434/545` 加 hook），这是方案 A
- **决策**：改用方案 B——在 `cmd/micro/micro.go` 的 DoEvent 里加一个专门分支
- **两个方案的本质对比**：
  - 方案 A（现状）：在每个可能开浮窗的入口（BufPane / notePane / 未来 TermPane）各放一段转发代码。**像每个门口各放一个保安**
  - 方案 B（采用）：在 micro 最顶层的事件总入口放一句"如果浮窗开着，事件直接给浮窗"。**像在总大门放一个保安**
- **取舍依据**：
  1. 两个方案**都要碰 micro 原生**（A 已碰 bufpane.go，B 要碰 micro.go），区别不是"侵入 vs 不侵入"，而是**分散 vs 集中**
  2. AGENTS.md 原则是"对 micro 原生代码侵入越小越好"——B 把分散的洞收拢成一处，符合这一原则
  3. **micro.go 里已经有 notePane 的专门分支作为先例**（539-553 行的 if-else 树里），B 是延续既有先例，不是开新风气
  4. modal 的语义本身就是"独占事件"，专门分支是"独占"最直接的表达；A 的宿主转发是用分散 hook 模拟本该集中的独占
  5. **（最关键）触发入口天然分散，A 会随入口数线性膨胀**：同一个浮窗可能有多个触发入口（如 selectReceiver 既可从主 BufPane 触发，也可从 notePane 触发）。方案 A 要求"每个可能开浮窗的宿主"都补转发 hook，入口越多洞越多、漏一个就有 bug；方案 B 一处封顶，与触发入口数量完全无关。详见 §六"触发层与路由层"
- **架构约束：分支优先级最高**：DoEvent 的 if-else 树中，Float 分支必须排在 notePane / InfoBar / Tabs 之前判断（modal = 最高优先级）。这保证"notePane 开着时从 notePane 触发浮窗"的场景自动正确——下一帧事件由 Float 分支接走，notePane 分支被跳过；Float 关闭后自动落回 notePane 分支。**零协调代码**，靠分支顺序 + 绘制顺序（Float 画在 notePane 之后）天然达成
- **代价**：要碰 micro.go 一处（事件分发 + 绘制序列各一行）
- **附带清理**：方案 B 落地后，`bufpane.go:434/545` 里 selectPane 的转发 hook 应**一并移除**（不再需要），把分散的洞清掉

### ADR-4：notePane 独立于本框架

- **背景**：notePane 看起来像浮窗，但本质是"BufPane 的分身"（嵌入 `*BufPane`，有完整编辑能力：光标 / buffer / 文本编辑 / 选择）。它和"轻量、单一职责"的弹窗本质不同
- **决策**：notePane 不进 Float 框架，不受本框架的任何约束（包括 no-nesting）
- **取舍依据**：研究文档 §3.3 已论证——notePane 嵌入 BufPane 的动机是白嫖编辑实现，不是加入某个浮窗俱乐部。强行纳入本框架是削足适履
- **实践后果**：notePane 保持现状（自己的路由分支、自己的生命周期、嵌入 BufPane）。"notePane 开着时能否开 Float" 这类问题在 v1 **不进入设计**——因为它们本就分属两个体系，互不感知；若未来真有具体场景需要协调，再单独处理

### ADR-5：滚动等业务能力归具体浮窗，不进 Float 容器

- **背景**：版本说明这类多行文本可能超出浮窗高度，需要滚动。问题：滚动逻辑放 Float 还是放 MessageBox？
- **决策**：放 MessageBox。Float 容器**不提供任何业务能力**，只提供容器能力（位置/边框/title/生命周期/路由/清屏）
- **取舍依据**：
  1. **不是所有浮窗都需要滚动**（Selector 通常不需要）。放 Float 是污染容器
  2. **不同浮窗的滚动语义可能不同**（MessageBox 滚动文本，未来的树形 outline 可能滚动节点）。容器无法预知
  3. **C5（框架要薄）的直接体现**
- **v1 决策**：MessageBox 早期版本**不实现滚动**（内容必须放得下，或截断）。滚动是 MessageBox 自己未来的演进项，与框架无关

### ADR-6：v1 只做同步、用户驱动的浮窗

- **背景**：讨论中曾考虑"浮窗里跑 .sh / 程序"的场景
- **决策**：v1 不纳入。异步/进程/PTY 驱动的场景全部排除在框架外
- **取舍依据**：异步把"生命周期由用户控制"变成"生命周期由进程控制"，引入 goroutine、PTY、跨线程 UI 更新等复杂度，与"轻量"定位冲突。真要跑脚本，未来用 micro 已有的 TermPane（完整的终端）才是正确归宿，而不是塞进浮窗框架

## 六、触发层与路由层（正交两轴）

> 讨论路由（ADR-3）时容易混淆"谁来打开浮窗"和"浮窗打开后事件去哪"。这是两个完全独立的轴，分清它们是理解整个框架的关键。

### 触发层（WHO / WHEN 打开）

- 逻辑天然分散在各入口，**不可也不该收口**——每个入口最懂自己的触发条件
- 共享的只有 `Float.Open()` 这个统一 API
- 已规划 / 已实现的触发入口：

  | 入口 | 触发条件 | 打开的浮窗 |
  |---|---|---|
  | 主 BufPane 热键 | 用户按 md大纲快捷键 | md大纲 Selector |
  | 命令行 | 用户输入 `set colorscheme` | theme Selector |
  | 程序启动 | 首次运行 | 版本说明 MessageBox |
  | 命令行 | 用户输入 `help` | help MessageBox |
  | 主 BufPane | alt-enter → discover 发现多 receiver | selectReceiver Selector |
  | notePane | alt-i（notePane 开着时） | **同一个** selectReceiver Selector，不同入口 |

### 路由层（打开后事件去哪）

- **必须统一**——modal 的定义就是独占
- 由框架的单一分支（ADR-3 方案 B）兜住，与触发点无关

### 两轴正交

- 触发层无论多花（上表 6 个入口、未来 60 个），路由层**零变化**
- 路由层只有一个分支，对触发层完全不感知"谁打开的"
- **每新增一个浮窗应用，路由层零改动**——这正是"不能搞成四处都要改"原则的落地

### 验证场景：selectReceiver 从 notePane 触发

这是"两轴正交 + 分支优先级"配合工作的完整证明：

1. notePane 里按 alt-i → `notePane.HandleEvent` → 调 `selectReceiver.Open()`
2. 下一帧：DoEvent 先看 `Float.IsOpen` → 命中 → 事件给 Float（**notePane 分支被跳过**）
3. Float 关闭：DoEvent 不再命中 Float 分支 → 落回 notePane 分支 → 事件回 notePane
4. **全程零协调代码**——notePane 不知道 Float 的存在，Float 不知道是谁打开的，靠分支顺序 + 绘制顺序自动达成

这个场景同时验证了 ADR-4（notePane 独立于框架）的正确性：notePane 不进框架、不感知 Float，但 selectReceiver 仍然能从 notePane 正常触发并正常收回——"互不感知"不等于"互相冲突"。

## 七、与 micro 原生代码的关系（侵入清单）

AGENTS.md 原则：**对 micro 原生代码侵入越小越好**。本框架对 micro 原生的侵入**全部集中在 `cmd/micro/micro.go` 一个文件**，共两处：

| 侵入点 | 位置 | 改动性质 | 改动量级 |
|---|---|---|---|
| 事件分发 | `micro.go` DoEvent 的 if-else 树（539-553 行附近） | 新增一个 Float 分支 | 约 2-3 行 |
| 绘制序列 | `micro.go` DoEvent 的绘制段（488-502 行附近） | 在 notePane.Display 之后追加 Float.Display | 约 2-3 行 |

**同时清理**：`internal/action/bufpane.go` 的 433-435 / 541-546 行（selectPane 的转发 hook）——方案 B 落地后这些 hook 失去存在理由，应一并移除。**净效果是分散的洞被收拢成集中的一处**，符合侵入最小化原则。

> 注：具体加在哪一行、resize 分支如何处理 Float，属于详细设计的范畴（分支优先级已在 ADR-3 定下：Float 排在 notePane 之前），本文档不规定。

**除上述外，框架不侵入任何其它 micro 原生文件**。Float 容器和 Selector / MessageBox 的代码全部在 microNeo 自己的文件里（按 AGENTS.md 约定，可以新建 `internal/action/float.go` / `selector.go` / `messagebox.go`，或沿用 selectpane.go 所在目录）。

## 八、框架边界（管什么 / 不管什么）

这张表是详细设计的依据——左边的必须做，右边的明确不做：

| Float 容器**管** | Float 容器**不管**（归具体浮窗） |
|---|---|
| 浮窗位置（x, y）+ 自适应展开算法 | 列表 items 管理、选中态 |
| 边框 + title 渲染 | 文本内容、换行 |
| 清屏（覆盖底层像素） | 滚动（任何形式） |
| 生命周期（Open / Close / IsOpen） | onSelect 回调的业务语义 |
| modal 事件路由（拦截、独占） | 具体按键的业务响应 |
| Reverse 样式等通用视觉元素 | 浮窗特有的视觉（图标、颜色、高亮规则） |

**判别准则**：如果一项能力"所有浮窗都需要且语义一致"——进容器；如果"只有某个浮窗需要"或"语义因浮窗而异"——归具体浮窗。

## 九、v1 范围与暂不做清单

### v1 做

1. Float 容器（位置/边框/title/生命周期/路由/清屏）
2. Selector（嵌入 Float，迁移现有 selectPane 的业务逻辑过来）
3. MessageBox 骨架（嵌入 Float，早期不滚动）
4. DoEvent 路由分支接入 + 清理 bufpane.go 旧 hook

### v1 明确不做（留给未来）

| 项 | 为什么不做 | 何时可能做 |
|---|---|---|
| 嵌套浮窗（A 里开 B） | 违反 C2，且 z-order / 焦点栈复杂度爆炸 | 永不（除非放弃轻量定位） |
| 多浮窗并存 | 违反 C1 | 永不 |
| modeless 浮窗（开着但不锁焦点） | 违反 C3 modal 约束 | 出现明确需求且能论证 modal 不够用时 |
| 滚动 | 归具体浮窗，非框架职责（ADR-5） | MessageBox 等具体浮窗自己演进 |
| 异步浮窗（跑程序 / 进程驱动） | 违反 C4，复杂度不匹配（ADR-6） | 用 TermPane 替代，不进本框架 |
| 第三个及以上的浮窗类型 | YAGNI——等真实需求出现 | 出现 ≥1 个无法用 Selector/MessageBox 表达的场景时 |

## 十、风险与待澄清

详细设计阶段需要回答、但本文档故意不替它回答的问题：

1. **Float 容器的具体接口形态**：嵌入字段 + 方法集的精确划分。本文档只定原则（容器薄、业务下沉），不定签名。
2. **Open 时已有 Float 的具体处理**：拒绝 + 记日志？还是其它？ADR-1 只定了"拒绝"，未定细节。
3. **resize 事件如何处理 Float**：DoEvent 的 resize 分支当前会通知 InfoBar / Tabs / notePane，Float 是否也需要在 resize 时重新定位？需要详细设计时验证。
4. **现有 selectPane 的迁移路径**：是"重写一个 Selector，旧 selectPane 弃用"，还是"原地改造 selectpane.go 让它嵌入 Float"？两种迁移路径对存量代码影响不同。
5. **测试策略**：弹窗的几何计算（自适应展开）是 selectPane 当前最复杂的部分，重构时如何保证行为不变（基线测试？）。

这些都不是架构问题，是落地问题，留给详细设计文档。
