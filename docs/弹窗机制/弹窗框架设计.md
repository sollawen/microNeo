# 弹窗框架设计

> **文档定位**：架构层方案，回答 WHY / WHAT / HOW。
>
> **配套文档**：见文末"文档索引"。

## 概述

microNeo 需要一个统一的弹窗机制来承载一系列临时、模态、带边框的浮层应用——包括现有的 `SelectPane`（receiver 选择），以及规划中的 fileList、mdOutline、theme 选择器、版本说明等。micro 原生没有弹窗框架；现有的 `SelectPane`（`internal/action/selectpane.go`）和 `NotePane`（`internal/action/notepane.go`）各自实现了一套边框、title、清屏与模态事件路由，彼此没有任何共享代码。

本设计提出一个轻量的 `FloatFrame` 容器作为弹窗框架。`FloatFrame` 本身是一个"空壳"——不持有任何业务逻辑，启动时创建、生命周期与进程相同；具体浮窗（v1 仅 `SelectPane`）在打开时把尺寸、锚点、标题、边框色、内容绘制函数和事件处理函数**一次性塞给** `FloatFrame`，由 `FloatFrame` 接管所有用户交互；关闭时清空内部状态，回到空壳。

设计原则有三条，构成后续所有取舍的依据：

- **框架要薄**：容器只做所有浮窗共享的事（锚点展开、画框、事件路由、生命周期）；业务能力（列表选择、滚动、动态刷新等）全部下沉到具体浮窗。
- **职责单向**：具体浮窗把"自己 + 回调"交给容器，容器调回调；两者是委托关系，不互相嵌入、不反向调用。
- **状态自有**：浮窗的几何、内容、业务状态都在自己身上；容器只是这些状态的临时持有者与绘制代理。

## 一、约束

约束 C1–C5 是本框架的硬性边界，任一放宽都会让复杂度非线性上升。

| # | 约束 | 含义 |
|---|---|---|
| C1 | 单浮窗 | 任意时刻全局最多一个浮窗打开 |
| C2 | 无嵌套 | 浮窗内不能再开浮窗 |
| C3 | 模态 | 浮窗打开后事件被锁住，不流向底层 |
| C4 | 同步用户驱动 | 不处理异步 / 进程 / PTY 驱动场景 |
| C5 | 框架要薄 | 容器只做容器能力，业务能力下沉到具体浮窗 |

C1 + C2 让 z-order、焦点栈、并发打开等问题从设计域消失；C3 + C1 让事件路由可做成 DoEvent 中的一条分支；C4 让框架不必处理 goroutine 与跨线程 UI 更新；C5 直接决定容器的代码量上限。

## 二、核心模型

`FloatFrame` 与具体浮窗之间是**单向委托**关系：具体浮窗不嵌入 `FloatFrame`，`FloatFrame` 也不持有具体浮窗的对象引用——后者只持有前者传入的两个函数值。这样既避免了 Go 中常见的双向引用与循环依赖，也让"v1 只有一个浮窗、未来加新浮窗"无任何耦合。

```
┌──────────────────────────────────────────────────────────┐
│  调用方（notepane.go / 命令 / 热键）                      │
│  1. new 一个具体浮窗（如 SelectPane）                     │
│  2. 调 SelectPane.Open(...)，准备好锚点、内容、回调       │
└────────────────────────┬─────────────────────────────────┘
                         │ SelectPane.Open 内部：
                         │   - 算出 w, h, anchor, title, frameColor
                         │   - 把 display() 和 handleEvent() 打包
                         │   - 一次性塞给 TheFloatFrame.Open(...)
                         ▼
┌──────────────────────────────────────────────────────────┐
│  FloatFrame（空壳容器）                                   │
│  接管后职责：                                             │
│    - 用 anchor + contentSize 算锚点展开 → (fx, fy)         │
│    - 每帧画边框 + title                                   │
│    - 每帧调 display(contentArea) 画内容                   │
│    - 每个事件调 handleEvent(event)                        │
│    - 关闭时清空所有字段、断回调引用                        │
└──────────────────────────────────────────────────────────┘
```

## 三、职责划分

每一项能力都有唯一的所有者，没有交叉、没有协议层：

| 能力 | 所有者 |
|---|---|
| 算浮窗**内容**尺寸（contentSize，纯内容，不含边框） | 具体浮窗 |
| 算触发锚点（anchor，通常来自光标位置） | 具体浮窗 |
| 提供 title 字符串和边框色 | 具体浮窗 |
| 实现"画内容"和"处理事件" | 具体浮窗 |
| 内部 padding（如有） | 具体浮窗 |
| **由内容尺寸派生边框外尺寸**（contentSize + 2，title 撑宽取 max） | **FloatFrame** |
| **锚点自适应展开**（向左/右、上/下展开，算最终 fx, fy） | **FloatFrame** |
| 画边框（4 角 + 上下左右边） | FloatFrame |
| 画 title（嵌入上边框） | FloatFrame |
| 清屏（浮窗矩形覆盖底层） | FloatFrame |
| 事件路由（DoEvent 分支） | FloatFrame |
| 处理 resize（决定取消还是重画） | 具体浮窗 |
| 生命周期（Open / Close / IsOpen） | FloatFrame |

划分依据是一条简单规则：**任何下沉到容器的能力，都必须证明"所有浮窗都需要"**。锚点展开是所有浮窗的公共几何需求，所以归容器；滚动、列表选中态、动态刷新等只有部分浮窗需要，所以一律留具体浮窗。这条规则让容器的代码量有天然上限，也避免容器随业务膨胀。

## 四、数据结构

### 4.1 FloatFrame

```go
// internal/action/floatframe.go

type Rect struct {
    X, Y, W, H int
}

// Pos 是屏坐标的一个点（用于锚点）
type Pos struct {
    X, Y int
}

// Size 是宽高（用于内容尺寸）
type Size struct {
    W, H int
}

type FloatFrame struct {
    isOpen bool

    // —— 从具体浮窗传入（均为纯内容语义，不含边框）——
    anchor      Pos           // 锚点（屏坐标）
    contentSize Size          // 内容尺寸
    title       string
    frameColor  tcell.Style   // 零值 = config.DefStyle
    display     func(contentArea Rect)
    handleEvent func(event tcell.Event)

    // —— FloatFrame 自己算 ——
    outerW, outerH int        // 含边框总尺寸（contentSize + 2，title 撑宽取 max）
    fx, fy         int        // 锚点展开后的最终左上角
}

func NewFloatFrame() *FloatFrame {
    return &FloatFrame{}
}

func (f *FloatFrame) Open(
    anchor Pos,
    contentSize Size,
    title string,
    frameColor tcell.Style,
    display func(Rect),
    handleEvent func(tcell.Event),
) bool {
    if f.isOpen {
        return false // C1: 拒绝
    }
    f.anchor = anchor
    f.contentSize = contentSize
    f.title = title
    if frameColor == (tcell.Style{}) {
        frameColor = config.DefStyle
    }
    f.frameColor = frameColor
    f.display, f.handleEvent = display, handleEvent

    // 由内容尺寸派生含边框外尺寸：宽度可能被 title 撑宽
    f.outerW = max(contentSize.W+2, len(title)+6) // 6 = 边框2 + 前后──各2
    f.outerH = contentSize.H + 2

    // 失败前置检查（D-1 修正）：屏幕完全放不下则放弃 open。
    // 不修改任何状态（isOpen 保持 false），不 set fx/fy，不 Redraw。
    if outerH > bottomLimit+1 || outerW > w {
        return false
    }

    f.fx, f.fy = expandAnchor(anchor.X, anchor.Y, f.outerW, f.outerH)
    f.isOpen = true
    screen.Redraw()
    return true
}

func (f *FloatFrame) Close() {
    if !f.isOpen {
        return
    }
    f.isOpen = false
    f.display, f.handleEvent = nil, nil // 断引用，便于 GC
    screen.Redraw()
}

func (f *FloatFrame) IsOpen() bool { return f.isOpen }

func (f *FloatFrame) Display() {
    if !f.isOpen {
        return
    }
    // 1. 清屏（边框 + 内容区）
    // 2. 画边框（4 角 + 上下左右），用 f.frameColor，宽 f.outerW，高 f.outerH
    // 3. 画 title（嵌入上边框）
    // 4. 委托画内容
    f.display(Rect{X: f.fx + 1, Y: f.fy + 1, W: f.contentSize.W, H: f.contentSize.H})
}

func (f *FloatFrame) HandleEvent(event tcell.Event) {
    if !f.isOpen {
        return
    }
    f.handleEvent(event) // 所有事件（含 resize）都转发给具体浮窗
}
```

**几何语义约定**：`Open` 入参 `contentSize` 是**纯内容尺寸**（不含边框、不含 title 撑宽）。`FloatFrame` 内部派生出含边框外尺寸 `outerW = max(contentSize.W + 2, len(title) + 6)`、`outerH = contentSize.H + 2`，再据此做锚点展开得到最外层包围盒的最终左上角 `fx, fy`。委托画内容时传入的 contentArea 始终是 `Rect{fx + 1, fy + 1, contentSize.W, contentSize.H}`——即使 title 撑宽了外框，内容区宽仍按原 `contentSize.W`，多出来的宽度留给右侧边框装饰。具体浮窗**完全不需要知道边框存在**。`expandAnchor` 是包内私有函数，签名 `(ax, ay, outerW, outerH) (fx, fy)`，承载所有锚点展开逻辑（见 §七）。

**Open 返回值语义**（D-1 修正）：`Open` 返回 bool 表达「是否成功 open」：

- `true` = 成功 open。
- `false` = 没开成（调用方不区分原因）：可能是 FloatFrame 已有浮窗在打开、依据约束 C1 拒绝再开；也可能是屏幕放不下（失败前置检查，几何归容器）。

「屏幕放不下」是几何问题，所有浮窗都会遇到，按 §三 末尾的简单规则归容器。具体浮窗（如 SelectPane）检查 `Open` 返回值，`false` 时直接 `onSelect(nil)` 返回，对调用方完全透明。失败前置检查逻辑（`outerH > bottomLimit+1 || outerW > w`）连同 `expandAnchor` 一起放在 `FloatFrame.Open` 内部——具体浮窗不读屏幕尺寸、不碰任何几何逻辑。

```go
for {
    // —— Display 阶段 ——
    // ... Tabs / InfoBar / notePane 正常画 ...
    if TheFloatFrame.IsOpen() {
        TheFloatFrame.Display()                    // 画边框 + title + 委托 sp.display 画内容
    }
    screen.Screen.Show()

    event := <-screen.Events

    // —— 事件分发 ——
    if TheFloatFrame.IsOpen() {                    // ← 最高优先级分支
        TheFloatFrame.HandleEvent(event)           // → sp.handleEvent(event)
        // 其它路径全部短路，模态
    } else {
        // ... 正常路由到 Tabs / notePane / InfoBar ...
    }
}
```

```go

// 用户交互恢复流向 BufPane / notePane / InfoBar
// sp 实例是否保留：见 §十 D2
```

### 两个要点

- **回调顺序固定**：关闭时永远先 `FloatFrame.Close()`（容器归位），再触发业务回调（`onSelect`）。这样回调里读到的容器状态已是关闭后，避免 "isOpen 还是 true" 的竞态感。
- **容器与具体浮窗的解耦**：阶段 2 的主循环只与 `TheFloatFrame` 对话，完全不知道打开的是 SelectPane 还是别的什么；SelectPane 只在阶段 1 和阶段 3 出现，运行期是透明的。

## 六、事件路由

DoEvent 是 micro 的主循环（`cmd/micro/micro.go`）。`FloatFrame` 在两处介入。

**背景知识：screen buffer 后写胜出**——同一格被多次 `SetContent` 写入时，最后一次写的留下来。Display 阶段所有写入完后 `screen.Screen.Show()` 一次性推到终端。所以"绘制顺序 = 覆盖顺序"，后画的盖在前面。

**Display 阶段**——`FloatFrame` **必须最后画**（硬约束，不是建议）：

```go
if action.TheFloatFrame.IsOpen() {
    action.TheFloatFrame.Display()
}
```

这条铁律由 FloatFrame 的模态语义（C3）派生：**用户看到的最上层 UI 必须等于用户事件实际作用的对象**。如果 FloatFrame 不是最后画，会出现"事件给 FloatFrame、视觉被 notePane / InfoBar 盖住"的撕裂，违反 GUI 一致性。未来调整 Display 顺序时必须保留这一条，FloatFrame 永远在最顶。

**事件分发**——在 if-else 树最前插入 `FloatFrame` 分支，赋予最高优先级（模态语义）。z-order 与事件优先级同向：FloatFrame 视觉最上 ↔ 事件最高优先级，这是 GUI 一致性的派生约束，二者必须同时调整：

```go
if event != nil {
    _, resize := event.(*tcell.EventResize)
    if resize {
        action.InfoBar.HandleEvent(event)
        action.Tabs.HandleEvent(event)
        if action.TheNotePane != nil && action.TheNotePane.IsOpen() {
            action.TheNotePane.HandleEvent(event)
        }
        if action.TheFloatFrame.IsOpen() {
            action.TheFloatFrame.HandleEvent(event) // 转发到具体浮窗；SelectPane 视为取消（见 ADR-9）
        }
    } else if action.TheFloatFrame.IsOpen() { // ← 最高优先级
        action.TheFloatFrame.HandleEvent(event)
    } else if action.TheNotePane != nil && action.TheNotePane.IsOpen() {
        action.TheNotePane.HandleEvent(event)
    } else if action.InfoBar.HasPrompt {
        action.InfoBar.HandleEvent(event)
    } else {
        action.Tabs.HandleEvent(event)
    }
}
```

由于 C1（单浮窗）+ C3（模态），事件路由只需一条分支；具体浮窗的 `handleEvent` 负责吞掉所有非业务键，模态性由此自然实现。

## 七、锚点展开

锚点展开算法（向右优先、向左次之、上下同构、兜底 + 边界保护）从 `selectpane.go` 现状的约 100 行原样迁移到 `FloatFrame`，改名为包内私有函数 `expandAnchor`。**算法逻辑不变，只改归属。**

这是 `FloatFrame` 唯一的几何算法——也正是因为它是所有浮窗的公共能力，才值得放进容器。任何"只此一家"的几何需求都应该留在具体浮窗里。具体浮窗只需要知道自己"多大、从哪触发"，不需要知道"该往哪个方向展开"——后者是容器的职责。

## 九、架构决策

### ADR-1：FloatFrame 为全局单例，启动时初始化

`TheFloatFrame` 在 `globals.go` 的 `InitGlobals` 里 new，与 `InfoBar` / `TheNotePane` 同生命周期，永远非 nil，进程退出时销毁。理由：`FloatFrame` 是程序级全局存在（DoEvent 每帧读它），启动开销约等于零（一个百字节结构体），全局单例与 `InfoBar` 既有惯例一致。

### ADR-2：具体浮窗以函数值方式接入，不走 interface

具体浮窗把 `display` 和 `handleEvent` 作为方法值（或闭包）连同几何参数一次性传给 `FloatFrame.Open`，而不是让 `FloatFrame` 持有具体浮窗的对象引用、通过 interface 反向调用。理由有三：

- 避免"双向引用 + is-a 语义"的耦合——`FloatFrame` 不需要知道"打开的是 SelectPane 还是其它浮窗"。
- 接入零抽象——具体浮窗不必 implement 一个 interface，只需要"有这两个方法"。
- 与"`FloatFrame` 是空壳"的定位一致——空壳只持有数据与函数指针，不持有对象。

### ADR-3：几何与展开的归属切分

具体浮窗负责算"自己多大、从哪触发"（contentSize / anchor / title / frameColor）；`FloatFrame` 负责"向哪展开、画在哪里"（锚点展开 + 画框 + 清屏）。前者是浮窗自有的内容属性，后者是所有浮窗共享的公共几何能力。切分依据见 §三末尾的简单规则。

### ADR-4：事件路由走 DoEvent 集中分支

所有浮窗的事件都通过 DoEvent 中的一条 `FloatFrame` 最高优先级分支接收，不在每个宿主（BufPane / TermPane / notePane）里放 forwarding hook。理由：notePane 已经在 `micro.go` 走这个模式，是既有先例；方案落地后清理 `bufpane.go` 的 forwarding hook，把分散的洞收拢成集中的一处，新加触发入口不再需要补 hook。

### ADR-5：notePane 独立于本框架

notePane 嵌入 `*BufPane`、有完整编辑能力，本质是 BufPane 的分身而非轻量弹窗，故不接入 `FloatFrame`。两者互不感知。"notePane 开着时能否开 FloatFrame"等交互问题 v1 不进入设计。

### ADR-6：v1 不提供滚动、嵌套、异步

容器不实现任何业务能力（滚动归具体浮窗）；C1+C2 让嵌套/多窗场景从设计域消失；C4 让异步驱动场景排除在外（真要跑脚本用 TermPane）。这些约束共同保证框架代码量有明确上限。

### ADR-7：frameColor 作为入参，零值表达 default

边框色由具体浮窗传入 `Open`；传 `tcell.Style{}` 零值时容器内部转成 `config.DefStyle`。不同浮窗可以指定不同边框色（如 SelectPane 默认、fileList 高亮），但常见情况是 default，零值语义让默认情况零负担。

### ADR-8：v1 砍掉 OnCancel

原始设计草拟过一个 `OnCancel` 接口方法，意图是"失败 / 强制关闭时通知浮窗收尾"。排查后发现 v1 没有任何调用点：Esc / Enter 由具体浮窗自己处理；Open 被拒（C1）由调用方自己收尾。唯一可能的真实场景是"resize 后屏幕放不下、容器强制关闭浮窗"，但 v1 不做 resize 重排。故 v1 接口仅 `display` + `handleEvent` 两项；待真出现强制关闭需求时再加，YAGNI。

### ADR-9：resize 转发给具体浮窗，由具体浮窗决定

**决策**：`FloatFrame.HandleEvent` 不对 resize 做任何特殊处理，所有事件（含 resize）一律转发给具体浮窗的 `handleEvent`。具体浮窗自己决定 resize 怎么处理——SelectPane v1 把 resize 视为取消（等同于 Esc）：`Close()` + `onSelect(nil)`。

**核心洞察**：resize 关闭浮窗在语义上**等同于用户按 Esc**——都是"结束当前浮窗交互"。既然 Esc 走的是 `SelectPane.handleEvent` 自己处理（Close + onSelect(nil)），resize 也应该走同一条路，不需要任何额外机制。

**为什么不重算几何（保留原 fx/fy）**：`anchor` 是屏坐标，通常来自调用方的光标位置。resize 后 bufPane 重排、视口滚动，光标的屏坐标**几乎一定会变**。FloatFrame 不知道锚点的语义（是光标位置还是屏角 0,0），也无法重算光标屏坐标——这是 bufPane / 调用方的知识。用 stale 的 `anchor` 重算出来的 `fx, fy` 会飘到奇怪的位置，体验比直接关更差。

**为什么不归 FloatFrame 处理（不归容器）**：resize 不是"几何事件"在 §三 意义上的——§三 的几何能力（锚点展开 / 画边框 / 清屏）都是所有浮窗共享的公共能力；resize 的处理方式因浮窗而异（SelectPane 取消、未来 textPane 可能重画），不是公共能力。把 resize 归具体浮窗符合 §三 末尾的简单规则。

**为什么 v1 关闭可接受**：resize 是低频事件（用户手动拖窗口）；SelectPane 是低频短交互（弹起选个 receiver 就关），不会在长时间开启状态下遇到 resize。用户重拖窗口后重新触发即可拿到新的锚点。

**与 ADR-8 的关系**：本 ADR 不引入"容器强制关闭"的概念——容器只转发，不做决定；关闭永远是具体浮窗自己发起的（无论是 Esc 还是 resize 触发）。因此 ADR-8 "v1 砍 OnCancel" 完全成立，无需修正。

**演进路径**：未来若出现"resize 后浮窗必须留存"的硬需求（如 fileList 长时间挂着），具体浮窗可以在自己的 `handleEvent` 里处理 resize 时不调 `Close()`，而是通过某个待设计的 `Reposition` 入口让 FloatFrame 重展开。届时需要调用方协作重算锚点。v1 不做。

## 十、已定决策

| # | 问题 | 决定 | 关键依据 |
|---|---|---|---|
| D1 | `Open` 签名是否打包几何参数 | 用 `Pos`（锚点）+ `Size`（内容尺寸）；display 入参仍用 `Rect` | 类型即文档，`Pos`/`Size` 比 `Rect` 二用语义清晰 |
| D2 | SelectPane 关闭后是否销毁 | S1'：每次 `NewSelectPane()`，用完即弃 | Go GC 自动回收；调用方拿到实例后丢弃即被回收，不会内存泄漏 |
| D3 | padding 归属 | FloatFrame 不关心，归具体浮窗 | `contentSize` 的"内部构成"是具体浮窗私有知识，FloatFrame 只看尺寸数字 |
| D4 | SelectPane 迁移的基线测试 | 手测 receiver 选择流程 | v1 浮窗仅此一处调用，自动化不划算 |

## 十一、风险

| 风险 | 应对 |
|---|---|
| `FloatFrame` 接管后 SelectPane 行为偏差 | 重写时保留原几何 / 绘制逻辑的语义，仅改归属；迁移后对照 receiver 选择流程手测 |
| 迁移期间 receiver 选择链路（notePane → SelectPane）出问题 | 先建框架、后切接入，分两个 commit |
| 多 goroutine 同时 Open 导致 `TheFloatFrame` 竞争 | v1 Open 全部在事件循环主 goroutine 串行执行，无并发 |
| DoEvent 加分支后影响其它 modal（notePane / prompt） | 分支顺序：FloatFrame > notePane > prompt > Tabs，互斥不冲突 |
