# 弹窗框架设计

> **文档定位**：microNeo 弹窗机制的架构说明。
>
> **阅读对象**：需要理解或扩展弹窗功能的开发者（人或 AI）。架构与设计原则读本文档；**具体实现读代码**，入口见文末「文件地图」。

## 概述

microNeo 需要一系列临时、模态、带边框的浮层应用——现有的 `SelectPane`（receiver 选择），以及规划中的 fileList、mdOutline、theme 选择器、命令面板等。micro 原生没有弹窗框架。

本设计用一个轻量的 `FloatFrame` 容器统一承载这些浮窗。`FloatFrame` 本身是"空壳"——不持有任何业务逻辑，启动时创建、生命周期与进程相同；具体浮窗在打开时把尺寸、锚点、标题、回调一次性塞给 `FloatFrame`，由它接管用户交互；关闭时清空状态，回到空壳。

## 设计原则

三条原则构成后续所有取舍的依据：

- **框架要薄**：容器只做所有浮窗共享的事（锚点展开、画框、事件路由、生命周期）；业务能力（列表选择、滚动、动态刷新等）下沉到具体浮窗。
- **职责单向**：具体浮窗把"自己 + 回调"交给容器，容器调回调；委托关系，不互相嵌入、不反向调用。
- **状态自有**：浮窗的几何、内容、业务状态都在自己身上；容器只是这些状态的临时持有者与绘制代理。

## 约束

| # | 约束 | 含义 |
|---|---|---|
| C1 | 单浮窗 | 任意时刻全局最多一个浮窗打开 |
| C2 | 无嵌套 | 浮窗内不能再开浮窗 |
| C3 | 模态 | 浮窗打开后事件被锁住，不流向底层 |
| C4 | 同步用户驱动 | 不处理异步 / 进程 / PTY 驱动 |
| C5 | 框架要薄 | 容器只做容器能力，业务下沉到具体浮窗 |

C1+C2 让 z-order、焦点栈、并发打开从设计域消失；C3+C1 让事件路由可做成主循环一条分支；C4 让框架不必处理 goroutine；C5 限定容器代码量上限。

## 核心模型

`FloatFrame` 与具体浮窗是**单向委托**：具体浮窗不嵌入 `FloatFrame`，`FloatFrame` 也不持有具体浮窗的对象引用——只持有前者传入的两个函数值（`display` / `handleEvent`）。这避免了双向引用与循环依赖。

```
调用方（notepane.go / 命令 / 热键）
  │  1. new 一个具体浮窗（如 SelectPane）
  │  2. 调 SelectPane.Open(...)，算好锚点、内容、回调
  ▼
FloatFrame（空壳容器）  ← Open(anchor, contentSize, title, frameColor, display, handleEvent)
  接管后：
    - 锚点展开 → (fx, fy)
    - 每帧画边框 + title + 委托 display 画内容
    - 每个事件转发给 handleEvent
    - 关闭时清空所有字段、断回调引用
```

## 具体浮窗范例：SelectPane

`SelectPane` 是当前唯一的具体浮窗，也是「业务能力下沉到具体浮窗」原则的范本——它自己持有滚动、循环、选中态，`FloatFrame` 完全不感知这些。调用方 new 一个实例，调 `Open` 把内容、锚点、回调、以及两个业务参数一次性塞进去：

```go
func (s *SelectPane) Open(
    items []string,
    title string,
    anchor Pos,
    frameColor tcell.Style,
    maxVisible int,            // 视口最大可见行数；<=0 时 fallback 到 len(items)（全部显示，不滚动）
    wrap bool,                 // true: 到顶/底循环；false: 到顶/底停住
    onSelect func(*string),
)
```

三个能力点，各自的设计依据：

- **视口滚动**（`maxVisible`）：候选数超过视口时只显示 `[topIdx, topIdx+visibleH)` 这一段，Up/Down 跟随视口。`contentSize.H = min(len(items), maxVisible)`，再由 FloatFrame 加边框。`maxVisible<=0` 表示「全部显示，不关心上限」，行为退化到无滚动。
- **循环/停住**（`wrap`）：调用方按场景选——receiver 选择是浏览性（2–5 个来回看）用 `true`，`:theme` 是定向选择（找特定 colorscheme）用 `false`。不设统一默认，免得失配。`selected` 始终是绝对索引，wrap 跳顶后按一下 Up 能精确回原位。
- **▲▼ 滚动指示符**：仅在内容溢出（`len(items) > visibleH`）时出现，画在右 padding 列（零加宽）。`▲` 在视口顶行、`▼` 在末行，style **跟随所在行**——若该行恰好选中则随高亮条一起反色，避免高亮条右端被「咬」出一个缺口。

**关于 `maxVisible` 过大的契约**：SelectPane 信任调用方传入的值，不跟屏幕高度做二次 clamp。如果传得过大导致 `contentSize.H` 超出屏幕，`FloatFrame.Open` 的失败前置检查会 `return false`，SelectPane 透明走 `onSelect(nil)`（等同 Esc）——**FloatFrame 不 open，SelectPane 也就跟着不 open**。判断「放不放得下」是 FloatFrame 的职责，SelectPane 不越界重复防御。

## 职责划分

| 能力 | 所有者 |
|---|---|
| 算浮窗**内容**尺寸（contentSize，纯内容不含边框） | 具体浮窗 |
| 算触发锚点（anchor） | 具体浮窗 |
| 提供 title 和边框色 | 具体浮窗 |
| 实现"画内容"和"处理事件" | 具体浮窗 |
| 内部 padding（如有） | 具体浮窗 |
| 由内容尺寸派生边框外尺寸（contentSize+2，title 撑宽取 max） | **FloatFrame** |
| 锚点自适应展开（向左/右、上/下，算最终 fx, fy） | **FloatFrame** |
| 画边框、画 title、清屏 | FloatFrame |
| 事件路由（主循环分支） | FloatFrame |
| 处理 resize（取消或重画） | 具体浮窗 |
| 生命周期（Open / Close / IsOpen） | FloatFrame |

划分依据：**任何下沉到容器的能力，都必须证明"所有浮窗都需要"**。锚点展开是公共几何需求，归容器；滚动、列表选中态等只部分浮窗需要，留具体浮窗。

## 几何语义

**坐标类型**：`Pos{X, Y}`（点，屏坐标）、`Size{W, H}`（尺寸）、`Rect{X, Y, W, H}`（矩形）。

**内容尺寸 vs 外尺寸**：`Open` 入参 `contentSize` 是**纯内容尺寸**（不含边框、不含 title 撑宽）。容器派生 `outerW = max(contentSize.W+2, len(title)+6)`、`outerH = contentSize.H+2`，据此做锚点展开。委托画内容时 contentArea 始终是 `Rect{fx+1, fy+1, contentSize.W, contentSize.H}`——具体浮窗**完全不需要知道边框存在**。

**锚点展开**：`expandAnchor(ax, ay, outerW, outerH)` 是包内私有纯函数，承载所有"往哪边展开"的逻辑（向右优先、向左次之、上下同构、兜底 + 边界保护）。具体浮窗只需知道"多大、从哪触发"，不需要知道"该往哪个方向展开"。

**锚点 sentinel**：`anchor.Y < 0` 时由 `Open` 归一化为相对 `bottomLimit`（statusLine 上方一行）的偏移——`-1` = 贴底、`-k` = statusLine 上方 k 行（公式 `ay = bottomLimit + anchor.Y + 1`）。归一化在 `Open` 内部完成（复用失败前置检查已算好的 `bottomLimit`），`expandAnchor` 始终收到真实屏坐标，保持纯函数性质。这让"贴底弹出"的浮窗（命令面板、fileList 等）调用方不必读屏宽高。X 方向暂不支持 sentinel（无需求，YAGNI）。

**失败前置检查**：屏幕放不下时（`outerH > bottomLimit+1 || outerW > w`）`Open` 返回 `false`，调用方按"没开成"处理（SelectPane 直接 `onSelect(nil)`）。`Open` 返回 `false` 也涵盖"已有浮窗打开被 C1 拒绝"。

## 生命周期

`TheFloatFrame` 是全局单例（`globals.go` 的 `InitGlobals` 里 new，与 `InfoBar` 同生命周期，永远非 nil）。

一次完整流程：

1. **调用方打开**：new 具体浮窗 → 调 `Open(...)` → 内部算 contentSize，连同回调一次性塞给 `TheFloatFrame.Open(...)`，后者算展开位置、置 isOpen、触发重画。
2. **运行**：主循环每帧 Display 阶段最后画 FloatFrame；事件分发因 isOpen 为真直接路由到 FloatFrame（模态）。
3. **关闭**：具体浮窗的 `handleEvent` 内部先调 `TheFloatFrame.Close()`（容器归位、断回调引用便于 GC、触发重画），再触发业务回调（如 `onSelect`）。**回调顺序固定**：先关容器，再回调——这样回调里读到的容器状态已是关闭后。

## 事件路由

主循环（`cmd/micro/micro.go` 的 `DoEvent`）两处介入：

**Display 阶段**——FloatFrame **必须最后画**（硬约束）：tcell 后写胜出，用户看到的最上层 UI 必须等于事件实际作用的对象，否则视觉与事件撕裂。

**事件分发**——在 if-else 树最前插入 FloatFrame 分支，优先级最高（模态语义）：

```
if FloatFrame.IsOpen()        → 转发给具体浮窗（最高优先级）
else if notePane.IsOpen()     → notePane
else if InfoBar.HasPrompt     → 命令行 prompt
else                          → Tabs → BufPane
```

resize 事件同时通知所有 modal（包括 FloatFrame）；具体浮窗自己决定 resize 怎么处理（SelectPane v1 视为取消，等同 Esc：`Close()` + `onSelect(nil)`）。

## 关键决策（ADR 浓缩）

| # | 决策 | 一句话理由 |
|---|---|---|
| 1 | FloatFrame 全局单例，启动时 new | 主循环每帧读它，与 InfoBar 惯例一致 |
| 2 | 具体浮窗以函数值接入，不走 interface | 避免双向引用；空壳只持有数据与函数指针 |
| 3 | 几何与展开的归属：内容归浮窗、展开归容器 | 前者是浮窗私有属性，后者是公共能力 |
| 4 | 事件路由走主循环集中分支 | 收拢 forwarding hook，新触发入口不再补 hook |
| 5 | notePane 独立于本框架 | notePane 嵌入 BufPane 有完整编辑能力，非轻量弹窗 |
| 6 | v1 不提供滚动、嵌套、异步 | 保证框架代码量上限 |
| 7 | frameColor 作为入参，零值 = default | 不同浮窗可指定不同边框色，默认零负担 |
| 8 | v1 砍 OnCancel | 无强制关闭场景，YAGNI |
| 9 | resize 转发给具体浮窗决定 | resize 处理方式因浮窗而异，不是公共能力 |
| 10 | SelectPane 滚动下沉到具体浮窗，FloatFrame 零改动 | ADR 6「v1 不提供滚动」只约束框架；滚动是部分浮窗的业务能力（已有实例验证） |
| 11 | `wrap` 不设统一默认，由调用方按场景定 | receiver 浏览用 true、theme 定向用 false；统一默认会失配 |
