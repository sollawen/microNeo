## 0. 总览

### 0.1 交付物（一句话）

把现有 `SelectPane` 自带的边框 / title / 锚点展开 / 清屏 / 事件转发（forwarding）能力，下沉到一个新建的 `FloatFrame` 空壳容器；`SelectPane` 退化为只持有列表状态与回调的薄业务对象；主循环通过 `DoEvent` 一条集中分支统一路由所有浮窗事件。

### 0.2 改动面（来自设计 §八，已与现状核对）

| 文件 | 类型 | 量级 | 说明 |
|---|---|---|---|
| `internal/action/floatframe.go` | 新增 | ~180 行 | 容器本体 + `expandAnchor`（从 selectpane.go 迁移） |
| `internal/action/selectpane.go` | 整文件重写 | 286 → ~100 行 | 只留业务逻辑 |
| `internal/action/notepane.go` | 改调用点 | ~5 行 | `TheSelectPane.Open(...)` → `NewSelectPane().Open(...)`，签名适配 |
| `internal/action/bufpane.go` | 删转发 hook（含整个 Display 重写） | -13 行 | 删 `HandleEvent` 顶部 forwarding（约 433–437）+ 删整个 `Display()` 重写（约 541–548，让 BufPane 回退到继承 BWindow.Display） |
| `internal/action/globals.go` | `+1 / -1`（分两阶段） | Phase 2 加 `TheFloatFrame`；Phase 3 删 `TheSelectPane`，净 0 行 |
| `cmd/micro/micro.go` | +7 行 | Display +2，事件分发 +5 |

**净侵入 micro 原生代码：`micro.go` +7、`bufpane.go` -13（其中 Display 重写整方法删除让 BufPane 回退继承）。其余改动全部落在 microNeo 自己的 `action` 包内。**

### 0.3 工作量估算

| 项 | 估算 |
|---|---|
| 纯编码 | 3–5 小时 |
| 手测调试（receiver 选择链路） | 1–2 小时 |
| 文档 / changelog 整理 | 0.5 小时 |
| **合计** | **~1 个工作日** |

前提：实施者已通读设计文档与本计划，且 `expandAnchor` 迁移保持算法不变（见 §3.1）。

### 0.4 阶段路线图

```
Phase 0  基线确认        ┐
Phase 1  FloatFrame 落地  ├► 检查点 ①：框架就位，管道通畅，行为零变化
Phase 2  管道接入         ┘
Phase 3  SelectPane 切换  ─► 检查点 ②：业务切到新框架，清理旧代码
Phase 4  验收收尾         ─► 全量测试通过 → 交付（由用户统一 commit）
```

检查点 ① 与 ② 是**硬性切分**（设计 §十一 风险表"先建框架、后切接入"）：每个检查点都应能独立编译、独立运行、独立手测，作为推进信心锚点。**全程不产生中间 commit**——所有改动完成后由用户一次性 commit。

## 2. Phase 1 — FloatFrame 容器落地（检查点 ① 上半）

**目标**：新建 `floatframe.go`，把 `expandAnchor` 从 `selectpane.go` 原样迁移过来，按设计 §四.1 实现 `FloatFrame`。**本阶段不接入主循环、不删任何旧代码。**

### 任务

- [ ] **T1.1 新建 `internal/action/floatframe.go`**，包含：
  - 类型 `Rect` / `Pos` / `Size`（设计 §四.1）
  - `FloatFrame` 结构体 + 字段（`isOpen` / `anchor` / `contentSize` / `title` / `frameColor` / `display` / `handleEvent` / `outerW` / `outerH` / `fx` / `fy`）
  - `NewFloatFrame()`
  - `Open(...)`：按设计 §四.1 实现（**含 D-1 修正**）：含 `outerW = max(contentSize.W+2, len(title)+6)`、`outerH = contentSize.H+2`。**先做失败前置检查**（迁移自 selectpane.go §4.6：outerW/outerH 超出屏幕 → 返回 false，不置 isOpen、不 Redraw）。检查通过后才调 `expandAnchor` 算 `fx,fy`、置 `isOpen`、`screen.Redraw()`、返回 true。返回 bool 语义：`true` = 成功 open；`false` = 没开成（两种失败，调用方不区分原因：① FloatFrame 已有浮窗在打开、依据约束 C1 拒绝再开；② 屏幕放不下）
  - `Close()`：置 `isOpen=false`、断 `display`/`handleEvent` 引用为 nil、`screen.Redraw()`
  - `IsOpen()`
  - `Display()`：清屏矩形 → 画 4 角 + 上下左右边 → 画 title（嵌入上边框）→ 委托 `f.display(contentArea)`，其中 `contentArea = Rect{fx+1, fy+1, contentSize.W, contentSize.H}`
  - `HandleEvent(event)`：`isOpen` 守卫后直接转发 `f.handleEvent(event)`
  - 包内私有 `expandAnchor(ax, ay, outerW, outerH) (fx, fy int)`

- [ ] **T1.2 迁移 `expandAnchor` + 失败前置检查**：从 `selectpane.go` 现状 `Open` 内迁移两段几何逻辑到 `floatframe.go`：
  - **失败前置检查**（§4.6）：`outerH > bottomLimit+1 ‖ outerW > w` → 不 open。这段逻辑**进 `FloatFrame.Open`**（返回 false），不进 `expandAnchor`——因为"放不下"是 Open 的失败语义。
  - **`expandAnchor`**（§4.3/§4.4 的 y/x 自适应 + 边界 clamp）：原样搬为包内私有函数。clamp 是防御性代码，正常路径靠前置检查保证不触发，但保留作为最后防线（绝对不该触发）。
  - 输入语义：原来接收 `paneW`/`paneH`（**含边框**），现在接收 `outerW`/`outerH`（**含边框**）。两者都是"含边框总尺寸"，语义一致，算法体不变。
  - **保留** `bottomLimit`（statusLine 上方）的计算逻辑——receiver 选择当前依赖此语义，迁移时不能丢。`expandAnchor` 内部自行读取 `screen.Screen.Size()` + `config.GetInfoBarOffset()`（与现状一致）。

- [ ] **T1.3 迁移绘制逻辑**：`selectpane.go` `Display()` 的清屏 + 边框 + title 绘制代码搬到 `FloatFrame.Display()`，颜色用 `f.frameColor`（现状全用 `config.DefStyle`，迁移后由入参决定，SelectPane 传零值 → default）。

- [ ] **T1.4 编译验证**：`make build-quick` 通过。此时 `FloatFrame` 无任何调用点，纯新增。

### 验收

- `floatframe.go` 编译通过，`FloatFrame` 行为可由一个临时调试调用点验证（验证后删除调试代码，不入最终交付）。
- `expandAnchor` 输出与旧 `selectpane.go` 在相同 `(ax, ay, outerW, outerH)` 输入下**逐字节一致**（可用测试函数对照，不入库）。

### 注意

- 本阶段**不**碰 `selectpane.go` / `bufpane.go` / `micro.go` / `globals.go`。新旧代码并存，互不引用。

## 4. Phase 3 — SelectPane 切换与旧代码清理（检查点 ②）

**目标**：把 `SelectPane` 改写为 `FloatFrame` 的委托方，删掉所有旧转发代码与全局单例。**这是唯一会改变用户可观测行为的阶段。**

### 任务

- [ ] **T3.1 重写 `internal/action/selectpane.go`**（整文件）：
  - 结构体精简为：`items` / `selected` / `title` / `onSelect`（删除 `x,y,width,height,maxHeight,isOpen` —— 几何归 FloatFrame）
  - 删除全局变量 `var TheSelectPane *SelectPane`（D2：用完即弃）
  - `NewSelectPane()` 返回空对象
  - `Open(items, title string, anchor Pos, frameColor tcell.Style, onSelect func(*string))`（**签名变化**：`x,y *int` → `anchor Pos`；新增 `frameColor`）：
    - 算 `contentSize = Size{W: maxItemLen(items)+2, H: min(len(items), 10)}`（左右 padding 各 1）
    - 调 `TheFloatFrame.Open(anchor, contentSize, title, frameColor, s.display, s.handleEvent)` 并**检查返回值**：`false`（FloatFrame 已有浮窗在打开被拒 / 屏幕放不下）则直接 `onSelect(nil)` 返回，不设置业务状态
    - **删除**原 `Open` 内全部锚点展开 / 边框宽度 / 失败前置检查代码（几何判断全部归 FloatFrame，见 D-1）——SelectPane 不读屏幕尺寸、不碰任何几何逻辑
  - `display(area Rect)`：只画 items 列表，当前选中项 `Reverse`；边框/title 归 FloatFrame
  - `handleEvent(event tcell.Event)`：
    - `KeyUp` / `KeyDown`：改 `selected`，`screen.Redraw()`
    - `KeyEnter`：先 `TheFloatFrame.Close()`，再 `onSelect(&picked)`
    - `KeyEscape`：先 `TheFloatFrame.Close()`，再 `onSelect(nil)`

### D-1：「屏幕放不下」语义

**背景**

现状 `selectpane.go` §4.6 在 `paneH > bottomLimit+1 ‖ paneW > w` 时**不 open**，直接 `onSelect(nil)`。但设计文档 §四.1 的 `FloatFrame.Open` 的 `false` 分支只表达一种情况——FloatFrame 已有浮窗在打开、依据约束 C1 拒绝再开——未提「屏幕放不下」——这是设计文档的 gap。

**选项**

- (a) 让 SelectPane 预判放不下，几何判断调到业务层 ❌
- (b) 几何判断全部归 FloatFrame，`Open` 返回 bool 语义扩展为「是否成功 open」（覆盖两种失败：FloatFrame 已有浮窗在打开 / 屏幕放不下），SelectPane 只看返回值决定是否回调 ✅

**决定：(b)**

「屏幕放不下」是几何问题，所有浮窗都会遇到，按设计 §三 铁律归容器。`FloatFrame.Open` 返回 bool 语义扩展：

- `true` = 成功 open
- `false` = 没开成（原因不区分）：可能是 FloatFrame 已有浮窗在打开、依据约束 C1 拒绝再开；也可能是屏幕放不下。调用方按「没开成」统一处理

SelectPane 检查返回值，`false` 时直接 `onSelect(nil)`——对 `notepane.go` 调用方完全透明，行为与旧实现一致。

> **这是对设计文档 §四.1 的小幅修正**（Open 的 false 分支不再只表达「依据 C1 拒绝再开」一种情况），但符合设计原则（几何归容器）。收尾时同步更新 [`弹窗框架设计.md`](./弹窗框架设计.md)。现状 `selectpane.go` §4.6 的失败前置检查逻辑连同 `expandAnchor` 一起迁移到 `FloatFrame.Open` 内部。

## 7. 风险与回滚

| 风险 | 概率 | 应对 | 回滚 |
|---|---|---|---|
| `expandAnchor` 迁移引入几何 bug（弹窗位置偏移） | 中 | D-3 临时单测对照；T1.2 强制"算法体不变" | 定位到检查点 ② 的几何改动，仅回退 `selectpane.go`/`floatframe.go` 的几何部分，恢复旧 `expandAnchor` 行为 |
| SelectPane 切换后 receiver 选择链路断裂（notePane ↔ SelectPane） | 中 | T3.2 调用点改动小且集中；§5.2 逐项验收 | 回退 Phase 3 改动，回到检查点 ① 的双轨态（旧 `SelectPane` + forwarding 仍在，系统可用） |
| DoEvent 分支顺序写错，导致 notePane / prompt 失效 | 低 | T2.3 严格按设计 §六 顺序：FloatFrame > notePane > prompt > Tabs | 回退 Phase 2 的事件分发改动 |
| 设计文档 §四.1 与 D-1 决策不一致（Open 的 false 分支只表达「依据 C1 拒绝」） | 低 | 实施 D-1 时同步修正设计文档 §四.1 的 Open bool 语义说明 | 在 PR 描述中标注"设计文档小幅修正"，同步更新 [`弹窗框架设计.md`](./弹窗框架设计.md) §四.1 |
| 多 goroutine 竞争 `TheFloatFrame` | 极低 | v1 所有 Open 都在主事件循环串行（设计 §十一） | 无需处理 |
| `make build`（含 generate）在 Phase 3 暴露问题 | 低 | T3.5 强制走 `make build` 而非 `build-quick` | 修 generate 产物 |

**回滚粒度**：由于全程不产生中间 commit，回滚依赖工作区快照（见下「工作区快照约定」）。Phase 3 出问题时，最稳妥的回退路径是恢复到检查点 ① 完成时的工作区状态——此时旧 `SelectPane` + forwarding 仍在，系统完全可用。

**工作区快照约定**（替代中间 commit 的安全网）：

到达**检查点 ①** 时（Phase 1+2 完成、行为零变化已验证），在工作区外做一次**物理备份**：

```bash
# 假设项目在 /path/to/microNeo
cp -r /path/to/microNeo /tmp/microNeo_cp1_backup
```

这是后续最关键的回退锚点。进入 Phase 3 前确认备份已建立。

**回退方法**（Phase 3 出问题时）：

```bash
# 从检查点 ① 完整恢复工作区
rsync -a --delete /tmp/microNeo_cp1_backup/ /path/to/microNeo/
# 验证：make build-quick 通过、旧 SelectPane 行为正常
```

为什么不用 `git stash create`：操作不直观、与检查点无自然关联。物理备份可验证、可追溯、与 git 工作流完全解耦。

若不想依赖快照，最低限度的保护是：Phase 3 的改动集中在 4 个文件（`selectpane.go` / `notepane.go` / `bufpane.go` / `globals.go`），可按文件粒度手动回退。
