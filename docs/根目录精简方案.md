# 仓库根目录精简方案

## 背景

GitHub 仓库根目录的文件/目录条目过多，会把 README 预览挤到第二屏。
访客从宣传链接进来，首屏看不到项目"卖相"（demo、一句话定位），影响 star 转化。

本方案目标：**在零/低风险前提下，减少 GitHub 仓库根目录显示的条目数**，让 README 重新回到首屏。

---

## 现状

仓库根目录实际提交在 git 里、GitHub 上会显示的条目共 **23 个**：

```
.editorconfig  .github/  .gitignore
AGENTS.md  CHANGELOG.md  LICENSE  LICENSE-THIRD-PARTY
Makefile  README.md
aibp-agents/  assets/  cmd/  data/  docs/
go.mod  go.sum  install.sh  internal/
mkdocs.yml  pkg/  runtime/  snapcraft.yaml  tools/
```

> 注：本地根目录还能看到 `microneo`（13MB 二进制）和 `site/`（mkdocs 构建产物），
> 但两者都已在 `.gitignore` 中，**GitHub 上不显示**，不属于本方案处理范围。

每条目在 GitHub 文件列表中约占 26px 高度，23 条 ≈ 600px，加上 repo header（标题/badges/about），
README 预览确实被推到第二屏才出现。

## 实施方案

共 **6 项改动**，根目录 **23 → 17 条目**（−2 目录删除、−1 文件删除、−2 文件移入现有目录、−1 目录移入现有目录）。

建议分 4 个 commit 提交，每步可独立验证、独立回滚。

---

### 步骤 1：删除零引用孤儿

无任何引用，删除零风险。

```bash
# 目录
git rm -r data/                      # micro 上游遗产（metainfo.xml + micro.json）

# 文件
git rm snapcraft.yaml                # 从未产出 snap 包，CI 未接

# assets 孤儿（仅被 data/ 引用，随 data/ 删除成孤儿）
git rm assets/micro-solarized.png    # 仅被 data/io.github.zyedidia.micro.metainfo.xml 引用
```

> `data/io.github.zyedidia.micro.metainfo.xml` 的 screenshot 标签指向**上游 URL** `https://raw.githubusercontent.com/micro-editor/micro/master/assets/micro-solarized.png`（非本地路径）。该 png 在本地仓库无任何引用，data/ 删除后彻底成孤儿，一并删除。

## 风险与回滚

- **风险**：极低。全部为路径迁移 + 引用更新，无逻辑变更。
- **回滚**：按 commit 粒度 `git revert`，可精确回退任一步骤。
- **外部影响**：
  - 已发布 release 的旧二进制不受影响（内含逻辑已编译）
  - 已 `curl | sh` 安装的用户不受影响
  - 新安装用户从新路径拉取，`master` 推送后 `raw.githubusercontent.com` 即时生效
  - `aibp-agents` 已配的开发者：路径名段 `aibp-agents` 保留，Go 侧识别不受影响；但绝对路径变了，需更新本地 `settings.json` / `tui.json` 里的路径
