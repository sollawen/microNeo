# 一、现在有个遗留的问题：

## 问题的现象
以docs/sample.md为例
- line55-59是个table，
	- 在原生编辑状态时只需要显示5 row
	- 但是在render模式时会显示9 row, 因为需要好几个表格frame的装饰row
	- 这是正确的
- 假设光标在line59,且line62是viewport显示的最后一行
	- 因为光标在line59，所以这个table一定是处在原生的编辑状态下，没有装饰行
	- 这时，我按下键，于是程序判断需要scroll up 1 row了
	- 于是整个buffer scrolled up 1 row, 光标移到了line60

这个时候，问题出现了
- 因为光标移到了line60，脱离了这个table了。所以整个table恢复到了渲染模式
- 而渲染模式会添加好几个装饰行，这导致了这个table的显示，一下子就变大了，下面的部份就跑到viewport的底部之外了
- 光标就更是在viewport底部的好几个row之外了，viewport里面看不见了。
- 然后再继续按下键的时候，光标就会在viewport的底边之外向下移动了。用户是看不见的。
- 除非用户按上键，把光标移回到viewport里面，才能看到

# 二、我的观点
- 我们的render把table添加装饰行进行渲染，是正确的，工作的很漂亮
- 光标在line59的时候，按下键后判断应该scroll up，也是非常正确的
- 但两个正确，得到的结果却是不正确的。

## 根本原因
在scrollup的时候，或者说在scrollup之前
- line55-59这个table是处在原生编辑状态下的，没有装饰行
- 所以scrollup 1 row的时候，是根据没有装饰行的viewportRowmap来计算应该向上滚动多少row

但是scrollup完成后
- 这个table变成了渲染状态，凭空多出来了几个装饰行
- 导致按原有数据计算后scrollup 的row数量不够，光标就跑出底部了

## 因此，有两个解决方案

### 方案A，临时补算

在scrollup 的时候，临时去计算那个table渲染后会多出多少装饰row，这样就scrollup 补算后的row

### 方案B，彻底重构

每次scroll up/down 的时候，都先渲染，再scroll

也就是说，当光标上下移动/或是用户输入了什么内容导致buffer变化的时候
- 先把viewport 从头开始 render, 把render后的数据(就是现在代码里render之后直接送进viewport去显示的的那堆数据)先暂存起来，不送进display
- 根据render后的数据，判断是否需要scroll, 需要scroll 几个row
- 把暂存的render后的数据(已经render过了，不需要重新render了)直接送给display，并告诉从是从第几个row开始显示的。于是display就从这些已经render好的数据里，找到start row, 并把后面的数据直接送进viewport显示出来

这样就彻底解决了row/line vs viewportRow 的计算和对应关系

换个说法：
- micro原生的时序是：
	- 用户做了些操作
	- 在 displayBuffer里面进行渲染，然后直接送进 display层
- 方案B的时序是
	- 用户做了些操作
	- 马上渲染但不显示，渲染后的用来显示的数据，保存在 renderedMap里面
	- 做一些必要的判断和处理
	- displayBufferMD()，里面不再render了，直接把renderedMap送进display层
- 这个改动不是特别为scroll服务的，而且根本改变
	- 用户各种对文件的改动，都会进行预渲染 
	- 预渲染是从buffer.startLine开始的，往下面完成整段的segment渲染。就是说如果遇到了一个代码块/表格，那么reder出来的是完整的segment，储存在renderedMap里
	- displayBufferMD()永远都是不渲染，只是把renderdMap中的准确的row送去显示

**本文档后续章节聚焦方案B。**

# 三、时序分析
	
## micro的原生时序

因为micro原生代码，是一个line editor，所以除了softwrap之外，根本不存在render的事情，line/row是精确的一一对应的关系。因为，micro原生代码做了一个非常简洁的处理
```
	while true {
	    display(startLine)            // 每帧都从头重算：从 startLine 开始逐行扫 buffer，算 tab/wrap/高亮，填入屏幕
	    screen.Show()
	    event = <-screen.Events
	    handleEvent(event) {
	        case A: 状态变更（编辑 / 移动光标 / 跳转）
	            → 改 buffer 或光标
	            → 调 Relocate()
	            → Relocate 判断光标是否出安全区：
	                出界 → 改 startLine
	                在安全区 → startLine 不变（这是常态）
	        case B: 纯视口（scroll / page / center / start / end）
	            → 直接设 startLine（必然变）
	            → 不碰 buffer / 光标，不调 Relocate
	    }
	}
```

**关键观察：原生 display 是一个纯函数 `f(startLine, buffer) → 屏幕内容`，每帧全量重算，没有任何缓存。** 所以用户连按三次右键，就会从头重算三次——这对 line editor 来说既便宜又正确（buffer 一改，下一帧自动反映，完全不用操心同步）。

## markdown 文件的特别之处
markdown文件渲染后，line/row之间不再是精确的一一对应关系了，某些segment会产生不同数量的装饰行。所以md文件需要有好几个不同的render来计算 buffer.line变成显示后的数据是什么样子

因此，方案B需要引入一个 renderedMap（实现上命名为 `bigViewportMap`）。
- 各render计算出来的结果并不直接送去displayer然后就扔掉了。而是以renderedMap为假装的显示viewport，把render后的数据送到这个map里面
- 所以，当各render运算完成后，这个map其实就是一个比当前viewport略长一点的bigViewportMap
	- 可以在现有的 viewportRowmap基础上改造一下，为每一个row，增加更多的字段
- 当displayBufferMD()的时候，就可以不用再render第二次了，
	- 直接把renderedMap拿过来，
	- 找到startRow，
	- 从startRow开始把每个row的数据直接填在displayer里面就行了

## 方案B的时序与伪代码

```
	while true {
	    display(startLine) // 直接把 bigViewportMap，从 startLine 开始，填入显示区域（纯 blit，不再 render）
	    screen.Show()
	    event = <-screen.Events
	    handleEvent(event) {
	        case A: 状态变更（编辑 / 移动光标 / 跳转）{
	            → 改 buffer 或光标
	            → startLine = Relocate(oldStartLine, true)    // 传 true：要保证光标被渲染进 map
	        }
	        case B: 纯视口（scroll / page / center / start / end）{
	            直接设 startLine（必然变）
	            不碰 buffer / 光标
	            preRender(startLine, false)    // 传 false：纯视口滚动，光标可能不在新视口内，按底边停止即可
	        }
	    }
	}
```

```
func preRender(startLine, showCursor bool) {
    line = startLine
    cursorShowed = false
    cursorRow = -1

    while true {
        processOneLine(line) {
            if 光标所在行 then {
                使用原生编辑代码生成渲染后的数据，填到 bigViewportMap 里面
                cursorShowed = true
                cursorRow = 当前最后一个 row 的序号
            }
            if 非光标所在行 then {
                使用 detect 来判断应该使用哪个 render
                使用正确的 render 生成渲染后的数据，填到 bigViewportMap 里面
                把这个 segment 全部计算完成
                // 比如是个 table，就要把这个 table 的整个渲染片都渲染完成
                // 不管是不是超出了 viewport 的底边
            }
        }

        // 停止条件（三条同时满足）：
        //   ① 旧光标所在段已整段渲染完整 —— 由上面"segment 全部计算完成"自动保证（概念清晰，实现免费）
        //   ② 新光标所在行已渲染        —— cursorShowed == true
        //   ③ 光标所在行之后，再渲染够 viewportMargin 行 —— 覆盖 display blit 的底边
        //      （viewportMargin 取 scrollmargin，即 display 真正需要的光标下方行数；见下方推导）
        if !showCursor {
            if 最后一个row 超过 viewport 底边 then break
        } else {
            if cursorShowed and (最后一个row >= cursorRow + viewportMargin) then break
        }
    }
}
```

```
func Relocate(oldStartLine, showCursor bool) {
    newStartLine = oldStartLine

    if 光标小于安全区的上边 {
        // 顶部分支：只服务单步↑。大跨度上移走 case B（scroll/page）或 GotoLoc 的 Scroll 预定位，不依赖此处
        newStartLine = oldStartLine - 1
        preRender(newStartLine, showCursor)
        return newStartLine
    }

    // 无条件先 preRender：既是下界分支"用有装饰行的新 map 算 scroll"的前提，
    // 更是【显示契约】的要求——安全区内编辑/段切换也要让 bigViewportMap 跟上 buffer（见下方原则1）
    preRender(newStartLine, showCursor)
    根据 cursor.line 和 bigViewportMap，计算光标所在的 viewport 里的 row

    if 光标大于安全区的下边 {
        // 下界分支：V2 问题的主战场。此时 map 已含展开后的 table，scroll 量算得准
        根据 cursor.line 和 bigViewportMap，想把光标所在的 row 显示在安全区最下边，newStartLine 应该是多少
        newStartLine = 刚才的计算结果
    }
    return newStartLine
}
```

# 四、设计原则与关键结论

> 以下是把方案 B 从直觉推向可落地设计的过程中，逐条确认的硬约束。每条都对应伪代码里的一处必然要求。

## 原则1：显示契约 —— Relocate 必须无条件 preRender

**这是方案 B 引入的新约束，原生 micro 没有这个要求。**

- 原生：`displayBuffer()` 每帧全量重算，直接读最新 buffer → 编辑后自动显示，不用操心。
- 方案B：`displayBufferMD()` 变成纯 blit，只读 `bigViewportMap`，**不再读 buffer**。所以如果 map 没刷新，display 就一直显示旧内容。

推论：**任何改变 buffer 或渲染内容的操作之后，都必须 preRender 刷新 bigViewportMap**，否则用户看不到自己的改动。体现在 `Relocate` 上，就是不能只在上界/下界 if 里 preRender，**安全区内也要 preRender**（光标在屏幕中间打字、左右移动、editMode 段切换，都靠这一条兜住显示）。

> 一个 MD 模式特有的点：光标移动会触发 editMode 切换（光标进出 table），导致**渲染内容本身**变化。所以 MD 模式下，光标移动是"渲染事件"，不只是"光标事件"——任何 case A 操作后都必须 preRender。

## 原则2：case A / case B 的精确划分

划分依据是 **action 是否调用 Relocate**，不是"用户按了什么键"（同一个键在不同绑定下走不同 case）：

- **case A（走 Relocate）**：编辑类（Insert/Backspace/Delete/Paste）、光标移动类（CursorUp/Down/...）、跳转类（FindNext→GotoLoc→Relocate）、混合类（CursorPageUp/Down：移动+滚动+Relocate 都做）。
- **case B（绕过 Relocate，直接设 startLine）**：纯视口类——滚轮 ScrollUp/Down、HalfPageUp/Down、Center、Start、End。

⚠️ **"scroll"有两种**，不要混淆：
- 用户**主动**滚的（滚轮/PageDown 的视口部分）→ case B，绕过 Relocate。
- 光标越界时 `Relocate` **被动**滚的 → case A 内部（V2 问题的发生场景）。

> case B 的处理：直接设好 startLine 后调 `preRender(startLine, false)`。因为纯视口滚动不改 buffer，所以这里 preRender 不是"刷新"，而是为新 startLine 生成 map（旧 map 的起点对不上了）。可选优化：小幅滚轮（scrollspeed=2）若新 startLine 仍在旧 map 范围内，可跳过 preRender 直接 blit——但这是性能优化，不影响正确性。

## 原则3：preRender 停止条件的数学依据

**display blit 的底边 = cursorRow + scrollmargin**（默认 +3）。推导：

```
下界分支把 cursor 放到安全区下沿：cursorRow_in_viewport = height - 1 - scrollmargin
→ newStartLine = cursorRow - (height - 1 - scrollmargin)
→ blit 底边 = newStartLine + height - 1 = cursorRow + scrollmargin
```

所以 display 真正需要 cursor 下方的行数是 `scrollmargin`。于是：

- **硬约束**：`viewportMargin ≥ scrollmargin`。
- **理论最小值**：`viewportMargin = scrollmargin`（默认 3）——一行都不多渲染，零浪费。

这也是伪代码里 break 锚点用「`cursorRow + viewportMargin`」而不是「viewport 固定底边」的根本原因：**blit 范围跟着 cursor 走，停止条件也必须跟着 cursor 走**。

## 原则4：缺口分析与设计演进（记录推导过程）

伪代码是经过下面几轮修正才定型的，记录下来避免以后踩同样的坑：

**缺口1 —— Relocate 只有边界 if，没有无条件 preRender**
- 现象：安全区内编辑/段切换不 preRender → 用户在屏幕中间打字看不到字符；光标移出 table 但没出安全区时，装饰行不出现。
- 根因：方案B 的纯 blit display 依赖 map 实时刷新，而原 Relocate 语义是"光标已可见就不动"。
- 修复：Relocate 改成无条件先 preRender，再判上下界（见原则1）。

**缺口2 —— preRender 停止条件的三次演进**
- v1「只看 viewport 底边就 break」：cursor（V2 的 line60）根本没进 map，Relocate 找不到光标 → 崩。
- v2「超过底边 且 cursorShowed」：cursor 进了 map，但 break 用的是**绝对锚点**（固定在 viewport 底边+M）。cursorRow 小时碰巧够用；cursorRow 大时（大段膨胀）cursor 刚渲染就超过固定锚点立即 break，cursor 下方一行都没有 → display blit 越界。
- v3「cursorRow + viewportMargin（相对锚点）」：锚点跟着 cursor 走，cursor 之后永远有 `viewportMargin` 行。配合 `viewportMargin = scrollmargin`，恰好覆盖 blit 底边，不多一行。

**顶部分支 `oldStartLine - 1` 为什么够用**
- 单步↑才进顶部分支。大跨度上移要么走 case B（直接设 startLine），要么 GotoLoc 先用 `Scroll(sloc, -height/4)` 把光标预定位到 height/4 处，Relocate 进来时最多差 1 行，`-1` 微调足够。

# 五、注意事项

当然，我们要寻找一个对原生代码侵入最小的方式来处理。方案B的所有改动都集中在 microNeo 自有的 `internal/display/bufwindow_md.go`，复用原生代码里已有的两处 MD 分发接缝（`bufwindow.go` 的 `relocateVertical` 和 `displayBuffer` 处各一行 `if w.Buf.IsMD`），不修改 micro 的原生逻辑。
