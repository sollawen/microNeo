# 光标滚动 V2 — 渲染时序重构的架构可行性评估

> 配套文档：`docs/光标滚动的问题V2.md`（问题陈述与两个候选方案）、
> `docs/光标滚动-修改总结.md`（v1.0.4 → v1.0.5 的现状）。
>
> 本文不是实施方案，而是对"方案 B（先渲染再滚动）"在当前 microNeo 技术架构下
> 的**可行性、侵入面、风险与边界**的技术评估。

## 1. 问题陈述（复述要点）

光标在 table 内容行（如 line 59）按 ↓ 离开段后，该段从"原生编辑态（5 行）"切回
"MD 渲染态（9 行，多出 frame 装饰行）"，导致**按旧布局算出来的滚动量不足**，
光标被段膨胀挤出 viewport 底部，下一帧用户看不见光标。

这个问题在 v1.0.5 的总结文档 §5 里已被标记为"已知限制：跨段切换时光标偶发消失一帧"，
但当时的定位是"光标段原生渲染特权"的衍生副作用。V2 文档把它升级为**时序层面**的
根因诊断，并提出了系统性的重构方向。本文评估这个升级是否成立、重构是否可行。

---

## 2. 根因的技术分析（含代码时序证据）

### 2.1 主循环的真实时序

`cmd/micro/micro.go` 的 `DoEvent` 是整个编辑器的帧驱动器。按调用顺序展开：

```
DoEvent()                                    // 每帧
├─ 1. action.Tabs.Display()                  // 渲染【本帧】
│      └─ BufWindow.Display()                // bufwindow.go:930
│           └─ displayBufferMD(editMode)     // bufwindow_md.go:700
│                └─ 写入 viewportRowmap       // 基于【当前光标位置】的段布局
├─ 2. screen.Show()                          // 上屏
├─ 3. event = <-screen.Events                // 取用户输入
└─ 4. Tabs.HandleEvent(event)
        └─ CursorDown()                      // actions.go:264
             ├─ MoveCursorDown(1)            // 光标已移动到【新位置】
             └─ Relocate()                   // bufwindow.go:249
                  └─ relocateVerticalMD(c)   // bufwindow_md.go:867
                       └─ 读【本帧】viewportRowmap   // ← 但光标已经是【新位置】
```

**断层出现在步骤 4 的最后一个动作**：`Relocate` 读的 `viewportRowmap` 是步骤 1
用**旧光标**渲染出来的，而它要判定的是**新光标**该不该滚动、滚多少。

这个时序断层是 micro 原生架构的固有特性（见总结文档 §2"时序坑"），**不是 microNeo
引入的 bug**。micro 原生之所以不受影响，是因为它假设"1 buffer 行 = 1 屏行"，
旧 map 和新 map 的行布局**永远一致**，用旧的算新的毫无误差。MD 装饰行打破了这个
1:1 假设，旧 map 才开始和新 map 产生偏差。

### 2.2 偏差为什么总是"滚动不足"

在 table 例子中：

- 旧光标在 line 59（table 内）→ 该段走 `renderSegmentNative` → table 显示 5 行。
- `MoveCursorDown` 后光标在 line 60（table 外）→ 若重渲染，该段走 `renderSegmentMD` → table 显示 9 行。
- 但 `Relocate` 读的是旧 map（5 行布局），算出 `delta = cursorRow - botMarginRow`
  基于的是 5 行布局。
- 实际上屏后 table 变 9 行，多出的 4 行把光标往下挤了 4 行 → 光标落在
  `viewport 底部之外 4 行`。

方向是单向的：**段膨胀总是让滚动量算少了**（光标比预期更靠下），所以现象是
"光标飞出底部"，而不是"光标飞出顶部"。这和用户实测一致。

### 2.3 为什么 v1.0.5 的 2D viewportRowmap 没解决它

v1.0.5 的 `viewportRowmap` 是精确的——它精确记录了**某一帧**的屏行 ↔ buffer 行映射。
问题不在于 map 不精确，而在于 **`Relocate` 用的是上一帧的精确 map 来判定下一帧的滚动**。
map 越精确，越只能精确地反映"旧布局"。这是时序问题，不是数据结构精度问题。
所以任何"在旧 map 上做更精细的计算"的尝试（包括方案 A）都治标不治本。

## 3. 方案 A（临时补算）评估

### 3.1 思路

`relocateVerticalMD` 在算 `delta` 之前，临时预测"光标离开/进入段后，目标段会多/少
几行装饰行"，把预测的行数差补进 `delta`。

### 3.2 可行性判断：能止血，但有结构性盲区

**优点**：改动局部（只动 `relocateVerticalMD` 一个函数），不引入新数据结构，
风险可控。作为方案 B 落地前的临时缓解是合理的。

**盲区 1 — 双向性**：补算不仅要处理"离开段（table 变大，正向补）"，还要处理
"进入段（table 变小，反向扣）"。两个方向的段切换走不同的渲染路径
（`renderSegmentNative` ↔ `renderSegmentMD`），行数差的计算逻辑不对称。

**盲区 2 — 跳转场景**：`Ctrl+G`、点击、`LocFromVisual` 等操作可能一次跨越多个段。
补算逻辑要枚举"光标从哪个段跳到哪个段"，跨多个段时要分段累加装饰行数差，逻辑会
膨胀成一堆特例。

**盲区 3 — 本质是"用旧 map 模拟新 map"**：每多一种段切换方式（列表、代码块、
引用块……），就要多打一个补丁。可维护性差。

**结论**：方案 A 是"在错误的时序上打补丁"，能掩盖最常见的单段切换场景，但无法
覆盖跳转和多段。**不推荐作为终局**。

## 4. 方案 B（预渲染 + blit）架构可行性评估

### 4.1 核心思想

把"渲染"和"显示"解耦：

- **渲染**：在 `Relocate` 阶段就执行，结果存入 `renderedMap`（一个比 viewport 略大的
  bigViewportMap），**不直接上屏**。
- **判定**：`Relocate` 基于 `renderedMap`（已经是新光标视角的布局）算滚动量，精确。
- **显示**：`displayBufferMD` 退化成"纯 blit"——从 `renderedMap` 里找到 `StartLine`
  对应的行，把后续 `bufHeight` 行直接拷贝到 screen，**不再渲染**。

### 4.2 时序对比

| 阶段 | 现状（v1.0.5） | 方案 B |
|------|----------------|--------|
| 按键后 | `MoveCursor` → `Relocate`（读旧 map）→ `Display`（渲染+上屏） | `MoveCursor` → `Relocate`（**先预渲染**，读新 map）→ `Display`（**仅 blit**） |
| `Relocate` 用的 map | 上一帧的 | 本帧刚渲染的 |
| `displayBufferMD` 做什么 | 重渲染一遍 + 上屏 | 从 renderedMap blit + 上屏 |

关键变化：**渲染时机从 `Display` 提前到 `Relocate`**。这样 `Relocate` 拿到的永远是
"当前光标视角"的布局，时序断层消失。

### 4.3 侵入面分析（这是可行性评估的核心）

方案 B 的改动是否触碰 micro 原生代码？逐一核对：

| 文件 | 归属 | 方案 B 是否需要改 | 说明 |
|------|------|-------------------|------|
| `cmd/micro/micro.go` `DoEvent` | 原生 | ❌ 不改 | 主循环顺序（Display → Show → Event → HandleEvent）保持不变。方案 B 只改各步骤**内部**的行为，不改步骤顺序。 |
| `internal/action/actions.go`（`CursorDown` 等） | 原生 | ❌ 不改 | `MoveCursor` + `Relocate` 的调用结构保持不变。 |
| `internal/display/bufwindow.go` `Relocate` | 原生（已有 1 行 MD 分发缝 `bufwindow.go:249`） | ❌ 不改 | 分发缝已存在，`relocateVerticalMD` 是自有函数，内部怎么改都行。 |
| `internal/display/bufwindow.go` `Display` | 原生（已有 1 行 MD 分发 `bufwindow.go:938-940`） | ❌ 不改 | 同上。 |
| `internal/display/softwrap.go`（`Scroll`/`SLocFromLoc`/`Diff`） | 原生 | ❌ 不改 | 全部复用。 |
| `internal/display/bufwindow_md.go` | **microNeo 自有** | ✅ 改 | `displayBufferMD` / `renderSegmentMD` / `renderSegmentNative` / `relocateVerticalMD` 全在此文件。 |
| `internal/action/bufpane_md.go` | **microNeo 自有** | 视情况 | editMode 切换逻辑可能微调，但结构不变。 |

**结论：方案 B 对 micro 原生代码零侵入**。所有改动集中在 `bufwindow_md.go` 这个
microNeo 自有文件里，完全符合 `AGENTS.md`"对原生代码侵入越小越好"的原则。
这也是方案 B 相对方案 A 的一个隐性优势——方案 A 要在 `relocateVerticalMD` 里塞
越来越多的特例逻辑，长期看会让这个函数变得复杂；方案 B 则是**一次性把架构摆正**，
后续维护反而更轻。

### 4.4 数据结构需求

需要一个 `renderedMap`，本质是"渲染好但还没上屏"的 bigViewport。每行需要存储：

- **cell 数据**：rune + style（含 gutter 列），用于 blit 到 screen。
- **SLoc 元信息**：复用现有 `viewportRowmap` 的语义（`Line=-1` 装饰行 / `Line=-2` 空白 /
  `Line>=0` 内容行 + `Row` softwrap 段号），用于 `Relocate` 判定。

形态上就是"现有 `viewportRowmap` 的每行再挂一个 cell 切片"。二维数组
`[rows][cols]`，rows = `bufHeight + 2M`，cols = `gutterOffset + bufWidth`。
分配一次复用，内存开销可忽略（终端典型 80×24，多渲染一两屏也就是几 KB 量级）。

**现有 `viewportRowmap` 可以直接演进成 `renderedMap` 的元信息部分**，不需要新建
两套数据结构。这是 v1.0.5 已经打好的地基。

### 4.5 关键约束：预渲染窗口必须大于一屏（bigViewportMap）

这是方案 B 成败的关键，也是最容易踩坑的地方。在 V2 文档的例子上验算：

- 旧 viewport 显示到 line 62，光标移到 line 60。
- 如果预渲染窗口**只有一屏**（bufHeight 行），从旧 `StartLine` 开始渲染：
  - table 恢复 9 行后，line 60 被往下挤，**掉出一屏范围之外**。
  - `LineToScreenRow(60)` 在 renderedMap 里找不到 → `relocateVerticalMD` 退化回 fallback。
  - 方案 B 失败，退化成 v1.0.5 的行为。

所以预渲染窗口必须向下延伸超过一屏。简单做法：从旧 `StartLine` 向下渲染
`bufHeight + M` 行（`M` 取一屏高度，即总量约 2×bufHeight）。为应对反向滚动
（光标上移导致上方段收缩）和上方段膨胀，向上也预留 `M` 行。

**为什么不需要渲染整个 buffer**：键击是用户节奏（人手每秒几次），每次按键多渲染
一两屏（几百行）的开销在毫秒级以下，无感知。只有跳转（`Ctrl+G`/点击）可能跨很远，
这种情况下 `w.StartLine` 会落在 renderedMap 范围外，触发"以新 StartLine 重新预渲染
一屏"的 fallback（等价于现状行为）——这条 fallback 路径天然存在，无需特殊设计。

### 4.6 边界场景的覆盖性

| 场景 | 方案 B 的处理 | 评估 |
|------|---------------|------|
| **单段切换（V2 的 table 例子）** | 预渲染捕获段膨胀，`delta` 精确 | ✅ 核心场景，彻底解决 |
| **多段连续切换** | 同上，renderedMap 覆盖范围内的所有段都被正确渲染 | ✅ |
| **反向（光标上移，上方段收缩/膨胀）** | 向上预留 M 行，symmetric 处理 | ✅ |
| **跳转（`Ctrl+G`/点击远处）** | StartLine 落 renderedMap 外 → fallback 重渲染一屏 | ⚠️ 该场景退化为现状行为，但不比现状差 |
| **首帧（map 为空）** | 沿用现有 `relocateVerticalNativeFallback` | ✅ 已有兜底 |
| **resize（宽度变化致 softwrap 重算）** | 触发整段重渲染，renderedMap 自然重建 | ✅ 无特殊处理 |
| **横向滚动** | 不涉及（本次问题纯属垂直） | ✅ 不动 |

## 5. 风险与权衡

### 5.1 主要风险

1. **`displayBufferMD` 的"纯 blit"改造**：当前 `renderSegmentMD` / `renderSegmentNative`
   是"边渲染边写 screen"的，要改成"边渲染边写 renderedMap"，再由 blit 阶段写 screen。
   这是一次中等规模的重构，需要仔细保证两阶段的 cell 写入语义一致（gutter、行号、
   diff 标志、softwrap 续行的行号留空等）。这是方案 B 工作量的大头，但都在自有文件内。

2. **预渲染窗口的起点选择**：从旧 `StartLine` 开始预渲染是自然选择，但要确认
   `editMode` 的判定（光标在哪个段）用的是**新光标位置**——否则预渲染出来的还是
   旧布局，方案 B 退化为 no-op。这一点需要在实施时明确。

3. **性能**：每次按键多渲染一两屏。实测前无法精确量化，但基于"渲染单段是纯内存操作、
   无 IO"的判断，预期无感知。可在实施后用大文件（万行级）压测验证。

### 5.2 方案 B vs 方案 A 的权衡

| 维度 | 方案 A（补算） | 方案 B（预渲染） |
|------|----------------|------------------|
| 解决彻底性 | 单段切换场景 | 所有段切换场景（含跳转 fallback） |
| 原生侵入 | 零 | 零 |
| 改动范围 | `relocateVerticalMD` 一个函数 | `bufwindow_md.go` 多个函数（中等重构） |
| 长期可维护性 | 差（特例会越来越多） | 好（架构摆正，后续维护轻） |
| 风险 | 低（局部） | 中（重构面较大，但都在自有文件） |
| 适用阶段 | 临时止血 | 终局方案 |

**建议**：以方案 B 为目标。若近期需要先止血，可以用方案 A 顶一版（仅处理最常见的
单段向下切换），但心里要清楚它是过渡，终局仍是方案 B。

### 5.3 一个可选的性能优化（不阻塞 v1）

行数变化**唯一**的来源是"editMode + 光标所在段"——内容是 content-static（事件驱动算好），
softwrap 仅随宽度变（resize 时才变）。所以常态下（光标在段内移动、或在普通文本里移动），
上一帧的 renderedMap 行布局**仍然有效**，可以加一个快速路径：

- `relocateVerticalMD` 入口先比较"光标所在段是否与上一帧相同"。
- 相同 → 复用上一帧 renderedMap（零额外渲染），直接 `LineToScreenRow` 走快速路径。
- 不同 → 走预渲染。

这样只有跨段那一帧才付预渲染代价，常态零开销。但这是优化，**不是正确性必需**，
建议方案 B 的 v1 先做"无条件预渲染"跑稳，再上 fast path。

---

## 6. 结论

1. **V2 文档的根因诊断成立**，且有代码时序证据（`DoEvent` 的步骤 1 与步骤 4 之间的
   map 时差）。这不是"光标段原生渲染特权"的衍生副作用，而是更底层的时序断层。

2. **方案 B 在当前架构下可行，且对原生代码零侵入**。所有改动集中在
   `internal/display/bufwindow_md.go`（microNeo 自有文件），符合项目的低侵入原则。
   v1.0.5 已经打好的 `viewportRowmap` 地基可以直接演进成 `renderedMap`。

3. **方案 B 的成败关键是一条**：预渲染窗口必须大于一屏（bigViewportMap），否则光标
   会被段膨胀挤出预渲染范围、重新触发 fallback。实施时务必守住这条。

4. **方案 A 可作为临时缓解**，但治标，不应作为终局。

5. **建议路径**：方案 B 为目标。若急需止血可先用方案 A 顶一版（仅单段向下切换），
   但终局仍是方案 B。
